# はじめに

<!-- Block: Role -->
## この docs の役割

OtomeKairo は、LLM を用いて人間を模した自律的な存在を目指すプロジェクトである。
本 `docs` は、その方向性をぶらさずに設計を詰めるための正本と、現在地を追うための実装計画を置く場所とする。

ここで扱うのは次の内容である。

- 何を目指すか
- 最初にどこまで作るか
- どの責務をどこに置くか
- どの契約を docs に残すか
- 現在どこまで実装が進んでいるか

<!-- Block: Structure -->
## docs の構成

この `docs` は、次の 3 層に分ける。

- `design/`
  - 長く残る設計の正本
- `design/api/`
  - 外向き API 仕様の詳細
- `plan/`
  - 現在地、未着手、次の実装順

入口としては、次の文書を使う。

- `design/14_API仕様.md`
  - API 仕様ファミリーの入口
- `design/memory/00_記憶設計方針.md`
  - 記憶設計ファミリーの入口
- `plan/01_MVP実装計画.md`
  - 現在の実装状況と次の順番

<!-- Block: Boundary -->
## docs に書くことと書かないこと

docs に書くのは、少なくとも次である。

- OtomeKairo と `CocoroConsole` の責務境界
- 状態や設定値の意味
- 記憶と更新操作の責務境界
- 外向き API の概念
- 外向き API の厳密な仕様
- デバッグ記録の保証内容
- 現在の実装状況と未着手

逆に、次はコードを正とする。

- 関数やクラスの分割
- モジュール間の細かな呼び出し順
- 一時オブジェクトの厳密な shape
- 実装途中の一時的な補助メモ

内部フローがコードから素直に読めるなら、同じ内容を別の docs に起こさない。

## 現時点の前提

- 最初のクライアントは `CocoroConsole` に限定する
- OtomeKairo は OtomeKairo 専用の API を持つ
- OtomeKairo は Linux サーバ上で常駐稼働する
- `CocoroConsole` は別マシンから接続する
- 生成系 LLM 連携は LiteLLM を前提にする
- `OpenRouter` の embedding は公式 `/api/v1/embeddings` を使う
- 日本語を最優先にする
- 実装はコード先行で進め、長く残る契約だけ docs に残す
- 外向き API 仕様は `design/14_API仕様.md` と `design/api/` 配下を正とする

<!-- Block: Principles -->
## 設計方針

- シンプルさを優先する
- 最初から全機能を抱え込まず、長期伴走に必要な核を先に定義する
- ルールベースで人格や行動を固定しすぎず、判断の中心は LLM に置く
- ただし、実行の入口と境界はコード側で明示的に管理する
- 長く残る契約だけ docs に残し、内部フローはコードを正とする
