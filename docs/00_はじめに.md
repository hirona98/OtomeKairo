# はじめに

## この docs の役割

OtomeKairo は、LLM を用いて人間を模した自律的な存在を目指すプロジェクトである。
本 `docs` は、その方向性をぶらさずに設計を詰めるための上位設計と長期に残る契約を置く場所とする。

ここで扱うのは次の内容である。

- 何を目指すか
- 最初にどこまで作るか
- どの責務をどこに置くか
- 何を docs に残し、何をコードへ寄せるか

ここでは DB カラムや関数分割のような内部実装までは扱わない。
ただし、責務、状態、更新操作、外部契約、保存責務のような長く残る前提と、外向き API 仕様はここで定める。

## docs の分割

この `docs` は、次の 2 系統に分ける。

- `design/`
  - 正本の設計と、その理解を支える補助設計文書
- `plan/`
  - 実装順と直近の進め方

`design/` は長く残る判断を置く場所であり、`plan/` は実装をどう進めるかを置く場所である。

## docs に書くことと書かないこと

docs に書くのは、少なくとも次である。

- OtomeKairo と `CocoroConsole` の責務境界
- 状態や設定値の意味
- 記憶と更新操作の責務境界
- 外向き API の概念
- 外向き API の厳密な仕様
- デバッグ記録の保証内容

逆に、次はコードを正とする。

- 関数やクラスの分割
- モジュール間の細かな呼び出し順
- 一時オブジェクトの厳密な shape
- 実装途中の一時的な補助メモ

内部フローがコードから素直に読めるなら、同じ内容を別の docs に起こさない。
ただし、外向き API 仕様だけは docs を正とする。

## 開発の進め方

OtomeKairo の開発は、次の順番で進める。

1. 基本設計
2. 機能設計
3. 実装

この順番の意図は、実装都合で全体設計を崩さないことである。
ただし、実装前に厚い追加設計 docs を積むことは前提にしない。
実装中に新しい境界や契約が固まった場合だけ、必要な docs を同じ変更内で追加または更新する。

## 現時点の前提

- 最初のクライアントは `CocoroConsole` に限定する
- OtomeKairo は `CocoroGhost` 互換ではなく、OtomeKairo 専用の API を持つ
- OtomeKairo は Linux サーバ上で常駐稼働する
- `CocoroConsole` は別マシンから接続する
- LLM 連携は LiteLLM を前提にする
- 日本語を最優先にする
- 実装はコード先行で進め、長く残る契約だけ docs に残す
- 外向き API 仕様は `design/14_API仕様.md` を正とする

## 設計方針

- シンプルさを優先する
- 最初から全機能を抱え込まず、長期伴走に必要な核を先に定義する
- ルールベースで人格や行動を固定しすぎず、判断の中心は LLM に置く
- ただし、実行の入口と境界はコード側で明示的に管理する
- 長く残る契約だけ docs に残し、内部フローはコードを正とする
- ただし、外向き API 仕様は docs を正とする
