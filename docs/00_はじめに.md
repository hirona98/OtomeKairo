# はじめに

このプロジェクトの目的は、自分の意志で行動し、人間とともに生活する、個として独立した存在 **OtomeKairo** を作ることである。

OtomeKairo では、人格設定と記憶により "現在の個" を構成する。

## 重要な用語

この docs では、少なくとも次を分けて扱う。

- 人格設定
  - 明示的に選ぶ基底定義
- 記憶
  - 経験の蓄積と、そこから育つ理解
- 現在の個
  - 人格設定と記憶から、その時点で成立する判断主体

この前提により、人格設定だけでは OtomeKairo の現在の個は定まらない。
記憶は検索補助ではなく、現在の個を構成する主要要素として扱う。
 
## 目指す存在像

OtomeKairo の存在像には次の性質を含む。

- 個々の人物や出来事を認識し、世界を自分なりに理解する
- 継続する人格設定と記憶を持ち、そこから現在の個が立ち上がる
- 記憶によって考え方や反応が変わる（記憶が現在の個を構成する）
- 移動可能で五感を持つ
- インターネットなどの外部アクセスが可能

ここでいう移動、五感、外部アクセスは、特定のハードウェアや特定の client 実装に固定しない。
OtomeKairo では、各種センサ、物理デバイス、ネットワーク接続を capability 拡張で扱う前提を採る。
設計正本では capability の意味境界と契約を固定し、具体的なハードウェア選定や接続機器の組み合わせは固定しない。

## この docs の役割

本 `docs` は、OtomeKairo の方向性をぶらさずに設計を詰めるための正本と、現在地を追うための実装計画を置く場所とする。

ここで扱うのは次の内容である。

- 何を目指すか
- 現時点でどこまで通っているか
- どの責務をどこに置くか
- どの契約を docs に残すか
- 現在どこまで実装が進んでいるか

## docs の構成

この `docs` は、主に次の 4 つのまとまりで読む。

- `design/`
  - OtomeKairo 全体の責務境界を定める上位設計
- `design/api/`
  - 外向き API 仕様の詳細
- `design/memory/`
  - `design/04_人格と記憶.md` を受けて記憶だけを詳細化する設計ファミリー
- `plan/`
  - 現在地、完了済み範囲、現行フロー

## `design/` 直下と `design/memory/` の関係

- `design/` 直下
  - OtomeKairo 全体で共有する意味境界と責務分担を定める
- `design/memory/`
  - そのうち記憶 subsystem の内部構造、想起、更新を詳細化する

特に `design/04_人格と記憶.md` は、人格設定、記憶、現在の個の上位関係を定める入口である。
`design/memory/` はその前提を変えずに、記憶の詳細契約だけを定める。

## docs に書くことと書かないこと

docs に書くのは、少なくとも次である。

- OtomeKairo と `CocoroConsole` の責務境界
- 状態や設定値の意味
- 記憶と更新操作の責務境界
- 外向き API の概念
- 外向き API の厳密な仕様
- デバッグ記録の保証内容
- 現在の実装状況と未完了

逆に、次はコードを正とする。

- 関数やクラスの分割
- モジュール間の細かな呼び出し順
- 一時オブジェクトの厳密な shape
- 実装途中の一時的な補助メモ

内部フローがコードから素直に読めるなら、同じ内容を別の docs に起こさない。

## 設計方針

- シンプルさを優先する
- ルールベースで人格設定や行動を固定しすぎず、判断の中心は LLM に置く
- ただし、実行の入口と境界はコード側で明示的に管理する
- 長く残る契約だけ docs に残し、内部フローはコードを正とする
