# はじめに

このプロジェクトの目的は、自分の意志で行動し、人間とともに生活する、個として独立した存在 **OtomeKairo** を作ることである。

OtomeKairo では、記憶が人格を構成する。
 
## 目指す存在像

OtomeKairo の存在像には次の性質を含む。

- 個々の人物や出来事を認識し、世界を自分なりに理解する
- 継続する人格と記憶を持つ
- 記憶によって考え方や反応が変わる（記憶が人格を構成する）
- 移動可能で五感を持つ
- インターネットなどの外部アクセスが可能

## この docs の役割

本 `docs` は、OtomeKairoの方向性をぶらさずに設計を詰めるための正本と、現在地を追うための実装計画を置く場所とする。

ここで扱うのは次の内容である。

- 何を目指すか
- 最初にどこまで作るか
- どの責務をどこに置くか
- どの契約を docs に残すか
- 現在どこまで実装が進んでいるか

## docs の構成

この `docs` は、主に次の 4 つのまとまりで読む。

- `design/`
  - 長く残る設計の上位文書
- `design/api/`
  - 外向き API 仕様の詳細
- `design/memory/`
  - 記憶設計の詳細
- `99_MVP.md`
  - 現在の MVP の範囲、成功条件、非目標
- `plan/`
  - 現在地、未完了、次の実装順

## docs に書くことと書かないこと

docs に書くのは、少なくとも次である。

- OtomeKairo と `CocoroConsole` の責務境界
- 状態や設定値の意味
- 記憶と更新操作の責務境界
- 外向き API の概念
- 外向き API の厳密な仕様
- デバッグ記録の保証内容
- 現在の実装状況と未完了

逆に、次はコードを正とする。

- 関数やクラスの分割
- モジュール間の細かな呼び出し順
- 一時オブジェクトの厳密な shape
- 実装途中の一時的な補助メモ

内部フローがコードから素直に読めるなら、同じ内容を別の docs に起こさない。

## 設計方針

- シンプルさを優先する
- ルールベースで人格や行動を固定しすぎず、判断の中心は LLM に置く
- ただし、実行の入口と境界はコード側で明示的に管理する
- 長く残る契約だけ docs に残し、内部フローはコードを正とする
