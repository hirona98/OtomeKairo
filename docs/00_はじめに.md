# はじめに

## この docs の役割

OtomeKairo は、LLM を用いて人間を模した自律的な存在を目指すプロジェクトである。
本 `docs` は、その方向性をぶらさずに設計を詰めるための上位設計を置く場所とする。

ここで扱うのは次の内容である。

- 何を目指すか
- 最初にどこまで作るか
- どの責務をどこに置くか
- 今後どの論点を詰めるべきか

ここでは API パス、DB テーブル定義、クラス設計のような実装詳細までは扱わない。
ただし、責務、状態、更新操作、最小契約のような実装前提の設計まではここで定める。

## 開発の進め方

OtomeKairo の開発は、次の順番で進める。

1. 基本設計
2. 機能設計
3. 詳細設計
4. 実装

この順番の意図は、実装都合で全体設計を崩さないためである。
そのため、実装は詳細設計が固まってから着手する。

また、仕様や設計の変更がある場合は、コードより先に `docs` を更新することを前提とする。

## 現時点の前提

- 最初のクライアントは `CocoroConsole` に限定する
- OtomeKairo は `CocoroGhost` 互換ではなく、OtomeKairo 専用の API を持つ
- OtomeKairo は Linux サーバ上で常駐稼働する
- `CocoroConsole` は別マシンから接続する
- LLM 連携は LiteLLM を前提にする
- 日本語を最優先にする
- 実装着手前に、まず設計を十分に詰める

## 設計方針

- シンプルさを優先する
- 最初から全機能を抱え込まず、長期伴走に必要な核を先に定義する
- ルールベースで人格や行動を固定しすぎず、判断の中心は LLM に置く
- ただし、実行の入口と境界はコード側で明示的に管理する
