# 想起と判断フロー図

<!-- Block: Role -->
## この文書の役割

この文書は、`docs/design/memory/` 配下の正本を Mermaid 図で俯瞰できるようにするための参考資料である。

ここでは次を図にする。

- 共通の同期想起
- 共通判断から会話 / 行動 / 何もしない への分岐
- 返答後の記憶更新
- 永続データ同士の関係
- `memory_units` の状態遷移

図はできるだけ細かく書くが、DB カラムや API の確定仕様ではない。
現時点では、設計理解とレビューのための概念図として扱う。

<!-- Block: ReadGuide -->
## 読み方

この文書は、次の順で読むと分かりやすい。

1. 全体ライフサイクル
2. 同期想起の詳細
3. 判断分岐の詳細
4. `RecallPack` への振り分け
5. 返答後 / 実行後の更新
6. 状態遷移
7. 永続データの関係

特に重要なのは、OtomeKairo の判断が 1 本線ではなく、次の 4 層に分かれていることである。

- 同期想起:
  - 判断直前に必要分だけ読む
- 判断分岐:
  - `reply / act / noop` を決める
- ターン更新:
  - 返答後または実行後に 1 ターン単位で記憶を育てる
- 反省整理:
  - 数ターンまたは一定時間ごとに長期変化を再評価する

## 全体ライフサイクル

```mermaid
sequenceDiagram
    autonumber
    actor Obs as 観測源
    participant RT as Runtime
    participant RH as RecallHintBuilder
    participant SR as 構造レーン
    participant VR as 連想レーン
    participant RP as RecallPackBuilder
    participant DJ as Decision LLM
    participant RL as Reply LLM
    participant AP as Action Planner
    participant TC as Turn Consolidation
    participant RC as Reflective Consolidation
    participant EV as events
    participant ED as episode_digests
    participant MU as memory_units
    participant AF as affect_state
    participant VX as vector_index

    Obs->>RT: 観測入力 or 定期起床
    RT->>RT: 直近3分以内・最大4-6発話をロード
    RT->>RH: 現在観測 + 直近会話 + 現在時刻
    RH->>RH: 軽量高速LLMで JSON 生成
    RH-->>RT: RecallHint

    par 構造レーン
        RT->>SR: RecallHint
        SR->>MU: scope / memory_type / status で取得
        SR-->>RT: hard candidates
    and 連想レーン
        RT->>VR: RecallHint
        VR->>MU: vector top-k
        VR->>ED: vector top-k
        VR-->>RT: associative candidates
    end

    RT->>RP: hard + associative candidates
    RP->>RP: section再配置
    RP->>RP: evidence付与
    opt 細部確認が必要
        RP->>EV: selected linked_event_ids を限定ロード
        EV-->>RP: raw event slices
        RP->>RP: slot型 event_evidence へ圧縮
    end
    RP->>RP: conflicts抽出
    RP-->>RT: RecallPack

    RT->>DJ: 人格設定 + 記憶文脈 + world_state + runtime_state + capability_state + 現在観測
    DJ-->>RT: reply / act / noop

    alt 会話
        RT->>RL: system + 固定ヘッダ + 直近会話 + 内部コンテキスト + 現在観測
        RL-->>RT: 返答
        RT-->>Obs: 応答返却
        RT->>TC: 観測 / 返答 / 判断結果
    else 行動
        RT->>AP: capability manifest + world_state + runtime_state + capability_state
        AP-->>RT: action plan
        RT->>RT: コード側で実行
        RT->>TC: 観測 / action plan / 実行結果
    else 何もしない
        RT->>TC: 観測 / noop 判断
    end

    TC->>EV: events 保存
    TC->>ED: MVP では 1イベント1digest 生成
    TC->>MU: create / reinforce / refine / supersede / revoke / dormant
    TC->>AF: 感情更新
    TC->>VX: 埋め込み更新要求

    opt トリガー到達
        RT->>RC: 8ターン / 24h / 訂正 / 関係変化
        RC->>ED: 複数 digest を横断
        RC->>MU: summary 生成 / confirmed 見直し / dormant 化
        RC->>VX: 再索引
    end
```

## `memory_units.status` の状態遷移

```mermaid
stateDiagram-v2
    [*] --> inferred : create(推測)
    [*] --> confirmed : create(明示 / 反復)

    inferred --> confirmed : 明示支持 / 反復支持 / reflective 再評価
    inferred --> superseded : 新しい現行理解へ置換
    inferred --> revoked : 明確な否定
    inferred --> dormant : 保持するが前面から下げる

    confirmed --> superseded : より新しい現行理解
    confirmed --> revoked : 誤りと明示
    confirmed --> dormant : salience 低下 / 今は前面に出さない

    dormant --> inferred : 再浮上したがまだ推測
    dormant --> confirmed : 再確認された

    superseded --> dormant : 履歴として保持
    revoked --> dormant : 履歴として保持

    note right of dormant
      delete はしない
      履歴保持を優先する
    end note
```

## 補足

この図群で特に確認したい設計判断は次である。

- `active_commitments` と `relationship_model` を、類似検索より先に取ること
- ベクトル検索は入れるが、主導権は持たせないこと
- `events` は監査正本であり、通常の想起注入では直接使わないこと
- `episode_digests` は証拠層、`memory_units` は継続理解層として分けること
- `reflective consolidation` は人格形成に効くが、同期経路へは入れないこと
- 会話と行動は想起まで共通にし、分岐は判断以降へ寄せること
