<!-- Block: Base Judgment -->
## 先に結論

- `docs/note/ココロゴースト_システムフロー図.md` は、**記憶の育成系**としてかなり良い土台になる
- 特に、`同期で思い出す` と `非同期で育てる` の分離は、そのまま人格コアの `短周期ループ` と `長周期ループ` に対応できる
- ただし、現状の図は `ユーザー入力 -> 返答` を前提にした会話中心の流れであり、そのままでは人格コア全体の中心ループにはならない
- 取り込むなら、`ユーザー入力` を `観測イベント` へ一般化し、`返答` を `行動` へ一般化し、`思い出し` を `文脈組み立て` に拡張する必要がある
- つまり、**記憶系の基盤としては使えるが、ランタイム全体の主導線としては作り直しが必要** という判断になる

<!-- Block: What Already Fits -->
## すでに噛み合っている部分

- `events`: 出来事ログとして、そのままエピソード記憶の母体に使える
- `event_links` と `event_threads`: 出来事どうしの連続性を持てるので、会話だけでなく行動連鎖の文脈にも流用できる
- `state`: 現在状態と育った情報を分けずに保持しているため、自己状態と世界状態の基礎に使える
- `event_affects`: 出来事単位の感情痕跡として、感情記憶の入口に使える
- `vec_items`: 記憶本体ではなく検索索引として扱っている点がよい
- `events_fts`: 文字一致検索の補助索引として、意味検索と役割分担しやすい
- `jobs`: 非同期の育成処理を明示しているため、長周期ループへ移植しやすい

<!-- Block: What Must Change -->
## そのままでは足りない部分

- 入力の起点が `ユーザー入力` 固定で、カメラ、マイク、ネットワーク応答、内部欲求が入口に入っていない
- 出力の終点が `返答` 固定で、移動、視線変更、検索、通知のような複数行動へ広がっていない
- `state` が広すぎて、`self_state`、`body_state`、`world_state`、`drive_state` に分割されていない
- `思い出す` は強いが、`反省して知識へ昇格する` 段が弱い
- `vec_items` はあるが、`working memory / episodic memory / semantic memory / archive` の階層がまだ明示されていない
- `user_preferences` は会話アプリ寄りの命名なので、人格個体設計では `preferences` または `relationship_memory` に寄せるべきである

<!-- Block: Tier Mapping -->
## 記憶階層への対応づけ

- `working memory`
  - 同期処理で使う一時文脈である
  - `LoadDialog`、`LoadState`、`Merge`、`Select`、`Pack` で組み上がる手前の材料が相当する
  - 永続化の正本ではなく、短周期ループだけで使う作業領域として扱う
- `episodic memory`
  - 出来事単位の記憶である
  - `events`、`event_links`、`event_threads`、`event_entities` が相当する
  - 行動結果や観測結果も、同じ出来事記録としてここへ統合できる
- `semantic memory`
  - 安定した知識と育った状態である
  - `state`、`state_links`、`state_entities` の一部が相当する
  - 現在状態と長期知識は、将来的には分離して持つべきである
- `affective memory`
  - 感情痕跡である
  - `event_affects` と、背景の気分としての `state` 更新が相当する
  - 記憶階層としては、エピソードに紐づく感情痕跡と、持続する気分を分けたほうがよい
- `retrieval indexes`
  - 想起の補助であり、記憶本体ではない
  - `vec_items` と `events_fts` が相当する
  - `sqlite-vec` はここに位置づけるのが自然である
- `consolidation`
  - 記憶の育成処理である
  - `jobs` の非同期処理が相当する
  - ここに反省、要約、重要度更新、忘却制御を追加するのが自然である

## 論文ごとの比較

- `Generative Agents`
  - 参照: https://arxiv.org/abs/2304.03442
  - 論点は `observation / planning / reflection` の循環である
  - `CocoroGhost_システムフロー図` は `observation` と `retrieval` は強いが、`reflection` が弱い
  - 相性はかなり良い
  - 取り込むべき点は、非同期ジョブに `reflection` を追加し、出来事から高次の知識を生成すること

- `MemGPT`
  - 参照: https://arxiv.org/abs/2310.08560
  - 論点は、限られたコンテキスト窓の内外で階層的に記憶を扱うことにある
  - `CocoroGhost_システムフロー図` にも `同期で使う材料` と `非同期で育てる保存領域` の分離はある
  - ただし、`working` と `archive` の役割分担がまだ明示されていない
  - 取り込むべき点は、`同期で使う作業領域` を明示し、長期記憶との出し入れ規則を固定すること

- `MemoryBank`
  - 参照: https://arxiv.org/abs/2305.10250
  - 論点は、長期対話での記憶保持と忘却の制御である
  - `CocoroGhost_システムフロー図` は、出来事、状態、感情を別経路で更新しており、土台としてかなり近い
  - 特に `event_affects` と `状態更新` の分離は相性がよい
  - 取り込むべき点は、記憶強度、時間減衰、再参照による再強化である

- `ReAct`
  - 参照: https://arxiv.org/abs/2210.03629
  - 論点は、推論と行動と観測を交互に回すことにある
  - `CocoroGhost_システムフロー図` は `思い出す -> 返答` の前処理としては強いが、行動途中での再観測が弱い
  - 取り込むべき点は、1 回の想起で終わらせず、行動後に再度 `思い出し` と `判断` を回せるようにすること

- `Reflexion`
  - 参照: https://arxiv.org/abs/2303.11366
  - 論点は、失敗や結果を言語的な反省として残し、次回の判断に効かせることにある
  - `CocoroGhost_システムフロー図` の `非同期で育てる` は、この受け皿としてそのまま使える
  - ただし、反省を独立した記憶種別として持っていない
  - 取り込むべき点は、`reflection_notes` のような反省メモを長周期で作ること

- `Voyager`
  - 参照: https://arxiv.org/abs/2305.16291
  - 論点は、探索しながらスキルを蓄積し、再利用することにある
  - `CocoroGhost_システムフロー図` の `event_threads` は連続性の保持には効くが、再利用可能な `skill` を持っていない
  - 取り込むべき点は、繰り返し成功した行動列を `skill` として昇格する層を追加すること

- `Code as Policies`
  - 参照: https://arxiv.org/abs/2209.07753
  - 論点は、高位の意図を実行可能な手続きへ落とすことにある
  - `CocoroGhost_システムフロー図` は記憶検索には強いが、行動手順の再利用までは持っていない
  - 取り込むべき点は、想起した記憶から直接返答するのではなく、`行動計画` へ変換する段を明示すること

- `VoxPoser`
  - 参照: https://arxiv.org/abs/2307.05973
  - 論点は、言語指示を空間制約と可到達性へ落とすことにある
  - `CocoroGhost_システムフロー図` には空間記憶や視点操作記憶の層がまだない
  - 取り込むべき点は、`world_state` に `見える / 近づける / 避ける` のような空間メモリを持たせること

- `OK-Robot`
  - 参照: https://arxiv.org/abs/2401.12202
  - 論点は、既存モジュールを統合して実世界タスクを行うことにある
  - `CocoroGhost_システムフロー図` のようにテーブルと処理段を明示している設計は、モジュール統合と相性がよい
  - 取り込むべき点は、記憶層をデバイス制御から切り離し、統合点を `gateway` 側へ寄せること

- `PaLM-E`
  - 参照: https://arxiv.org/abs/2303.03378
  - 論点は、視覚や状態を言語モデルへ統合するマルチモーダル化である
  - `CocoroGhost_システムフロー図` はテキスト対話中心なので、そのままだとマルチモーダル記憶になっていない
  - 取り込むべき点は、`events` と `observation_frame` に入力元種別とモダリティ情報を持たせること

- `RT-2`
  - 参照: https://arxiv.org/abs/2307.15818
  - 論点は、視覚言語知識を行動トークンへつなぐことにある
  - `CocoroGhost_システムフロー図` の `返答` は自然言語出力だが、人格コアでは `行動命令` へ一般化すべきである
  - 取り込むべき点は、返答結果を `action_command` として構造化し、実行履歴も行動記憶として残すこと

- `OK-Robot` と `PaLM-E` と `RT-2` の共通点
  - 実世界系では、マルチモーダル観測と行動の構造化が必須である
  - `CocoroGhost_システムフロー図` を採用するなら、`events` を会話ログではなく `観測・行動イベント` へ拡張しないと足りない

<!-- Block: Concrete Changes -->
## この比較から入れるべき具体変更

- `CocoroGhost_システムフロー図` の `ユーザー入力` を `観測イベント` に置き換える
- `返答` を `行動` に置き換える
- `state` を `self_state / body_state / world_state / drive_state` に分割する
- `jobs` に `reflection`、`importance update`、`memory decay`、`skill promotion` を追加する
- `vec_items` は `episodic_memory` と `semantic_memory` で索引を分ける
- `event_affects` は、出来事に紐づく感情痕跡として残し、背景気分は別の持続状態で持つ
- `user_preferences` は `preferences` または `relationship_memory` に改名する
- `events` はテキストだけでなく、カメラ、マイク、ネットワーク、行動結果も同じイベント形式で保存する

<!-- Block: Final Verdict -->
## 最終判断

- `docs/note/ココロゴースト_システムフロー図.md` は、**記憶の検索と育成の設計原型として採用できる**
- 特に、`同期で思い出す` と `非同期で育てる` の二層構造は、そのまま使う価値がある
- ただし、現在の人格コアへ入れるには、`会話中心` から `観測・判断・行動中心` へ一般化する必要がある
- 先行研究との比較では、もっとも足す価値が高いのは `反省`、`記憶階層の明示`、`忘却制御`、`行動再利用の明示`、`空間メモリ` である
