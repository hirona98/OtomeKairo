# 現行計画

## この文書の役割

この文書は、OtomeKairo の現在地、現行基盤の到達点、残っている拡張テーマだけを簡潔に管理する。

- 設計判断は `docs/design/` を正とする
- 実装状態はコードを正とする
- 実装や仕様を変えたら、同じ変更内で docs も更新する
- 旧仕様互換だけの移行レイヤーや検索フォールバックは扱わない

## 計画文書の分担

- `01_現行計画.md` は、現在地、実装済み範囲、解消対象の実装制約、維持する境界、直近マイルストーン、優先順位、次の主作業を正本として管理する
- `02_現行処理フロー.md` は、現行コードの主要フローだけを補助資料として管理する
- 現在地、未完了、優先順位、マイルストーンを変更するときは、先に `01_現行計画.md` を更新する

## 解消対象の実装制約

次は現時点で残っている実装上の制約であり、後続の実装で解消対象にする。

- `world_state` は source pack 第二段、real source `summary_source` 区別第一段、source-aware TTL と real schedule multi-slot 第一段まで到達しているが、TTL の精密化、専用 capability や外部連携の実データ導入はまだ入っていない
- `drive_state` は `support_strength / scope_alignment / signal_strength / persona_alignment / stability_hint` を含む集約 shape へ安定化し、`initiative_context` は時刻帯 / foreground thinness / suppression summary を持つが、精密根拠確認はまだ限定的である
- 自律判断は `pending_intent` の再評価中心から抜け、候補系統比較と `reply / noop / capability_request` の分岐理由は inspection で比較できる状態になったが、capability family 全体の自発実行品質はまだ限定的である
- background wake は interval 起床、基本的な cooldown、時刻帯 / foreground thinness を含む比較文脈と低リスク probe 優先を持つが、capability family 全体の起床制御はまだ限定的である
- `ongoing_action` は汎用 capability request の開始、結果待ち、完了、timeout 終了と、accepted async capability result 後の継続 / 保留 / 中断、`transition_kind` を含む共通 trace 語彙まで入っているが、対応 capability family 自体はまだ少ない
- capability manifest、binding、`capability_state` の責務分離は availability inspection、汎用 capability request dispatcher、busy / cooldown / last result / last failure / temporary unavailable の runtime state 第一段まで実装済みであり、family ごとの細かな前提条件や pause 運用は未整備である
- `event_evidence` は圧縮根拠生成まで入っているが、必要時だけ `events` を限定ロードして精密根拠確認する段はまだ入っていない

## 現行方針として維持する境界

次は現行方針として維持する境界であり、直近マイルストーンで解消対象にしない。

- `event_evidence` は短い圧縮根拠であり、逐語引用や正確引用の保証を目的にしない
- `RecallHint` は長期記憶を読まず、現在入力と直近文脈だけから作る
- `RecallHint` は `interaction_mode` と `primary_recall_focus` を主軸にし、完全な同時最適化を目的にしない
- 通常会話 API は `text` と `client_context` に加えて任意の `images` を受けるが、raw image payload の保存や OCR 全文抽出は扱わない
- `desktop_watch` は `vision.capture` の image payload を短い観測要約へ変換して判断へ入れるが、raw image payload の保存、OCR 全文抽出、UI 構造化は扱わない
- `wake` と `desktop_watch` は共有判断パイプラインへ入り、`reply`、保留意図更新、capability failure、意味のある `world_state` 変化が起きたサイクルだけ `turn consolidation` を走らせる。条件に当てはまらない自律・観測 `noop` は `memory_trace=skipped` に留める
- `world_state` は foreground summary を判断入力へ渡すが、生の画像、音声、長い payload は直接渡さず、必要なら短い補助要約だけを使う
- 個別記憶行は OtomeKairo 自身の記憶更新処理だけが変更し、管理 API、retention 実行、redaction 実行、export は追加しない
- `memory_set` 全体の clone/delete は設定定義操作として扱い、UI の未確定操作は client 側の下書きで保持する
- `editor-state` の全体置換は、記憶集合の実体 clone/delete を暗黙実行しない
- `editor-state` の秘密値返却例外は認証済み編集操作として扱い、read/write の audit event を秘密値なしで記録する
- background wake の外界行動実行は直近マイルストーンの対象に含めない
- 回帰テスト整備は現行計画の完了条件に含めない
