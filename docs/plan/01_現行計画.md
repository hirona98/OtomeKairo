# 現行計画

## この文書の役割

この文書は、OtomeKairo の現在地、現行基盤の到達点、残っている拡張テーマだけを簡潔に管理する。

- 設計判断は `docs/design/` を正とする
- 実装状態はコードを正とする
- 実装や仕様を変えたら、同じ変更内で docs も更新する
- 旧仕様互換だけの移行レイヤーや検索フォールバックは扱わない

## 計画文書の分担

- `01_現行計画.md` は、現在地、実装済み範囲、既知の制約、直近マイルストーン、優先順位、次の主作業を正本として管理する
- `02_現行処理フロー.md` は、現行コードの主要フローだけを補助資料として管理する
- 現在地、未完了、優先順位、マイルストーンを変更するときは、先に `01_現行計画.md` を更新する

## 現在地の詳細

- 会話基盤:
  通常会話 1 サイクル、通常会話の任意画像入力、bootstrap、設定取得、設定変更、catalog、判断サイクル inspection API
- 記憶基盤:
  `SQLite + sqlite-vec`、`turn consolidation`、`RecallHint`、`memory_interpretation`、`RecallPack`、`reflective consolidation`、`drive_state` 集約第二段（`drive_kind / support_count / freshness_hint` と `self / relationship` signal 補助）、非会話サイクルの選択的 episode 化
- 自発判断基盤:
  runtime-only の `pending_intent` キュー、wake observation API、background 起床スケジューラ、`reply / noop / pending_intent / capability_request`、accepted async capability result の共通 follow-up pipeline 第一段と `ongoing_action` 継続 / 保留 / 中断、`world_state` 第一段の前景参照、`initiative_context` 候補系統比較第二段と自発 `vision.capture` probe 優先
- 観測基盤:
  `desktop_watch` 第一段、`client_context` の正規化、`vision.capture` capability、`external.status` capability、`vision.capture` 対応 client 自動選択、画像観測要約の判断入力反映、capability availability inspection、`logs/stream`
- 外界状態基盤:
  `world_state` source pack 第二段、`client_context` / state-type 別 `screen_context`・`social_context_context`・`environment_context`・`location_context`・`external_service_context`・`body_context`・`device_context`・`schedule_context` / capability result summary / wake pending-intent 時刻ヒントからの短期外界状態更新、`vision.capture` と `external.status` result の state-type 別 summary 反映、body/device/schedule の capability-derived summary 反映、state_type 別 TTL と integration key 第一段、pending-intent ごとの schedule slot、foreground summary の判断入力反映、inspection の `world_state_trace` / `source_pack_state_type_hooks` / `normalized_candidate_policies`
- 後段処理:
  durable な memory postprocess worker、`status.runtime_summary`、inspection の `memory_trace`、`ongoing_action_exists`
- 運用確認:
  `scripts/run_long_smoke.py` による `smoke / soak` 実行、isolated data dir、主要な capture failure case と `external.status` follow-up / multi-service integration、`desktop_watch` の pending-intent schedule slot の確認
- 意味判断での LLM 採用:
  `reflective consolidation` の `summary_text`、`event_evidence`、`RecallPack` の意味的選別、pending-intent 候補選択、`vision.capture` と通常会話画像の観測要約、`capability_decision_view`、`capability_request` 判断、accepted async capability result 後の後続判断と `ongoing_action` 継続判断、`initiative_context` の initiative_baseline / runtime / recent turn / candidate family priority / intervention suppression summary 第二段

この前提では、会話、記憶、自発判断、`desktop_watch` 方針と `vision.capture` capability の第一段、`drive_state` 第一段、長時間 smoke までを現行基盤の到達点として扱う。
これは設計上の完成を意味しない。
以降は、設計上の完成形との差分を埋める拡張段階として扱う。

## 既知の制約

次は現時点の設計上の制約であり、未整理のバグ一覧ではない。
この制約を前提として、後続のマイルストーンを定義する。

- `event_evidence` は短い圧縮根拠であり、逐語引用や正確引用は保証しない
- `RecallHint` は長期記憶を読まず、現在入力と直近文脈だけから作る
- `RecallHint` は `interaction_mode` と `primary_recall_focus` を主軸にし、完全な同時最適化は行わない
- 通常会話 API は `text` と `client_context` に加えて任意の `images` を受けられるが、raw image payload の保存や OCR 全文抽出はまだ行わない
- `desktop_watch` は `vision.capture` の image payload を短い観測要約へ変換して判断へ入れるが、raw image payload の保存、OCR 全文抽出、UI 構造化はまだ行わない
- `wake` と `desktop_watch` は共有判断パイプラインへ入り、`reply`、保留意図更新、capability failure、意味のある `world_state` 変化が起きたサイクルだけ `turn consolidation` を走らせる。条件に当てはまらない自律・観測 `noop` は `memory_trace=skipped` に留める
- `world_state` は `client_context` と capability result summary に加え、短い `social_context / environment / location / external_service / body / device / schedule` summary と wake の pending-intent 時刻ヒントを source pack に入れ、state_type 別 TTL と integration key 第一段まで入っている。専用 capability や外部連携の実データ拡張は後続拡張で扱う
- `world_state` は foreground summary を判断入力へ渡すが、生の画像、音声、長い payload は直接渡さず、必要なら短い補助要約だけを使う
- 個別記憶行は OtomeKairo 自身の記憶更新処理だけが変更し、管理 API、retention 実行、redaction 実行、export は追加しない
- `memory_set` 全体の clone/delete は設定定義操作として扱い、UI の未確定操作は client 側の下書きで保持する
- `editor-state` の全体置換は、記憶集合の実体 clone/delete を暗黙実行しない
- `editor-state` の秘密値返却例外は認証済み編集操作として扱い、read/write の audit event を秘密値なしで記録する
- background wake は複雑な時間帯制御や外界行動実行をまだ持たない
- 自律判断はまだ `pending_intent` の再評価中心であり、`drive_state` は `reflective consolidation` から `drive_kind / support_count / freshness_hint` を持つ集約 shape へ拡張したが、生成品質と活用範囲はまだ限定的である
- 自律判断入力には `initiative_context` の initiative_baseline / runtime / recent turn / candidate family priority / intervention suppression summary 第二段を入れ、wake / background_wake は `pending_intent` が空でも `drive_state / world_state / ongoing_action` に前景があれば shared pipeline へ進める。現在文脈が薄いときの自発 `vision.capture` probe 優先は入ったが、候補系統ごとの意味判断品質と capability family 全体の自発実行品質改善は後続拡張で扱う
- `ongoing_action` は汎用 capability request の開始、結果待ち、完了、timeout 終了と、accepted async capability result 後の継続 / 保留 / 中断の第一段まで入っているが、public result endpoint と capability ごとの意味付け hook はまだ `vision.capture` 中心である
- capability manifest、binding、`capability_state` の責務分離は、availability inspection と汎用 capability request dispatcher まで実装済みであり、能力ファミリー全体の状態最適化は後続拡張で扱う
- 回帰テスト整備は現行計画の完了条件に含めない

## 直近設計変更の実装同期

次は docs 側の設計を更新済みであり、コード同期時に確認する。

- bootstrap:
  `register-first-console` は token 未発行時だけ発行し、発行済み状態では既存 token を返さない
- capability:
  `CapabilityManifest / CapabilityBinding / CapabilityState` の分離と availability の意味境界は [../design/17_capability_manifest.md](../design/17_capability_manifest.md) を正とし、コード同期時にその境界を確認する
- memory interpretation:
  `candidate_memory_units` は意味骨格 hint を含む記憶候補メモに留め、半構造化した `memory_units` 正本はコードが作る
