# 現行計画

## この文書の役割

この文書は、OtomeKairo の現在地、到達済み範囲、残っている制約、次の主作業だけを管理する。

- 設計判断は `docs/design/` を正とする
- 実装状態はコードを正とする
- 実装や仕様を変えたら、同じ変更内で docs も更新する
- 完了済みの実行計画や作業履歴はこの文書に残さない
- 到達済みの内容は、現在地要約と対応する design docs へ吸収する

## 計画文書の分担

- `01_現行計画.md` は、現在地、到達済み範囲、既知の制約、維持する境界、次の主作業を正本として管理する
- `02_現行処理フロー.md` は、現行コードの主要フローだけを補助資料として管理する
- 完了済みの詳細手順、検証ログ、過去の作業分割は git 履歴で追う

## 残っている制約

次は現時点で残っている実装上の制約であり、後続の実装で解消対象にする。

- `world_state` は source pack 第二段、real source `summary_source` 区別第一段、source-aware TTL と `schedule.status` result slot 投影まで到達しているが、外部連携の実データ導入はまだ限定的である
- 自律判断は候補系統比較と `reply / noop / capability_request` の分岐理由を inspection で比較できる状態になったが、実 source capability を増やした後の自発実行品質確認はまだ限定的である
- background wake は interval 起床、基本的な cooldown、時刻帯 / foreground thinness を含む比較文脈と低リスク probe 優先を持つが、実 source capability を増やした後の起床制御確認はまだ限定的である
- `ongoing_action` は汎用 capability request と accepted async capability result 後の継続 / 保留 / 中断まで入っているが、対応 capability family 自体はまだ少ない
- capability manifest、binding、`capability_state` の責務分離は runtime state 第一段まで実装済みだが、family ごとの細かな前提条件は未整備である

## 維持する境界

次は現行方針として維持する境界であり、次の主作業で解消対象にしない。

- `event_evidence` は短い圧縮根拠であり、逐語引用や正確引用の保証を目的にしない
- `RecallHint` は長期記憶を読まず、現在入力と直近文脈だけから作る
- 通常会話 API は `text` と `client_context` に加えて任意の `images` を受けるが、raw image payload の保存や OCR 全文抽出は扱わない
- `desktop_watch` は `vision.capture` の image payload を短い観測要約へ変換して判断へ入れるが、raw image payload の保存、OCR 全文抽出、UI 構造化は扱わない
- `wake` と `desktop_watch` は共有判断パイプラインへ入り、`reply`、保留意図更新、capability failure、意味のある `world_state` 変化が起きたサイクルだけ `turn consolidation` を走らせる。条件に当てはまらない自律・観測 `noop` は `memory_trace=skipped` に留める
- `world_state` は foreground summary を判断入力へ渡すが、生の画像、音声、長い payload は直接渡さず、短い補助要約だけを使う
- 個別記憶行は OtomeKairo 自身の記憶更新処理だけが変更し、管理 API、retention 実行、redaction 実行、export は追加しない
- `memory_set` 全体の clone/delete は設定定義操作として扱い、UI の未確定操作は client 側の下書きで保持する
- `editor-state` の全体置換は、記憶集合の実体 clone/delete を暗黙実行しない
- `editor-state` の秘密値返却例外は認証済み編集操作として扱い、read/write の audit event を秘密値なしで記録する
- 旧仕様互換だけの移行レイヤーや検索フォールバックは扱わない
- background wake の外界行動実行は次の主作業に含めない
- 回帰テスト整備は次の主作業の完了条件に含めない

## 次の主作業

次の主作業は、`world_state` が取り込む実 source capability の横展開である。

完了条件は次とする。

- 導入する実 source capability を絞り、capability manifest、result schema、必要な `client_context` 境界を定める
- 実 source result を `world_state` source pack に取り込み、state_type ごとの summary、TTL、integration key を source に応じて分ける
- real schedule source の複数 slot を `schedule:<slot_key>` として統合し、generic schedule summary と併存させる
- inspection で source pack、normalized candidate policies、TTL 上限、integration key を比較できる
- `smoke` profile で通常会話、wake、desktop_watch、capability result、memory worker、failure case matrix が壊れていないことを確認する

着手順は次とする。

1. 導入対象の実 source capability を 1 つに絞る
2. 対応する design/API 文書へ wire 契約と意味境界を追加する
3. capability result から `world_state` source pack への投影を実装する
4. inspection と smoke harness に比較点を追加する
