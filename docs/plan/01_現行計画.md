# 現行計画

## この文書の役割

この文書は、OtomeKairo の現在地、現行基盤の到達点、残っている拡張テーマだけを簡潔に管理する。

- 設計判断は `docs/design/` を正とする
- 実装状態はコードを正とする
- 実装や仕様を変えたら、同じ変更内で docs も更新する
- 旧仕様互換だけの移行レイヤーや検索フォールバックは扱わない

## 現在地

現在の実装は、主要な会話基盤と観測基盤までは通っている。

- 会話基盤:
  通常会話 1 サイクル、bootstrap、設定取得、設定変更、catalog、判断サイクル inspection API
- 記憶基盤:
  `SQLite + sqlite-vec`、`turn consolidation`、`RecallHint`、`memory_interpretation`、`RecallPack`、`reflective consolidation`、`drive_state` 第一段
- 自発判断基盤:
  runtime-only の `pending_intent` キュー、wake observation API、background 起床スケジューラ、`reply / noop / pending_intent`
- 観測基盤:
  `desktop_watch` 第一段、`client_context` の正規化、`vision.capture` capability、`vision.capture` 対応 client 自動選択、`logs/stream`
- 後段処理:
  durable な memory postprocess worker、`status.runtime_summary`、inspection の `memory_trace`、`ongoing_action_exists`
- 運用確認:
  `scripts/run_long_smoke.py` による `smoke / soak` 実行、isolated data dir、主要な capture failure case の確認
- 意味判断での LLM 採用:
  `reflective consolidation` の `summary_text`、`event_evidence`、`RecallPack` の意味的選別、pending-intent 候補選択

この前提では、会話、記憶、自発判断、`desktop_watch` 方針と `vision.capture` capability の第一段、`drive_state` 第一段、長時間 smoke までを現行基盤の到達点として扱う。
これは設計上の完成を意味しない。
以降は、設計上の完成形との差分を埋める拡張段階として扱う。

## 既知の制約

次は現時点の設計上の制約であり、未整理のバグ一覧ではない。

- `event_evidence` は短い圧縮根拠であり、逐語引用や正確引用は保証しない
- `RecallHint` は長期記憶を読まず、現在入力と直近文脈だけから作る
- `RecallHint` は `interaction_mode` と `primary_recall_focus` を主軸にし、完全な同時最適化は行わない
- 通常会話 API は `text` と `client_context` だけを受け、画像入力をまだ持たない
- `desktop_watch` は `vision.capture` の image payload を取得できるが、まだ画像内容自体は判断に入れず、`client_context` 主体で扱う
- `wake` と `desktop_watch` は共有判断パイプラインへ入るが、現行実装では記憶更新を同期実行せず、意味変化がある自律・観測サイクルの episode 化は後続拡張で扱う
- `world_state` の詳細設計は追加済みだが、現行実装では外界状態の正本化と失効管理は後続拡張で扱う
- 記憶本体は OtomeKairo 自身の記憶更新処理だけが変更し、管理 API、retention 実行、redaction 実行、export は追加しない
- `editor-state` の秘密値返却例外は認証済み編集操作として条件を明文化済みだが、audit 実装の確認は後続拡張で扱う
- background wake は複雑な時間帯制御や外界行動実行をまだ持たない
- 自律判断はまだ `pending_intent` の再評価中心であり、`drive_state` は `reflective consolidation` から作る第一段だけで、生成品質と活用範囲はまだ限定的である
- `pending_intent` が空の状況でも `drive_state` と現在文脈から自発系評価へ進む initiative loop は設計済みだが、現行コードへの反映は後続拡張で扱う
- `ongoing_action` は `vision.capture` request / response に対する第一段の作成 / 完了 / timeout 終了までは入っているが、汎用 capability 実行や複数段の継続更新にはまだ一般化していない
- capability manifest、binding、`capability_state` の責務分離は設計済みだが、能力ファミリー全体の実装一般化は後続拡張で扱う
- `GET /api/inspection/capabilities` の wire 契約は設計済みだが、現行コードへの反映は後続拡張で扱う
- 回帰テスト整備は現行計画の完了条件に含めない

## 直近設計変更の実装同期

次は docs 側の設計を更新済みであり、コード同期時に確認する。

- bootstrap:
  `register-first-console` は token 未発行時だけ発行し、発行済み状態では既存 token を返さない
- capability:
  `CapabilityManifest / CapabilityBinding / CapabilityState` の分離と availability の意味境界は [../design/17_capability_manifest.md](../design/17_capability_manifest.md) を正とし、コード同期時にその境界を確認する
- timestamp:
  正本 timestamp は OtomeKairo のローカルタイムゾーンに属する offset 付き timestamp とし、LLM には生活文脈向けの自然文で渡す
- memory interpretation:
  `candidate_memory_units` は意味骨格 hint を含む記憶候補メモに留め、半構造化した `memory_units` 正本はコードが作る

## 次の拡張

現時点で保留している主要な拡張は次である。

- 通常会話で画像入力を受け、共有判断パイプラインへ入れられるようにする
- `vision.capture` の image payload を判断材料へ入れる
- capability manifest、binding、`capability_state` の分離を汎用 capability 実行へ反映する
- `GET /api/inspection/capabilities` を実装し、server 派生の capability availability を確認できるようにする
- `ongoing_action` を汎用 capability 実行へ広げ、継続 / 完了 / 中断の状態遷移を一般化する
- `wake` / `desktop_watch` のうち、reply、保留意図更新、capability failure、意味のある観測変化が起きたサイクルを選択的に episode 化する
- `world_state` を実装し、観測結果と外部サービス結果から短期外界状態を更新する
- `editor-state` の audit と秘密値非露出を実装で確認する
- `relationship / self` 要約品質を改善する
- 人格設定と記憶から派生する `drive_state` の生成品質と活用範囲を拡張し、自律判断を短期候補再評価だけに寄せないようにする
- `initiative_context` を自律判断入力へ入れ、`pending_intent` が無い場合の自発系評価を扱う
- 時刻帯や前景状況を使う判断補助を絞って足す
- `events` の限定ロードを使う精密根拠確認を追加する
- `--profile soak` を土台に、より長い実時間運用と追加 failure case を確認する

上の拡張は後続の拡張テーマとして扱い、現行基盤の到達点とは分ける。
