# 現行計画

## この文書の役割

この文書は、OtomeKairo の現在地、到達済み範囲、残っている制約、次の主作業だけを管理する。

- 設計判断は `docs/design/` を正とする
- 実装状態はコードを正とする
- 実装や仕様を変えたら、同じ変更内で docs も更新する
- 完了済みの実行計画や作業履歴はこの文書に残さない
- 到達済みの内容は、現在地要約と対応する design docs へ吸収する

## 計画文書の分担

- `01_現行計画.md` は、現在地、到達済み範囲、既知の制約、維持する境界、次の主作業を正本として管理する

- 完了済みの詳細手順、検証ログ、過去の作業分割は git 履歴で追う

## 残っている制約

次は現時点で残っている実装上の制約であり、後続の実装で解消対象にする。

- 記憶品質 compact digest と想起品質 compact digest は mock smoke に実装済みであり、`smoke` profile の標準 failure injection で確認済みである。`short-soak / long-soak` の継続確認が残る
- `memory_links` の自動保存 label は `derived_from / contradicts` に限定している。`supports / about_same_scope / affects` は成立ルールが未固定である
- 想起品質 compact digest は直近窓外の曖昧参照、`active_commitments`、`conflicts`、標準 / precise `event_evidence`、判断 trace 反映を一続きで確認する
- `world_state` は source pack 第二段、real source `summary_source` 区別第一段、source-aware TTL、`schedule.status` result slot 投影、`device.status / body.status / environment.status / location.status / social.status` result 投影まで到達しているが、外部連携の実データ導入はまだ限定的である
- 自律判断は候補系統比較と `reply / noop / capability_request` の分岐理由を inspection と実 LLM scenario matrix の compact digest で比較する状態になった。新鮮な `schedule / social_context / body / external_service / device / environment / location` の negative 境界、古い `schedule / external_service` と不足 `social_context / device` の positive 境界は固定済みであり、`body / environment / location` の positive compact digest は未固定である
- background wake は interval 初回起床、interval 未到達 skip、cooldown skip、弱い前景の `noop`、grounded 前景の `reply` を実 LLM 構造 assert へ固定済みであり、外界行動実行はまだ扱わない
- `ongoing_action` は汎用 capability request と accepted async capability result 後の継続 / 保留 / 中断まで入っているが、対応 capability family 自体はまだ少ない
- capability manifest、binding、`capability_state` の責務分離は runtime state 第一段まで実装済みだが、family ごとの細かな前提条件は未整備である

## 維持する境界

次は現行方針として維持する境界であり、次の主作業で解消対象にしない。

- `event_evidence` は短い圧縮根拠であり、逐語引用や正確引用の保証を目的にしない
- `RecallHint` は長期記憶を読まず、現在入力と直近文脈だけから作る
- 通常会話 API は `text` と `client_context` に加えて任意の `images` を受けるが、raw image payload の保存や OCR 全文抽出は扱わない
- `desktop_watch` は `vision.capture` の image payload を短い観測要約へ変換して判断へ入れるが、raw image payload の保存、OCR 全文抽出、UI 構造化は扱わない
- `wake` と `desktop_watch` は共有判断パイプラインへ入り、`reply`、保留意図更新、capability failure、意味のある `world_state` 変化が起きたサイクルだけ `turn consolidation` を走らせる。条件に当てはまらない自律・観測 `noop` は `memory_trace=skipped` に留める
- `world_state` は foreground summary を判断入力へ渡すが、生の画像、音声、長い payload は直接渡さず、短い補助要約だけを使う
- 個別記憶行は OtomeKairo 自身の記憶更新処理だけが変更し、管理 API、retention 実行、redaction 実行、export は追加しない
- `memory_set` 全体の clone/delete は設定定義操作として扱い、UI の未確定操作は client 側の下書きで保持する
- `editor-state` の全体置換は、記憶集合の実体 clone/delete を暗黙実行しない
- `editor-state` の秘密値返却例外は認証済み編集操作として扱い、read/write の audit event を秘密値なしで記録する
- 旧仕様互換だけの移行レイヤーや検索フォールバックは扱わない
- background wake の外界行動実行は次の主作業に含めない
- pytest 等の網羅的な新規テスト整備は次の主作業の完了条件に含めない。既存 smoke の compact digest 拡張で完了判定する

## 次の主作業

次の主作業は、記憶品質 / 想起品質 digest の soak 確認と実 LLM trace shape 確認である。

完了条件は次とする。

- `short-soak` で記憶品質 compact digest が `create / reinforce / refine / supersede / revoke / dormant`、`commitment_state=open / waiting_confirmation / on_hold / done / cancelled`、`memory_links`、`affect_state`、`drive_state` を summary に残す
- `short-soak` で想起品質 compact digest が直近窓外の曖昧参照、`active_commitments`、`conflicts`、標準 / precise `event_evidence`、判断 trace 反映を一つの summary に残す
- `drive_state` が `summary`、`commitment`、`mood_state`、`affect_state` の signal を使い、`scope_supports` と `drive_summaries` に根拠を残す状態を smoke summary で確認する
- `real-llm-smoke` は実 LLM の通常動作確認を維持し、記憶品質の完了判定は mock smoke の deterministic compact digest を主確認にする
