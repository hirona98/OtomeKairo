# 現行計画

## この文書の役割

この文書は、OtomeKairo でこれから実施する作業だけを管理する。

- 設計判断は `docs/design/` を正とする
- 実装状態は `src/` のコードを正とする
- 実装や仕様を変える変更では、同じ変更内で docs も更新する

## 優先順

1. capability family readiness の実行時 digest を整備する
2. capability family ごとの concrete client 実データ接続を広げる
3. 記憶基盤の `long-soak` を実施する
4. `long-soak` の結果に基づき、記憶関連の残作業を次の実装項目へ分解する

## 1. capability family readiness digest

目的は、manifest の `decision_readiness` が実行時の capability input、result summary、fresh `world_state` 再利用判定と一致していることを継続検証することである。

完了条件は次とする。

- `scripts/run_long_smoke.py` の compact digest が、各 capability request input と `decision_readiness.input_keys` の対応を記録する
- compact digest が、各 capability result summary と `decision_readiness.result_summary_keys / result_item_keys` の対応を記録する
- status refresh positive / no-request digest が、`readiness.world_state_type` と foreground `world_state.state_type` の対応を記録する
- `smoke` と `real-llm-smoke` が readiness digest の shape を確認する
- readiness digest に token、credential、内部 URL、transport 詳細、raw payload を入れない

着手順は次とする。

1. manifest の `decision_readiness` から検証用の期待値を作る helper を追加する
2. capability request summary に input key 対応の compact digest を追加する
3. capability result / observation summary に result key 対応の compact digest を追加する
4. initiative / status refresh probe の digest に `world_state_type` 対応を追加する
5. `smoke` と `real-llm-smoke` の assertion を更新する
6. docs を更新する

## 2. concrete client 実データ接続

目的は、status 系 capability が mock だけでなく実際の client context または安全な設定経由のデータを使って、短い現在状態要約を返すことである。

完了条件は次とする。

- `external.status` が service 名ごとの短い状態要約を返す
- `schedule.status` が近い予定要約と deterministic な slot を返す
- `device.status` が端末、接続、電源、バッテリー、ネットワークの短い状態要約を返す
- `body.status` が身体、疲労、眠気、姿勢、体調の短い状態要約を返す
- `environment.status` が周囲、部屋、騒音、明るさ、作業環境の短い状態要約を返す
- `location.status` が場所、移動中か、作業場所の短い状態要約を返す
- `social.status` が会話、連絡、通知、会議文脈の短い状態要約を返す
- 認証情報は環境変数または安全な設定経由で扱い、ログ、docs、サンプル、test data に入れない
- 実データが無い family は、未接続理由を inspection trace で判別できる状態にする

着手順は次とする。

1. family ごとに利用する client context field と result summary field を決める
2. secret を必要としない family から実データ接続を追加する
3. secret を必要とする family は設定境界とログ redaction を先に整備する
4. `real-llm-smoke` の status follow-up trace で実データ接続の shape を確認する
5. docs を更新する

## 3. 記憶基盤 long-soak

目的は、記憶 worker、memory link 想起、capability follow-up の長時間運用時の欠落を検出することである。

完了条件は次とする。

- `long-soak` が memory worker backlog lag を記録する
- `long-soak` が startup orphan `ongoing_action` cleanup を記録する
- `long-soak` が memory link 想起利用の compact digest を記録する
- `long-soak` が capability follow-up missing の有無を記録する
- pending memory job が drain する
- active `ongoing_action` が残らない
- stale `world_state` が残らない

着手順は次とする。

1. `long-soak` を実行する
2. artifact の summary と server log から backlog lag、orphan cleanup、memory link 想起、follow-up missing を読む
3. 失敗または不安定箇所があれば、コードまたは smoke assertion を修正する
4. 修正後に `smoke` と必要な範囲の `real-llm-smoke` を実行する
5. docs を更新する

## 4. 記憶関連の残作業分解

目的は、`long-soak` の結果を次の小さな実装単位へ分けることである。

完了条件は次とする。

- 記憶 worker lag の対処が必要か判断する
- memory link 想起の精度調整が必要か判断する
- capability follow-up missing の対処が必要か判断する
- 専用 graph UI、管理 API、retention、redaction、export を次の実装対象に入れない
- 次の実装項目が、1 コミット単位で扱える大きさに分かれている

着手順は次とする。

1. `long-soak` の artifact から問題候補を抽出する
2. 問題候補を bug、観測不足、仕様未定のいずれかに分類する
3. bug は修正計画へ入れる
4. 観測不足は digest または inspection 拡張へ入れる
5. 仕様未定は design docs の判断項目へ入れる
