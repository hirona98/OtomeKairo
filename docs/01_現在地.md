# 現在地

## この文書の役割

この文書は、設計 docs と現行コードを読むときの前提、既知の制約、確認入口を管理する。
実装状態の正本は `src/` のコードと smoke の結果である。
この文書は、機能一覧や実装済み項目を正本として持たない。

- 長く残る意味境界は `docs/design/` を正とする
- HTTP / WebSocket API 仕様は `docs/design/api/` を正とする
- 記憶 subsystem の詳細契約は `docs/design/memory/` を正とする
- 関数分割や処理順の最終正本は `src/` のコードとする
- 実装済み項目の一覧、一時的な実装計画、次の作業列挙は、この文書の責務に含めない

## 現行実装の確認方針

現行実装を確認するときは、次の順に見る。

- 意味境界と設計意図は `docs/design/` を読む
- HTTP / WebSocket API の形は `docs/design/api/` を読む
- 記憶 subsystem の詳細は `docs/design/memory/` を読む
- 実装済みかどうか、処理順、関数分割は `src/` を読む
- 回帰確認は `scripts/run_long_smoke.py` の profile と smoke 結果を見る
- 設計 docs と現行コードが食い違う場合は、この文書の既知の制約を確認する

## 既知の制約

現時点では次を前提に扱う。

- `docs/design/api/` には現行設計が目指す HTTP / WebSocket API 仕様を含むため、設計済みだがコードへ未反映の項目がありうる。厳密な実装範囲は `src/` と smoke 結果で判断する
- 現行フロー専用の補助文書は置かず、内部フローの把握はコードを正とする
- `/api/events/stream` は client -> server message として `hello` だけを受け付ける。`vision.capture` が accepted された client は、`hello.vision_sources` に少なくとも 1 件の source を載せ、各 `vision_source_id` は `vision_source:` で始める必要がある
- `/api/conversation` の `images` は image Data URI だけを受け付け、最大 1 件に制限する
- `/api/capability/result` は capability family ごとに `result` の必須 field を追加検証する
- capability dispatcher は、配送先 binding が一意に定まる capability だけを実行対象にする
- `pending_intent_candidates` は runtime-only で保持し、永続化しない
