<!-- Block: Premise -->
## 進め方の前提

API は、OtomeKairo がすでに持っている概念と操作を外へ露出する層である。
そのため、API の前に設定項目、設定責務、設定変更の最小操作を固める。

また、次を前提に進める。

- `CocoroConsole` は設定の正本ではなく、編集 UI として振る舞う
- 記憶設計の主線は `docs/design/memory/` にあり、ここには未解決の次段だけを書く
- クライアント都合ではなく、OtomeKairo 側の責務に沿って順番を決める

<!-- Block: Step1 -->
## 1. 設定項目と設定責務を固定する

まず、OtomeKairo が何を設定として保持するかを固める。

ここで詰めること:

- OtomeKairo 側が正本として保持する設定項目の範囲
- `runtime_state` に入る動作条件と、永続設定として持つ項目の境界
- `selected_persona_id` と `selected_memory_set_id` の保持方法
- `wake_policy` や起床間隔のような自発判断設定の保持方法
- モデル役割の差し替えを設定として持つか、設計固定にするか
- `CocoroConsole` が編集できる対象と、OtomeKairo が内部でのみ保持する対象の境界

<!-- Block: Step2 -->
## 2. 設定操作の最小契約を固定する

次に、設定をどう変えるかの操作意味を固める。
この段階では API パスではなく、操作の意味だけを確定する。

ここで詰めること:

- 人格切り替え
- 記憶切り替え
- 自発判断設定の更新
- 必要ならモデル関連設定の更新
- 変更時の validator と反映単位

<!-- Block: Step3 -->
## 3. OtomeKairo 専用 API の概念整理

設定項目と操作契約が固まった後で、API を整理する。
`CocoroGhost` 互換ではなく、OtomeKairo に自然な API を切る。

ここで詰めること:

- クライアントが送る観測の種類
- サーバが返す応答と状態の種類
- 設定操作をどの操作面に載せるか
- 人格切り替えや記憶切り替えをどう露出するか

<!-- Block: Step4 -->
## 4. 人格設定の粒度

ここで詰めること:

- 性格と口調の関係
- 行動判断傾向の持たせ方
- 人格更新の扱い

<!-- Block: Step5 -->
## 5. モデル役割の具体的な分け方

LiteLLM を前提に、役割ごとのモデル境界を詰める。

ここで詰めること:

- 会話と判断を同じモデルにするか
- 想起入口と行動判断を分けるか
- コストと品質の折り合い

<!-- Block: Step6 -->
## 6. デバッグ可能性の設計

OtomeKairo は長期伴走を目指すため、後から挙動を検証できることが重要である。

ここで詰めること:

- 応答や行動の根拠追跡の粒度
- 保存量と見通しやすさの両立
- 観測ログと判断ログの分け方

<!-- Block: Step7 -->
## 7. 自発行動の扱い

自発行動は本プロジェクトの特徴のひとつである一方、設計上の難所でもある。

ここで詰めること:

- `reply / act / noop` の閾値
- 文脈と頻度の関係
- 自発会話から外界行動へ広げる順序
- 複雑な外界行動の実行設計をどこから別責務に切り出すか

<!-- Block: Step8 -->
## 8. 記憶設計の次段

記憶設計の主線は `docs/design/memory/` を正としつつ、次段で検討する項目だけをここに残す。

ここで詰めること:

- 複合意図の同時最適化
- 直近窓の外にある曖昧参照
- 長期変化の分析をどの層で扱うか
- 正確引用や詳細時系列を責務に含めるか
- 複雑な外界行動の実行設計をどこから別責務に切り出すか

最後の項目は、`7. 自発行動の扱い` と連動して詰める。

<!-- Block: Step9 -->
## 9. 将来の身体性と外界接続

会話以外の行動能力をどう設計に収めるかは、中長期の拡張性に関わる。

ここで詰めること:

- 観測能力と作用能力をどう分けるか
- 能力追加時に人格や記憶とどう接続するか
- 人間互換に近づけるための拡張順序

<!-- Block: Gate -->
## API 設計へ進んでよい条件

少なくとも次が揃った時点で、API 設計へ進む。

- OtomeKairo 側が保持する設定項目の一覧がある
- 各設定項目が、設定なのか状態なのか別層なのか説明できる
- 設定変更の最小操作が、API 名に依存せずに説明できる
- `CocoroConsole` は編集 UI であり、設定の正本ではないと docs 全体で一致している
