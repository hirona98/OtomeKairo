## 現在地

いまの OtomeKairo は、基本設計と機能設計をほぼ閉じた段階にある。

この時点で固定済みの主な論点は次である。

- 設定と `runtime_state` の境界
- 人格設定の粒度
- モデル役割とモデル設定資源
- 接続 bootstrap と認証
- デバッグ記録
- 自発判断の `future_act` 候補キュー
- 記憶設計の MVP 境界

したがって、直近の本命は追加の上位設計ではなく、MVP の詳細設計である。

<!-- Block: Step1 -->
## 1. MVP 詳細設計

次に書くべき正本 docs は、MVP の縦切り詳細設計である。

ここで閉じる対象は次である。

- `観測入力 -> RecallHint -> 想起 -> reply / noop / future_act -> 記憶更新 -> debug trace` の 1 本の流れ
- 同期経路と非同期経路の責務分担
- 各段階で読む状態と書く状態
- 内部失敗の畳み方
- MVP 実装で最初に通す最小ユースケース

この段階では、将来拡張のために抽象化を増やすより、MVP の 1 サイクルを具体化することを優先する。

<!-- Block: Step2 -->
## 2. MVP 実装へ渡す順序

詳細設計のあとに、実装は次の順で進めるのがよい。

1. 会話観測の 1 サイクルを通す
2. `events / episode_digests / memory_units` 更新を通す
3. `future_act` 候補キューを通す
4. `cycle_summary` と段階トレースを通す
5. 設定切り替えと接続 bootstrap を実運用に寄せる

この順にすると、最初に会話主体 MVP を成立させたまま、自発性と運用性を後から積める。

<!-- Block: Deferred -->
## 3. 後段へ送る論点

次は、MVP の詳細設計と実装を止めない前提で後段へ送る。

- 将来の身体性と外界接続
- 観測能力と作用能力の拡張
- 能力追加時に人格や記憶とどう接続するか
- 人間互換に近づけるための拡張順序

これらは重要だが、いまの MVP 直前段階で先に閉じなくても進められる。

<!-- Block: Gate -->
## 実装へ進んでよい条件

少なくとも次が揃った時点で、MVP 実装へ進む。

- 応答や自発判断の根拠をどの粒度で追うか説明できる
- `reply / noop / future_act` の判断境界と `future_act` 候補キューの扱いを説明できる
- 記憶設計の次段で何を MVP に含め、何を後段へ送るか説明できる
- 正確引用、曖昧参照、複合意図、`future_act` 候補キュー、長期変化分析、外界行動境界の扱いが docs 全体で一致している
- 段階トレースで何を保存し、何を保存しないか説明できる
- 接続 bootstrap と通常 API の境界が docs 全体で一致している
- `CocoroConsole` は編集 UI であり、設定の正本ではないと docs 全体で一致している
- MVP の 1 判断サイクルを、どの順で、どの状態を読み書きして通すか説明できる
