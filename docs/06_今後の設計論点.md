# 今後の設計論点

## この文書の役割

ここでは、初版の上位設計で意図的に保留した論点を整理する。
この文書にある内容は未決事項であり、今後の設計作業で優先的に詰める対象とする。

## 論点 1: OtomeKairo 専用 API の概念整理

`CocoroGhost` 互換ではなく、OtomeKairo に自然な API をどう切るかは今後の重要論点である。

ここで詰めるべきこと:

- クライアントが送るべき観測の種類
- サーバが返すべき応答と状態の種類
- 人格切り替えや記憶切り替えの操作概念をどう表すか

## 論点 2: 記憶の育ち方

会話や観測の生ログをどう保持し、そこから何を育てるかは本プロジェクトの中心論点である。

初期検討メモとして `07_記憶検討メモ.md` を追加した。
ここで出した方向性を踏まえて、今後この論点を設計へ落とし込む。
また、具体化の入口として `08_記憶ユニット設計メモ.md` を追加した。

ここで詰めるべきこと:

- 生の出来事と派生した理解の関係
- 嗜好、人物像、状況理解、関係性をどう分けるか
- 何を継続し、何を更新し、何を忘れるか

## 論点 3: 人格設定の粒度

人格設定ファイルに何を含めるべきかは、会話品質と保守性の両方に影響する。

ここで詰めるべきこと:

- 性格と口調の関係
- 行動判断の傾向をどこまで人格側に持たせるか
- 人格更新と互換性をどう考えるか

## 論点 4: モデル役割の具体的な分け方

LiteLLM を前提に、役割ごとにモデルを分けられるようにするが、その境界は今後詰める。

ここで詰めるべきこと:

- 会話と判断を同じモデルにするか
- 記憶解釈を別役割に分けるか
- コストと品質の折り合いをどう取るか

## 論点 5: デバッグ可能性の設計

OtomeKairo は長期伴走を目指すため、後から挙動を検証できることが重要である。

ここで詰めるべきこと:

- 応答や行動の根拠をどこまで追えるようにするか
- 保存量と見通しやすさをどう両立するか
- 広めに保存した情報をどの基準で絞り込むか

## 論点 6: 自発行動の扱い

自発行動は本プロジェクトの特徴のひとつである一方、設計上の難所でもある。

ここで詰めるべきこと:

- 発話しない判断をどう扱うか
- 文脈と頻度の関係
- 自発会話から外界行動へ拡張する順序

## 論点 7: 将来の身体性と外界接続

会話以外の行動能力をどう設計に収めるかは、中長期の拡張性に関わる。

ここで詰めるべきこと:

- 観測能力と作用能力をどう分けるか
- 能力追加時に人格や記憶とどう接続するか
- 人間互換に近づけるための拡張順序
