# 設定操作契約

<!-- Role -->
## この文書の役割

この文書は、OtomeKairo が受け付ける設定変更と関連設定資源更新の最小操作契約を定める正本である。

ここで固定したいのは次の内容である。

- 設定変更をどの単位で行うか
- 各操作が何を変え、何を変えないか
- validator が何を確認するか
- 反映単位と失敗時の扱い

ここでは API パスや wire 形式の細部は `14_API仕様.md` と `design/api/` 配下で定める。
操作の意味だけを固定する。

<!-- Premise -->
## 基本前提

設定変更と関連設定資源更新では、OtomeKairo 側の意味単位をそのまま操作単位にする。
そのため、通常操作では UI の都合で複数の責務を 1 操作に混ぜない。

MVP の最小操作は次の操作群とする。

- `select_persona(persona_id)`
- `select_memory_set(memory_set_id)`
- `patch_current(current_patch)`
- `update_wake_policy(wake_policy)`
- `replace_persona(persona_id, persona_definition)`
- `replace_memory_set(memory_set_id, memory_set_definition)`
- `delete_persona(persona_id)`
- `delete_memory_set(memory_set_id)`
- `select_model_preset(model_preset_id)`
- `replace_model_preset(model_preset_id, model_preset_definition)`
- `replace_model_profile(model_profile_id, model_profile_definition)`
- `delete_model_preset(model_preset_id)`
- `delete_model_profile(model_profile_id)`
- `replace_editor_state(editor_state)`

<!-- OperationKinds -->
## 操作の種類

MVP の設定操作は、次の 2 種類に分ける。

- 通常操作
  - 1 つの設定値、または 1 つの設定資源を意味単位で変える操作
- bundle 操作
  - `current` スナップショットや `editor-state` bundle を、UI の都合でまとめて扱う明示的な操作

ここでいう bundle 操作は例外を曖昧に許すためのものではない。
`patch_current` と `replace_editor_state` だけを、役割を限定した補助操作として持つ。

<!-- CommonRule -->
## 共通ルール

設定操作と関連設定資源更新では、次を共通ルールにする。

- validator を通らなかった場合は部分反映しない
- 指定していない他設定や他資源は自動で変えない
- 成功時は、永続設定の正本を先に更新する
- 反映は単純さを優先し、永続設定更新後の次回 request / cycle から新しい値を使う
- 通常操作は 1 つの設定値または 1 つの設定資源だけを変える
- `patch_current(current_patch)` は current 設定スナップショットだけを対象にする
- `replace_editor_state(editor_state)` は editor bundle だけを対象にする
- 任意の複数操作を混ぜる汎用 transaction 面は作らない

人格、記憶、モデルの選択値や、それらに対応する選択中資源の中身が変わる場合は、`future_act` 候補キューのような内部保留状態をクリアする。
`replace_editor_state(editor_state)` の成功時も同様にクリアする。
MVP では、このキューを再起動越しに保持しない。

このルールにより、人格切り替え、記憶切り替え、自発判断設定更新、モデルプリセット切り替えを独立に扱える。

<!-- PatchCurrent -->
## `patch_current(current_patch)`

`patch_current(current_patch)` は、現在設定スナップショットの一部だけをまとめて更新する補助操作である。

この操作の変更対象は次だけである。

- `selected_persona_id`
- `selected_memory_set_id`
- `selected_model_preset_id`
- `memory_enabled`
- `desktop_watch`
- `wake_policy`

この操作で変えないのは次である。

- `persona` 定義
- `memory_set` 定義
- `model_preset` 定義
- `model_profile` 定義
- `runtime_state`
- `world_state`

validator が確認する項目は次とする。

- 指定した `selected_persona_id` が存在すること
- 指定した `selected_memory_set_id` が存在すること
- 指定した `selected_model_preset_id` が存在し、必須 role を満たすこと
- `memory_enabled` が boolean であること
- `desktop_watch` が最小契約を満たすこと
- `wake_policy` が最小契約を満たすこと

この操作では、複数の current 項目を 1 回で送る。
ただし、それは「current 設定スナップショットを部分更新する」という 1 種類の操作として扱う。
任意の複数資源をまたぐ汎用 transaction にはしない。

<!-- SelectPersona -->
## `select_persona(persona_id)`

`select_persona(persona_id)` は、現在選択中の人格だけを切り替える操作である。

この操作で変えるのは次だけである。

- `selected_persona_id`

この操作で変えないのは次である。

- `selected_memory_set_id`
- `wake_policy`

validator が確認する項目は次とする。

- `persona_id` が空でないこと
- 指定した `persona_id` が選択対象の人格として存在すること

MVP では、成功後の次回 request / cycle から新しい選択値を使う。
ただし、人格切り替えの意味は常に「人格だけを変える」で固定する。
実装上は `patch_current({"selected_persona_id": ...})` の単項目版として扱う。

<!-- SelectMemory -->
## `select_memory_set(memory_set_id)`

`select_memory_set(memory_set_id)` は、現在選択中の記憶だけを切り替える操作である。

この操作で変えるのは次だけである。

- `selected_memory_set_id`

この操作で変えないのは次である。

- `selected_persona_id`
- `wake_policy`

validator が確認する項目は次とする。

- `memory_set_id` が空でないこと
- 指定した `memory_set_id` が選択対象の記憶として存在すること

ここでは、人格との自動連動は行わない。
人格と記憶は独立に選ぶという前提を守るためである。
実装上は `patch_current({"selected_memory_set_id": ...})` の単項目版として扱う。

<!-- UpdateWakePolicy -->
## `update_wake_policy(wake_policy)`

`update_wake_policy(wake_policy)` は、自発判断の機会をどう与えるかだけを更新する操作である。

この操作で変えるのは次だけである。

- `wake_policy`

この操作で変えないのは次である。

- `selected_persona_id`
- `selected_memory_set_id`

MVP で validator が確認する項目は次とする。

- `wake_policy.mode` が `disabled` または `interval` であること
- `mode=interval` のとき `interval_minutes` が 1 以上の整数であること
- `mode=disabled` のとき、余分な起床条件を要求しないこと

`wake_policy` は全体置き換えで扱う。
部分 patch にすると UI と validator が先に複雑になるためである。
実装上は `patch_current({"wake_policy": ...})` の単項目版として扱う。

## `select_model_preset(model_preset_id)`

`select_model_preset(model_preset_id)` は、現在選択中のモデルプリセットだけを切り替える操作である。

この操作で変えるのは次だけである。

- `selected_model_preset_id`

この操作で変えないのは次である。

- `selected_persona_id`
- `selected_memory_set_id`
- `wake_policy`

validator が確認する項目は次とする。

- `model_preset_id` が空でないこと
- 指定した `model_preset_id` が選択対象のプリセットとして存在すること
- そのプリセットが必須役割を満たしていること
- 参照先の `model_profile` が存在し、実行に必要な構成が成立していること

MVP では、成功後の次回 request / cycle から新しい選択値を使う。
ただし、意味は常に「モデルプリセットだけを切り替える」で固定する。
実装上は `patch_current({"selected_model_preset_id": ...})` の単項目版として扱う。

<!-- ReplacePersona -->
## `replace_persona(persona_id, persona_definition)`

`replace_persona(persona_id, persona_definition)` は、OtomeKairo が保持する人格定義を全体置換で作成または更新する操作である。

この操作で変えるのは次である。

- 指定した `persona_id` の人格定義

この操作で自動で変えないのは次である。

- `selected_persona_id`
- `selected_memory_set_id`
- `selected_model_preset_id`
- `wake_policy`

validator が確認する項目は次とする。

- `persona_id` と定義本文の ID が一致すること
- `display_name` が空でないこと
- `core_persona` が object であること
- `expression_style` が object であること
- `expression_style.tone` が空でないこと
- `persona_text`、`second_person_label`、`addon_text` を含む場合は文字列であること

部分 patch は行わない。
人格定義は全体置換で扱う。

<!-- DeletePersona -->
## `delete_persona(persona_id)`

`delete_persona(persona_id)` は、指定した人格定義だけを削除する操作である。

この操作で変えるのは次である。

- 指定した `persona_id` の削除

この操作で自動で変えないのは次である。

- `selected_persona_id`
- 他の人格定義
- `selected_memory_set_id`
- `selected_model_preset_id`

validator が確認する項目は次とする。

- 指定した `persona_id` が存在すること
- 選択中の人格ではないこと
- 最後の 1 件ではないこと

人格が選択中なら、削除ではなく先に選択変更を要求する。

<!-- ReplaceEditorState -->
## `replace_editor_state(editor_state)`

`replace_editor_state(editor_state)` は、`CocoroConsole` の編集 UI が扱う editor bundle 全体を置き換える補助操作である。

この操作で変えるのは次である。

- `current`
- `personas`
- `memory_sets`
- `model_presets`
- `model_profiles`

この操作で変えないのは次である。

- `runtime_state`
- `world_state`
- 記憶 DB の内部テーブル

validator が確認する項目は次とする。

- `current` が object であること
- `personas`、`memory_sets`、`model_presets`、`model_profiles` が配列であり、それぞれ空でないこと
- 各配列の要素が ID を持ち、重複しないこと
- 各 `persona`、`memory_set`、`model_profile` が個別契約を満たすこと
- 各 `model_preset` が正規化後に個別契約を満たすこと
- `current.selected_persona_id`、`current.selected_memory_set_id`、`current.selected_model_preset_id` が bundle 内の資源を指していること
- `current.memory_enabled`、`current.desktop_watch`、`current.wake_policy` が個別契約を満たすこと

この操作は、通常の意味単位操作を一般化したものではない。
editor UI の一括保存だけを対象にした、明示的な bundle 置換として扱う。
成功時は bundle 全体を整合性つきで置き換え、部分反映しない。

<!-- Failure -->
## 失敗時の扱い

設定操作では、失敗時に部分反映しない。

失敗として返す主な条件は次である。

- 指定 ID が存在しない
- 読み込み対象が壊れている
- `wake_policy` が契約を満たさない
- `memory_enabled` または `desktop_watch` が契約を満たさない
- `persona` または `memory_set` の定義が契約を満たさない
- `model_preset` が必須役割を満たさない
- `model_profile` が必要な設定契約を満たさない
- `editor_state` の bundle 整合性が崩れている
- 永続設定の更新に失敗した
- 反映に必要な再起動または再読み込みを完了できなかった

ここで重要なのは、validator 失敗を UI 側で都合よく補正しないことである。
OtomeKairo 側が受け付ける操作意味を保ったまま、成功か失敗かを返す。

<!-- Reflection -->
## 反映単位

MVP では、設定変更の反映単位を「操作 1 件 = 反映 1 件」とする。

つまり、次のように扱う。

- `select_persona` が成功したら、その反映単位は人格切り替え 1 件
- `select_memory_set` が成功したら、その反映単位は記憶切り替え 1 件
- `update_wake_policy` が成功したら、その反映単位は自発判断設定更新 1 件
- `select_model_preset` が成功したら、その反映単位はモデルプリセット切り替え 1 件
- `patch_current` が成功したら、その反映単位は current 設定スナップショット更新 1 件
- `replace_model_preset` が成功したら、その反映単位はプリセット定義更新 1 件
- `replace_model_profile` が成功したら、その反映単位はプロファイル定義更新 1 件
- `replace_editor_state` が成功したら、その反映単位は editor bundle 置換 1 件

ここで入れないのは、任意の複数意味操作を自由に束ねる汎用 transaction である。
`patch_current` と `replace_editor_state` は、その例外を曖昧に広げるためではなく、対象範囲を固定した補助操作として扱う。

<!-- Excluded -->
## MVP でまだ入れない操作

MVP では、次の操作は最小契約に含めない。

- 記憶内部の直接編集
- `runtime_state` の直接変更
- `world_state` の直接書き換え
- 任意の複数意味操作を束ねる汎用 transaction
- 役割ごとの部分 patch 更新

これらは責務が重く、先に入れると設定面と判断面の境界が崩れやすい。

<!-- DesignDecision -->
## ここで固定したい設計判断

この文書で固定したいのは次である。

- 人格切り替え、記憶切り替え、自発判断設定更新を独立操作にする
- モデルプリセット切り替えも独立操作にする
- モデルプロファイル更新も独立操作にする
- `persona` と `memory_set` も設定資源として全体置換で編集できる
- `patch_current` は current 設定スナップショットだけを対象にした補助 patch 面にする
- `replace_editor_state` は editor bundle だけを対象にした補助置換面にする
- 通常操作は 1 つの設定値または 1 つの設定資源だけを変える
- validator 失敗時は部分反映しない
- 反映は MVP では永続設定更新後の次回 request / cycle から行う
- モデルプリセット定義とモデルプロファイル定義は全体置換で編集する
