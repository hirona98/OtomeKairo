# 設定操作契約

<!-- Block: Role -->
## この文書の役割

この文書は、OtomeKairo が受け付ける設定変更と関連設定資源更新の最小操作契約を定める正本である。

ここで固定したいのは次の内容である。

- 設定変更をどの単位で行うか
- 各操作が何を変え、何を変えないか
- validator が何を確認するか
- 反映単位と失敗時の扱い

ここでは API パスや wire 形式の細部は `14_API仕様.md` で定める。
操作の意味だけを固定する。

<!-- Block: Premise -->
## 基本前提

設定変更と関連設定資源更新では、OtomeKairo 側の意味単位をそのまま操作単位にする。
そのため、UI の都合で複数の責務を 1 操作に混ぜない。

MVP の最小操作は次の操作群でよい。

- `select_persona(persona_id)`
- `select_memory_set(memory_set_id)`
- `patch_current(current_patch)`
- `update_wake_policy(wake_policy)`
- `replace_persona(persona_id, persona_definition)`
- `replace_memory_set(memory_set_id, memory_set_definition)`
- `delete_persona(persona_id)`
- `delete_memory_set(memory_set_id)`
- `select_model_preset(model_preset_id)`
- `replace_model_preset(model_preset_id, model_preset_definition)`
- `replace_model_profile(model_profile_id, model_profile_definition)`
- `delete_model_preset(model_preset_id)`
- `delete_model_profile(model_profile_id)`
- `replace_editor_state(editor_state)`

<!-- Block: CommonRule -->
## 共通ルール

設定操作と関連設定資源更新では、次を共通ルールにする。

- 1 回の操作で変える対象は 1 種類だけにする
- validator を通らなかった場合は部分反映しない
- 成功時は、永続設定の正本を先に更新する
- 反映は単純さを優先し、自動再起動または再読み込みで行ってよい
- 指定していない他設定は自動で変えない

`replace_editor_state(editor_state)` は `CocoroConsole` の一括編集保存を受ける補助操作として持ってよい。
この場合も、OtomeKairo 側で整合性を検証し、部分反映しない。

反映のために自動再起動または再読み込みを行う場合、`future_act` 候補キューのような内部保留状態はクリアしてよい。
MVP では、このキューを再起動越しに保持しない。

このルールにより、人格切り替え、記憶切り替え、自発判断設定更新、モデルプリセット切り替えを独立に扱える。

<!-- Block: SelectPersona -->
## `select_persona(persona_id)`

`select_persona(persona_id)` は、現在選択中の人格だけを切り替える操作である。

この操作で変えるのは次だけである。

- `selected_persona_id`

この操作で変えないのは次である。

- `selected_memory_set_id`
- `wake_policy`

validator が確認する項目は次とする。

- `persona_id` が空でないこと
- 指定した `persona_id` が選択可能な人格として存在すること
- その人格が読み込み可能な設定として成立していること

MVP では、成功後に自動再起動または再読み込みで反映してよい。
ただし、人格切り替えの意味は常に「人格だけを変える」で固定する。

<!-- Block: SelectMemory -->
## `select_memory_set(memory_set_id)`

`select_memory_set(memory_set_id)` は、現在選択中の記憶だけを切り替える操作である。

この操作で変えるのは次だけである。

- `selected_memory_set_id`

この操作で変えないのは次である。

- `selected_persona_id`
- `wake_policy`

validator が確認する項目は次とする。

- `memory_set_id` が空でないこと
- 指定した `memory_set_id` が選択可能な記憶として存在すること
- その記憶が読み込み可能な保存対象として成立していること

ここでは、人格との自動連動は行わない。
人格と記憶は独立に選ぶという前提を守るためである。

<!-- Block: UpdateWakePolicy -->
## `update_wake_policy(wake_policy)`

`update_wake_policy(wake_policy)` は、自発判断の機会をどう与えるかだけを更新する操作である。

この操作で変えるのは次だけである。

- `wake_policy`

この操作で変えないのは次である。

- `selected_persona_id`
- `selected_memory_set_id`

MVP で validator が確認する項目は次とする。

- `wake_policy.mode` が `disabled` または `interval` であること
- `mode=interval` のとき `interval_minutes` が 1 以上の整数であること
- `mode=disabled` のとき、余分な起床条件を要求しないこと

`wake_policy` は全体置き換えで扱う。
部分 patch にすると UI と validator が先に複雑になるためである。

<!-- Block: SelectModelPreset -->
## `select_model_preset(model_preset_id)`

`select_model_preset(model_preset_id)` は、現在選択中のモデルプリセットだけを切り替える操作である。

この操作で変えるのは次だけである。

- `selected_model_preset_id`

この操作で変えないのは次である。

- `selected_persona_id`
- `selected_memory_set_id`
- `wake_policy`

validator が確認する項目は次とする。

- `model_preset_id` が空でないこと
- 指定した `model_preset_id` が選択可能なプリセットとして存在すること
- そのプリセットが必須役割を満たしていること
- 参照先の `model_profile` が存在し、実行可能な構成になっていること

MVP では、成功後に自動再起動または再読み込みで反映してよい。
ただし、意味は常に「モデルプリセットだけを切り替える」で固定する。

<!-- Block: ReplaceModelPreset -->
## `replace_model_preset(model_preset_id, model_preset_definition)`

`replace_model_preset(model_preset_id, model_preset_definition)` は、OtomeKairo が保持するモデルプリセット定義を全体置換で作成または更新する操作である。

この操作で変えるのは次である。

- 指定した `model_preset_id` のプリセット定義

この操作で自動で変えないのは次である。

- `selected_model_preset_id`
- `selected_persona_id`
- `selected_memory_set_id`
- `wake_policy`

validator が確認する項目は次とする。

- `model_preset_id` と定義本文の ID が一致すること
- 必須役割がすべて定義されていること
- 各役割が必要な `model_profile_id` を持つこと
- 参照先の `model_profile` が存在すること
- 役割と `model_profile.kind` が整合すること

部分 patch は行わない。
MVP では、プリセットは全体置換で扱う。

<!-- Block: ReplaceModelProfile -->
## `replace_model_profile(model_profile_id, model_profile_definition)`

`replace_model_profile(model_profile_id, model_profile_definition)` は、OtomeKairo が保持するモデルプロファイル定義を全体置換で作成または更新する操作である。

この操作で変えるのは次である。

- 指定した `model_profile_id` のプロファイル定義

この操作で自動で変えないのは次である。

- `selected_model_preset_id`
- `selected_persona_id`
- `selected_memory_set_id`
- `wake_policy`

validator が確認する項目は次とする。

- `model_profile_id` と定義本文の ID が一致すること
- `kind` が `generation` または `embedding` であること
- `provider` と `model_name` が空でないこと
- 必要な接続先情報が定義されていること
- 必要な認証情報が定義されていること

部分 patch は行わない。
MVP では、プロファイルも全体置換で扱う。

<!-- Block: Failure -->
## 失敗時の扱い

設定操作では、失敗時に部分反映しない。

失敗として返す主な条件は次である。

- 指定 ID が存在しない
- 読み込み対象が壊れている
- `wake_policy` が契約を満たさない
- `model_preset` が必須役割を満たさない
- `model_profile` が必要な接続条件または認証条件を満たさない
- 永続設定の更新に失敗した
- 反映に必要な再起動または再読み込みを完了できなかった

ここで重要なのは、validator 失敗を UI 側で都合よく補正しないことである。
OtomeKairo 側が受け付ける操作意味を保ったまま、成功か失敗かを返す。

<!-- Block: Reflection -->
## 反映単位

MVP では、設定変更の反映単位は「操作 1 件 = 反映 1 件」でよい。

つまり、次のように扱う。

- `select_persona` が成功したら、その反映単位は人格切り替え 1 件
- `select_memory_set` が成功したら、その反映単位は記憶切り替え 1 件
- `update_wake_policy` が成功したら、その反映単位は自発判断設定更新 1 件
- `select_model_preset` が成功したら、その反映単位はモデルプリセット切り替え 1 件
- `replace_model_preset` が成功したら、その反映単位はプリセット定義更新 1 件
- `replace_model_profile` が成功したら、その反映単位はプロファイル定義更新 1 件

複数設定の一括トランザクションは、MVP では入れない。
まずは操作意味の単純さを優先する。

<!-- Block: Excluded -->
## MVP でまだ入れない操作

MVP では、次の操作は最小契約に含めない。

- 記憶内部の直接編集
- `runtime_state` の直接変更
- `world_state` の直接書き換え
- 複数設定の同時一括更新
- 役割ごとの部分 patch 更新

これらは責務が重く、先に入れると設定面と判断面の境界が崩れやすい。

<!-- Block: DesignDecision -->
## ここで固定したい設計判断

この文書で固定したいのは次である。

- 人格切り替え、記憶切り替え、自発判断設定更新を独立操作にする
- モデルプリセット切り替えも独立操作にする
- モデルプロファイル更新も独立操作にする
- `persona` と `memory_set` も設定資源として全体置換で編集できる
- `patch_current` で現在設定をまとめて更新できる
- `replace_editor_state` を補助操作として持てる
- 各操作は 1 種類の設定だけを変える
- validator 失敗時は部分反映しない
- 反映は MVP では自動再起動または再読み込みでよい
- モデルプリセット定義とモデルプロファイル定義は全体置換で編集する
