# 自発判断と自発行動

<!-- Role -->
## この文書の役割

この文書は、OtomeKairo が自分から前に出る判断をどう扱うかを定める正本である。

ここで固定したいのは次の内容である。

- 自発判断の内部結果をどう定義するか
- `wake_policy`、`initiative_baseline`、現在文脈をどう分担させるか
- `future_act` を何として保持するか
- 連続起床時の過剰反応をどう抑えるか
- 外向き API と内部判断の境界をどう保つか

ここでは、DB テーブル名までは定めない。
外向き API に関わる厳密な仕様は `14_API仕様.md` と `design/api/` 配下を正とする。
ただし、自発判断の責務と最小構造はここで固定する。

<!-- Premise -->
## 基本前提

MVP の主対象は会話である。
そのため、自発性もまずは自発会話を中心に成立させる。

一方で、OtomeKairo では将来の外界行動まで見据える。
そのため、起床時の判断を単純な `reply / noop` の 2 択で閉じず、「今は話さないが、後で触れる価値がある」という内部結果を持てるようにする。

この文書では、この内部結果を `future_act` と呼ぶ。

<!-- InternalExternalBoundary -->
## 内部結果と外向き結果を分ける

自発判断では、内部結果と外向き結果を分けて扱う。

MVP では、内部判断の結果は次の 3 つでよい。

- `reply`
- `noop`
- `future_act`

一方で、通常 API の外向き結果は引き続き次でよい。

- `reply`
- `noop`
- `internal_failure`

ここで重要なのは、`future_act` を通常 API の外向き結果にしないことである。
`future_act` は「今すぐ外へ出す行動」ではなく、「後で扱う内部候補」である。

<!-- WakeRelation -->
## `wake_policy`、`initiative_baseline`、現在文脈の関係

自発判断では、少なくとも次の 3 つを分けて扱う。

- `wake_policy`
  - いつ判断機会を与えるか
- `initiative_baseline`
  - 判断機会が来たときに、その人格がどれくらい前に出やすいか
- 現在文脈
  - 今この瞬間に介入する自然さがあるか

この分離により、次のように整理できる。

- `wake_policy` だけでは、自発会話の有無は決まらない
- `initiative_baseline` だけでも、今話すべきかは決まらない
- 実際の `reply / noop / future_act` は、その回の現在文脈を加えて決まる

つまり、`wake_policy` は機会生成、`initiative_baseline` は人格傾向、現在文脈はその場の適否を持つ。

<!-- ResultMeaning -->
## `reply / noop / future_act` の意味

MVP では、自発判断の各結果を次のように定義する。

- `reply`
  - 今この瞬間に話しかける自然さと価値が十分にある
- `future_act`
  - 触れる価値はあるが、今すぐ割り込むより少し待つほうが自然である
- `noop`
  - 介入根拠が弱い、重複している、または今は待つべきである

ここでいう `future_act` は、将来の外界行動実行そのものではない。
まずは「自発会話候補を内部で保留する」ための結果として扱う。

<!-- FutureActDefinition -->
## `future_act` は保留候補キューとして持つ

MVP では、`future_act` を単なる一時メモではなく、判断間をまたぐ保留候補キューとして持つ。

このキューは、次の性質を持てばよい。

- OtomeKairo 内部で保持する
- 通常 API の外向き結果にはしない
- 後続の起床判断で再評価できる
- 期限切れや重複整理ができる
- プロセス再起動では引き継がなくてよい

このキューは、記憶の正本そのものではない。
また、`runtime_state` の公開要約そのものでもない。
位置づけとしては、判断間をまたぐ内部保留状態である。
MVP では、通常の起床サイクル間だけ持ち越す短期運用状態として扱う。

<!-- FutureActCandidate -->
## `future_act` 候補の最小構造

MVP では、1 件の `future_act` 候補は少なくとも次を持てばよい。

| 項目 | 役割 |
|------|------|
| `candidate_id` | 候補自体の識別子 |
| `intent_kind` | 何の種類の保留候補か |
| `intent_summary` | 後で何をしたいかの短い要約 |
| `reason_summary` | なぜ今は保留にしたかの要約 |
| `source_cycle_id` | どの判断サイクルから生まれたか |
| `not_before` | これより前には再評価しない時刻 |
| `expires_at` | これを過ぎたら候補を捨ててよい時刻 |
| `dedupe_key` | 同じ候補の重複を抑えるためのキー |

MVP では、`intent_kind` はまず自発会話向けの候補だけを持てばよい。
将来の外界行動候補は、この枠組みを広げて扱う。

<!-- QueueLifecycle -->
## 候補キューの扱い

MVP では、候補キューの扱いを次のように固定してよい。

1. 起床判断で `future_act` を選んだら、新しい候補を作る
2. 同じ `dedupe_key` の未失効候補がある場合は、重複追加せず更新または据え置きにする
3. `not_before` に達するまでは再評価しない
4. 再評価で `reply` に昇格したら、その候補は消費してキューから外す
5. `expires_at` を過ぎた候補は捨てる
6. プロセス再起動では候補を引き継がなくてよい
7. 人格、記憶、モデル設定の反映で自動再起動または再読み込みが起きた場合も、候補をクリアしてよい

この方針により、「同じ話題を何度も保留し続ける」「毎回同じ介入を試みる」といったノイズを抑えやすくなる。

<!-- WakeFlow -->
## 起床時の基本フロー

起床時の判断フローは、MVP では次でよい。

1. `wake_policy` に基づいて起床機会を得る
2. 失効していない `future_act` 候補を読み、再評価対象を絞る
3. 現在観測、`world_state`、記憶、人格設定、`initiative_baseline` を加味する
4. 今すぐ話すなら `reply` を選ぶ
5. 今は待つが価値が続くなら `future_act` を選び、候補を作るか更新する
6. 根拠が弱いなら `noop` を選ぶ

ここでは、既存候補の再評価を先に見る。
そのうえで、その回の新しい文脈が強ければ、新規の `reply` や新規候補生成へ進んでよい。

<!-- DecisionFactors -->
## 判断で強く見る要素

自発判断では、少なくとも次を強く見る。

- 前回の自発会話からどれだけ時間が空いたか
- 同じ `dedupe_key` の候補や直近介入があるか
- 未完了の話題や継続意図があるか
- 関係記憶や嗜好記憶から見て、その話題を持ち出す自然さがあるか
- 現在の軽量クライアント文脈に割り込みにくさがあるか
- `initiative_baseline` がどの程度前向きか

ここで重要なのは、単一の数値閾値だけで決めると固定しないことである。
MVP では、上の要素をまとめて見て `reply / noop / future_act` を選ぶ方針を固定すればよい。

<!-- OverreactionSuppression -->
## 連続起床時の過剰反応を抑える

連続起床時の過剰反応は、`wake_policy` ではなく判断側で抑える。

MVP では、少なくとも次を持てばよい。

- `last_spontaneous_at`
- 同じ `dedupe_key` に対する直近の `reply` 有無
- 同じ `dedupe_key` の未失効候補有無
- `cooldown_until`

この抑制により、起床機会が来ても毎回同じ話題で前に出ることを避けやすくなる。

`wake_policy` は判断機会の頻度だけを持ち、過剰反応抑制はこの内部判断側で扱う。

<!-- RelationToMemory -->
## 記憶との関係

`future_act` 候補キューは、記憶の正本と同一視しない。

記憶が持つのは、関係、継続話題、未完了、嗜好、出来事から育った理解である。
一方で `future_act` 候補キューが持つのは、「次の数回の判断で再評価したい内部保留候補」である。

つまり、記憶は長めの継続知識であり、`future_act` 候補キューは短めの運用状態である。
この短さには、再起動や設定反映で失ってよい一時状態であることも含む。

<!-- ApiAndDebug -->
## API とデバッグ記録への見え方

通常 API では、`future_act` を外向き結果にしない。
クライアントが受け取るのは、引き続き `reply / noop / internal_failure` でよい。

一方で、デバッグ記録には `future_act` を内部結果として残してよい。
MVP では、少なくとも次が追えればよい。

- その回に `future_act` を選んだか
- どの `candidate_id` を作ったか、または更新したか
- `intent_summary`
- `reason_summary`
- `dedupe_key`

この分離により、通常運用の API を単純に保ちつつ、内部判断は後から追える。

<!-- NonGoals -->
## MVP でまだやらないこと

MVP では、次はまだ入れない。

- `future_act` の外界行動への直接実行
- 候補キューの複雑な優先度計算
- 時間帯別の細かな行動ポリシー
- 候補キューのユーザー編集
- 多数候補の同時最適化

まずは、自発会話の保留と再評価を成立させることを優先する。

<!-- DesignDecision -->
## ここで固定したい設計判断

この文書で固定したいのは次である。

- 自発判断の内部結果は `reply / noop / future_act` とする
- `future_act` は通常 API の外向き結果ではなく、内部保留候補とする
- `future_act` は判断間をまたぐ候補キューとして保持する
- 候補は `candidate_id`、`intent_kind`、`intent_summary`、`reason_summary`、`source_cycle_id`、`not_before`、`expires_at`、`dedupe_key` を最小構造にする
- `wake_policy` は判断機会、`initiative_baseline` は人格傾向、現在文脈はその場の適否を持つ
- 連続起床時の過剰反応は、判断側の `last_spontaneous_at`、`dedupe_key`、`cooldown_until` で抑える
- `future_act` 候補キューは記憶の正本ではなく、判断間をまたぐ内部保留状態として扱う
- `future_act` 候補キューは永続設定にも `runtime_state` 公開要約にも入れず、再起動や設定反映時にクリアしてよい
