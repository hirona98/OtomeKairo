# 設定変更

## 基本前提

設定変更では、UI や通信都合ではなく、OtomeKairo 側の意味単位をそのまま変更単位にする。
選択変更、動作設定変更、設定定義の差し替えは、それぞれ別の意味操作である。

そのため、通常は複数の責務をひとつの曖昧な変更に混ぜない。

## 変更の種類

設定変更は、少なくとも次の種類に分ける。

- 現在設定変更
  - 現在どの人格設定、記憶、モデルプリセットを使うか、またはどう動くかを変える
- 設定定義更新
  - 人格設定、記憶集合、モデルプリセットの定義そのものを作成または差し替える
- 設定定義削除
  - 参照されていない定義を取り除く
- 設定一式の一括更新
  - 対象範囲が明確な現在設定と設定定義群だけをまとめて扱う

ここでいう一括更新は、任意の複数責務を束ねる汎用 transaction ではない。
対象範囲と意味が固定されている場合に限る。

## 共通原則

設定変更では、次を共通原則とする。

- validator を通るまで部分更新しない
- 指定していない他の現在設定や他の設定定義を暗黙に変えない
- 更新に失敗したら成功扱いにしない
- 失敗をクライアント側で都合よく補正しない

この原則により、どの変更が何を意味していたかを後から追いやすくする。

## 現在設定変更

現在設定変更は、選択変更と動作設定変更を含む。

### 選択変更

人格設定、記憶、モデルプリセットの選択変更は独立の意味操作として扱う。

ここでは次を守る。

- 人格設定変更が記憶を暗黙に変えない
- 記憶変更が人格設定を暗黙に変えない
- モデルプリセット変更が人格設定や記憶を暗黙に変えない
- 記憶集合の選択は人格設定に従属させない

ただし、選択変更によって現在の作業文脈や `drive_state`、短期内部状態が無効になる場合は、それらの状態を再構築する。
人格設定の途中切り替えと記憶集合の途中切り替えは、どちらも独立に行う。

### 動作設定変更

動作設定変更は、どの設定定義を使うかではなく、本体がどう振る舞うかの条件を変える操作である。

たとえば次を含む。

- 記憶利用方針の変更
- 観測源利用方針の変更
- 自律判断方針の変更

具体的には、`desktop_watch` や `wake_policy` のような現在設定項目の変更がこれに当たる。

動作設定変更も、選択変更や設定定義更新とは別の意味操作として扱う。

## 設定定義更新

人格設定、記憶集合、モデルプリセットのような設定定義更新では、意味単位を壊さないことを優先する。

そのため、少なくとも次を守る。

- 定義の整合性を validator で確認する
- 参照関係が成立しない定義を受け付けない
- 一部だけ壊れた中途半端な定義を正本にしない

部分更新を許すとしても、それは明示的な部分契約がある場合に限る。
定義の意味が見えなくなる細かな patch を前提にしない。

`memory_set` では、空の集合を新規作成するか、既存の `memory_set` を複製して新しい集合として作成する。
`memory_set` の更新では、記憶集合の識別情報だけでなく、その集合に紐づく埋め込み設定も定義の一部として扱う。
`memory_set` の埋め込み設定を変更したときは、その集合に属する vector index を作り直す。
複製後の `memory_set` は、元の集合と独立に更新する。

## 設定定義削除

設定定義削除では、システムを不整合や無効設定へ落とさないことを優先する。

少なくとも次を満たす必要がある。

- 現在参照中の定義を削除しない
- 削除後も有効な設定が保てる
- ぶら下がる内部記録や参照関係の扱いが定義されている

削除は単なるデータ消去ではなく、参照整合性を伴う意味操作として扱う。
`memory_set` の削除は、その集合全体の削除を指す。
個別の `events`、`episodes`、`memory_units` を標準の設定操作として削除しない。

## 正本更新

設定変更では、validator を通った更新内容を、そのまま現在有効な設定として扱う。
設定されたがまだ未確定の中間状態、という概念は標準では持たない。

設計上は次の順序を前提にする。

1. 変更要求を妥当性検証する
2. 正本を更新する
3. 変更後の設定を前提に作業文脈、`drive_state`、短期内部状態を再構築または破棄する

これにより、変更の意味と、その後に再構築すべき短期状態を切り分けて扱いやすくする。

## 一時状態の再構築

設定変更は、長期の正本だけでなく派生する内部状態にも影響する。
特に次のような状態は、変更後の設定に合わせて再構築または破棄する。

- 作業文脈
- `drive_state`
- 保留意図
- 旧設定を前提にした読み込み済み参照

逆に、外界条件そのものや記憶の出来事履歴までを無差別に消さない。

## 失敗時の扱い

設定変更の失敗では、少なくとも次を保証する。

- 部分更新しない
- 失敗理由を観測できる
- 旧設定を保ったまま終える

詳細な wire 契約は `14_API仕様.md` と `design/api/` 配下で定めるが、ここで固定するのは「意味を壊さない変更単位」と「原子性」である。
