# デバッグ可能性

## 基本前提

OtomeKairo は長期伴走を目指すため、後から「なぜその応答や判断になったか」を追える必要がある。

MVP では、完全再生よりも、見通しのよい段階トレースを優先する。
そのため、トークン単位の全記録ではなく、判断サイクルごとの構造化要約を正本として残す。

この文書では、この考え方を B 案として採用する。

## 判断サイクルを記録単位にする

MVP では、デバッグ記録の基本単位は 1 回の判断サイクルである。

1 判断サイクルとは、少なくとも次のどちらかを起点に開始される 1 回の処理を指す。

- 会話観測
- 起床観測
- `desktop_watch` 観測

この単位ごとに `cycle_id` を払い、観測から結果までをひとまとまりとして残す。

## 段階トレースに分ける

MVP では、1 判断サイクルを少なくとも次の 4 段階に分けて記録する。

- `observation_trace`
- `recall_trace`
- `decision_trace`
- `result_trace`

この分離により、次が追いやすくなる。

- 何を受け取ったか
- 何を想起候補にしたか
- 何を根拠に最終判断したか
- 外向き結果として何を返したか

## `cycle_summary`

各判断サイクルには、段階トレースとは別に一覧向けの `cycle_summary` を持つ。

MVP では、少なくとも次を含める。

- `cycle_id`
- `server_id`
- `trigger_kind`
- `started_at`
- `finished_at`
- `selected_persona_id`
- `selected_memory_set_id`
- `selected_model_preset_id`
- 最終結果種別
- 失敗有無

`cycle_summary` は、`CocoroConsole` が最近の判断を一覧で眺めるための最小要約である。

## `observation_trace`

`observation_trace` には、「何を受け取って判断を始めたか」を残す。

MVP では、少なくとも次を含める。

- `trigger_kind`
- ユーザー入力の要約
- 軽量クライアント文脈の要約
- 正規化した観測の要約
- 観測時点の `runtime_state` 要約

ここで重要なのは、生入力の巨大な payload をそのまま丸ごと残すことではなく、後から読める単位に要約して残すことである。

## `recall_trace`

`recall_trace` には、「何を思い出そうとし、何を採ったか」を残す。

MVP では、少なくとも次を含める。

- `RecallHint` の要約
- 想起候補の件数
- 採用した記憶の ID 一覧
- 採用理由の短い要約
- 代表的に落とした候補の短い要約

ここでは、全候補を完全再現できることまでは求めない。
「なぜその記憶が前に出たか」を説明できる程度の痕跡を残す。

## `decision_trace`

`decision_trace` には、「なぜその結論になったか」を残す。

MVP では、少なくとも次を含める。

- 判断結果種別
- 判断理由の要約
- 強く効いた人格要素の要約
- 強く効いた記憶要素の要約
- 強く効いた現在文脈の要約
- `reply / noop / future_act` のどれを主候補にしたか

ここでは、自由文の長い思考ログではなく、後から追える短い構造化要約を優先する。

## `result_trace`

`result_trace` には、最終的に外へ出た結果と、その実行結果を残す。

MVP では、少なくとも次を含める。

- 最終結果種別
- 返答文の要約
- `noop` の理由要約
- `future_act` を選んだ場合の候補 ID と意図要約
- 内部失敗の要約
- 所要時間要約

ここでは、ユーザーへ返した全文を唯一のデバッグ正本にしない。
追跡上の意味を持つ要約を、段階トレースの一部として残す。

## どの時点の設定を残すか

デバッグ記録では、判断開始時点の設定と反映状態を残す。

MVP では、少なくとも次を開始時点でスナップショットとして記録する。

- `server_id`
- `selected_persona_id`
- `selected_memory_set_id`
- `selected_model_preset_id`
- `wake_policy`
- 読み込み済みの人格設定参照
- 読み込み済みの記憶セット参照
- 読み込み済みのモデルプリセット参照

これにより、後から「どの選択値で動いていたか」と「実際に何が反映されていたか」を切り分けて確認できる。

## MVP で保存しないもの

MVP では、次はデバッグ正本として保存しない。

- LLM への完全プロンプト全文
- LLM の生レスポンス全文
- トークン単位の思考過程
- 想起候補の完全一覧
- クライアントから来た巨大 payload の完全複製

これらを先に入れると、保存量と見通しが悪化しやすい。
MVP では、完全再生よりも判断の説明可能性を優先する。

## 通常 API と inspection 面の境界

デバッグ記録の参照は、通常 API の 4 面とは別の inspection 面として扱う。

つまり、MVP では次のように分ける。

- 通常 API
  - 観測面
  - 状態面
  - 設定面
  - 列挙面
- inspection 面
  - 最近の `cycle_summary` 一覧を読む
  - 特定 `cycle_id` の段階トレースを読む
- debug stream 面
  - `logs/stream` で短い段階要約ログを live に読む

この分離により、通常運用の API 概念を崩さずに、運用上の検証経路を追加できる。

## `CocoroConsole` から見える形

`CocoroConsole` からは、少なくとも次を読めるようにする。

- 最近の `cycle_summary` 一覧
- 特定 `cycle_id` の `observation_trace`
- 特定 `cycle_id` の `recall_trace`
- 特定 `cycle_id` の `decision_trace`
- 特定 `cycle_id` の `result_trace`
- `logs/stream` に流れる live の短い段階要約ログ

一方で、`CocoroConsole` からデバッグ記録そのものを編集させない。
デバッグ記録は、OtomeKairo が生成する検証用の正本である。

`logs/stream` は inspection の正本ではなく、`cycle_trace` を live に観測しやすくした派生表示である。
ここでは少なくとも次の種類を流す。

- `Observation`
- `RecallHint`
- `RecallStructured`
- `RecallAssociation`
- `RecallResult`
- `Decision`
- `Result`
- `Memory`

## 保存量の扱い

MVP では、保存量は無制限にしない。

最小方針は次とする。

- `cycle_summary` は比較的長く残す
- 段階トレース詳細は要約中心に残す
- 古い詳細を削る場合でも、一覧要約は残す

具体的な保持件数や保持期間は実装と運用設定で決める。
この文書では、保存量を抑えつつ一覧性を保つ方針だけを固定する。
