## 優先課題

1. `autonomous_run` の完成度向上
   - 継続監視、待機、capability 連鎖、ユーザー割り込み、pause / resume / cancel を実運用で扱える状態にする。
   - `autonomous_run` は、あとで続ける責務、条件付き通知、見守り、複合行動を保持する中核状態として完成させる。
   - 固定 step 数や固定観測回数ではなく、`autonomous_step_generation` の判断で次の一手を決める。

2. capability 実体の拡張
   - `schedule.status`、`device.status`、`body.status`、`environment.status`、`location.status`、`social.status`、`external.status` の実行実体を増やす。
   - 予定、端末状態、作業環境など、判断に効く現在文脈から優先して connector を整備する。
   - provider 側検索 option と OtomeKairo の capability を混同しない。

3. 記憶品質の評価と調整
   - `turn consolidation`、`reflective consolidation`、`drive_state`、`affect_state` の品質確認ケースを増やす。
   - 会話応答の自然さだけでなく、数日後の判断に効く記憶になっているかを評価する。
   - 何を覚えるか、何を強めるか、何を dormant にするかを代表ケースで確認する。

4. 運用・秘密情報・資格ライフサイクル
   - token、API key、camera account、MCP env の表示例外を endpoint 単位で守る。
   - 資格の発行、更新、失効、復旧の運用手順を明確にする。
   - bootstrap、inspection、status、logs、cycle trace に秘密値の実値を出さない。

5. `CocoroConsole` 側の操作面
   - 設定編集、capability 状態、`autonomous_run` 管理、cycle trace 確認を一貫して扱う操作面を整える。
   - 外部接点は判断や設定の正本ではなく、OtomeKairo 本体の操作面として振る舞う。
   - inspection は通常運用面と分け、判断経路の確認に使う。

6. スキーマ変更時の運用方針
   - `memory.db` と `config.db` の schema 変更時に、どのデータを保持し、どの手順で再作成するかを明確にする。
   - 旧仕様との互換レイヤーを追加せず、現行 schema を正として扱う。
   - 長期運用前に、破壊的変更時のバックアップ、検証、復旧手順を定める。

## 運用方針

この文書は、優先順位と着手先を示す入口として更新する。
各課題の詳細仕様は、該当する正本文書へ書く。
