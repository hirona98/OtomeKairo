# 判断と行動

## 基本方針

OtomeKairo では、伝達、外界への作用、待機を別々の仕組みに分けない。
すべてを「今この状況で何をするべきか」を判断する同じ枠組みの中で扱う。

このとき、会話は多数ある行動形態のひとつであり、設計の中心を占有しない。
記憶利用の共通本体と判断文脈の組み立ては `memory/03_想起と判断.md` を正とする。

## 判断に渡す入力

判断では、少なくとも次を分けて扱う。

- 現在の個の文脈
- 記憶文脈
- 現在観測
- 世界状態（`world_state`）
- 実行状態（`runtime_state`）
- 能力マニフェスト
- 能力状態（`capability_state`）

ここでいう現在の個の文脈は、人格設定と記憶から組み上がる判断主体の文脈を指す。

ここで重要なのは、記憶だけで行動を決めないことである。
OtomeKairo は「何を知っているか」だけでなく、「いま何が見えているか」「何ができるか」「どの条件で動いているか」を合わせて判断する。

## 判断結果の種類

判断の結果は、大きく次の 4 つに分ける。

- 伝達
  - 人への返答、提案、確認、通知
- 能力実行
  - 外部サービス利用、身体や機器への作用、情報取得
- 保留意図
  - 今回は前へ出さず、後で再評価する短期の内部意図を残す
- 見送り
  - 今回は前へ出ず、内部意図も残さない

外向き結果と内部結果は同一にしない。

## LLM と能力の関係

判断の中心は LLM に置く。
ただし、LLM が実行境界を直接支配する形にはしない。
意味判断は可能な限り LLM が担い、コードは capability 境界と安全側の制御を担う。

役割分担は次のとおりである。

- コード
  - どの能力が存在するかを定義する
  - 実行条件と接続境界を管理する
  - 実行そのものを担う
  - validator、権限、状態遷移、監査を担う
- LLM
  - その時点の文脈から意図を組み立てる
  - 宣言された能力の中から何を使うべきかを選ぶ
  - 伝達内容や実行要求の意味を整える
  - 可能なら候補の意味的優先付けも担う

つまり、OtomeKairo は「できること」を宣言された範囲の中で意図形成する。
repo 全体の原則は [20_LLM判断優先方針.md](20_LLM判断優先方針.md) を正とする。

## 能力マニフェスト

能力は、個別実装の詳細ではなく、OtomeKairo にとっての行動可能性を定義する契約として外部化する。

この設計により、次が成立する。

- OtomeKairo は、いま使える能力を明示的に把握できる
- 能力が増減しても、判断の枠組みを変えずに済む
- 実装都合の推測ではなく、宣言された能力境界に基づいて選択できる

## 能力マニフェストの最小契約

能力マニフェストは、少なくとも次の概念を持つ。

- 能力の識別子
- 何ができるかの説明
- 前提条件と禁止条件
- 実行に必要な入力の概念
- 実行によって起こりうる作用の概念
- 成功結果と失敗結果の概念
- 現在有効かどうか

LLM には、この契約に基づいて「何が可能か」と「どこまで許されるか」を渡す。
一方で、内部接続方法や秘密情報の扱いまで見せない。

## 自律判断

OtomeKairo の自律性とは、人からの直近入力がなくても、自分の判断機会を持ち、必要なら前へ出ることである。
ただし、自律判断は独立した別系統ではなく、通常の判断と同じ枠組みの延長にある。

## 機会生成と判断を分ける

自律判断では、少なくとも次の 3 つを分けて扱う。

- 自律判断方針（`wake_policy`）
  - いつ判断機会を与えるか
- 自発傾向（`initiative_baseline`）
  - 機会が来たとき、その人格設定でどれくらい前へ出やすいか
- 現在文脈
  - いま介入する自然さや必要性があるか

`wake_policy` は機会生成、自発傾向は人格設定由来の傾向、現在文脈はその場の適否を持つ。
この 3 つを混ぜないことで、自律性を頻度だけの設定にしない。

## 保留意図

保留意図は、「いま外へ出さないが、後で再評価する価値がある内部意図」である。
長期記憶でも現在設定でもなく、判断間をまたぐ短期の内部状態として扱う。

少なくとも次の性質を持つ。

- OtomeKairo 内部で保持する
- 後続の自律判断で再評価できる
- 期限切れや重複整理ができる
- 設定変更時には再構築または破棄できる

## 保留意図の最小構造

1 件の保留意図は、少なくとも次を持つ。

| 項目 | 役割 |
|------|------|
| `candidate_id` | 候補自体の識別子 |
| `intent_kind` | 何の種類の意図か |
| `intent_summary` | 後で何をしたいかの短い要約 |
| `reason_summary` | なぜ今は保留にしたかの要約 |
| `source_cycle_id` | どの判断サイクルから生まれたか |
| `not_before` | これより前には再評価しない時刻 |
| `expires_at` | これを過ぎたら捨てる時刻 |
| `dedupe_key` | 同じ候補の重複を抑えるためのキー |

## 自律判断の基本フロー

自律判断の流れは、少なくとも次を含む。

1. 自律判断方針に基づいて判断機会を得る
2. 未失効の保留意図を読み、再評価対象を絞る
3. 現在観測、世界状態、記憶、人格設定、能力状態を加味する
4. いま前へ出るなら伝達または行動を選ぶ
5. 今ではないが価値が続くなら保留意図を作るか更新する
6. 根拠が弱いなら見送る

既存候補の再評価を先に見つつ、その回の新しい文脈が強ければ新規の前進判断へ進む。

## 過剰介入を抑える

起床機会が多くても、毎回同じ話題で前へ出るべきではない。
そのため、自律判断側で少なくとも次を持つ。

- `last_spontaneous_at`
- 同じ `dedupe_key` の直近介入有無
- 同じ `dedupe_key` の未失効候補有無
- `cooldown_until`

ここで重要なのは、過剰反応抑制を機会生成側だけに背負わせないことだ。

## 記憶との関係

保留意図は記憶の正本と同一視しない。

- 記憶
  - 関係、継続話題、未完了、嗜好、出来事から育った理解を持つ
- 保留意図
  - 次の数回の判断で再評価したい短期の内部候補を持つ

つまり、記憶は長めの継続知識であり、保留意図は短めの運用状態である。

## 実行結果の扱い

行動は、選んだ時点で終わりではない。
実行後の結果は、少なくとも次へ戻す必要がある。

- 現在観測
  - 今回返ってきた結果そのもの
- 世界状態
  - 次回以降にも持ち越すべき外界条件
- 実行状態
  - 実行中かどうか、再試行可否、接続状態
- 記憶
  - 経験として残すべき出来事と理解

失敗も「なかったこと」にしない。
失敗は判断や記憶更新に効く出来事として扱う。
後から返る実行結果は、その時点の新しい経験として別 `episode` に残し、同じ継続中の実行に属する場合だけ同じ `episode_series_id` で束ねる。

## 拡張方針

観測能力、情報取得能力、外界作用能力が増えても、次の構造は維持する。

- 判断入力は現在の個の文脈、記憶、状態、能力から組む
- 能力は宣言される
- LLM はその中から選ぶ
- 実行境界はコード側が管理する
- 結果は状態、記憶、デバッグ記録へ戻る
