# プロンプト文脈分離方針

## 目的

OtomeKairo は、現在入力、ユーザー発話、内部コンテキスト、検索用クエリ、記憶更新入力を別概念として扱う。
LLM に渡す入力では、ユーザーが実際に発話した内容と、システムが補助的に観測・要約した内容を混ぜない。

この文書は、LLM prompt に渡す文脈の意味境界を定める。

## 現在入力と内部コンテキスト

会話判断と返信生成では、その判断サイクルの入口を `current_input` として扱う。
`current_input` は、誰から来た入力か、どの起点か、誰への直接応答として扱うか、本文を分けて持つ。

`current_input` は少なくとも次を持つ。

| 項目 | 役割 |
|------|------|
| `sender` | `user / system / capability / assistant` のいずれか。入力の発信主体を表す |
| `source_kind` | `user_message / wake / background_wake / capability_result` などの処理起点を表す |
| `response_target` | `user / none` のいずれか。今回 reply になった場合の直接応答先を表す |
| `text` | 入力本文。`sender=user` ではユーザー原文、非ユーザーでは観測・起床・実行結果の短い説明を入れる |

`sender=user` かつ `response_target=user` の `text` だけを、ユーザーが実際に発話した本文として扱う。
`sender=system / capability / assistant` の `text` は判断材料であり、ユーザー発話ではない。
`sender=user` かつ `response_target=user` の `text` が非空の場合、明示的な返信不要表現を除き `decision.kind=noop` を契約違反として repair する。

画像要約、時刻、想起結果、現在状態、capability result は、`internal_context` の構造化データとして渡す。

LLM から見た役割境界は次の形にする。

```text
system: 役割、人格、出力制約
user/internal_context: システムが注入する補助文脈
user/current_input: その判断サイクルの入口を表す current_input JSON
```

`current_input.text` に `添付画像では、...` のような内部要約を連結しない。
ユーザー発話と内部要約を連結すると、LLM は「ユーザーが言った内容」と「システムが見た内容」を区別できない。

会話中核 prompt では、役割・制約の system message と、内部注入専用の context message と、現在入力専用の input message を分ける。
内部注入データは system role に置かず、`<<<OTOMEKAIRO_INTERNAL_CONTEXT>>>` のような reserved sentinel で囲った compact JSON とする。
現在入力専用の input message は `<<<OTOMEKAIRO_CURRENT_INPUT>>>` で囲った compact JSON とし、同じ message に `recent_turns` や `recall_hint` を混ぜない。

## 検索用クエリ

想起、検索、埋め込み、根拠解決では、画像要約や視覚要約を含む拡張クエリを使う。
この文字列は `augmented_query_text` として扱い、`current_input.text` とは別名にする。

```text
current_input.text: 現在入力の本文
augmented_query_text: 検索用に組み立てた現在入力本文 + 画像要約・視覚要約
```

検索用に混ぜた情報を、会話入力の正本として扱わない。
検索用の内部要約を文字列へ追加する場合は、通常のユーザー入力と衝突しにくい次の区切りを使う。

```text
<<<OTOMEKAIRO_INTERNAL_CONTEXT conversation_attachment_visual_summary>>>
...
<<<END_OTOMEKAIRO_INTERNAL_CONTEXT>>>
```

`[要約]` のような自然に入力され得る短い区切りは使わない。

recall、embedding、evidence resolution のうち、検索や根拠抽出に画像要約が必要な経路だけ `augmented_query_text` を使う。
world_state source pack は会話添付画像の `visual_summary_text` を受け取らない。

`augmented_query_text` は内部派生値であり、ユーザー発話ではない。
inspection では `current_input` と `augmented_query_text` を別フィールドとして表示する。

## 画像要約

会話添付画像は `visual_observation_context` として扱う。
`visual_observation_context.source=conversation_attachment` かつ `image_interpreted=true` のとき、画像は `visual_summary_text` として解釈済みである。

decision / reply は、raw image が prompt に無いことを理由に画像欠落と判断しない。
`visual_summary_text` に無い細部は断定せず、短い確認質問へ寄せる。

decision / reply には `current_input` と、構造化された `visual_observation_context` を別々に渡す。
input_interpretation でも、ユーザー原文と補助観測を別フィールドにする。

LLM prompt では、`visual_observation_context` が内部文脈であり、ユーザー発話ではないことを明示する。

画像要約に失敗した場合、画像付き会話は `internal_failure` として残す。
silent fallback で画像なし会話へ丸めない。

## プロンプト種別

会話用の人格表現、装飾タグ、口調、文字数指示は、返信生成だけに適用する。
input interpretation、decision、world_state、memory、visual_observation の JSON 生成には、会話本文向けの装飾指示を入れない。

内部 JSON 生成では、スキーマ、禁止事項、根拠境界、作ってはいけない事実を優先する。

## 外部入力

background wake、capability result、通知、定期観測、vision.capture result はユーザー発話ではない。
これらは専用の trigger context として扱い、通常の user message と同じ `current_input` 解釈へ寄せない。

capability result では、source capability の結果を受けた follow-up であることを明示する。
background wake では、自律判断トリガーであることを明示する。
capability request は dispatch 時点の `current_input` を `source_current_input` として request record に保持する。
capability result は `sender=capability` とし、`response_target` は source request の `source_current_input.response_target` を引き継ぐ。
ユーザー入力から始まった capability result は `response_target=user`、system 起点の wake / background_wake から始まった capability result は `response_target=none` とする。
capability が発信主体であることと、ユーザーへ返すべき follow-up であることを分ける。
background wake と wake は `sender=system` かつ `response_target=none` を既定とする。

## RecallPack 境界

RecallPack は想起・検索結果だけを表す。
TimeContext、VisualObservationContext、ForegroundWorldState、CapabilityResultContext、InitiativeContext は RecallPack に混ぜない。

現在ターンの観測と過去記憶を同じ pack に混ぜると、デバッグ時に根拠の出所が曖昧になる。

## ログと inspection

LLM 入出力ログでは、内部コンテキストを構造化データとして読める形にする。
data URI、base64、API key、token、authorization はログに出さない。

ログの目的は、LLM が何を現在入力として受け取り、そのうち何をユーザー発話として扱い、何を内部文脈として受け取ったかを確認できることにある。

LLM 送信ログでは、`current_input`、`augmented_query_text`、`visual_observation_context` の区別が見えるようにする。
core conversation prompt では、internal_context 専用 block と current_input 専用 block を役割ごとに追えるようにする。
ログには raw image、base64、API key、token を出さない。

inspection では `current_input.sender / source_kind / response_target / text`、内部視覚文脈、検索用拡張クエリの差を追えるようにする。

## trace 名

trace では `input_context_addition_summary` と `augmented_query_summary` を使い、内部補助文脈と検索・想起用拡張文字列を分ける。

## memory / event 保存境界

user event の本文は `current_input.sender=user` の原文だけを保存する。
画像要約や視覚要約は、観測要約または内部 context として別保存する。

記憶更新では、画像要約を根拠材料として渡すが、ユーザーが発話した本文として扱わない。
