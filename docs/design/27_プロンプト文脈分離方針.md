# プロンプト文脈分離方針

## 目的

OtomeKairo は、ユーザー発話、内部コンテキスト、検索用クエリ、記憶更新入力を別概念として扱う。
LLM に渡す入力では、ユーザーが実際に発話した内容と、システムが補助的に観測・要約した内容を混ぜない。

この文書は、`CocoroGhost` のプロンプト関連処理を参考に、OtomeKairo へ取り込む方針と取り込まない方針を定める。

## 参考にすべき点

### ユーザー発話と内部コンテキストを分離する

会話判断と返信生成では、ユーザー発話を原文のまま `input_text` として扱う。
画像要約、時刻、想起結果、現在状態、capability result は、`internal_context` の構造化データとして渡す。

LLM から見た役割境界は次の形にする。

```text
system: 役割、人格、出力制約
assistant/internal_context: システムが注入する補助文脈
user: ユーザーが実際に発話した本文
```

`input_text` に `添付画像では、...` のような内部要約を連結しない。
ユーザー発話と内部要約を連結すると、LLM は「ユーザーが言った内容」と「システムが見た内容」を区別できない。

### 検索用クエリだけ拡張する

想起、検索、埋め込み、根拠解決では、画像要約や視覚要約を含む拡張クエリを使う。
この文字列は `augmented_query_text` として扱い、`input_text` とは別名にする。

```text
input_text: ユーザー原文
augmented_query_text: ユーザー原文 + 検索用の画像要約・視覚要約
```

検索用に混ぜた情報を、会話入力の正本として扱わない。

### 画像要約を内部コンテキストとして渡す

会話添付画像は `visual_observation_context` として扱う。
`visual_observation_context.source=conversation_attachment` かつ `image_interpreted=true` のとき、画像は `visual_summary_text` として解釈済みである。

decision / reply は、raw image が prompt に無いことを理由に画像欠落と判断しない。
`visual_summary_text` に無い細部は断定せず、短い確認質問へ寄せる。

### 会話用プロンプトと内部 JSON 生成プロンプトを分ける

会話用の人格表現、装飾タグ、口調、文字数指示は、返信生成だけに適用する。
input interpretation、decision、world_state、memory、visual_observation の JSON 生成には、会話本文向けの装飾指示を入れない。

内部 JSON 生成では、スキーマ、禁止事項、根拠境界、作ってはいけない事実を優先する。

### 外部入力とユーザー発話を分ける

background wake、capability result、通知、定期観測、vision.capture result はユーザー発話ではない。
これらは専用の trigger context として扱い、通常の user message と同じ `input_text` 解釈へ寄せない。

capability result では、source capability の結果を受けた follow-up であることを明示する。
background wake では、自律判断トリガーであることを明示する。

### 現在文脈と検索結果を混ぜない

RecallPack は想起・検索結果だけを表す。
TimeContext、VisualObservationContext、ForegroundWorldState、CapabilityResultContext、InitiativeContext は RecallPack に混ぜない。

現在ターンの観測と過去記憶を同じ pack に混ぜると、デバッグ時に根拠の出所が曖昧になる。

### ログでは内部コンテキストを構造化して読めるようにする

LLM 入出力ログでは、内部コンテキストを構造化データとして読める形にする。
data URI、base64、API key、token、authorization はログに出さない。

ログの目的は、LLM が何をユーザー発話として受け取り、何を内部文脈として受け取ったかを確認できることにある。

### 画像処理は best effort で継続する

画像要約に失敗した場合でも、会話全体を落とさない。
有効画像があるのに要約が空になった場合はログに残し、返信では画像を断定せず確認へ寄せる。

ただし、OtomeKairo の capability result で必須契約を満たせない場合は、既存の capability failure として扱う。

## 取り込まない点

### CocoroGhost の検索実装そのものは移植しない

`CocoroGhost` の rule-based retrieval、SearchPlan 廃止、SearchResultPack の候補選別方式は、OtomeKairo の RecallPack 設計とは別物である。
OtomeKairo は既存の RecallPack、evidence resolution、memory link 設計を正とする。

### SSE ストリーミング前提の実装は移植しない

`CocoroGhost` の SSE 応答生成、逐次 sanitizer、stream delta 処理は、OtomeKairo の現在 API とは前提が違う。
OtomeKairo では必要になった時点で別途設計する。

### 複数画像最大 5 枚の仕様は急いで合わせない

`CocoroGhost` は複数画像を順序付き配列として扱う。
OtomeKairo は現在の対話画像入力仕様を正とし、複数画像対応は別変更で扱う。

### デスクトップウォッチ仕様は復活させない

`CocoroGhost` の desktop_watch は参考実装として読む。
OtomeKairo では desktop_watch を独立機能として復活させず、視覚機能と wake_policy の統合設計を正とする。

## OtomeKairo で次にやるべき修正

### 1. `effective_input_text` の全段階利用を廃止する

`input_text` はユーザー原文だけを表す。
会話添付画像や視覚観測の要約を `input_text` に連結しない。

既存の `effective_input_text` は廃止し、検索専用の `augmented_query_text` へ置き換える。

### 2. `augmented_query_text` を導入する

recall、embedding、evidence resolution のうち、検索や根拠抽出に画像要約が必要な経路だけ `augmented_query_text` を使う。
world_state source pack は会話添付画像の `visual_summary_text` を受け取らない。

`augmented_query_text` は内部派生値であり、ユーザー発話ではない。
inspection では `input_text` と `augmented_query_text` を別フィールドとして表示する。

### 3. decision / reply / input_interpretation を構造化 context 化する

decision / reply にはユーザー原文の `input_text` と、構造化された `visual_observation_context` を別々に渡す。
input_interpretation でも、ユーザー原文と補助観測を別フィールドにする。

LLM prompt では、`visual_observation_context` が内部文脈であり、ユーザー発話ではないことを明示する。

### 4. trace 名を整理する

`effective_input_addition_summary` は廃止する。
trace では `input_context_addition_summary` と `augmented_query_summary` を使い、内部補助文脈と検索・想起用拡張文字列を分ける。

旧名との互換レイヤーは入れない。

### 5. memory / event 保存の境界を確認する

user event の本文はユーザー原文だけを保存する。
画像要約や視覚要約は、観測要約または内部 context として別保存する。

記憶更新では、画像要約を根拠材料として渡すが、ユーザーが発話した本文として扱わない。

### 6. prompt preview とログを改善する

LLM 送信ログでは、`input_text`、`augmented_query_text`、`visual_observation_context` の区別が見えるようにする。
ログには raw image、base64、API key、token を出さない。

### 7. ドキュメントを正本へ反映する

`24_対話画像入力.md` は会話添付画像の API と保存境界を定める。
この文書は、LLM prompt に渡す文脈分離方針を定める。
`memory/03_想起と判断.md` は recall / evidence で使う `augmented_query_text` の扱いを定める。

同じ仕様を複数文書に正本として重複記述しない。

## 完了条件

- `input_text` はすべての会話判断・返信でユーザー原文だけを指す
- 画像要約は `visual_observation_context` として input_interpretation / decision / reply へ渡る
- 検索用の拡張文字列は `augmented_query_text` として `input_text` から分離される
- inspection でユーザー原文、内部視覚文脈、検索用拡張クエリの差が追える
- 画像付き会話で「画像データが欠落している」という誤った `noop` が発生しない
