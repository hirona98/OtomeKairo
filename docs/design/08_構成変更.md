# 構成変更

## 基本前提

構成変更では、UI や通信都合ではなく、OtomeKairo 側の意味単位をそのまま変更単位にする。
人格の切り替え、記憶の切り替え、方針変更、定義差し替えは、それぞれ別の意味操作である。

そのため、通常は複数の責務をひとつの曖昧な変更に混ぜない。

## 変更の種類

構成変更は、少なくとも次の種類に分ける。

- 選択変更
  - 現在どの人格、記憶、モデルを使うかを変える
- 方針変更
  - 記憶利用、自律判断、観測源利用などの動作条件を変える
- 資源更新
  - 人格定義やモデル定義そのものを作成または差し替える
- 資源削除
  - 参照されていない定義を取り除く
- 構成束の一括反映
  - 対象範囲が明確な構成束だけをまとめて扱う

ここでいう一括反映は、任意の複数責務を束ねる汎用 transaction ではない。
対象範囲と意味が固定されている場合に限る。

## 共通原則

構成変更では、次を共通原則とする。

- validator を通るまで部分反映しない
- 指定していない他設定や他資源を暗黙に変えない
- 正本の更新と反映状態の更新を分けて考える
- 反映に失敗したら成功扱いにしない
- 失敗をクライアント側で都合よく補正しない

この原則により、どの変更が何を意味していたかを後から追いやすくする。

## 選択変更

人格、記憶、モデルの選択変更は独立の意味操作として扱う。

ここでは次を守る。

- 人格変更が記憶を暗黙に変えない
- 記憶変更が人格を暗黙に変えない
- モデル変更が人格や記憶を暗黙に変えない

ただし、選択変更によって現在の作業文脈や短期内部状態が無効になる場合は、その短期状態を再構築する。

## 方針変更

方針変更は、どの構成資源を使うかではなく、本体がどう振る舞うかの条件を変える操作である。

たとえば次を含む。

- 記憶利用方針の変更
- 観測源利用方針の変更
- 自律判断方針の変更

方針変更も、選択変更や資源更新とは別の意味操作として扱う。

## 資源更新

人格、記憶集合、モデル構成のような資源更新では、意味単位を壊さないことを優先する。

そのため、少なくとも次を守る。

- 定義の整合性を validator で確認する
- 参照関係が成立しない定義を受け付けない
- 一部だけ壊れた中途半端な定義を正本にしない

部分更新を許すとしても、それは明示的な部分契約がある場合に限る。
定義の意味が見えなくなる細かな patch を前提にしない。

## 削除

資源削除では、システムを不整合や無効構成へ落とさないことを優先する。

少なくとも次を満たす必要がある。

- 現在参照中の資源を削除しない
- 削除後も有効な構成が保てる
- ぶら下がる内部記録や参照関係の扱いが定義されている

削除は単なるデータ消去ではなく、参照整合性を伴う意味操作として扱う。

## 正本更新と反映

構成変更では、「設定上そうなったこと」と「実際にそう反映されたこと」を分ける。

設計上は次の順序を前提にする。

1. 変更要求を妥当性検証する
2. 正本を更新する
3. 次回以降の判断や実行へ反映する
4. 実行状態やデバッグ記録から反映結果を追えるようにする

これにより、何を変えたかったのかと、いつから実際に効いたのかを切り分けられる。

## 一時状態の再構築

構成変更は、長期の正本だけでなく短期の内部状態にも影響する。
特に次のような状態は、変更後の構成に合わせて再構築または破棄する。

- 作業文脈
- 保留意図
- 旧構成を前提にした読み込み済み参照

逆に、外界条件そのものや記憶の出来事履歴までを無差別に消さない。

## 失敗時の扱い

構成変更の失敗では、少なくとも次を保証する。

- 部分反映しない
- 失敗理由を観測できる
- 旧構成を保ったまま終える

詳細な wire 契約は `14_API仕様.md` と `design/api/` 配下で定めるが、ここで固定するのは「意味を壊さない変更単位」と「原子性」である。
