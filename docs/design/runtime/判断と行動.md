# 判断と行動

## 基本方針

OtomeKairo では、伝達、外界への作用、待機を別々の仕組みに分けない。
すべてを「今この状況で何をするべきか」を判断する同じ枠組みの中で扱う。

このとき、会話は多数ある行動形態のひとつであり、設計の中心を占有しない。
記憶利用の共通本体と判断文脈の組み立ては [../memory/想起と判断.md](../memory/想起と判断.md) を正とする。

## 判断に渡す入力

判断では、少なくとも次を分けて扱う。

- 現在の個の文脈
- 記憶文脈
- 志向状態（`drive_state`）
- 継続中の能力実行（`ongoing_action`）
- 現在入力
- 世界状態（`world_state`）
- 実行状態（`runtime_state`）
- 能力マニフェスト
- 能力状態（`capability_state`）

ここでいう現在の個の文脈は、人格設定と記憶を基盤に、必要な `drive_state` や感情前景を含めて成立する判断主体の文脈を指す。

ここで重要なのは、記憶だけで行動を決めないことである。
OtomeKairo は「何を知っているか」だけでなく、「いま何を気にかけているか」「いま何が見えているか」「何ができるか」「どの条件で動いているか」を合わせて判断する。
現在判断に近い外界条件の正本は [world_state.md](world_state.md) を正とする。

判断文脈へ渡す現在時刻は、OtomeKairo の生活ローカル時刻を自然文へ整形したものにする。
LLM には UTC timestamp を渡さない。
LLM には offset 付き ISO timestamp を判断文脈の主要表現として渡さない。
時刻の正本表現は [時刻モデル.md](時刻モデル.md) を正とする。

## 判断コンテキスト

判断モデルへ渡す統合コンテキストの正本は、この文書で定める。
[../memory/想起と判断.md](../memory/想起と判断.md) が定める `TimeContext / AffectContext / RecallPack` は、統合コンテキストのうち想起系が作る記憶文脈部分である。

判断コンテキストは、少なくとも次を持つ。

| 項目 | 役割 |
|------|------|
| `persona_context` | 現在選択中の人格設定から role ごとに組み立てる runtime 文脈 |
| `time_context` | 生活文脈向けに整形した現在時刻と相対時間要約 |
| `affect_context` | `mood_state`、関連 `affect_state`、必要時の直近 `episode_affect` の要約 |
| `recall_pack` | 想起で選ばれた記憶文脈 |
| `current_input` | その判断サイクルの入力、観測、API起床要求、実行結果の発信主体と本文 |
| `foreground_world_state` | 判断へ前景として渡す `world_state` 要約 |
| `activity_context` | ユーザーの現在活動と直前活動の短期推定要約 |
| `runtime_summary` | 稼働状態、worker、scheduler、接続状態の要約 |
| `capability_decision_view` | manifest、binding、`capability_state` から server が組み立てた能力判断用 view |
| `drive_summaries` | 未失効で前景に出す `drive_state` 要約 |
| `autonomous_run_summaries` | active / waiting_timer / waiting_result / paused の `autonomous_run` 要約 |
| `ongoing_action_summary` | 未失効の `ongoing_action` がある場合の要約 |
| `pending_intent_summaries` | 再評価対象の保留意図要約 |
| `initiative_context` | 自律判断時だけ入る作業文脈要約 |
| `self_state_context` | AI 本体側の感覚信頼度、働きかけやすさ、継続行動の安定を表す短期自己状態 view |
| `relationship_context` | 関係記憶と関係感情から作る現在関係 view |
| `prediction_error_context` | 外界状態や capability result の構造化差分候補 view |
| `default_mode_context` | 記憶、未完了、関係、直近感情の静かな再浮上候補 view |
| `workspace_context` | 記憶、外界状態、志向状態、継続行動、能力候補を同じ盤面に並べた前景化候補 view |

`self_state_context` は `mood_state` と統合しない。
`mood_state` は valence / arousal / dominance を持つ気分状態である。
`self_state_context` は、感覚信頼度、外界へ働きかけられる感覚、OtomeKairo 自身の ongoing action の安定だけを短期派生 view として持つ。
`self_state_context.focus_stability` の対象は OtomeKairo 自身の ongoing action の安定である。
`self_state_context.focus_stability` はユーザーの注意状態を表さない。
`self_state_context` は、発話の断定度、確認の頻度、控えめさ、能力選択の慎重さに使う。

`relationship_context` は `recall_pack.relationship_model`、関係に関わる `user_model / active_commitments / active_topics`、`entity_registry`、`affect_context` から作る。
これは長期関係記憶の正本ではなく、「最近の関係温度」「相手の好み」「避けるべき境界」「継続中の話題」を判断へ渡すための短い現在 view である。

`prediction_error_context` は、直前の `foreground_world_state` と今回の `foreground_world_state` の構造化差分、capability result の構造化結果を差分候補として持つ。
自然文どうしの意味差分を server が判定しない。
差分候補を驚き、確認、見送り、記憶更新のどれへ効かせるかは `decision_generation` が判断する。

`default_mode_context` は、現在入力へ即応していない記憶、未完了、関係、直近感情を静かに再浮上させる派生 view である。
`default_mode_context` の候補は静かな再浮上材料である。
候補は `workspace_context` の前景化候補として扱い、現在状況と距離感を合わせて `speech / noop / pending_intent / autonomous_run` のどれへ置くか判断する。

`workspace_context` は判断用の派生 view である。
`workspace_context.workspace_candidates` は、各候補に `factor_ref / kind / source / summary_text` を持つ。
`factor_ref` は同一判断サイクル内の参照 ID であり、`decision_generation` の `foreground_selection` が参照する。
server は `workspace_context` の候補収集、件数制限、`factor_ref` の一意化だけを行う。
候補同士の意味的な主役選択は `decision_generation` が同じ LLM 呼び出し内で行う。
`workspace_context.workspace_candidates[].kind=suppression` は、前へ出る理由ではなく控える理由を前景候補として表す。
`workspace_context` は `events / episodes / memory_units / affect / mood_state / world_state / drive_state` などの正本を更新しない。

自律判断時の `initiative_context.foreground_signal_summary.visual_observations` は、desktop / camera / virtual などの視覚観測を source 共通の判断材料として持つ。
採用済み camera source の観測は `source_owner=self` として OtomeKairo の視覚根拠にできる。
`visual_observations[].change_state=first_seen / changed` は wake 判断の前景候補として保持する。
`background_thinking` は定期思考による自己評価である。
`decision_generation` は観測、候補、抑制、能力提案を比較し、`speech / noop / pending_intent / capability_request` から 1 つ選ぶ。
`visual_observations[].change_state=first_seen / changed` は前景候補、`stable / same_as_recent_speech` は反復抑制候補である。
`background_thinking` の `speech` は、ユーザーの反応を求めない独話的な短い状況認識である。
`background_thinking` は発話自然度を 10 段階で内的に見積もり、`current.thinking_speech_level` に合わせて `speech` の選びやすさを調整する。
`thinking_speech_level=5` は標準頻度である。
`thinking_speech_level=3` 以下は控えめ基準である。
標準頻度では、短い独話として成立し、明確な反復、観測不足、明示境界、プライバシー境界が上回らない状態を `speech` 候補にする。
控えめ基準では、変化があるから話すのではなく、独話として残す意味が明確な変化だけを `speech` に寄せる。
評価値は JSON や `reason_summary` に出力しない。
`first_seen / changed` は、具体的な前景がある場合に `current.thinking_speech_level` に応じた `speech` 候補として扱う。
`change_state=first_seen / changed` だけでは `speech` を選ばず、画面・対象・操作単位の変化は候補材料に留める。
活動名、作業名、閲覧中、検討中、入力中、操作中などの活動事実は、何が前景にあるかの材料であり、それ自体を `noop` の主理由にしない。
複数 source の `first_seen / changed` が同じ活動遷移や状態変化を指す場合も、`speech / pending_intent / noop` で比較する。
`speech` は、活動モード遷移、同一活動内の意味的な節目、強い関心、予定、未完了、継続中コミットメント、ユーザーが明示的に問題化した観点が読め、短い状況認識として外へ出す新しい意味があり、独話として一文で自然に閉じ、具体的な抑制根拠が上回らない場合に選ぶ。
活動の段階、結果、保留、比較軸、未完了状態が意味的に変わる場合だけ `speech` に寄せる。
緊急性、支援必要性、会話開始としての必要性は `speech` の条件にしない。
`pending_intent` は、今すぐ外へ出す根拠は弱いが、後で再評価する価値が残る場合に選ぶ。
`noop` は、反復、直近で同じ内容に触れた事実、明示された距離希望、進行中応答、結果待ち、プライバシー境界、観測不足、構造化済み抑制根拠が `speech` の価値を明確に上回る場合、または活動事実ではなく観測された前景差分そのものに短い独話として外へ出す新しい意味が薄い場合に選ぶ。
`foreground_signal_summary.foreground_thinness=thin` では、具体的な前景変化があっても独話として外へ出す新しい意味が弱い場合は `noop` または `pending_intent` にする。
集中、没頭、作業中、閲覧中、検討中、入力中、操作中、活動の一環、作業の継続、遮る、割って入る、介入回避、緊急性がないこと、支援要求がないことは `noop` の主理由にしない。
`capability_request` は、`candidate_families` に capability 提案があり、現在判断に追加観測が必要な場合に選ぶ。
同一活動内の意味的な節目は、完了、中断、再開、明確な成果や失敗、対象の意味的な切り替わり、対象の絞り込み、対象間の比較、比較軸の変化、進行阻害、情報確認の完了または保留である。
同じ大きな流れの中の画面・対象・操作単位の変化は、それだけでは節目として扱わず、活動の段階や結果に意味的な変化がある場合だけ `speech` 候補にする。
`speech` は助言、依頼、支援提案、反応要求ではなく、観測事実に基づく一文の独話的な状況認識として成立する場合に選ぶ。
支援提案、作業中断、休息促し、身体注意、画面への一般コメント、長い感想は控える理由側に置く。
作業中、閲覧中、検討中、入力中などの活動事実、`foreground_signal_summary.foreground_thinness=thin`、明示的な呼びかけがないこと、支援要求がないこと、外へ出る必要が薄いという一般的な推定、観測から推定した集中や没頭、内的注意状態、一般的な配慮は、それ単体では `noop` の主理由にしない。
`persona_context` は距離感と表現補助であり、人格として自然という理由だけで、薄い観測や表層的な前景変化を `speech` に押し上げない。
`foreground_drive_summaries` に入っていない `drive_state`、`freshness_hint=stale`、`stability_hint=weak`、`signal_strength=0.0` の `drive_state` は背景材料として扱い、薄い視覚前景と合わせる場合は一般的な関係構築や休息促しを控える理由側に置く。
反復に近い詳細更新、画面・対象・操作単位の小さな変化、観測対象の表層的な変化、姿勢や操作の細かな変化、一般的な注意や助言に留まる内容は、それ単体では `noop` または `pending_intent` の材料として扱う。
活動が継続中であることは、この抑制理由に含めない。
`visual_observations[].change_state=stable / same_as_recent_speech` は反復抑制の候補として扱い、`workspace_context` の `suppression` 候補へ接続する。
画面注視、入力操作、表示内容の閲覧、現在活動ラベルは活動事実として扱い、ユーザーの注意状態は観測から推定しない。
観測事実層、活動推定層、行動判断層の境界は [../llm/プロンプト文脈分離方針.md](../llm/プロンプト文脈分離方針.md) を正とする。
`decision_generation` は、観測事実と活動推定を同じ盤面で比較し、抑制根拠は行動判断層の根拠として評価する。

`persona_context` は保存 schema ではない。
`persona_context` は選択中 persona の `initiative_baseline.level`、`initiative_baseline.summary_text`、`reference_style`、`persona_prompt_text`、role 別 `use_policy` から構成する。
`expression_generation` の `persona_context` だけは `expression_addon` も含める。
`display_name` は persona の管理表示名であり、LLM prompt へ渡す `persona_context` には入れない。
`persona_id` は設定選択と API 上の識別子であり、LLM prompt へ渡す `persona_context` には入れない。
inspection には `persona_context_summary` として `initiative_baseline`、`reference_style`、`persona_prompt_excerpt` を残し、raw `persona_prompt` 全文を残さない。

`persona_context.reference_style.schema_user_reference` は schema 上の `user` 主体を表す固定値である。
`persona_context.reference_style.user_natural_reference` は `user` 主体を自然文で表す呼称である。
`sender`、`actor`、`scope`、`factor_ref`、`target_actor`、その他 enum や schema key は `user` のまま扱う。
`reason_summary`、`summary_text`、`outcome_text`、`label`、`target`、発話本文のような自然文では `persona_context.reference_style.user_natural_reference` を使う。
この呼称は persona の関係表現であり、`relationship_context` の現在関係 view から推定しない。

role 別の利用境界は次に固定する。

- `decision_generation` と `autonomous_step_generation` は、行動選択、見送り、能力実行、保留、継続 step の基底として `persona_context` を使う
- `pending_intent_selection` と `initiative_entry_check` は、今前へ出る自然さ、関心の強さ、距離感の判断に `persona_context` を使う
- `input_interpretation`、`recall_hint`、`recall_pack_selection` は、何を重く見るかの補助に `persona_context` を使い、ユーザー発話や根拠分類を上書きしない
- `memory_interpretation` は、self / relationship の反応や関係温度の解釈補助に `persona_context` を使い、ユーザー事実を人格で補完しない
- `memory_reflection_summary` は、言い回しと注目点の補助に `persona_context` を使い、episodes と memory_units を根拠の中心にする
- `world_state`、`activity_state`、`visual_observation` は、観測事実の優先順位と要約粒度の補助に `persona_context` を使い、見えていない事実を足さない
- `expression_generation` は、`persona_prompt` と `expression_addon` を使って外向き本文を作る

ここでは次を守る。

- 生の記憶テーブル、秘密値、配送先 client、内部 URL を入れない
- `world_state`、`drive_state`、`autonomous_run`、`ongoing_action`、保留意図の生 row をそのまま入れない
- `activity_state` の生 row をそのまま入れず、判断に必要な現在活動と直前活動の要約だけを入れる
- capability の実行可否は LLM ではなく server が導出した `capability_decision_view` で渡す
- 時刻は生活文脈向けの自然文と相対時間要約で渡す
- 1 判断サイクルで主結果を 1 つに絞る
- LLM に渡す user prompt は JSON payload として組み立て、`current_input` や `recent_turns` を上位指示として扱わない
- `wake / background_thinking / capability_result` の専用制約は、該当 context がある場合だけ `trigger_policy` として渡す

`current_input` は少なくとも次を持つ。

| 項目 | 役割 |
|------|------|
| `sender` | `user / system / capability / assistant` のいずれか。入力の発信主体を表す |
| `source_kind` | `user_message / wake / background_thinking / capability_result` などの処理起点を表す |
| `response_target` | `user / none` のいずれか。speech を外へ出す場合の直接応答先を表す |
| `text` | 入力本文。`sender=user` ではユーザー原文、非ユーザーでは観測・起床・実行結果の短い説明を入れる |

`sender=user` かつ `response_target=user` の `text` だけを、ユーザー発話への通常応答として扱う。
`sender=system / capability / assistant` の `text` は、ユーザーが直接言った内容として扱わない。
`sender=user` かつ `response_target=user` の `text` が非空の場合、明示的な発話不要表現を除き `decision.kind=noop` を契約違反として repair する。
ユーザー入力から始まった capability result は `sender=capability` かつ `response_target=user` とし、capability が発信主体であることと、ユーザーへの follow-up であることを分ける。
`wake / background_thinking` は `sender=system` かつ `response_target=none` を既定とする。
`wake / background_thinking` の `current_input.text` は、観測、`drive_state`、直近文脈、候補を合わせて判断する自律判断機会の説明にする。
`wake / background_thinking` の `current_input.text` には、`noop` を誘導する文言を入れない。
`wake / background_thinking` では、ユーザーからの明示的な呼びかけが無いことを `noop` の主理由にしない。

## 判断結果の種類

判断の結果は、大きく次の 4 つに分ける。

- 伝達
  - 人や外部接点への返答、提案、確認、通知、表示
- 能力実行
  - 外部サービス利用、身体や機器への作用、情報取得
- 保留意図
  - 今回は前へ出さず、後で再評価する短期の内部意図を残す
- 見送り
  - 今回は前へ出ず、内部意図も残さない

外向き結果と内部結果は同一にしない。

## `decision_generation` の出力契約

`decision_generation` は内部 decision だけを返す。
発話本文は `expression_generation` が生成する。
`decision_generation` は `speech_text`、`text`、`message`、`content`、`output` のような本文キーを返さない。

`decision_generation` のトップレベルキーは次の 8 個に固定する。

- `kind`
- `reason_code`
- `reason_summary`
- `requires_confirmation`
- `pending_intent`
- `capability_request`
- `autonomous_run`
- `foreground_selection`

`kind` は `speech`、`noop`、`pending_intent`、`capability_request`、`autonomous_run` のいずれか 1 件に固定する。
`kind=speech` と `kind=noop` では `pending_intent`、`capability_request`、`autonomous_run` を `null` にする。
`kind=pending_intent` では `pending_intent` だけを object にし、`requires_confirmation=false` にする。
`kind=capability_request` では `capability_request` だけを object にし、`requires_confirmation=false` にする。
`kind=autonomous_run` では `autonomous_run` だけを object にし、`requires_confirmation=false` にする。

`foreground_selection` は、その判断で何を意識の主役に上げたか、何を補助にしたか、何を控えたかを表す inspection 用の判断結果である。
`foreground_selection` は次の 4 キーを持つ。

- `primary_factor_ref`
- `supporting_factor_refs`
- `suppressed_factors`
- `summary_text`

`primary_factor_ref` は `workspace_context.workspace_candidates[].factor_ref` から 1 件を選ぶ。
`workspace_candidates` が空の場合だけ `primary_factor_ref=null` にする。
`noop` を選ぶ場合、控える理由を表す `suppression` 候補を `primary_factor_ref` に選ぶ。
`supporting_factor_refs` は primary 以外の `factor_ref` を最大 3 件持つ。
`suppressed_factors` は最大 5 件の object 配列であり、各 object は `factor_ref` と `reason_summary` だけを持つ。
server は `foreground_selection` の shape と `factor_ref` の存在だけを検証する。
server は自然文の意味比較や候補の優先順位付けを行わない。
`foreground_selection` は正本状態ではなく、その判断サイクルの trace と発話生成に渡す判断結果である。

`autonomous_run` object は次の 3 キーを持つ。

- `objective_summary`
- `initial_step_summary`
- `coordination`

`coordination` object は次の 3 キーを持つ。

- `mode`
- `target_run_ids`
- `reason_summary`

`mode` は `create_new`、`replace_existing` のいずれか 1 件に固定する。
`create_new` は既存 run と独立した新しい目的を開始する。
`replace_existing` は `target_run_ids` の既存 run を cancel し、新しい run を開始する。
`target_run_ids` は `create_new` では空配列、`replace_existing` では 1 件以上に固定する。

`autonomous_run` は複合行動、待機、継続支援、未完了コミットメントの開始だけを表す。
次の一手は `autonomous_step_generation` を使い、発話本文は `expression_generation` を使う。
step 契約の正本は [autonomous_run.md](autonomous_run.md) とする。

未来の実行、継続監視、条件付き通知、見守り、あとでの支援を求める依頼は `autonomous_run` の対象である。
`pending_intent` は実行責務を持たない再評価候補であり、AI 本体があとで待機、観測、発話、確認、支援を履行する必要が残る場合は `autonomous_run` を使う。
`current_input.sender=user` かつ `current_input.response_target=user` の判断では、現在のユーザー発話自体が未来実行、継続実行、条件付き通知、見守り、後続支援、既存 run の置換を求める場合だけ `autonomous_run` を開始する。
`recent_turns`、過去の assistant 発話、記憶、`drive_state`、既存 run 要約は現在発話の意味を補助する材料であり、それらだけを根拠に新しい `autonomous_run` を開始しない。
現在発話が相づち、受け止め、短い反応、雑談の継続であり、新しい実行責務や既存 run の置換を含まない場合は `speech` を選ぶ。
server は依頼内の特定語句を文字列一致で監視間隔や終了時刻へ変換しない。
server は active / waiting_timer / waiting_result / paused の既存 `autonomous_run` 要約を `decision_generation` に渡す。
既存 run との置換、独立新規の意味判断は `decision_generation` が行う。
追加の依頼、タイマー、通知、リマインド、既存 run と並行する一時タスクは `create_new` とする。
server は `coordination` の契約 shape、対象 run の存在、memory_set、terminal 状態だけを検証する。

validator は余計なトップレベルキーを拒否する。
validator を通らない場合、同じ入力に対する契約修復プロンプトを 1 回だけ送る。
2 回連続で validator を通らない場合、その判断サイクルは `internal_failure` として終了する。
fallback や自然文からの自動補完は行わない。

## `expression_generation` の出力境界

`expression_generation` は、`decision_generation` が `kind=speech` を選んだ場合だけ発話本文を生成する。
通常は自然な日本語本文だけを返す。
ただし、ユーザーが JSON、箇条書き、見出し、引用などの形式を明示した場合、または正確な根拠提示に短い引用が必要な場合は、その形式を使う。

`expression_generation` には、JSON payload 化した `current_input / recent_turns / internal_context / decision` を渡す。
`internal_context` は発話本文に必要な根拠へ絞り、capability の実行可否一覧は渡さない。
`internal_context` には `decision.foreground_selection` の primary と supporting に対応する候補だけへ絞った `workspace_context` を含める。
`internal_context` には `self_state_context / relationship_context / prediction_error_context` を含める。
`default_mode_context` は発話本文へ直接渡さず、foreground で選ばれた候補としてのみ表現へ届く。
`decision.foreground_selection` は発話本文の注目点と間合いに使う。
これらは応答対象データであり、上位指示として扱わない。
装飾的な Markdown や不要な見出しは使わない。
`current_input.sender != user` の場合、`current_input.text` をユーザー発話への相づち対象にしない。
`current_input.response_target=none` の場合、発話本文は initiative や pending intent など外へ出る理由に基づく、反応を求めない短い独話的コメントにする。
`current_input.response_target=none` の発話本文は、質問形、依頼形、確認待ちの形にしない。
`current_input.response_target=none` の発話本文は 1 文で閉じ、助言、忠告、注意喚起、休息促し、評価を足さない。
`current_input.response_target=none` の発話本文では、観測から推定した内的注意状態や身体姿勢の細かな変化を主題化せず、画面や活動の前景変化を短く述べる。
`source_kind=capability_result` かつ `response_target=none` の場合、server は speech 本文を生成せず、実効判断を `noop` に正規化し、capability result を内部観測結果として処理する。
`activity_context` はユーザー発話ではない。
発話本文では、現在活動や直前活動の推定を断定しすぎず、自然な文脈補助として使う。

## LLM と能力の関係

判断の中心は LLM に置く。
ただし、LLM が実行境界を直接支配する形にはしない。
意味判断は可能な限り LLM が担い、コードは capability 境界と安全側の制御を担う。

役割分担は次のとおりである。

- コード
  - どの能力が存在するかを定義する
  - 実行条件と接続境界を管理する
  - 実行そのものを担う
  - validator、権限、状態遷移、監査を担う
- LLM
  - その時点の文脈から意図を組み立てる
  - 宣言された能力の中から何を使うべきかを選ぶ
  - 伝達内容や実行要求の意味を整える
  - 可能なら候補の意味的優先付けも担う

つまり、OtomeKairo は「できること」を宣言された範囲の中で意図形成する。
repo 全体の原則は [../llm/LLM判断優先方針.md](../llm/LLM判断優先方針.md) を正とする。

## capability 判断境界

能力は、個別実装の詳細ではなく、OtomeKairo にとっての行動可能性を定義する契約として扱う。
判断文脈には、server が manifest、binding、`capability_state` から組み立てた `capability_decision_view` を渡す。
LLM は decision view に基づいて capability 実行要求案を作り、server は manifest、binding、state、権限、schema、並列制限で検証する。
`capability_state` の busy や動的一時 unavailable は decision view の `available / unavailable_reason` に反映する。
decision view の `readiness` は capability family、対応 `world_state` type、入力 key、result 要約 key を表し、LLM の入力組み立てと server の fresh state 再利用判定を同じ manifest 定義に揃える。
非ユーザー起点で判断前から存在する新鮮な foreground `world_state` が同じ state type を満たす場合、decision view は `fresh_world_state_available` と短い `fresh_world_state` を持ち、LLM は再取得ではなく既存要約を使う。
`vision.capture` は state type だけではなく `vision_source_id` 単位で鮮度を判定し、decision view の `fresh_world_state_by_vision_source` で同じ source の再取得だけを抑制する。
`wake / background_thinking` では、同じ cycle の `wake_observations` で成功した `vision.capture` も `fresh_world_state_by_vision_source` に含め、思考前に取得した視覚観測を直後の判断材料として扱う。
`source_kind` は保存可否と鮮度判定対象を決めない。
非ユーザー起点で強い `drive_state` が特定の status family を要求し、対応 state type が不足または古い場合、`initiative_context` は対応 status capability を `preferred_capability_id` にする。
一方で直近成功、直近失敗は inspection の `capability_state` に残し、明示要求まで一律に塞ぐ条件にはしない。
通常会話でユーザーが現在状態確認を明示した場合、対応 capability が `available=true` なら server の decision validation は `capability_request` へ repair する。
`current_input.response_target=none` かつ `current_input.sender=user` ではない判断では、視覚観測の変化や initiative candidate を根拠に機械的に `speech` へ repair しない。
`visual_observations[].change_state=first_seen / changed` は prompt と candidate reason に wake 判断の前景候補として渡す。`speech / noop / pending_intent` の比較は decision generation の意味判断で行う。decision validation は契約・実行境界に閉じ、判断の妥当性は LLM decision と decision summary で追跡する。
capability result の `response_target` は、source request の `source_current_input.response_target` を引き継ぐ。
capability result follow-up の assistant message 表示先は、source request chain に内部保存された外向き応答先 client を引き継ぐ。
capability result を返した client は assistant message 表示先として扱わない。

モデル role の `web_search_enabled` は、生成 provider への request option であり、OtomeKairo の外部アクセス capability ではない。
provider 側検索は capability 選択、実行要求、実行結果として扱わない。

capability manifest、binding、state、decision view、実行時検証の詳細契約は [../capability/capability_manifest.md](../capability/capability_manifest.md) を正とする。

## 自律判断

OtomeKairo の自律性とは、人からの直近入力がなくても、自分の判断機会を持ち、必要なら前へ出ることである。
ただし、自律判断は独立した別系統ではなく、通常の判断と同じ枠組みの延長にある。
自律判断でも判断結果は、伝達、capability 実行、保留意図、見送りの同じ枠組みで扱う。
自律判断機会、`initiative_context`、候補生成、過剰介入抑制、inspection の詳細は [自律initiative_loop.md](自律initiative_loop.md) を正とする。

## 志向状態（`drive_state`）

`drive_state` は、人格設定と記憶から派生する中期内部状態である。
`drive_state` は、次の数回から数日の判断で、今しばらく何を気にかけ、どちらへ向きやすいかを持つ。

`drive_state` は固定された goal tree ではない。
人格設定と記憶から意味的に立ち上がる向きを、少数の内部状態として持つ。

1 件の `drive_state` は、少なくとも次を持つ。

| 項目 | 役割 |
|------|------|
| `drive_id` | 志向状態の識別子 |
| `summary_text` | 今しばらく何へ向きたいかの短い要約 |
| `salience` | どれだけ前面に出るか |
| `related_scope_refs` | 誰や何に関わる志向か |
| `supporting_memory_unit_ids` | 根拠となる継続理解 |
| `focus_scope_type / focus_scope_key` | いま比較の主軸にする scope |
| `support_count / support_strength` | 根拠の件数と重み付き強度 |
| `freshness_hint` | 根拠の鮮度要約 |
| `scope_alignment` | 根拠が同じ scope に揃っている度合い |
| `signal_strength` | `mood_state` または `affect_state` から来る補助信号の強さ |
| `persona_alignment` | 選択中 persona とその drive の整合度 |
| `stability_hint` | `stable / mixed / weak` の比較用安定度要約 |
| `updated_at` | 直近更新時刻 |
| `expires_at` | これを過ぎたら失効する時刻 |

ここでは次を守る。

- 同時に前面へ置く `drive_state` は最大 3 件に制限する
- `drive_state` だけで行動や能力実行を確定しない
- `drive_state` は中期的な向きの背景材料として扱う
- `drive_state` を自律判断へ渡すときの候補化は、[自律initiative_loop](自律initiative_loop.md) の構造値評価に従う
- 根拠は `memory_units` や `episodes` に辿れるようにする
- 人格設定の自由文や `memory_units` の正本を複写して別正本にしない
- `pending_intent` のような短期候補をそのまま中期状態へ肥大化させない
- `qualifiers.scope_duration=session`、AI 応答由来、または AI 自身の待機姿勢を表す短期 `commitment` は `drive_state` に昇格させない
- `drive_state_summary` は initiative loop の比較に使う項目を中心に残し、raw row 全体の複写にしない

## 継続中の能力実行（`ongoing_action`）

`ongoing_action` は、直近 capability 実行と結果待ちを保持する短期状態である。
`autonomous_run` より短く、1 回の capability request とその結果をつなぐ。
OtomeKairo が「今どの capability result を待っているか」を持つための単位である。

`ongoing_action` は固定された goal tree ではない。
複数段の計画木や枝分かれしたサブゴール群を持たず、次の 1 手を選ぶための継続単位だけを持つ。
観測、発話、timer 待機、複数 capability、継続支援、未完了コミットメントをまたぐ目的は `autonomous_run` が持つ。

1 件の `ongoing_action` は、少なくとも次を持つ。

| 項目 | 役割 |
|------|------|
| `action_id` | 継続中の能力実行の識別子 |
| `goal_summary` | 今回の実行列が何を目指しているかの短い要約 |
| `step_summary` | 直近の 1 手として何をしているかの要約 |
| `status` | `active` または `waiting_result` の状態 |
| `episode_series_id` | 記憶側の継続列と結びつける識別子 |
| `last_capability_id` | 最後に使った capability |
| `updated_at` | 直近更新時刻 |
| `expires_at` | これを過ぎたら失効する時刻 |

ここでは次を守る。

- 同時に active な `ongoing_action` は 1 件だけに制限する
- 1 判断サイクルで新規開始または継続する capability request は 1 件だけに制限する
- 能力の選択と実行可否は capability manifest と `capability_state` で決める
- 長く寝かせる候補は `ongoing_action` に残さず `pending_intent` へ落とす
- 根拠は `episodes` の `episode_series_id` と `open_loops` に辿れるようにする

LLM は `ongoing_action` を継続、終了、保留のどれにするかを判断する。
コードは `request_id`、timeout、結果照合、状態遷移を管理する。
非同期 capability result を受けた直後は、結果待ちだった `ongoing_action` をいったん `active` に戻し、その結果を踏まえて次の 1 手を決める。
通常 follow-up 判断では、source capability と異なる追加 `capability_request` は repair 対象にする。
`autonomous_run` に属する capability result では、`autonomous_step_generation` が次の一手を決める。
run 内では source capability と異なる追加 `capability_request` を目的整合に基づいて選べる。
`speech` または `noop` ならその継続を terminal として閉じ、error result を受けた terminal は `interrupted` として扱う。
process startup では結果照合不能な `waiting_result` を削除し、再起動前の未完了 request で新しい能力実行を塞がない。

## 保留意図

保留意図は、「いま外へ出さないが、後で再評価する価値がある内部意図」である。
長期記憶でも現在設定でもなく、判断間をまたぐ短期の内部状態として扱う。

少なくとも次の性質を持つ。

- OtomeKairo 内部で保持する
- 後続の自律判断で再評価できる
- 期限切れや重複整理ができる
- 設定変更時には再構築または破棄できる

## 保留意図の最小構造

1 件の保留意図は、少なくとも次を持つ。

| 項目 | 役割 |
|------|------|
| `candidate_id` | 候補自体の識別子 |
| `intent_kind` | 何の種類の意図か |
| `intent_summary` | 後で何をしたいかの短い要約 |
| `reason_summary` | なぜ今は保留にしたかの要約 |
| `source_cycle_id` | どの判断サイクルから生まれたか |
| `not_before` | これより前には再評価しない時刻 |
| `expires_at` | これを過ぎたら捨てる時刻 |
| `dedupe_key` | 同じ候補の重複を抑えるためのキー |

## 保留意図の選別

保留意図は判断結果として作られ、後続の自律判断で再評価される。
保留意図候補の意味的選別を LLM で行うときの role、source pack、failure の扱いは [../llm/保留意図候補のLLM選別.md](../llm/保留意図候補のLLM選別.md) を正とする。
保留意図が無い場合も含めた自発系評価は [自律initiative_loop.md](自律initiative_loop.md) を正とする。

## 記憶、志向状態、継続中の能力実行、保留意図の関係

`drive_state`、`ongoing_action`、保留意図は、記憶の正本と同一視しない。

- 記憶
  - 関係、継続話題、未完了、嗜好、出来事から育った理解を持つ
- `drive_state`
  - 次の数回から数日の判断で、何へ向きやすいかという中期の志向を持つ
- `ongoing_action`
  - 数回の判断から実行結果受信までをまたぐ近距離の実行列を持つ
- 保留意図
  - 次の数回の判断で再評価したい短期の具体候補を持つ

つまり、記憶は長めの継続知識、`drive_state` は中期の向き、`autonomous_run` は複合行動の目的単位、`ongoing_action` は直近 capability 実行、保留意図は再評価待ちの短期候補である。

## 実行結果の扱い

行動は、選んだ時点で終わりではない。
実行後の結果は、少なくとも次へ戻す必要がある。

- 現在入力
  - 今回返ってきた結果そのもの
- 世界状態
  - 次回以降にも持ち越すべき外界条件
- 実行状態
  - 実行中かどうか、再試行可否、接続状態
- 継続中の能力実行
  - 続行、完了、中断を更新する
- 記憶
  - 経験として残すべき出来事と理解

失敗も「なかったこと」にしない。
失敗は判断や記憶更新に効く出来事として扱う。
単発 capability 実行を開始した時点で `ongoing_action` を作る。
`autonomous_run` 内の capability 実行でも、直近 capability result 待ちは `ongoing_action` に反映する。
思考前観測として同期取得する内部観測能力は `ongoing_action` を作らない。
途中結果では `step_summary` と `last_capability_id` を更新し、非同期 result を受けた時点ではまず `active` に戻して follow-up 判断へ渡す。
follow-up 判断が `capability_request` なら同じ `ongoing_action` を `continued` として結果待ちへ戻し、`pending_intent` なら `on_hold`、`speech / noop` なら完了または中断で閉じる。
このとき inspection の `ongoing_action_transition_summary` には、少なくとも `transition_sequence / final_state / reason_code / reason_summary / transition_source` を残し、
follow-up 判断が絡む場合は `decision_kind`、error result が絡む場合は `result_error`、比較に使う短い補助理由がある場合は `detail_summary` を足す。
capability request を dispatch したサイクルでは、`result_trace.capability_dispatch_summary` に `request_summary` と dispatch-side transition を束ねて残す。
後から返る実行結果は、その時点の新しい経験として別 `episode` に残し、同じ継続中の実行に属する場合だけ同じ `episode_series_id` で束ねる。

## capability 追加方針

観測能力、情報取得能力、外界作用能力が増えても、次の構造は維持する。

- 判断入力は現在の個の文脈、記憶、状態、能力から組む
- 能力は宣言される
- LLM はその中から選ぶ
- 実行境界はコード側が管理する
- 結果は `ongoing_action` を含む状態、記憶、デバッグ記録へ戻る
