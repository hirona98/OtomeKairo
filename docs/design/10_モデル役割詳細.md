# モデル役割詳細

<!-- Block: Role -->
## この文書の役割

この文書は、OtomeKairo がどのモデル役割を持ち、それをどう設定として管理するかを定める正本である。

ここで固定したいのは次の内容である。

- モデル役割をどの単位で分けるか
- 役割ごとの論理境界
- モデルプリセットとモデルプロファイルをどう分けるか
- モデル役割をどの設定値で選ぶか
- プリセットとプロファイルをどう編集し、どう切り替えるか
- `CocoroConsole` がどこまで触れるか

ここでは、LiteLLM への接続実装の細部までは定めない。
ただし、モデル設定をどう保持し、どう送受信するかの責務境界はここで固定する。
現実装では、`model=mock` は開発用の内蔵ロジックを使い、`model!=mock` の `generation` プロファイルは LiteLLM Python SDK へ `model` をそのまま渡して呼び出す。

<!-- Block: Premise -->
## 基本前提

OtomeKairo では、モデル役割をその場の都度指定するのではなく、サーバ側で保持するモデル設定資源として管理する。

この前提により、次が成立する。

- 現在どのモデル構成で動いているかを追跡しやすい
- 人格や記憶と同じく、ID で選択できる
- `CocoroConsole` からプリセットやプロファイルを編集しても、正本は OtomeKairo 側に残る
- `CocoroConsole` からモデル設定全体を送受信できる
- ローカルネットワーク内利用を前提にしつつ、経路は HTTPS に固定できる
- デバッグ時に「どのプリセットでその応答が出たか」を辿りやすい

MVP では、この考え方を B 案として採用する。

<!-- Block: LogicalRoles -->
## モデル役割の論理単位

MVP では、少なくとも次の論理役割を持てばよい。

| 役割 | 役割名 | 主な責務 |
|------|------|------|
| 会話 | `reply_generation` | 会話返答を生成する |
| 判断 | `decision_generation` | 自発判断と将来の行動判断を行う |
| 想起入口 | `recall_hint_generation` | `RecallHint` を生成する |
| 記憶解釈 | `memory_interpretation` | 記憶更新や再整理で使う解釈を行う |
| 埋め込み | `embedding` | ベクトル化を行う |

ここで重要なのは、論理役割とモデル実体を同一視しないことである。
1 つのプリセットの中で、複数役割が同じモデルを参照していてもよい。

<!-- Block: WhyPreset -->
## プリセットで持つ理由

モデル役割を個別設定としてばらばらに持つと、次が難しくなる。

- どの組み合わせで動かしているかを把握すること
- 安定した構成を切り替えること
- 比較実験を単純に行うこと
- `CocoroConsole` 側で編集 UI を作ること

そのため、OtomeKairo では役割ごとの割り当てをひとまとめにした `model_preset` を持ち、現在選択中のものを `selected_model_preset_id` で指すほうがよい。

<!-- Block: SelectedModelPreset -->
## `selected_model_preset_id`

MVP では、OtomeKairo の永続設定として次を保持する。

- `selected_model_preset_id`

これは人格や記憶と同じく、現在どのモデルプリセットを使うかを表す選択値である。
意味としては次のとおりである。

- 正本は OtomeKairo 側にある
- `CocoroConsole` はこの選択値を編集できる
- 再起動後も同じプリセットを引き継ぐ
- `runtime_state` には現在反映中のモデルプリセット参照を出してよい

<!-- Block: ModelProfile -->
## `model_profile` の最小構造

MVP では、実際に LiteLLM へ渡す接続設定は `model_profile` として OtomeKairo 側に保持する。
1 つの `model_profile` は少なくとも次の構造を持てばよい。

```yaml
model_profile_id: model_profile:gemini_reply
display_name: OpenRouter Gemini Reply
kind: generation
model: openrouter/google/gemini-3.1-flash-lite-preview
auth:
  type: bearer
  token: ""
```

ここでの `model_profile` は、外部プロバイダの名前そのものではなく、OtomeKairo が管理するモデル設定資源である。
`CocoroConsole` はこの定義を編集できる。
`auth` を含む各項目は、他の設定値と同じく送受信対象として扱ってよい。
`base_url` は省略してよく、ローカル接続や OpenAI 互換 API を明示したい場合だけ使えばよい。

`kind` は少なくとも次のどちらかであればよい。

- `generation`
- `embedding`

<!-- Block: ModelPreset -->
## `model_preset` の最小構造

MVP では、1 つのモデルプリセットは少なくとも次の構造を持てばよい。

```yaml
model_preset_id: model_preset:default
display_name: Default OpenRouter Gemini Preset
roles:
  reply_generation:
    model_profile_id: model_profile:gemini_reply
    reasoning_effort: low
  decision_generation:
    model_profile_id: model_profile:gemini_decision
  recall_hint_generation:
    model_profile_id: model_profile:gemini_recall
  memory_interpretation:
    model_profile_id: model_profile:gemini_memory
  embedding:
    model_profile_id: model_profile:gemini_embedding
```

ここでの `model_profile_id` は、OtomeKairo が保持している `model_profile` を指す ID である。
つまり、役割ごとに「どの接続設定を使うか」をプリセット側で束ねる形にする。

また、同じ `model_profile_id` を複数役割で共有してよい。
役割は分けるが、実体は共有できる、という形で扱う。
`reasoning_effort` は任意項目であり、空欄や空文字列は保存時に削除して無効扱いにしてよい。

<!-- Block: EditablePreset -->
## プリセットとプロファイル編集の扱い

`model_preset` と `model_profile` は、どちらも OtomeKairo が保持する編集可能な資源として扱う。
MVP では、細かな patch ではなく、定義全体の置き換えで編集してよい。

この方針により、次を保てる。

- どの版のプリセットが有効かを追いやすい
- 一部だけ壊れた中途半端な設定状態を避けやすい
- UI から編集しても validator を単純に保てる

MVP では、少なくとも次ができれば十分である。

- `select_model_preset(model_preset_id)`
- `replace_model_preset(model_preset_id, model_preset_definition)`
- `replace_model_profile(model_profile_id, model_profile_definition)`

<!-- Block: Transport -->
## 通信前提

`model_preset` と `model_profile` は、他の設定資源と同じく `CocoroConsole` と OtomeKairo のあいだで送受信してよい。

MVP では、少なくとも次を前提にする。

- 設定資源は値種別で特別扱いしない
- API 通信の経路は HTTPS に固定する
- 利用範囲はローカルネットワーク内に限定する

API 上の具体的な wire 形式は `14_API仕様.md` で定めるが、この通信前提はここで固定する。

<!-- Block: Validator -->
## validator の最小ルール

`model_preset` の validator は、少なくとも次を確認する。

- `model_preset_id` が空でないこと
- 必須役割がすべて存在すること
- 各役割が `model_profile_id` を持つこと
- 参照先の `model_profile` が存在すること
- 役割と `model_profile.kind` が整合すること
- 余計なトップレベル責務を持ち込まないこと

`model_profile` の validator は、少なくとも次を確認する。

- `model_profile_id` が空でないこと
- `kind` が `generation` または `embedding` であること
- `model` が空でないこと
- `base_url` を持つ場合は文字列であること
- `auth` を持つ場合は object であること

ここでは、最適なモデルの良し悪しは validator の責務にしない。
契約を満たしているかだけを確認する。

<!-- Block: SelectionAndReflection -->
## 選択と反映

`selected_model_preset_id` の変更は、人格や記憶の切り替えと同じく、ID 単位の選択変更として扱う。

MVP では、次の考え方を取る。

- 選択値の変更は永続設定へ先に反映する
- 実際の読み替えは再読み込みまたは再起動でよい
- `runtime_state` には現在反映中のモデルプリセット参照を出してよい

これにより、「設定上は何を選んでいるか」と「いま実際に何が反映されているか」を分けて観測できる。

<!-- Block: ApiBoundary -->
## API と UI から見た境界

`CocoroConsole` からは、少なくとも次を扱えるようにしてよい。

- 現在の `selected_model_preset_id` を読む
- `selected_model_preset_id` を切り替える
- 利用可能な `model_preset` 一覧を読む
- 利用可能な `model_profile` 一覧と要約を読む
- `model_profile` の詳細を読む
- `model_preset` を全体置換で編集する
- `model_profile` を全体置換で編集する
- `model_profile` を送受信する

つまり、編集可能なのは役割割り当てだけではなく、実接続設定そのものまで含む。

<!-- Block: Excluded -->
## MVP でまだ入れないもの

MVP では、次は入れない。

- リクエストごとの `model` 文字列上書き
- 役割ごとの部分 patch 更新
- HTTPS 以外での API 通信
- 自動チューニング
- 実行中リクエスト単位の動的切り替え

これらを先に入れると、設定責務と運用責務が崩れやすい。

<!-- Block: DesignDecision -->
## ここで固定したい設計判断

この文書で固定したいのは次である。

- モデル役割は `model_preset` としてまとめて持つ
- 現在選択中の値は `selected_model_preset_id` で保持する
- 実接続設定は `model_profile` として OtomeKairo 側で保持する
- `CocoroConsole` はプリセットとプロファイルの両方を編集できる
- `model_profile` の各項目は他の設定値と同じく送受信してよい
- API 通信の経路は HTTPS に固定し、利用範囲はローカルネットワーク内を前提にする
- 役割は分けるが、同じモデルプロファイルを複数役割で共有してよい
- プリセット編集とプロファイル編集は MVP では全体置換で扱う
