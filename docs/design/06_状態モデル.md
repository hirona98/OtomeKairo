# 状態モデル

## 基本前提

OtomeKairo では、その場で変わるものを状態として扱う。
状態は、設定定義や現在設定とは分けて考える。

ここで重要なのは、「変わるもの」をすべて一枚に押し込めないことだ。
少なくとも、入力、外界条件、稼働状態、能力状態、中期内部状態、短期内部状態を分ける。

## 状態の層

OtomeKairo の状態は、少なくとも次の層に分ける。

- 現在入力（`current_input`）
- 世界状態（`world_state`）
- 実行状態（`runtime_state`）
- 能力状態（`capability_state`）
- 中期内部状態（`drive_state`）
- 継続中の能力実行（`ongoing_action`）
- 作業文脈
- 短期内部状態

## 現在入力（`current_input`）

現在入力は、その回の入力と結果を扱う一時的な層である。
ここには少なくとも次が入る。

- 人からの入力
- 感覚入力
- 外界イベント
- 実行結果
- 自律判断の起点となった起床要求

これは判断サイクルの材料であり、そのまま現在設定や長期記憶の正本にはしない。

## 世界状態（`world_state`）

世界状態は、次の判断にも持ち越したい外界条件を保持する。
ここには、単発の生入力ではなく、正規化して残す意味単位を入れる。

たとえば次を含みうる。

- いま継続している状況
- 周囲の対象や場所の状態
- 外部サービスや身体の最新条件

世界状態は記憶の代替ではない。
記憶より短く、現在判断に近い継続状態を持つ。
`world_state` の最小構造、更新責務、失効、記憶との境界は `22_world_state.md` を正とする。

## 実行状態（`runtime_state`）

実行状態は、OtomeKairo 自身の現在の稼働条件だけを持つ。
少なくとも次の系統を含む。

- 自律判断機構の稼働状態
- 実行中の行動有無
- 接続状態
- 内部ジョブや再整理処理の進行状態

実行状態は正本ではなく、現在の稼働状態要約である。
そのため、外部から直接書き換える対象にはしない。
何を続けているかの意味単位は `runtime_state` に入れず、`ongoing_action` で持つ。

## 能力状態（`capability_state`）

能力状態は、能力マニフェストそのものではなく、その時点での実行条件を持つ。
どの client がどの能力を実行できるかは capability binding で扱い、能力状態は binding を踏まえた利用可否を保持する。

たとえば次を含みうる。

- 現在利用可能か
- 一時停止中か
- 制約や上限があるか
- 前提条件が満たされているか
- 権限不足か
- 直近失敗や cooldown があるか

能力定義は安定契約であり、能力状態はその瞬間の運用状態である。
capability manifest、binding、state、availability 判定、外向き確認 API の詳細契約は `17_capability_manifest.md` を正とする。
通常の `status` は運用要約だけを返す。

## 中期内部状態（`drive_state`）

`drive_state` は、人格設定と記憶から派生し、次の数回から数日の判断で前面に出やすい志向を持つ状態である。

ここには少なくとも次が入る。

- 今しばらく気にかける対象や話題
- 前へ出やすい方向の要約
- その向きの強さ
- いつ更新され、いつ失効するか

`drive_state` は長期記憶の代替ではない。
`drive_state` は継続理解を保存する層ではなく、判断の向きを持つ派生状態である。
`drive_state` の意味境界は `05_判断と行動.md` を正とする。

## 継続中の能力実行（`ongoing_action`）

`ongoing_action` は、最終目的へ向かう近距離の能力実行列を持つ状態である。
これは `drive_state` と 1 回の capability request のあいだに置く継続単位である。

ここには少なくとも次が入る。

- 何を目指している実行列か
- 直近の 1 手として何をしているか
- 実行待ちか結果待ちか
- どの `episode_series_id` に属するか
- いつ更新され、いつ失効するか

`ongoing_action` は goal tree ではない。
複数段の計画木を持たず、今続けている 1 本の実行列だけを持つ。
同時に active な `ongoing_action` は 1 件だけに制限する。
文脈が変わったときは更新、終了、`pending_intent` への移し替えのいずれかを行う。

## 作業文脈

作業文脈は、その回の判断のために一時的に組み立てる文脈である。

ここには少なくとも次が入る。

- 今回の主題
- 参照中の記憶や状態の要約
- 一時的な優先順位
- 今回だけ有効な解釈結果
- 自律判断で作る `initiative_context`

作業文脈は判断のための足場であり、長期の正本にはしない。
`initiative_context` の責務境界は `21_自律initiative_loop.md` を正とする。

## 短期内部状態

短期内部状態は、`ongoing_action` とは別に、判断間をまたいで持ち越す軽量状態であり、長期記憶や現在設定には入れない。

たとえば次を含みうる。

- 保留意図
- 直近の自律介入抑制情報
- 再評価待ちの短期候補

保留意図の意味境界は `05_判断と行動.md` を正とする。

## 状態に含めないもの

次は、状態と混同しない。

- 現在設定
- 人格設定、記憶集合、モデルの設定定義
- クライアントのローカル UI 状態
- デバッグ表示用の派生要約
- 想起や更新の途中だけで使う中間値
- 長期記憶の内部テーブルそのもの

内部処理の中間物や表示都合の派生物を状態の正本へ昇格させないことが、設計の単純さを保つ。
