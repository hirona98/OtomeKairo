# 人格と記憶

## 基本原則

OtomeKairo では、人格と記憶を独立したものとして扱う。

この分離は本プロジェクトの重要な前提である。

記憶設計は `memory/01_記憶システム概要.md` から `memory/04_記憶更新と再整理.md` で扱う。

この分離の主目的は、同じ人格に対して複数の記憶を切り替えながら検証できるようにすることである。

人格と記憶を個別にしているのは、「一つの人格に対して複数の記憶」という扱いを特別な仕組みにせず、操作と実装を単純に保つためである。

記憶を別の人格で再利用することは、この分離の主目的ではない。

- 人格は「どう考え、どう振る舞うか」を決める
- 記憶は「何を経験し、何を覚えているか」を決める

ただし、実際の応答時にはこの二つは分離して現れるわけではない。
人格設定は基本姿勢を与え、記憶は自己像、他者理解、関係、気分、継続中の話題を育てる。
そのため、実際の挙動は「人格設定と記憶から組み上がる現在の人格文脈」で決まるものとして扱う。

## 人格

人格は設定ファイルとして表現し、切り替える。
人格設定の内部粒度は `09_人格設定詳細.md` を正とする。

人格には少なくとも次の要素が含まれる。

- 基本的な性格
- 話し方
- 価値観や判断傾向
- 自己像
- 他者との関わり方
- 行動判断時の姿勢

人格はアップデートされうるが、その更新は記憶の継承とは別問題として扱う。

## 記憶

記憶は人格と独立に選択できる。

記憶には少なくとも次の前提を置く。

- 日をまたぐ継続性を持つ
- 会話や観測の出来事を保持する
- 将来的には、出来事から育った理解も含む
- 削除や差し替えを行う

人格を切り替えても、記憶は自動で引き継がれない。
また、記憶を切り替えても、人格は自動で切り替わらない。

## 人格と記憶の組み合わせ

OtomeKairo では、人格と記憶を別々に選べることを前提にする。

この設計により、主に次のような使い方を単純に扱える。

- 同じ人格に対して、複数の記憶を切り替えて比較や検証を行う
- 人格だけを更新し、記憶はそのままにする
- 記憶だけを初期化し、人格は保つ

人格と記憶を独立に選べるのは操作単位を分けるためであり、同じ記憶を複数の人格で使い回すことを積極的に前提にするものではない。

## ライフサイクル

OtomeKairo の動作単位は、現在選択中の人格と現在選択中の記憶の組である。

この前提のもとで、次の方針を置く。

- 人格切り替えでは、記憶は自動で変わらない
- 記憶切り替えでは、人格は自動で変わらない
- どちらも独立した選択対象とする

## 最小操作契約

人格と記憶の独立切り替えは、MVP の中心機能として概念契約まで固定する。
ここで定めるのは API パスではなく、OtomeKairo が受け付ける操作の意味である。

人格と記憶の選択値をどこに保持するかは `06_設定と状態.md` を正とし、設定操作全体の契約は `07_設定操作契約.md` を正とする。

最小操作は次の 2 つとする。

- `select_persona(persona_id)`
- `select_memory_set(memory_set_id)`

この契約では、次を前提にする。

- 人格切り替えは `selected_persona_id` だけを変える
- 記憶切り替えは `selected_memory_set_id` だけを変える
- 指定していない側は自動で変わらない
- 無効な対象を指定した場合は、部分反映せず失敗として返す

反映方法はまず単純さを優先する。
MVP では、切り替え要求を受けたら自動再起動で反映する。
ただし、「独立に選べる」という操作意味そのものはこの段階で固定する。

- 人格または記憶の切り替え要求を受けた場合、自動再起動で反映する
- 初期段階では、ランタイム中の即時差し替えを必須としない

このとき、`future_act` 候補キューのような短期の内部保留状態はクリアする。
人格と記憶の切り替えは、保留候補を引き継ぐよりも、選択内容の切り替えを優先する。

また、削除の意味も分けて考える。

- 人格削除は、人格設定のみを対象にする
- 記憶削除は、その記憶に属する出来事と派生記憶を対象にする

この分離により、人格の更新と記憶の整理を別々に行えるようにする。

## 世界認識の前提

OtomeKairo は、ユーザー本人だけでなく、そのほかの人や対象も認識する必要がある。

そのため、記憶は少なくとも次を扱える必要がある。

- 自分
- ユーザー
- その他の人物
- 物や場所や出来事
- それらの関係

## デバッグと観測

本プロジェクトでは、デバッグ可能性を重視する。

そのため、次の方針を置く。

- 会話や観測の生の出来事は保存対象とする
- 判断や変化の痕跡も、できるだけ追跡可能にする
- 何を根拠にその応答や変化が起きたのかを後から見返せるようにする
- まずは簡単に実装できる範囲で広めに出し、不要なものを後から削る

初期方針としては、次の考え方を取る。

- 保存対象は狭く絞り込まず、観測、応答、判断痕跡を広めに扱う
- デバッグ容易性を優先し、後から不要な保存を減らす

ただし、最終的にどの粒度までを恒久的な保存対象とするかは、今後さらに詰める。
