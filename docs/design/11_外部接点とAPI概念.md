# 外部接点と API 概念

## 基本前提

OtomeKairo の外部接点は、クライアント都合ではなく本体側の責務境界に沿って分ける。
外部接点は観測を送り、結果や状態を受け取り、変更要求を送るが、判断や設定の正本にはならない。

この文書では、path や wire 形式ではなく、外向きにどんな面を持つかを定義する。
具体的な API 仕様は `14_API仕様.md` と `design/api/` 配下を正とする。

## 接点の分類

OtomeKairo の外部接点は、少なくとも次の面に分ける。

- bootstrap 面
  - 接続確立、相手識別、認証開始
- 観測面
  - 人、感覚入力、外界イベント、実行結果を送る
- 実行連携面
  - 能力実行要求やその結果のやりとりを行う
- 状態面
  - 現在の要約状態を読む
- 設定面
  - 現在設定と設定定義を読む、変更する
- 列挙面
  - 選択対象や利用可能資源を一覧する
- inspection 面
  - デバッグ記録や判断履歴を読む
- stream 面
  - 継続的な通知や live 観測を行う

## 観測面

観測面では、少なくとも次を受け取れる必要がある。

- 対話入力
- 感覚入力
- 外界イベント
- 実行結果
- 自律判断の起点となる観測

ここで重要なのは、観測面に設定変更を混ぜないことだ。
観測は「今回の判断材料」であり、設定変更は別責務として扱う。

## 実行連携面

OtomeKairo は、必要に応じて外部接点へ実行要求を出し、その結果を受け取る。

この面では次を分けて考える。

- 何をしたいかという意図
- どの能力を使うかという選択
- 実行要求として何を渡すか
- 実行結果として何が返るか

内部で保留した意図を、そのまま外向き結果に露出する必要はない。
外向きに返す結果と、内部で持ち越す判断結果は分けてよい。

## 状態面

状態面は、現在の要約状態を読むための面である。
少なくとも次の 2 系統を返せる必要がある。

- 現在の設定要約
  - どの人格設定、記憶、モデルプリセット、方針が選ばれているか
- 現在の稼働要約
  - 接続や実行がどうなっているか
  - 自律判断機構がどう動いているか

ここでは、長期記憶の内部テーブルや中間計算結果をそのまま返す面を作らない。

## 設定面

設定面では、現在設定と設定定義を扱う。

少なくとも次を行える必要がある。

- 現在設定の取得
- 選択値や方針の変更
- 人格設定、記憶集合、モデルプリセットなどの設定定義の取得
- 設定定義の作成、複製、差し替え、削除

変更単位の意味境界は `08_設定変更.md` を正とする。

## 列挙面

列挙面は、設定や能力の選択肢を一覧するための面である。

少なくとも次を列挙できる必要がある。

- 人格設定
- 記憶集合
- モデルプリセット
- 利用可能能力

列挙面は、正本のコピーではなく、選択に必要な要約を返す。

## inspection 面

inspection 面は、通常運用とは別に、後から判断や更新を追うための面である。

少なくとも次を扱う。

- 判断サイクルの一覧要約
- 特定サイクルの詳細トレース
- 設定変更や記憶更新の観測用情報

inspection 面は読み取り中心であり、運用上の正本や編集面とは分ける。

## stream 面

stream 面は、継続的な往復や live 観測が必要な場面で使う。

たとえば次を含みうる。

- server-driven の event/control
- live の段階要約ログ
- 長めの処理の進行通知

stream 面は request/response API の代替ではなく、継続的な連携を補うための面である。

## bootstrap 面との分離

接続確立のための bootstrap 面は、通常運用の各面とは別に扱う。

この分離により、次を保てる。

- 接続確立の都合を通常の意味単位へ持ち込まない
- 認証開始と通常操作を切り分ける
- 相手識別や資格発行の責務を独立させる

接続と権限の意味境界は `12_接続と権限境界.md` を正とする。
