# 外部接点と API 概念

## 基本前提

OtomeKairo の外部接点は、クライアント都合ではなく本体側の責務境界に沿って分ける。
外部接点は対話入力、観測結果、起床要求、実行結果、変更要求を送り、結果や状態を受け取るが、判断や設定の正本にはならない。

この文書では、path や wire 形式ではなく、外向きにどんな面を持つかを定義する。
具体的な API 仕様は `14_API仕様.md` と `design/api/` 配下を正とする。

## 接点の分類

OtomeKairo の外部接点は、少なくとも次の面に分ける。

- bootstrap 面
  - 接続確立、相手識別、認証開始
- 対話面
  - 人との明示的なやりとりを送受信する
- 観測面
  - 本体が能動取得した観測結果を受け渡す
- 自律面
  - 自律判断の起動要求を送る
- 実行連携面
  - 能力実行要求やその結果のやりとりを行う
- 状態面
  - 現在の要約状態を読む
- 設定面
  - 現在設定と設定定義を読む、変更する
- 列挙面
  - 選択対象や利用可能資源を一覧する
- inspection 面
  - デバッグ記録や判断履歴を読む
- stream 面
  - 継続的な通知や live 連携を行う

## 対話面

対話面では、少なくとも次を扱う。

- 対話入力
- 対話への返答

対話面は、人との明示的な往復だけを扱う。
設定変更や観測取得は別責務として扱う。

## 観測面

観測面では、本体が能動取得した観測結果だけを扱う。

少なくとも次を受け取れる必要がある。

- 感覚入力
- 外界の取得結果
- 能動取得したクライアント文脈

ここでいう観測は、OtomeKairo が自分から取りに行った情報に限定する。
外から届く対話入力や起床要求は観測面へ入れない。
`desktop_watch` は観測機会を作る現在設定であり、実際の capture は実行連携面の capability を使って行う。
将来的にカメラ、マイク、位置取得、各種センサが増えても、観測面では機器名ではなく capability の結果として受ける。

## 自律面

自律面では、自律判断を始める機会だけを扱う。

少なくとも次を受け取れる必要がある。

- wake 起床要求
- 自律判断の再評価トリガー

自律面は観測面ではない。
ここで受け取るのは判断開始のきっかけであり、観測結果そのものではない。

## 実行連携面

OtomeKairo は、必要に応じて外部接点へ実行要求を出し、その結果を受け取る。
実行連携の基本単位は capability とする。

この面では次を分けて考える。

- 何をしたいかという意図
- どの能力を使うかという選択
- capability ごとの request / result 契約
- 実行要求として何を渡すか
- 実行結果として何が返るか

`desktop_watch` 自体は capability ではない。
`desktop_watch` は観測方針として `vision.capture` のような capability を起動し、その結果を観測面へ戻す。
移動機構やアクチュエータのような物理デバイスも、同じく capability として実行連携面で扱う。
ここでは具体的なハードウェア名ではなく、request / result 契約を正とする。

内部で保留した意図を、そのまま外向き結果に露出する必要はない。
外向きに返す結果と、内部で持ち越す判断結果は分ける。

## 状態面

状態面は、現在の要約状態を読むための面である。
少なくとも次の 2 系統を返せる必要がある。

- 現在の設定要約
  - どの人格設定、記憶、モデルプリセット、方針が選ばれているか
- 現在の稼働要約
  - 接続や実行がどうなっているか
  - 自律判断機構がどう動いているか

ここでは、長期記憶の内部テーブル、`drive_state` の生 row、中間計算結果をそのまま返す面を作らない。

## 設定面

設定面では、現在設定と設定定義を扱う。

少なくとも次を行える必要がある。

- 現在設定の取得
- 選択値や方針の変更
- 人格設定、記憶集合、モデルプリセットなどの設定定義の取得
- 設定定義の作成、複製、差し替え、削除

変更単位の意味境界は `08_設定変更.md` を正とする。

## 列挙面

列挙面は、設定の選択肢を一覧するための面である。

少なくとも次を列挙できる必要がある。

- 人格設定
- 記憶集合
- モデルプリセット

列挙面は、正本のコピーではなく、選択に必要な要約を返す。
現行の `GET /api/catalog` は、人格設定、記憶集合、モデルプリセットだけを返す。
`vision.capture` のような接続中 client の capability は、汎用 catalog ではなく `GET /api/events/stream` の `hello.caps` で扱う。

## inspection 面

inspection 面は、通常運用とは別に、後から判断や更新を追うための面である。

少なくとも次を扱う。

- 判断サイクルの一覧要約
- 特定サイクルの詳細トレース
- 設定変更や記憶更新の観測用情報

ここでは、`episode_series_id` や `open_loops` のような追跡用情報を返す。
ただし、通常の状態面や設定面には持ち込まない。

inspection 面は読み取り中心であり、運用上の正本や編集面とは分ける。

## stream 面

stream 面は、継続的な往復や live 連携が必要な場面で使う。

たとえば次を含みうる。

- server-driven の event/control
- live の段階要約ログ
- 長めの処理の進行通知

stream 面は request/response API の代替ではなく、継続的な連携を補うための面である。

## bootstrap 面との分離

接続確立のための bootstrap 面は、通常運用の各面とは別に扱う。

この分離により、次を保てる。

- 接続確立の都合を通常の意味単位へ持ち込まない
- 認証開始と通常操作を切り分ける
- 相手識別や資格発行の責務を独立させる

接続と権限の意味境界は `12_接続と権限境界.md` を正とする。
