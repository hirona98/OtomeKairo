# モデル構成詳細

## 基本前提

OtomeKairo では、モデル利用をその場の都度指定するのではなく、本体が保持する構成資源として管理する。

この前提により、次が成立する。

- どの構成で動いていたかを追跡しやすい
- 人格や記憶と同じく、選択値で切り替えられる
- 役割ごとの責務を分けたまま、実体の共有もできる
- デバッグ時に、どのモデル構成が判断へ効いたかを追いやすい

## 論理役割

モデル役割は、少なくとも次の論理単位に分けて考える。

| 役割 | 主な責務 |
|------|------|
| 観測解釈 | 観測や想起入口を整理する |
| 意図判断 | 何をするべきかを判断する |
| 表現生成 | 伝達内容を整える |
| 記憶解釈 | 記憶更新や再整理に必要な解釈を行う |
| 埋め込み | ベクトル化を行う |

ここで重要なのは、論理役割とモデル実体を同一視しないことである。
ひとつの実体を複数役割で共有してよい。

## `selected_model_preset_id`

現在どのモデル構成を使うかは `selected_model_preset_id` で表す。
これは永続設定に置き、`runtime_state` には現在反映中のモデル構成参照を出す。

この分離により、次を切り分けられる。

- 設定上どの構成を選んでいるか
- 実際にどの構成で稼働しているか

## `model_profile`

`model_profile` は、実際のモデル接続定義を表す構成資源である。
1 つの `model_profile` は少なくとも次の概念を持つ。

- `model_profile_id`
- `display_name`
- `kind`
  - `generation` または `embedding`
- `provider`
- `model`
- `endpoint_ref`
- `auth_ref`

ここで重要なのは、秘密情報を定義へ直書きしないことである。
`auth_ref` は安全な保管先を指す参照として扱い、秘密値そのものを構成の正本へ埋め込まない。

例:

```yaml
model_profile_id: model_profile:default_generation
display_name: Default Generation Profile
kind: generation
provider: provider_name
model: model_name
endpoint_ref: endpoint:primary
auth_ref:
  type: env
  env_var: PROVIDER_API_KEY
```

## `model_preset`

`model_preset` は、役割ごとのモデル割り当てを束ねる構成資源である。
1 つの `model_preset` は少なくとも次を持つ。

```yaml
model_preset_id: model_preset:default
display_name: Default Preset
roles:
  observation_interpretation:
    model_profile_id: model_profile:default_generation
  decision_generation:
    model_profile_id: model_profile:default_generation
  expression_generation:
    model_profile_id: model_profile:default_generation
  memory_interpretation:
    model_profile_id: model_profile:default_generation
  embedding:
    model_profile_id: model_profile:default_embedding
```

役割ごとに同じ `model_profile_id` を共有してもよい。
役割を分けるのは責務を見通しよくするためであり、物理的なモデル実体の分離を強制するためではない。

## プリセットで持つ理由

役割ごとの割り当てをばらばらに持つと、次が難しくなる。

- どの組み合わせで動いているかを把握すること
- 安定した構成を単位として切り替えること
- 比較実験を追うこと

そのため、役割割り当ては `model_preset` として束ね、現在選択中のものを `selected_model_preset_id` で指す。

## validator の原則

`model_preset` と `model_profile` の validator では、少なくとも次を確認する。

- 必須役割が存在すること
- 各役割が有効な `model_profile` を参照していること
- 役割と `kind` が整合すること
- 秘密情報参照が安全な形で解決可能であること
- 構成全体として意味が崩れていないこと

ここでは、モデル品質の優劣を validator の責務にしない。
契約が成立しているかを確認する。

## 更新と反映

モデル構成の更新は、人格や記憶の更新と同様に明示的な構成変更として扱う。

ここでは次を守る。

- 選択値変更と資源更新を分ける
- 中途半端な部分更新を正本にしない
- 変更後の反映状態を `runtime_state` から追えるようにする

変更単位の原則は `08_構成変更.md` を正とする。

## 役割と外部接点の境界

モデル構成は、外向き API の都合より先に、OtomeKairo の内部責務に合わせて決める。
詳細な wire 契約は `14_API仕様.md` と `design/api/` 配下で定める。

一方で、接続経路や資格情報の信頼境界は `12_接続と権限境界.md` を正とする。
