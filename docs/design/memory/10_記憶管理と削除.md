# 記憶管理と削除

## 目的

この文書は、記憶の通常更新とは別に、管理上の削除、伏せ字化、保持期限、export の境界を定める。

OtomeKairo の記憶は、利用者が通常 API で行単位編集するプロフィールデータベースではない。
一方で、生活する存在として扱う以上、秘密情報、誤投入、不要な raw payload、利用者の削除要求に対する管理境界を持つ。

## 境界

記憶に対する操作は次の 3 種類に分ける。

| 種類 | 役割 |
|------|------|
| 通常入力による訂正 | 会話や出来事として受け、`supersede / revoke / dormant` などの記憶更新へ落とす |
| 記憶集合操作 | `memory_set` 全体の複製、切り替え、削除を設定定義操作として扱う |
| 管理操作 | privacy、security、retention、incident response のために raw payload や記憶行を削除または伏せ字化する |

この文書が定めるのは管理操作である。
通常入力による意味的訂正や、`memory_set` 全体の削除とは混同しない。

## 管理操作の原則

管理操作では、少なくとも次を守る。

- 通常会話 API や通常設定 API には混ぜない
- 管理権限を要求する
- 操作対象、理由、実行者、時刻を audit に残す
- audit へ秘密値や削除対象の raw payload を残さない
- 削除や伏せ字化の結果、根拠を失った派生記憶を再評価する
- 失敗時に部分更新を成功扱いしない

管理操作は、人格設定や LLM 判断から自動実行しない。
LLM は削除権限を持たない。

## 採用する管理操作

採用する管理操作は次とする。

| 操作 | 対象 | 役割 |
|------|------|------|
| `export_memory_set` | `memory_set` | 管理者が記憶集合の内容を確認するために export する |
| `redact_event_payload` | `events.raw_text_or_payload` | 根拠 event の raw payload を伏せ字化し、必要な監査 metadata だけ残す |
| `redact_trace_payload` | cycle trace / logs | デバッグ用 payload から秘密情報や不要な詳細を伏せ字化する |
| `delete_memory_scope` | `scope_type / scope_key` | 特定対象に属する記憶と派生状態を管理削除する |
| `delete_memory_set` | `memory_set` | 記憶集合全体を削除する |
| `apply_retention_policy` | retention 対象 | 保持期限を過ぎた raw payload や詳細 trace を整理する |

`delete_memory_set` は設定定義削除としても現れる。
管理操作として扱う場合は、privacy や incident response の audit 要件も満たす。

## `redact_event_payload`

`redact_event_payload` は、`events` の識別子、時刻、種別、source、最小 metadata を残し、`raw_text_or_payload` を伏せ字化する操作である。

この操作では次を行う。

1. 対象 event を特定する
2. `raw_text_or_payload` を削除済み marker へ置き換える
3. `metadata.redaction` に、redaction の種類、実行時刻、理由コードを残す
4. その event を根拠に持つ `episodes`、`memory_units`、`episode_affect`、`retrieval_runs` を再評価対象にする
5. 根拠を失った派生記憶は `revoke`、`dormant`、または再要約の対象にする

伏せ字化後の marker に元の秘密値や原文を含めない。
削除理由の自由文にも秘密値を入れない。

## `delete_memory_scope`

`delete_memory_scope` は、特定の `scope_type / scope_key` に属する記憶を管理削除する操作である。
これは利用者が通常会話で「その話は違う」と訂正する操作ではない。

対象は次を含む。

- 対象 scope の `memory_units`
- 対象 scope を主対象にする `episodes`
- 対象 scope の `episode_affect`
- 対象 scope の `affect_state`
- 対象 scope に紐づく `drive_state`
- 対象 scope に紐づく `pending_intent`
- 対象 scope に紐づく `world_state`
- 対象 scope の vector index

`events` は、対象 scope だけでなく他 scope の根拠を兼ねる場合がある。
そのため、`delete_memory_scope` では `events` を無条件に消さず、対象 scope の raw payload と派生参照を個別に整理する。

## retention

retention は、保存量とプライバシーのために raw payload と詳細 trace を整理する方針である。

少なくとも次を分ける。

| 対象 | 方針 |
|------|------|
| `cycle_summary` | 長めに保持する |
| 段階 trace 詳細 | 要約中心で保持し、古い詳細は整理する |
| raw event payload | retention 方針に従って伏せ字化または削除する |
| vector index | 元データの削除や伏せ字化に合わせて削除または再生成する |
| audit log | 操作事実を保持し、秘密値を含めない |

retention により raw payload を削除しても、削除済みである事実と最小監査情報は残す。
監査情報には秘密値を含めない。

## export

`export_memory_set` は、管理者が記憶集合の内容を確認するための読み取り操作である。

export では次を守る。

- 秘密値を含む設定値を混ぜない
- raw payload を含む export と、要約だけの export を分ける
- export 実行を audit に残す
- export 結果を通常の会話文脈や LLM prompt へ自動投入しない

export は記憶更新ではない。
export により `memory_units`、`episodes`、`events` を変更しない。

## 派生状態の扱い

管理削除や伏せ字化の後は、派生状態をそのまま残さない。

少なくとも次を再評価する。

- `memory_units`
- `memory_links`
- `revisions`
- `vector_index_entries`
- `drive_state`
- `pending_intent`
- `ongoing_action`
- `world_state`
- `episode_affect`
- `mood_state`
- `affect_state`
- `retrieval_runs`
- `cycle_trace`

根拠を失った状態は、削除、失効、再生成、または `dormant` 化する。
どの処理を行ったかを audit に残す。

## API 境界

管理操作の wire 契約は、通常の設定面や会話面へ混ぜない。
管理 API を追加する場合は、`design/api/` 配下に専用の管理面を追加し、認証、管理権限、audit、秘密値非露出を同じ変更内で定義する。

通常の記憶訂正は会話入力として扱う。
管理削除は管理権限を持つ明示操作として扱う。

## やらないこと

次は採らない。

- 利用者の通常入力をそのまま物理削除命令として扱うこと
- LLM 判断だけで hard delete を実行すること
- 管理削除を `supersede / revoke / dormant` と同じ意味更新に混ぜること
- audit に削除対象の秘密値を残すこと
- raw payload を削除したあとも、派生記憶を根拠確認なしに残すこと
