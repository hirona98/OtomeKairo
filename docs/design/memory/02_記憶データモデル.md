# 記憶データモデル

<!-- Block: Role -->
## この文書の役割

この文書は、OtomeKairo の記憶データモデルを定める正本である。

ここで固定したいのは次の 4 点である。

- 記憶の最小単位を何にするか
- `scope` をどう切るか
- 想起時にどの形で LLM へ渡すか
- 更新をどの粒度で回すか

ここでは DB や API の詳細までは確定しない。
ただし、実装がぶれないための中核形はここで固定する。

<!-- Block: SetVsScope -->
## 記憶セットと `scope` は別物とする

OtomeKairo では、記憶切り替えのために「選択可能な記憶全体」が必要である。
一方で、その記憶の中で「誰について」「何について」の理解かを区別する整理軸も必要になる。

この 2 つは分けて考えたほうがよい。

- 記憶セット:
  - ユーザーが切り替える単位
  - 1 つの人格に対して複数持てる全体の入れ物
- `scope`:
  - 1 つの記憶セットの内部で、どの対象についての記憶かを整理する単位

たとえば同じ記憶セットの中に、次のような `scope` が共存する。

- 自分についての理解
- ユーザーについての理解
- 特定の人物についての理解
- 特定の話題についての理解
- 自分とユーザーの関係についての理解

この分離により、記憶切り替えの操作は単純に保ちつつ、内部では人格形成に必要な整理を細かく持てる。

<!-- Block: Layers -->
## 記憶の層を分ける

現時点では、次の 4 層に分けるのがよい。

1. `events`
2. `episode_digests`
3. `memory_units`
4. `affect_state`

それぞれの役割は次のとおり。

- `events`:
  - 観測された生ログの正本
  - 消さない
- `episode_digests`:
  - 想起しやすい出来事単位
  - 「何があったか」の要約
- `memory_units`:
  - 継続理解の最小単位
  - 比較、更新、矛盾管理、想起の中心
- `affect_state`:
  - 瞬間感情と背景感情
  - 返答トーンと salience に効く

この中で、OtomeKairo の人格形成の中核になるのは `memory_units` である。

<!-- Block: Events -->
## `events` の最小契約

`events` は観測された生ログの正本である。
MVP では、少なくとも次を持つ前提にする。

| 区分 | 項目 | 役割 |
|------|------|------|
| 識別 | `event_id` | 出来事の識別子 |
| 種別 | `event_type` | 観測、返答、判断、実行結果などの分類 |
| 時間 | `occurred_at` | その出来事が起きた時刻 |
| 出所 | `source` / `role` | ユーザー入力、システム返答、内部判断などの区別 |
| 内容 | `raw_text_or_payload` | 生の文字列または構造化 payload |
| 補足 | `metadata` | クライアント文脈や実行結果補足 |

ここでは、整形や解釈を `events` に混ぜない。
`events` は後続の `episode_digests` と `memory_units` が辿る根拠層とする。

<!-- Block: EpisodeDigests -->
## `episode_digests` の最小契約

`episode_digests` は、`events` を想起しやすい出来事単位へ圧縮した層である。
MVP では、少なくとも次を持つ前提にする。

| 区分 | 項目 | 役割 |
|------|------|------|
| 識別 | `episode_digest_id` | 要約単位の識別子 |
| 種別 | `episode_type` | 会話、相談、約束確認などの大まかな分類 |
| 対象 | `primary_scope_type` / `primary_scope_key` | 主にどの対象領域の出来事か |
| 要点 | `summary_text` | 何があったかの要約 |
| 結果 | `outcome_text` | その場で決まった方針や結果 |
| 未完了 | `open_loops` | 続きや確認待ちとして残る点 |
| 根拠 | `linked_event_ids` | この digest が参照する `event_id` 群 |
| 品質 | `salience` | 想起時の前面度 |
| 時間 | `formed_at` | digest を作った時刻 |

`episode_digests` は、回想と根拠提示の主軸に使う。
逐語再現ではなく、「その場で何が話題になり、何が決まったか」を持つ層として扱う。
MVP の `turn consolidation` では、標準を `1イベント = 1digest` とする。
そのため、`linked_event_ids` は初期段階では長さ 1 が基本であり、複数件を持つのは将来の圧縮や再整理を入れたときの拡張余地として扱う。

<!-- Block: AffectState -->
## `affect_state` の最小契約

`affect_state` は感情の短期と長期を分けて持つ層である。
MVP では、少なくとも次を持つ前提にする。

| 区分 | 項目 | 役割 |
|------|------|------|
| 識別 | `affect_state_id` | 感情状態の識別子 |
| 層 | `layer` | 瞬間感情か背景感情か |
| 対象 | `target_scope_type` / `target_scope_key` | 何に対する感情か |
| 内容 | `affect_label` | 不安、安心、苛立ち、親しみなど |
| 強度 | `intensity` | 強さ |
| 時間 | `observed_at` / `updated_at` | 観測時刻と更新時刻 |

`affect_state` は `RecallPack` の一部ではない。
想起後の `AffectContext` を構成する別枠の内部コンテキストとして扱う。

<!-- Block: WhyUnit -->
## `memory_units` を中核にする理由

`state` のような広い自由文だけで記憶を持つと、次が難しくなる。

- 何が同じ対象についての理解かを判定すること
- 新しい理解が旧理解を更新したのか、別件なのかを判定すること
- 想起時に役割ごとに分けること
- 関係や矛盾を追跡すること

そのため、OtomeKairo では「人間が読める自然文」は残しつつ、更新と比較ができる半構造化の単位を持つほうがよい。

<!-- Block: Contract -->
## `memory_units` の最小契約

`memory_units` は、少なくとも次の情報を持つ前提にする。

| 区分 | 項目 | 役割 |
|------|------|------|
| 識別 | `memory_unit_id` | 単位の識別子 |
| 種別 | `memory_type` | 何の理解かを表す |
| 対象 | `scope_type` / `scope_key` | どの対象領域についての記憶かを表す |
| 意味骨格 | `subject_ref` / `predicate` / `object_ref_or_value` | 更新・比較・関係付けの軸 |
| 本文 | `summary_text` | 人間と LLM が読む短い本文 |
| 状態 | `status` | 現在どういう扱いの理解かを表す |
| 品質 | `confidence` / `salience` | どれだけ信じるか、どれだけ前に出すか |
| 時間 | `formed_at` / `last_confirmed_at` / `valid_from` / `valid_to` | 継続性と変化の管理 |
| 根拠 | `evidence_event_ids` | どの出来事から来た理解か |
| 補足 | `qualifiers` | 数値、条件、参加者などの追加情報 |

ここで重要なのは、`summary_text` を主にしつつも、それだけに依存しないことである。

意味骨格を別に持つことで、次がやりやすくなる。

- 同じ対象かどうかの比較
- 訂正時の置き換え判断
- 関係リンクの生成
- 想起パックへの振り分け

## `memory_units` の概念例

```json
{
  "memory_unit_id": 184,
  "memory_type": "preference",
  "scope_type": "user",
  "scope_key": "user",
  "subject_ref": "user",
  "predicate": "likes",
  "object_ref_or_value": "food:spicy",
  "summary_text": "あなたは辛い食べ物をかなり好む。",
  "status": "confirmed",
  "confidence": 0.94,
  "salience": 0.82,
  "formed_at": "2026-03-20T19:22:00",
  "last_confirmed_at": "2026-03-27T21:41:00",
  "valid_from": null,
  "valid_to": null,
  "evidence_event_ids": [120, 188],
  "qualifiers": {
    "polarity": "positive"
  }
}
```

<!-- Block: Types -->
## `memory_type` の最小セット

MVP 時点では、`memory_type` は増やしすぎないほうがよい。
最小セットとしては次で十分である。

| `memory_type` | 用途 | 例 |
|------|------|------|
| `fact` | 継続的な事実理解 | プロフィール、生活状況、設定 |
| `preference` | 好み、苦手、傾向 | 食べ物、話題、会話スタイル |
| `relation` | 対象同士の関係 | 親しい、距離がある、協力している |
| `commitment` | 約束、未完了、保留 | 後で話す、やる予定、確認待ち |
| `interpretation` | その場を越えて残す解釈 | 最近疲れていそう、今この話題は重い |
| `summary` | 複数単位を束ねる反省や圧縮 | 最近の関係変化、月間の話題整理 |

この段階では、感情は `memory_units` に統合しない。
感情は別層で持ち、必要に応じて `memory_units.salience` や想起順へ効かせる。

<!-- Block: Scope -->
## `scope` の切り方

`scope` は「この記憶が主にどの対象領域に属するか」を表す。
MVP の最小セットは次とする。

| `scope_type` | 用途 | `scope_key` の例 |
|------|------|------|
| `self` | 自分についての理解 | `self` |
| `user` | ユーザーについての理解 | `user` |
| `entity` | その他の人物、物、場所、道具 | `person:tanaka`, `place:office`, `tool:obsidian` |
| `topic` | 継続話題やテーマ | `topic:game_dev`, `topic:health` |
| `relationship` | 複数対象の関係 | `self|user`, `self|person:tanaka` |
| `world` | 特定対象に閉じない全体理解 | `world` |

この設計では、`scope` は検索のためだけでなく、人格形成の整理単位でもある。

`relationship` の `scope_key` は、同じ関係が複数キーに割れないように正規化する。
初期ルールとしては、`self` が含まれる場合は常に先頭に置き、それ以外は正規化済み参照名の辞書順で並べる。

たとえば次のように使う。

- `self`: 自己像、最近の自分の傾向、行動姿勢
- `user`: ユーザー理解、嗜好、状態推定
- `relationship`: 自分とユーザーの距離感、信頼、摩擦、約束
- `topic`: 継続中の話題や未解決のテーマ

<!-- Block: Refs -->
## 参照表現の最小ルール

`subject_ref` と `object_ref_or_value` は、比較しやすい最小ルールを持たせる。

- `self`
- `user`
- `person:<normalized_name>`
- `place:<normalized_name>`
- `tool:<normalized_name>`
- `topic:<normalized_name>`
- リテラル値
  - 例: `food:spicy`, `status:waiting_confirmation`

内部で識別子が確立したら、将来的に内部 ID へ寄せてもよい。
ただし、初期段階では「人間が見ても分かる正規化文字列」を優先する。

<!-- Block: ScopeNotes -->
## `scope` と `subject_ref` の違い

`scope` と `subject_ref` は似ているが、役割が違う。

- `scope`:
  - どの棚に置く記憶か
  - 想起パックへの振り分けに効く
- `subject_ref`:
  - 文の主語として、何についての理解か
  - 更新判定と関係判定に効く

例:

- 「私は最近、あなたに少し甘えやすくなっている」
  - `scope_type = relationship`
  - `scope_key = self|user`
  - `subject_ref = self`
- 「あなたは最近、睡眠不足気味かもしれない」
  - `scope_type = user`
  - `scope_key = user`
  - `subject_ref = user`

<!-- Block: Status -->
## `status` の最小セット

`status` は、記憶を消さずに現在の扱いを変えるために必要である。
`memory_units` では次を最小セットとする。

| `status` | 意味 |
|------|------|
| `inferred` | 推測で得た理解。通常会話に使ってよいが、強く断定しない |
| `confirmed` | 明示発言や反復で強く支持された理解 |
| `superseded` | 新しい理解により旧版になったもの |
| `revoked` | 明確に否定されたもの |
| `dormant` | 現在は前面に出さないが保持するもの |

ここでは `observed` を入れない。
観測そのものは `events` と `episode_digests` 側で扱うからである。

<!-- Block: Quality -->
## `confidence` と `salience` は分ける

この 2 つは似て見えるが、役割が違う。

- `confidence`:
  - その理解がどれだけ正しそうか
- `salience`:
  - その理解を今どれだけ前に出すべきか

たとえば次のような違いがある。

- 昔から確かな好み:
  - `confidence` は高い
  - `salience` は会話文脈次第
- 直近の強い出来事:
  - `confidence` は中程度でもよい
  - `salience` は高くなりやすい

OtomeKairo では、忘却を「消える」ではなく「salience が落ちる」で表現したほうがよい。

<!-- Block: Links -->
## `memory_links` の最小セット

`memory_units` は孤立させず、少数の関係を持てるようにする。
MVP の最小リンクは次でよい。

| `label` | 意味 |
|------|------|
| `supports` | この理解を支持する |
| `contradicts` | この理解と矛盾する |
| `derived_from` | この理解から派生した |
| `about_same_scope` | 同じ対象領域を別角度から扱う |
| `affects` | この理解が別理解や関係に影響する |

これで最低限、次が追える。

- 関係の補強
- 訂正の追跡
- 反省がどの事実から育ったか
- 同じ対象に関する理解の束

<!-- Block: RetrievalRuns -->
## `retrieval_runs` の最小契約

`retrieval_runs` は、同期想起が何を根拠に `RecallPack` を組んだかを追うための監査層である。
MVP では、少なくとも次を持つ前提にする。

| 区分 | 項目 | 役割 |
|------|------|------|
| 識別 | `retrieval_run_id` | 想起実行の識別子 |
| 時間 | `created_at` | 想起を実行した時刻 |
| 入力 | `observation_summary` | 現在観測の要約 |
| 入口 | `recall_hint_snapshot` | その回の `RecallHint` |
| 選定 | `selected_memory_unit_ids` | 採用した `memory_units` |
| 選定 | `selected_episode_digest_ids` | 採用した `episode_digests` |
| 根拠 | `selected_event_ids` | 例外参照で開いた `events` |
| 結果 | `recall_pack_summary` | 最終 `RecallPack` の要約 |

`retrieval_runs` は長期記憶そのものではない。
あくまで、想起の理由と選定結果を後から検証するための観測ログとして扱う。

<!-- Block: RetrievalPack -->
## 想起パックは役割別に組む

返答生成に渡す記憶は、フラット一覧ではなく役割別パックにしたほうがよい。

最小形は次を推奨する。

```json
{
  "self_model": [],
  "user_model": [],
  "relationship_model": [],
  "active_topics": [],
  "active_commitments": [],
  "episodic_evidence": [],
  "conflicts": []
}
```

各セクションの役割は次のとおり。

- `self_model`:
  - 自己像、最近の自分の傾向
- `user_model`:
  - ユーザー理解
- `relationship_model`:
  - 自分とユーザー、または重要人物との関係
- `active_topics`:
  - いま継続中の話題
- `active_commitments`:
  - 約束、保留、未完了
- `episodic_evidence`:
  - 根拠になる出来事
- `conflicts`:
  - 矛盾や注意点

`RecallPack` の外側には、次を別で持つ。

- 直近会話
- `TimeContext`
- `AffectContext`

<!-- Block: RetrievalBudget -->
## 候補収集はセクションごとに仮上限を持つ

人格を構成する記憶では、1 種類の記憶が全部を食うと崩れやすい。
そのため、候補収集の段階でセクションごとの仮上限を持つほうがよい。

初期の目安は次程度で十分である。
この値は議論用の目安ではなく、MVP の候補収集時の既定値として扱う。

- `self_model`: 3〜4件
- `user_model`: 4〜6件
- `relationship_model`: 3〜4件
- `active_topics`: 2〜4件
- `active_commitments`: 3〜5件
- `episodic_evidence`: 4〜6件
- `conflicts`: 0〜2件

その後、最終的な `RecallPack` は全体上限 `8〜14 件` に収まるように再圧縮する。
これにより、候補の偏りを防ぎつつ、注入サイズも小さく保てる。

<!-- Block: RetrievalRule -->
## 想起の基本ルール

想起では、少なくとも次の順で組むのがよい。

1. `RecallHint` から `primary_intent` と `focus_scopes` を取る
2. その `scope` に近い `memory_units` を引く
3. `self` / `user` / `relationship` の基礎記憶を少量ずつ混ぜる
4. `active_commitments` を必ず別枠で引く
5. 必要な根拠 `episode_digests` を添える
6. 矛盾候補があれば `conflicts` へ入れる

この構成なら、「人格に効く記憶」と「答えの根拠」を混ぜずに扱える。

<!-- Block: UpdateFlow -->
## 更新フロー

更新は次の 2 段階に分けるのがよい。

### `turn consolidation`

1ターンごとに行う。
ここでやるのは次である。

- `events` を保存する
- `episode_digests` を作る
- 候補となる `memory_units` を生成する
- 既存の同系統 `memory_units` と比較する
- `inferred` / `confirmed` / `superseded` / `revoked` を更新する
- `salience` と `confidence` を更新する
- `memory_links` を少量だけ張る
- `affect_state` を更新する

### `reflective consolidation`

数ターンまたは一定時間ごとに行う。
ここでやるのは次である。

- 複数の `episode_digests` を横断して `summary` を作る
- `self` / `user` / `relationship` の長期変化を再評価する
- 重複や近縁の `memory_units` を整理する
- `dormant` 化の判断を行う
- 反省や行動傾向の学びを作る

OtomeKairo では、この後者が人格形成に特に効く。

<!-- Block: Compare -->
## 更新時の比較単位

新しい `memory_units` を既存と比較するときは、少なくとも次を見る。

- `memory_type`
- `scope_type` / `scope_key`
- `subject_ref`
- `predicate`
- `object_ref_or_value`

この 5 軸が近いものを「更新候補」とみなす。
逆に、本文が似ていてもこの軸が違うなら、別記憶として扱うほうがよい。

<!-- Block: Conflict -->
## 矛盾の扱い

OtomeKairo では、矛盾を即削除しないほうがよい。
原則は次とする。

- 古い理解は `superseded` へ落とす
- 明確な否定は `revoked` にする
- どちらが現行か曖昧なら両方を残し、`conflicts` として想起側へ渡せるようにする

これにより、誤更新で人格が急変するのを防ぎやすい。

<!-- Block: MVP -->
## MVP でまず入れるべき範囲

MVP では、欲張りすぎず次を先に成立させるのがよい。

- `events`
- `episode_digests`
- `memory_units`
  - `fact`
  - `preference`
  - `relation`
  - `commitment`
- `affect_state`
- 役割別の想起パック
- `revisions`
- `retrieval_runs`

逆に、次は後回しでよい。

- 大規模知識グラフ
- 複雑な自動忘却
- 手続き記憶の自動書き換え
- 多段の反省ループ
