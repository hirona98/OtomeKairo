# 記憶データモデル

## 記憶セットと `scope` は別物とする

OtomeKairo では、記憶切り替えのために「選択対象の記憶全体」が必要である。
一方で、その記憶の中で「誰について」「何について」の理解かを区別する整理軸も必要になる。

この 2 つは分けて考える。

- 記憶セット
  - ユーザーが切り替える単位
  - 人格設定と独立に選べる記憶全体の入れ物
  - 同じ人格設定に対して複数持て、既存の記憶セットを複製して新しい記憶セットを作ってよい
- `scope`
  - 1 つの記憶セットの内部で、どの対象についての記憶かを整理する単位

この分離により、記憶切り替えの操作は単純に保ちつつ、内部では現在の個の構成に必要な整理を細かく持てる。
同じ `memory_set` を異なる人格設定と組み合わせて選んでもよい。

## 記憶の 4 層

OtomeKairo の記憶は、次の 4 層で構成する。

1. `events`
2. `episodes`
3. `memory_units`
4. `affect_state`

それぞれの役割は次のとおりである。

- `events`
  - 観測された生ログの正本
- `episodes`
  - 経験のまとまり
- `memory_units`
  - 継続理解の最小単位
- `affect_state`
  - 瞬間感情と背景感情

`memory_units` は継続理解の中心だが、それだけでは現在の個は立たない。
`episodes` は経験の連続性を支え、`events` は根拠層として残り、`affect_state` は判断の前景を変える。

## `events` の最小契約

`events` は観測された生ログの正本である。
少なくとも次を持つ。

| 区分 | 項目 | 役割 |
|------|------|------|
| 識別 | `event_id` | 出来事の識別子 |
| 種別 | `event_type` | 観測、返答、判断、実行結果などの分類 |
| 時間 | `occurred_at` | その出来事が起きた時刻 |
| 出所 | `source` / `role` | ユーザー入力、システム返答、内部判断などの区別 |
| 内容 | `raw_text_or_payload` | 生の文字列または構造化 payload |
| 補足 | `metadata` | 文脈や実行結果の補足 |

ここでは、整形や解釈を `events` に混ぜない。
`events` は後続の `episodes` と `memory_units` が辿る根拠層とする。

## `episodes` の最小契約

`episodes` は、`events` を経験単位へ束ねた層である。
逐語再現ではなく、「その場で何が起き、何が残ったか」を持つ。

少なくとも次を持つ。

| 区分 | 項目 | 役割 |
|------|------|------|
| 識別 | `episode_id` | 経験単位の識別子 |
| 種別 | `episode_type` | 会話、相談、約束確認、行動結果などの大まかな分類 |
| 対象 | `primary_scope_type` / `primary_scope_key` | 主にどの対象領域の経験か |
| 要点 | `summary_text` | 何が起きたかの要約 |
| 結果 | `outcome_text` | その場で決まったことや残ったこと |
| 未完了 | `open_loops` | 続きや確認待ちとして残る点 |
| 根拠 | `linked_event_ids` | 同一判断サイクルを構成する `event_id` 群 |
| 品質 | `salience` | 想起時の前面度 |
| 時間 | `formed_at` | `episode` を作った時刻 |

この設計では、`1判断サイクル = 1episode` を基本単位とする。
ただし、短すぎる処理や極端に小さい観測では、1 件の `event` とほぼ同義になることもある。

## `affect_state` の最小契約

`affect_state` は感情の短期と長期を分けて持つ層である。
ここでいう `layer` は、記憶全体の 4 層とは別であり、感情の時間層を表す。

少なくとも次を持つ。

| 区分 | 項目 | 役割 |
|------|------|------|
| 識別 | `affect_state_id` | 感情状態の識別子 |
| 時間層 | `layer` | 瞬間感情か背景感情か |
| 対象 | `target_scope_type` / `target_scope_key` | 何に対する感情か |
| 内容 | `affect_label` | 不安、安心、苛立ち、親しみなど |
| 強度 | `intensity` | 強さ |
| 時間 | `observed_at` / `updated_at` | 観測時刻と更新時刻 |

`layer` の値は次に固定する。

- `surface`
  - その場で前面に出ている短期感情
- `background`
  - 数ターン以上にまたがりうる背景的な感情傾向

同じ `target_scope_type / target_scope_key` の中では、`layer` ごとに最大 2 件まで保持する。

- 同じ `affect_label` は別件として増やさず、同じ感情状態を更新する
- 異なる `affect_label` は少数だけ並存してよい
- 上限を超える場合は、強度と最近性が低いものから落とす

`target_scope_type=self` は、その時点の自分の気分を表す。
`target_scope_type=relationship` は、特定相手との関係感情を表す。
`target_scope_type=topic`、`entity`、`world` は、話題、対象、全体状況に対する感情を表してよい

`self` と `relationship` は、現在の個に直接効く主要な感情対象として扱う。
`topic`、`entity`、`world` は保持してよいが、背景感情へ育てる条件と減衰はより慎重にする。

`affect_state` は `RecallPack` の一部ではない。
想起後の `AffectContext` を構成する別枠の内部コンテキストとして扱う。

## `memory_units` を中核にする理由

自由文だけで記憶を持つと、次が難しくなる。

- 何が同じ対象についての理解かを判定すること
- 新しい理解が旧理解を更新したのか、別件なのかを判定すること
- 想起時に役割ごとに分けること
- 関係や矛盾を追跡すること

そのため、OtomeKairo では「人間が読める自然文」は残しつつ、更新と比較ができる半構造化の単位を持つ。

## `memory_units` の最小契約

`memory_units` は、少なくとも次の情報を持つ。

| 区分 | 項目 | 役割 |
|------|------|------|
| 識別 | `memory_unit_id` | 単位の識別子 |
| 種別 | `memory_type` | 何の理解かを表す |
| 対象 | `scope_type` / `scope_key` | どの対象領域についての記憶か |
| 意味骨格 | `subject_ref` / `predicate` / `object_ref_or_value` | 更新、比較、関係付けの軸 |
| 本文 | `summary_text` | 人間と LLM が読む短い本文 |
| 状態 | `status` | 現在どういう扱いの理解か |
| 進行 | `commitment_state` | `memory_type=commitment` のときの履行状態 |
| 品質 | `confidence` / `salience` | どれだけ信じるか、どれだけ前に出すか |
| 時間 | `formed_at` / `last_confirmed_at` / `valid_from` / `valid_to` | 継続性と変化の管理 |
| 根拠 | `evidence_event_ids` | どの出来事から来た理解か |
| 補足 | `qualifiers` | 数値、条件、参加者などの追加情報 |

`summary_text` を主にしつつも、それだけに依存しないことが重要である。
意味骨格を別に持つことで、比較、訂正、関係付け、想起パックへの振り分けが安定する。

## `memory_units` の概念例

```json
{
  "memory_unit_id": 184,
  "memory_type": "preference",
  "scope_type": "user",
  "scope_key": "user",
  "subject_ref": "user",
  "predicate": "likes",
  "object_ref_or_value": "food:spicy",
  "summary_text": "あなたは辛い食べ物をかなり好む。",
  "status": "confirmed",
  "confidence": 0.94,
  "salience": 0.82,
  "formed_at": "2026-03-20T19:22:00",
  "last_confirmed_at": "2026-03-27T21:41:00",
  "valid_from": null,
  "valid_to": null,
  "evidence_event_ids": [120, 188],
  "qualifiers": {
    "polarity": "positive"
  }
}
```

## 採用する `memory_type`

`memory_type` の値は次とする。

| `memory_type` | 用途 | 例 |
|------|------|------|
| `fact` | 継続的な事実理解 | プロフィール、生活状況、設定 |
| `preference` | 好み、苦手、傾向 | 食べ物、話題、会話スタイル |
| `relation` | 対象同士の関係 | 親しい、距離がある、協力している |
| `commitment` | 約束、未完了、保留 | 後で話す、やる予定、確認待ち |
| `interpretation` | その場を越えて残す解釈 | 最近疲れていそう、今この話題は重い |
| `summary` | 複数単位を束ねる長期変化の要約 | 最近の関係変化、月間の話題整理 |

感情は `memory_units` に統合しない。
感情は別層で持ち、`memory_units.salience` や想起順へ反映する。

## `scope` の切り方

`scope` は「この記憶が主にどの対象領域に属するか」を表す。
`scope_type` の値は次とする。

| `scope_type` | 用途 | `scope_key` の例 |
|------|------|------|
| `self` | 自分についての理解 | `self` |
| `user` | ユーザーについての理解 | `user` |
| `entity` | その他の人物、物、場所、道具 | `person:tanaka`, `place:office`, `tool:obsidian` |
| `topic` | 継続話題やテーマ | `topic:game_dev`, `topic:health` |
| `relationship` | 複数対象の関係 | `self|user`, `self|person:tanaka` |
| `world` | 特定対象に閉じない全体理解 | `world` |

この設計では、`scope` は検索のためだけでなく、現在の個を構成する整理単位でもある。

`relationship` の `scope_key` は、同じ関係が複数キーに割れないように正規化する。
`self` が含まれる場合は常に先頭に置き、それ以外は正規化済み参照名の辞書順で並べる。

## 参照表現の最小ルール

`subject_ref` と `object_ref_or_value` は、比較しやすい最小ルールを持たせる。

- `self`
- `user`
- `person:<normalized_name>`
- `place:<normalized_name>`
- `tool:<normalized_name>`
- `topic:<normalized_name>`
- リテラル値
  - 例: `food:spicy`, `commitment_state:waiting_confirmation`

内部で識別子が確立している場合は、内部 ID を使ってよい。
ただし、人間が見ても分かる正規化文字列を優先する。

## `scope` と `subject_ref` の違い

`scope` と `subject_ref` は似ているが、役割が違う。

- `scope`
  - どの棚に置く記憶か
  - 想起パックへの振り分けに効く
- `subject_ref`
  - 文の主語として、何についての理解か
  - 更新判定と関係判定に効く

## 採用する `status`

`status` は、記憶を消さずに現在の扱いを変えるために必要である。
値は次とする。

| `status` | 意味 |
|------|------|
| `inferred` | 推測で得た理解。使うが強く断定しない |
| `confirmed` | 明示発言や反復で強く支持された理解 |
| `superseded` | 新しい理解により旧版になったもの |
| `revoked` | 明確に否定されたもの |
| `dormant` | 現在は前面に出さないが保持するもの |

観測そのものは `events` と `episodes` 側で扱うため、ここに `observed` は入れない。

## 採用する `commitment_state`

`commitment` は、確からしさとは別に履行状態を持つ必要がある。
そのため、`memory_type=commitment` のときだけ次を使う。

| `commitment_state` | 意味 |
|------|------|
| `open` | 未完了で進行中 |
| `waiting_confirmation` | 相手や条件の確認待ち |
| `on_hold` | 一時保留 |
| `done` | 完了した |
| `cancelled` | 取消または立ち消え |

ここで重要なのは、`status` と `commitment_state` を混ぜないことである。

- `status`
  - その理解がどれだけ支持されているか
- `commitment_state`
  - 約束そのものが今どういう進行状態にあるか

## `confidence` と `salience` は分ける

この 2 つは似て見えるが、役割が違う。

- `confidence`
  - その理解がどれだけ正しそうか
- `salience`
  - その理解を今どれだけ前に出すべきか

OtomeKairo では、忘却を「消える」ではなく「`salience` が落ちる」で表現する。

## 補助構造

### `memory_links`

`memory_units` は孤立させず、少数の関係を持てるようにする。
最小リンクは次とする。

| `label` | 意味 |
|------|------|
| `supports` | この理解を支持する |
| `contradicts` | この理解と矛盾する |
| `derived_from` | この理解から派生した |
| `about_same_scope` | 同じ対象領域を別角度から扱う |
| `affects` | この理解が別理解や関係に影響する |

### `revisions`

`revisions` は、`memory_units` の変化を後から監査するための履歴層である。
少なくとも次を持つ。

| 区分 | 項目 | 役割 |
|------|------|------|
| 識別 | `revision_id` | 変更履歴の識別子 |
| 時間 | `occurred_at` | 変更を記録した時刻 |
| 対象 | `memory_unit_id` | 変更対象の `memory_units` |
| 操作 | `operation` | `create` / `reinforce` / `refine` / `supersede` / `revoke` / `dormant` |
| 関連 | `related_memory_unit_ids` | 置換元や関連先の `memory_unit_id` 群 |
| 変更前 | `before_snapshot` | 変更前の要約スナップショット |
| 変更後 | `after_snapshot` | 変更後の要約スナップショット |
| 理由 | `reason` | なぜその更新を行ったか |
| 根拠 | `evidence_event_ids` | 根拠になった `event_id` 群 |

### `retrieval_runs`

`retrieval_runs` は、同期想起が何を根拠に `RecallPack` を組んだかを追うための監査層である。
少なくとも次を持つ。

| 区分 | 項目 | 役割 |
|------|------|------|
| 識別 | `retrieval_run_id` | 想起実行の識別子 |
| 時間 | `created_at` | 想起を実行した時刻 |
| 入力 | `observation_summary` | 現在観測の要約 |
| 入口 | `recall_hint_snapshot` | その回の `RecallHint` |
| 状態 | `result_status` | `success` または `failed` |
| 理由 | `failure_reason` | 失敗時の理由 |
| 選定 | `selected_memory_unit_ids` | 採用した `memory_units` |
| 選定 | `selected_episode_ids` | 採用した `episodes` |
| 根拠 | `selected_event_ids` | 例外参照で開いた `events` |
| 結果 | `recall_pack_summary` | 最終 `RecallPack` の要約 |

### `vector_index_entries`

`vector_index_entries` は、連想レーンが `memory_units` と `episodes` を曖昧検索するための補助索引である。
少なくとも次を持つ。

| 区分 | 項目 | 役割 |
|------|------|------|
| 識別 | `vector_entry_id` | index エントリの識別子 |
| 出所 | `source_kind` | `memory_unit` または `episode` |
| 出所 | `source_id` | 元データの識別子 |
| 埋め込み | `embedding_preset` | どの埋め込み設定で作ったか |
| 検索文 | `source_text` | 埋め込みに使った正規化済み文字列 |
| 対象 | `scope_type` / `scope_key` | 想起時の絞り込み補助 |
| 種別 | `source_type` | `memory_type` または `episode_type` |
| 状態 | `status` | 元データの現在状態 |
| 品質 | `salience` | 想起時の再整列補助 |
| 補助 | `has_open_loops` | 続き物かどうかの補助フラグ |
| 時間 | `updated_at` | 最後に index を更新した時刻 |
| 整合 | `text_hash` | 差分更新判定用のハッシュ |

index 対象は次に限定する。

- `memory_units.summary_text`
- `episodes.summary_text`
- `episodes.outcome_text`
- `episodes.open_loops`

`events` の生ログ全文は index しない。
細部確認は `events` の例外参照で扱い、連想レーンの主索引には入れない。
