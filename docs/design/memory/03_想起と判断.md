# 想起と判断

<!-- Block: Role -->
## この文書の役割

この文書は、OtomeKairo が記憶をどう想起し、その結果をどう会話と行動の判断へ接続するかを定める正本である。

ここでは、次の 4 点を正本として定める。

- 記憶をどう使って現在の判断文脈を作るか
- 会話と行動で何が共通で、どこから違うか
- 行動判断に入ったときに、会話と違って何を追加で見るか
- 返答後 / 実行後に何を記憶更新へ返すか

個別の図や検証は参考資料へ分けるが、記憶利用の主線はこの文書に寄せる。

<!-- Block: Guide -->
## この文書で定める範囲

この文書では、次を正本として定める。

- `RecallHint` の役割と最小契約
- 同期想起の基本フロー
- `RecallPack` の構成
- `reply / act / noop` の判断分岐
- `events` の例外参照の使い方

<!-- Block: Premise -->
## 基本前提

OtomeKairo では、会話と行動を別々の知能として扱わない。
両方とも、「観測を受け、記憶を踏まえて、何をするべきかを判断する」という同じ中心ループの分岐として扱う。

この前提により、次が成立する。

- 会話だけが人格の表現ではない
- 行動も、記憶から組み上がる人格文脈の延長で決まる
- 会話用の記憶と行動用の記憶を別系統に分けずに済む

ただし、同じなのは判断の土台までである。
判断後に求められる出力契約は、会話と行動で異なる。

<!-- Block: SharedLoop -->
## 共通本体

会話でも行動でも、共通本体は次の流れにする。

1. 現在の観測を受け取る
2. `RecallHint` を作る
3. 構造レーンと連想レーンで必要な記憶を集める
4. `RecallPack` を作る
5. `reply / act / noop` を判断する
6. 分岐先ごとの処理を実行する
7. 結果を `events` に保存し、記憶更新へ返す

ここで重要なのは、会話と行動で最初から別の想起系を作らないことである。
まずは同じ想起系で現在の人格文脈を作り、その後で分岐する。

<!-- Block: Observation -->
## 共通本体で扱う入力

共通本体が受けるものは、厳密には「ユーザー入力」ではなく「現在の観測」である。

MVP で主に入る観測は次である。

- 最新のユーザー入力
- 直近会話
- 定期起床という事実
- 軽量クライアント文脈
- 現在時刻

会話主体の初版では `RecallHint` を最新ユーザー入力中心で定義しているが、統合設計ではこれを「現在観測を中心とした想起入口」として読む。
つまり、会話時は `最新入力` が主観測になり、自発判断時は `定期起床 + 軽量文脈` が主観測になる。

<!-- Block: RecallHint -->
## `RecallHint`

`RecallHint` は軽量高速 LLM が、現在観測と直近文脈だけを見て生成する入口情報である。

入力は次に限定する。

- 現在観測
- 直近 3 分以内
- 最大 4-6 発話
- 現在時刻

この段階では長期記憶を読まない。

`RecallHint` の最小契約は次とする。

- `primary_intent`
- `secondary_intents`
- `confidence`
- `time_reference`
- `focus_scopes`
- `mentioned_entities`
- `mentioned_topics`

`primary_intent` は必ず 1 つにする。
`secondary_intents` は補助扱いとし、最大 2 件までに留める。
`focus_scopes`、`mentioned_entities`、`mentioned_topics` は、それぞれ最大 4 件までに留める。

MVP で使う `primary_intent` の enum は次で固定する。

- `smalltalk`
- `reminisce`
- `commitment_check`
- `consult`
- `check_state`
- `preference_query`
- `fact_query`
- `meta_relationship`

`secondary_intents` も同じ enum を使う。

`time_reference` の enum は次で固定する。

- `none`
- `recent`
- `past`
- `future`
- `persistent`

役割は次のとおりである。

- どの `scope` を優先するか
- どの `section` を重くするか
- 回想、確認、相談、行動検討のどれに近いかを粗く決める
- 時間軸を決める

会話との違いは、`primary_intent` の解釈先である。

- 会話では:
  - 返答の温度、断定度、話題継続に効く
- 行動では:
  - 介入すべきか、何もしないべきか、確認を挟むべきかに効く

つまり、`RecallHint` は共通だが、後段での重み付け先が違う。

<!-- Block: CompoundIntent -->
## 複合意図は `primary_intent` 主軸で扱う

MVP では、複合意図を同時最適化しない。
`primary_intent` を 1 本の主軸とし、`secondary_intents` は軽い補助としてだけ使う。

ここで固定するのは次である。

- 想起と判断の主ルーティングは `primary_intent` で決める
- `secondary_intents` は `primary_intent` を上書きしない
- 1 判断サイクルで複数の主結果を同時に出さない

この方針により、複合入力を拾いつつも、MVP の判断契約を単純に保てる。

<!-- Block: SecondaryUse -->
## `secondary_intents` の効き先

MVP では、`secondary_intents` の効き先を次に限定してよい。

- 想起候補の rerank
- `RecallPack` の section boost
- 返答方針の補助
- 確認質問の必要性

逆に、次には使わない。

- 別の主結果を追加で出すこと
- 並列の会話出力や並列行動を選ぶこと
- `primary_intent` を置き換えること

つまり、`secondary_intents` は「補助の重み」であり、「第二の主命令」ではない。

<!-- Block: IntentConflict -->
## 意図が衝突するときの扱い

`primary_intent` と `secondary_intents` が緊張する場合でも、MVP では `primary_intent` を優先する。

このときの倒し先は次でよい。

- 会話では、確認寄りにする
- 行動では、`noop` または保留寄りにする
- 自発判断では、`future_act` に倒してよい

ここで重要なのは、衝突を解こうとして複雑な同時最適化へ進まないことである。

<!-- Block: RecallHintValidator -->
## `RecallHint` の validator

`RecallHint` の validator は、意味補正を行う層ではない。
MVP では、軽量高速 LLM が返した出力が契約を満たしているかだけを確認する。

validator が確認する項目は次とする。

- JSON として解釈できること
- 必須キーがすべて存在すること
- `primary_intent` が enum 8 種のいずれか 1 件であること
- `secondary_intents` が同じ enum を使い、重複なし、最大 2 件であること
- `secondary_intents` に `primary_intent` と同じ値を入れないこと
- `confidence` が `0.0-1.0` の数値であること
- `time_reference` が固定 enum のいずれかであること
- `focus_scopes`、`mentioned_entities`、`mentioned_topics` が配列であり、各最大 4 件であること
- 余計なトップレベルキーを持たないこと

ここでは、コード側で意味を補正しない。
たとえば、intent の置き換え、scope の推測補完、曖昧値の自動修正は行わない。

<!-- Block: RecallHintRetry -->
## `RecallHint` の再試行条件

MVP では、`RecallHint` の再試行は 1 回までに限定する。

再試行するのは、次のように validator を通過しなかったときだけである。

- JSON でない
- 必須キーが欠けている
- enum が不正である
- 型が不正である
- 配列上限を超えている
- `primary_intent` と `secondary_intents` が重複している

1 回目が失敗した場合は、同じ入力で `RecallHint` を 1 回だけ再生成する。
2 回目も validator を通らなかった場合は、その回の `RecallHint` は生成失敗として扱う。

ここで重要なのは、コード側で代替 `RecallHint` を捏造しないことである。
MVP では、`RecallHint` の判定主体を軽量高速 LLM に置き、コード側は契約確認と失敗判定だけを担う。

<!-- Block: RecallHintFailure -->
## `RecallHint` 生成失敗時の扱い

`RecallHint` が 2 回連続で validator を通らなかった場合、その判断サイクルはそこで打ち切る。
失敗後に、代替の想起や自然文フォールバックを作らない。

会話要求だった場合は、`reply / act / noop` 判断へ進まず、内部失敗として扱う。
ユーザー向けにどう返すかという wire の詳細は API 論点で定めるが、少なくとも自然な会話返答を捏造しない。

自発起床や非会話観測だった場合は、その回を内部失敗として終了し、外向きの `reply` や `act` は行わない。
実質的には「失敗により畳まれた判断サイクル」であり、通常の意味での `noop` とは分けて扱う。

ログとしては、少なくとも次を残す。

- `events`
  - `recall_hint_failure` 相当の失敗イベント
- `retrieval_runs`
  - `result_status=failed`
  - `failure_reason`

この設計により、失敗を隠さず、後から観測・検証できるようにする。

<!-- Block: RecallFlow -->
## 同期想起の基本フロー

同期想起は、次の 2 レーンで行う。

1. 構造レーン
2. 連想レーン

構造レーンでは、`scope`、`memory_type`、`status` を軸に厳密候補を取る。
連想レーンでは、現在観測をクエリ化して `memory_units` と `episode_digests` をベクトル検索し、曖昧な連想を補助する。

ここでの原則は次である。

- 骨格は構造レーンで作る
- 主ルーティングは `primary_intent` で決める
- `secondary_intents` は rerank と section boost に留める
- 連想レーンは補助に留める
- 連想レーンの index 対象は `memory_units` と `episode_digests` に限定する
- `commitment` と `relationship` は構造レーン優先にする
- `vector-only` の候補は断定根拠にしない

<!-- Block: RecallPack -->
## `RecallPack`

`RecallPack` の主 section は、会話でも行動でも共通でよい。

- `self_model`
- `user_model`
- `relationship_model`
- `active_topics`
- `active_commitments`
- `episodic_evidence`
- `conflicts`

この構成を分けない理由は、会話でも行動でも「今の人格文脈」を支える材料が同じだからである。

違いは、各 section の効き方にある。

`RecallPack` の組み立てでは、次を守る。

- 候補収集時は section ごとの仮上限を持つ
- 最終 `RecallPack` は全体 `8〜14 件` に収める
- `episodic_evidence` は短い要点に圧縮する
- `active_commitments` には `status in (inferred, confirmed)` かつ active な `commitment_state` のものだけを入れる
- `conflicts` は別枠で渡す

<!-- Block: EpisodeUse -->
## `episode_digests` と `events` の使い方

想起の標準経路は、`memory_units + episode_digests` とする。

- `memory_units`
  - 継続理解の主軸
- `episode_digests`
  - 回想と根拠の主軸

`events` は標準経路には入れない。
ただし、次のときだけ例外的に使ってよい。

- 「そのとき何を話したか」を確認したい
- 「どうするって言ったか」を確認したい
- 実行の由来を短く確かめたい

このときも、`events` 全文を広く再読しない。
先に選ばれた `episode_digests` の `linked_event_ids` や `memory_units` の `evidence_event_ids` を最大 1-3 件だけ読み、短い `event_evidence` に圧縮して使う。

<!-- Block: QuoteScope -->
## MVP で正確引用と詳細時系列を責務にしない

MVP では、想起系の標準責務に正確引用や詳細時系列の再構成を含めない。

ここで優先するのは次である。

- 何が話題だったか
- 何が決まったか
- 何が継続しているか
- いまの判断に何が効くか

そのため、通常想起では `memory_units` と `episode_digests` を主に使い、必要時だけ `events` を狭く証拠参照する。

MVP の段階では、次は標準責務に含めない。

- 発話の逐語的な正確引用
- 長い会話の詳細な時系列再現
- 広い event 群からの完全な会話再構成

これらが必要な場面では、通常想起で無理に再構成せず、確認寄りに倒すか、証拠参照を最小限に留める。

<!-- Block: EventEvidence -->
## `event_evidence` の圧縮ルール

`event_evidence` は、`events` の永続表現ではない。
細部確認が必要になったときだけ、その回の想起の中で一時的に作る短い証拠表現として扱う。

MVP では、`event_evidence` を 1 event につき 1 件だけ作る。
同一想起で扱う件数は最大 1-3 件に留める。

形式は、次のスロット型とする。

- `anchor`
  - 場所、時期、出来事名などの短いアンカー
- `topic`
  - その event で主に話題になっていたこと
- `decision_or_result`
  - その場で決まったこと、確認されたこと、実行結果
- `tone_or_note`
  - 雰囲気や補足。ただし 断定に効く主根拠にはしない

各スロットは 1 文までに留める。
空欄は省略してよい。

圧縮時の優先順は次とする。

1. `decision_or_result`
2. `topic`
3. `anchor`
4. `tone_or_note`

重要なのは、逐語再現を目指さないことである。
MVP の `event_evidence` は「何が話題で、何が決まったか」を短く返すためのものであり、長い引用や正確引用の責務は持たない。

圧縮時は、次を守る。

- 生ログの言い回しを長くそのまま再掲しない
- 断片をつなげて新しい事実を捏造しない
- 雑音になりやすい相槌や言い直しは落としてよい
- 判断に効くなら `decision_or_result` を優先し、回想に効くなら `anchor` と `topic` を残す

<!-- Block: AmbiguousReference -->
## 直近窓の外にある曖昧参照は限定対応にする

MVP では、直近窓の外にある曖昧参照を全面的には解かない。
ここでは、限定対応だけを責務に含める。

基本方針は次である。

- まず構造レーンで `active_topics`、`active_commitments`、関係、嗜好の候補を絞る
- 次に `memory_units` と `episode_digests` の通常想起で近い候補を取る
- 候補が狭まった場合だけ、関連 `event_id` を最大 1-3 件開いて短い `event_evidence` を作る
- それでも断定できない場合は、会話では確認寄りに倒し、行動では `noop` や保留寄りに倒す

つまり、MVP では広域検索や高価な再構成器で曖昧参照を全面解決しない。
通常想起で自然に拾える範囲と、その周辺の狭い証拠参照までを責務にする。

<!-- Block: InternalContext -->
## 返答前に注入する内部コンテキスト

返答モデルや判断モデルには、次の 2 つを分けて渡す。

- 直近会話
- 内部コンテキスト

内部コンテキストの最小形は次である。

- `TimeContext`
- `AffectContext`
- `RecallPack`

ここで重要なのは、生ログ全文をそのまま並べないことである。
会話の流れは直近会話で持ち、長期記憶は役割別に圧縮して渡す。

<!-- Block: SectionDiff -->
## 各 section の効き方の違い

| section | 会話での役割 | 行動での役割 |
|------|------|------|
| `self_model` | 話し方や返答姿勢の補正 | 行動スタイルや慎重さの補正 |
| `user_model` | 話題選び、説明の仕方 | 誰に何をいつするべきかの判断 |
| `relationship_model` | 距離感、踏み込み方、トーン | どこまで介入してよいかの上限 |
| `active_topics` | 会話継続の軸 | 今処理すべき対象の軸 |
| `active_commitments` | 前の続きの自然さ | 実行候補や確認候補の最優先根拠 |
| `episodic_evidence` | 思い出の根拠、自然な回想 | 実行前確認や説明責任の根拠 |
| `conflicts` | 断定を止めて確認寄りにする | 実行を抑制し、確認や noop に倒す |

ここで重要なのは、行動では `conflicts` が会話より強いブレーキになることである。

<!-- Block: CommonVsDiff -->
## 会話と行動の違いは、想起後に何を決めるか

共通本体まで終わったら、判断は次の 3 分岐になる。

- `reply`
  - 今は会話で返す
- `act`
  - 能力を使った行動を選ぶ
- `noop`
  - 今回は何もしない

つまり、差分は `RecallPack` の有無ではなく、`RecallPack` を見たあとに何を出力として決めるかにある。
ただし、MVP の自発判断では `act` を通常 API の外向き結果としてはまだ使わず、`13_自発判断と自発行動.md` で定めた内部結果 `future_act` を保留候補として扱ってよい。

<!-- Block: ReplyBranch -->
## 会話分岐

会話分岐では、次を決めればよい。

- 何を返すか
- どこまで断定するか
- 思い出をどのくらい自然に混ぜるか
- 確認質問にするかどうか

この分岐では、`relationship_model` と `conflicts` が返答の温度に強く効く。
また、`episodic_evidence` は自然な「そういえば」の材料になる。

会話では、多少曖昧でも確認寄りに落とせば破綻を避けられることがある。
この点が、行動との大きな違いである。

<!-- Block: ActionBranch -->
## 行動分岐

行動分岐では、記憶だけでは足りない。
少なくとも次を追加で見る必要がある。

- 能力マニフェスト
- 現在利用可能な能力状態
- ランタイム状態
- 世界状態
- 現在観測

行動分岐で決めるべきことは次である。

- そもそも介入してよいか
- 何の能力を使うか
- いつやるか
- 何を入力として渡すか
- 実行失敗時にどう扱うか

ここでの基本原則は次とする。

- 記憶は「なぜその行動候補が出るか」を支える
- 能力定義は「本当にできるか」を制限する
- `conflicts` や低信頼は、会話より強く抑制に使う
- 曖昧なら確認か `noop` に倒す

つまり、行動分岐は会話分岐より保守的であるべきである。

<!-- Block: ActionStateBoundary -->
## 行動判断に渡す状態の境界

MVP では、行動判断に渡す状態を次の 4 つに分ける。

- `current_observation`
  - 今回その場で入ってきた生の観測
- `world_state`
  - 次の判断にも持ち越したい外界の継続状態
- `runtime_state`
  - OtomeKairo 自身の現在の稼働条件
- `capability_state`
  - 各能力の現在の実行可能性

それぞれの境界は次のとおりである。

| 状態 | 何を入れるか | 例 |
|------|------|------|
| `current_observation` | 生の現在入力や直近取得結果 | ユーザー入力、GPS の直近値、画面の直近観測、定期起床 |
| `world_state` | 正規化済みの外界条件で、次の判断にも効くもの | `current_location`、`current_place_semantic`、`active_app`、`network_state`、`destination_context` |
| `runtime_state` | OtomeKairo 自身の現在の稼働条件 | 読み込み済み人格設定参照、読み込み済み記憶セット参照、起床スケジューラ状態、`connection_state`、`ongoing_action_exists` |
| `capability_state` | 能力ごとの可用性や制約 | `available_now`、`auth_ready`、`dependency_health`、`cooldown_until`、`last_error_summary` |

ここで重要なのは、現在地のような情報を `runtime_state` に入れないことである。
現在地は OtomeKairo 自身の状態ではなく、外界の状態なので `world_state` に属する。

また、`selected_persona_id`、`selected_memory_set_id`、`wake_policy` の正本は永続設定側にあり、`runtime_state` にはその現在反映中の参照または実体要約だけを見せればよい。

また、1 回の判断だけで消えてよい生入力は `current_observation` に残し、複数回の判断で参照したい外界条件だけを `world_state` に上げる。
この分離により、観測の生さと行動判断の安定性を両立しやすくなる。

<!-- Block: NoopBranch -->
## `noop` 分岐

`noop` も正規の判断結果として扱う。

`noop` になる主な理由は次である。

- 介入根拠が弱い
- 記憶が衝突している
- 能力が利用不可である
- 今は話す / 行動するより待つほうが自然である

会話では「何か返す」が選ばれやすいが、行動では `noop` の価値が高い。
不用意な介入を避けること自体が、人格の一貫した振る舞いになるからである。

<!-- Block: EventUse -->
## `events` の例外参照は両分岐で共有する

`events` の例外参照は、会話だけのものではない。
行動でも、次のような細部確認に使ってよい。

- 本当にその約束があったか
- その時点で何を決めたか
- 実行の由来を短く確認したいか

ただし、会話でも行動でもルールは同じである。

- `events` を通常候補として広く読まない
- 先に `memory_units` と `episode_digests` を選ぶ
- その `linked_event_ids` / `evidence_event_ids` だけを狭く開く
- 生ログ全文ではなく `event_evidence` へ圧縮する

この設計により、細部確認と人格文脈を分離しやすくなる。

<!-- Block: ActionSpecificDifference -->
## 行動で特に追加される判断軸

会話と違って、行動では次の軸が新しく入る。

| 軸 | 会話 | 行動 |
|------|------|------|
| 能力の有無 | ほぼ不要 | 必須 |
| 実行失敗 | 軽い言い換えで済むことがある | 明確な結果管理が必要 |
| 介入コスト | 低い | 高いことがある |
| 誤り時の影響 | 誤答や違和感 | 誤作動や不要介入 |
| `conflicts` の重み | 確認寄りへ調整 | 実行抑制へ直結 |

この差分があるため、行動設計では「会話の延長」ではあっても、「会話と同じ軽さ」で決めてはいけない。

<!-- Block: UpdateBack -->
## 結果をどう記憶更新へ返すか

会話でも行動でも、最終的には `events -> episode_digests -> memory_units` の更新系へ返す。
ただし、返すべき結果の意味が少し違う。

- 会話:
  - 何を言ったか
  - ユーザーがどう反応したか
  - 何が未完了として残ったか
- 行動:
  - 何をやろうとしたか
  - 実際に成功したか失敗したか
  - ユーザーが受け入れたか拒否したか
  - 次回の再試行や抑制にどう効くか

行動結果は、次の判断に与える影響が大きい。
そのため、会話よりも `result` と `status` を意識して保存する必要がある。

<!-- Block: DesignDecision -->
## ここで固定したい設計判断

この統合設計で固定したいのは次である。

- 想起までは会話と行動で共通本体にする
- 差分は `reply / act / noop` の判断以降に寄せる
- 行動分岐では、記憶に加えて能力定義と能力状態を必須にする
- `events` は通常想起の主役にしない
- 複合意図は `primary_intent` 主軸、`secondary_intents` 軽補助で扱う
- `secondary_intents` は rerank、section boost、返答方針補助、確認必要性にだけ効かせる
- 正確引用や詳細時系列の再構成は MVP の標準責務に含めない
- 直近窓の外にある曖昧参照は、通常想起と狭い証拠参照までの限定対応にする
- 行動では `conflicts` と低信頼を強いブレーキにする
- `noop` を正式な判断結果として扱う

これにより、会話だけに閉じた記憶設計を避けつつ、行動のために記憶系を二重化しないで済む。
