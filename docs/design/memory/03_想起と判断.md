# 想起と判断

## この文書で定める範囲

この文書では、次を正本として定める。

- `RecallHint` の役割と最小契約
- 同期想起の基本フロー
- `RecallPack` の構成
- `episodes` と `events` の使い分け
- `episodes` の継続列の扱い
- `events` の例外参照の使い方

## 基本前提

OtomeKairo では、会話と行動を別々の知能として扱わない。
両方とも、「入力を受け、記憶を踏まえて、何をするべきかを判断する」という同じ中心ループの分岐として扱う。

この前提により、次が成立する。

- 会話だけが現在の個の表現ではない
- 行動も、人格設定と記憶から組み上がる現在の個の文脈の延長で決まる
- 会話用の記憶と行動用の記憶を別系統に分けずに済む

ただし、同じなのは判断の土台までである。
判断後に求められる出力契約は、会話と行動で異なる。

## 共通本体

会話でも行動でも、共通本体は次の流れにする。

1. 現在入力を受け取る
2. `RecallHint` を作る
3. 構造レーンと連想レーンで必要な記憶を集める
4. `RecallPack` を作る
5. 判断系が結果種別を選ぶ
6. 分岐先ごとの処理を実行する
7. 結果を `events` に保存し、記憶更新へ返す

ここで重要なのは、会話と行動で最初から別の想起系を作らないことである。
同じ想起系で現在の個の文脈を作り、その後で分岐する。

## 共通本体で扱う入力

共通本体が受けるものは、厳密には「ユーザー入力」ではなく「現在入力」である。

主に入る入力は次である。

- 最新の対話入力
- 直近会話
- 定期起床や外界変化の事実
- 軽量なクライアント文脈
- 現在時刻

会話時は `最新入力` が主入力になり、自発判断時は `定期起床 + 軽量文脈` が主入力になる。

## `RecallHint`

`RecallHint` は軽量高速 LLM が、現在入力と直近文脈だけを見て生成する入口情報である。

入力は次に限定する。

- 現在入力
- 直近 3 分以内
- 最大 4-6 発話
- 生活文脈向けに整形した現在時刻

この段階では長期記憶を読まない。
現在時刻は `2026年4月25日 土曜日 9時00分（日本時間）` のような自然文で渡す。
LLM には UTC timestamp や offset 付き ISO timestamp を主要表現として渡さない。

`RecallHint` の最小契約は次とする。

- `interaction_mode`
- `primary_recall_focus`
- `secondary_recall_focuses`
- `confidence`
- `time_reference`
- `focus_scopes`
- `mentioned_entities`
- `mentioned_topics`
- `risk_flags`

`interaction_mode` は必ず 1 つにする。
`primary_recall_focus` は必ず 1 つにする。
`secondary_recall_focuses` は補助扱いとし、最大 2 件までに留める。
`focus_scopes`、`mentioned_entities`、`mentioned_topics` は、それぞれ最大 4 件までに留める。
`risk_flags` は主軸の危うさを表す補助扱いとし、最大 3 件までに留める。

`focus_scopes` の対象は次に留める。

- `self`
- `user`
- `relationship:<key>`
- `topic:<key>`

第三者名や固有名は `focus_scopes` へ広げず、`mentioned_entities` で扱う。
`mentioned_entities` は `person:<normalized_name>`、`place:<normalized_name>`、`tool:<normalized_name>` の正規化済み参照を返す。
`mentioned_topics` は `topic:<normalized_name>` の正規化済み参照を返す。
`world` は `focus_scopes` へ入れない。
特定対象に閉じない世界理解は、`primary_recall_focus=fact` または `primary_recall_focus=state` で、現在入力が世界条件を主題にしている場合だけ構造レーンで扱う。

`interaction_mode` の enum は次で固定する。

- `conversation`
- `action`
- `observation`
- `autonomous`

`primary_recall_focus` の enum は次で固定する。

- `self`
- `user`
- `relationship`
- `commitment`
- `topic`
- `preference`
- `fact`
- `state`
- `episodic`

`secondary_recall_focuses` も同じ enum を使う。

`time_reference` の enum は次で固定する。

- `none`
- `recent`
- `past`
- `future`
- `persistent`

`risk_flags` の enum は次で固定する。

- `mixed_intent`
- `ambiguous_reference`
- `weak_memory_cue`
- `time_ambiguous`

役割は次のとおりである。

- 会話、行動、観測、自律再評価のどの入口として扱うかを粗く決める
- どの `scope` を優先するか
- どの `RecallPack` section を重くするか
- どの記憶焦点を主軸にするかを決める
- 時間軸を決める
- 主軸だけでは危うい点を `risk_flags` で後段へ渡す

ここで重要なのは、複合意図や曖昧参照をこの段階で解き切ったことにしないことである。
主軸は `primary_recall_focus` で持ち、危うさは `risk_flags` で持つ。

## 複合焦点の扱い

複合焦点を同時最適化しない。
`primary_recall_focus` を 1 本の主軸とし、`secondary_recall_focuses` は軽い補助としてだけ使う。

固定するのは次である。

- 想起と判断の主ルーティングは `interaction_mode` と `primary_recall_focus` の組で決める
- `secondary_recall_focuses` は `primary_recall_focus` を上書きしない
- `risk_flags` は主ルーティングを上書きしない
- 1 判断サイクルで複数の主結果を同時に出さない

`secondary_recall_focuses` と `risk_flags` の効き先は次に限定する。

- 想起候補の rerank
- `RecallPack` の section boost
- `conflicts` の確認補助
- 返答方針や確認質問の補助

## `RecallHint` の validator

`RecallHint` の validator は、意味補正を行う層ではない。
軽量高速 LLM が返した出力が契約を満たしているかだけを確認する。

validator が確認する項目は次とする。

- JSON として解釈できること
- 必須キーがすべて存在すること
- `interaction_mode` が固定 enum のいずれか 1 件であること
- `primary_recall_focus` が固定 enum のいずれか 1 件であること
- `secondary_recall_focuses` が同じ enum を使い、重複なし、最大 2 件であること
- `secondary_recall_focuses` に `primary_recall_focus` と同じ値を入れないこと
- `confidence` が `0.0-1.0` の数値であること
- `time_reference` が固定 enum のいずれかであること
- `risk_flags` が固定 enum の配列であり、重複なし、最大 3 件であること
- `focus_scopes`、`mentioned_entities`、`mentioned_topics` が配列であり、各最大 4 件であること
- `focus_scopes` が `self`、`user`、`relationship:<key>`、`topic:<key>` のいずれかであること
- `mentioned_entities` が `person:<normalized_name>`、`place:<normalized_name>`、`tool:<normalized_name>` のいずれかであること
- `mentioned_topics` が `topic:<normalized_name>` であること
- 余計なトップレベルキーを持たないこと

ここでは、コード側で意味を補正しない。
mode や focus の置き換え、scope の推測補完、曖昧値の自動修正は行わない。

一方で、想起全体の設計方針としては、意味的な候補選別や要約表現では LLM を第一候補とする。
ただし、件数上限、契約検証、状態遷移、失敗監査はコード側へ残す。
repo 全体の原則は [../20_LLM判断優先方針.md](../20_LLM判断優先方針.md) を正とする。

## `RecallHint` の再試行と失敗時の扱い

`RecallHint` の再試行は 1 回までに限定する。
再試行するのは validator を通過しなかったときだけである。

2 回連続で validator を通らなかった場合、その判断サイクルはそこで打ち切る。
失敗後に、代替の想起や自然文フォールバックを作らない。

ログとしては、少なくとも次を残す。

- `events`
  - `recall_hint_failure` 相当の失敗イベント
- `retrieval_runs`
  - `result_status=failed`
  - `failure_reason`

## 同期想起の基本フロー

同期想起は、次の 2 レーンで行う。

1. 構造レーン
2. 連想レーン

構造レーンでは、`scope`、`memory_type`、`status` を軸に厳密候補を取る。
連想レーンでは、現在入力をクエリ化して `memory_units` と `episodes` をベクトル検索し、曖昧な連想を補助する。

ここでの原則は次である。

- 骨格は構造レーンで作る
- 主ルーティングは `interaction_mode` と `primary_recall_focus` で決める
- `mentioned_entities` は `scope_type=entity` の構造候補取得に使う
- `mentioned_topics` は `scope_type=topic` の構造候補取得に使う
- `world` 候補は `scope_type=world` かつ `scope_key=world` に限定し、広域の世界知識探索には使わない
- `secondary_recall_focuses` は rerank と section boost に留める
- `risk_flags` は断定しすぎないための補助信号としてだけ使う
- 連想レーンは補助に留める
- 連想レーンの index 対象は `memory_units` と `episodes` に限定する
- `commitment` と `relationship` は構造レーン優先にする
- `vector-only` の候補は断定根拠にしない

## `RecallPack`

`RecallPack` の主 section は、会話でも行動でも共通とする。

- `self_model`
- `user_model`
- `relationship_model`
- `active_topics`
- `active_commitments`
- `episodic_evidence`
- `conflicts`

この構成を分けない理由は、会話でも行動でも「その回の現在の個の文脈」を支える材料が同じだからである。

`RecallPack` の組み立てでは、次を守る。

- 候補収集時は section ごとの仮上限を持つ
- 最終 `RecallPack` は全体 `8〜14 件` に収める
- `episodic_evidence` は短い要点に圧縮する
- `active_commitments` には `status in (inferred, confirmed)` かつ active な `commitment_state` のものだけを入れる
- `conflicts` は別枠で渡す

`RecallPack` 候補の意味的選別、section 配置、`conflicts` 文面を LLM で行うときの
role、入力契約、failure の扱いは
[09_RecallPackのLLM選別.md](09_RecallPackのLLM選別.md) を正とする。

`memory_type=summary` は、独立 section を作らず既存 section に混ぜる。
対応は次で固定する。

- `scope_type=self`
  - `self_model`
- `scope_type=user`
  - `user_model`
- `scope_type=relationship`
  - `relationship_model`
- `scope_type=topic`
  - `active_topics`

`scope_type=entity` と `scope_type=world` も独立 section を作らない。
対応は次で固定する。

- `scope_type=entity`
  - その対象自体の継続理解は `active_topics`
  - その対象を含む出来事の根拠は `episodic_evidence`
  - `self` や `user` との関係理解は `relationship_model`
- `scope_type=world`
  - 現在入力の事実確認や状態判断に効くものだけ `active_topics`
  - 出来事根拠が主であるものは `episodic_evidence`

## `episodes` と `events` の使い方

想起の標準経路は、`memory_units + episodes` とする。

- `memory_units`
  - 継続理解の主軸
- `episodes`
  - 経験のまとまりと回想の主軸

`events` は標準経路には入れない。
ただし、次のときだけ例外的に使う。

- そのとき何を話したかを確認したい
- その場で何を決めたかを確認したい
- 実行の由来を短く確かめたい

このときも、`events` 全文を広く再読しない。
先に選ばれた `episodes` の `linked_event_ids` や `memory_units` の `evidence_event_ids` を最大 1-3 件だけ読み、短い `event_evidence` に圧縮して使う。

## 正確引用と詳細時系列を標準責務にしない

想起系の標準責務に、正確引用や詳細時系列の再構成を含めない。

ここで優先するのは次である。

- 何が話題だったか
- 何が決まったか
- 何が継続しているか
- いまの判断に何が効くか

そのため、通常想起では `memory_units` と `episodes` を主に使い、必要時だけ `events` を狭く証拠参照する。

## `episodes` の継続列の扱い

通常想起では、まず `episode` 単体を選ぶ。
継続性が必要な場合だけ、選ばれた `episodes` の `episode_series_id` を使って近接する `episodes` を追加で見る。

継続列をたどるのは、次のような場合に限る。

- `open_loops` の続きや確認待ちが現在の判断に効く
- 約束確認や進行中タスクの途中経過を見たい
- 後から返った実行結果の由来を短く確かめたい

単に同じ話題が続いているだけなら、継続列を標準経路にしない。

継続列をたどる場合も、同じ `episode_series_id` を持つ `episodes` 全件を広く並べない。
現在の判断に近いものを最大 1-2 件だけ追加し、通常の `RecallPack` に混ぜて使う。

## `event_evidence` の圧縮ルール

`event_evidence` は、`events` の永続表現ではない。
細部確認が必要になったときだけ、その回の想起の中で一時的に作る短い証拠表現として扱う。

形式は、次のスロット型とする。

- `anchor`
  - 場所、時期、出来事名などの短いアンカー
- `topic`
  - その `event` で主に話題になっていたこと
- `decision_or_result`
  - その場で決まったこと、確認されたこと、実行結果
- `tone_or_note`
  - 雰囲気や補足。ただし断定に効く主根拠にはしない

各スロットは 1 文までに留める。
空欄は省略する。

圧縮時は、次を守る。

- 生ログの言い回しを長くそのまま再掲しない
- 断片をつなげて新しい事実を捏造しない
- 雑音になりやすい相槌や言い直しは落とす
- 判断に効くなら `decision_or_result` を優先し、回想に効くなら `anchor` と `topic` を残す

`event_evidence` の圧縮を LLM で行うときの role、入力契約、failure の閉じ込め方は
[08_event_evidenceのLLM圧縮.md](08_event_evidenceのLLM圧縮.md) を正とする。

## 直近窓の外にある曖昧参照は限定対応にする

直近窓の外にある曖昧参照を全面的には解かない。
ここでは、限定対応だけを責務に含める。

基本方針は次である。

- まず構造レーンで `active_topics`、`active_commitments`、関係、嗜好の候補を絞る
- 次に `memory_units` と `episodes` の通常想起で近い候補を取る
- 候補が狭まった場合だけ、関連 `event_id` を最大 1-3 件開いて短い `event_evidence` を作る
- それでも断定できない場合は、確認寄りに倒す

## 返答前に注入する内部コンテキスト

返答モデルや判断モデルには、次の 2 つを分けて渡す。

- 直近会話
- 内部コンテキスト

内部コンテキストの最小形は次である。

- `TimeContext`
- `AffectContext`
- `RecallPack`

ここで重要なのは、生ログ全文をそのまま並べないことである。
会話の流れは直近会話で持ち、長期記憶は役割別に圧縮して渡す。

## `AffectContext`

`AffectContext` は、`mood_state`、関連 `affect_state`、必要時だけ直近 `episode_affect` を判断用に圧縮した内部コンテキストである。

少なくとも次を守る。

- 全 `episode_affect` と全 `affect_state` をそのまま渡さない
- `mood_state` は常に含める
- `mood_state` は数値状態だけを持ち、自然文は持たない
- `affect_state` は現在入力と `RecallHint` に関係する `target_scope` だけを見る
- 直近 `episode_affect` は原因追跡や当時感情の参照が必要な場合だけ少数を渡す
- `self` の機嫌と主要な `relationship` を優先する

ここで重要なのは、感情を単独で判断させないことである。
`AffectContext` は `RecallPack` を補う前景情報であり、記憶や現在入力の代替ではない。

## 想起後の判断との境界

想起が作るのは現在の個の文脈までである。
その文脈を見て何を返すか、行動するか、保留するか、何もしないかを決める責務は `05_判断と行動.md` に置く。

この分離により、想起系は「何を思い出すか」に集中し、行動系は「何を選ぶか」に集中できる。
