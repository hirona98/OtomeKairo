# 記憶設計方針

## なぜ記憶を重く扱うか

OtomeKairo では、人格設定と記憶を操作上は独立に扱う。
ただし、実際の応答や自発判断は人格設定だけで決まらない。

現行設計では、次を前提にする。

- 人格設定は基本姿勢を与える
- 記憶は自己像、他者理解、関係、未完了、感情傾向を育てる
- 応答や自発判断は、その時点の人格設定と記憶から構成される現在の個で決まる

そのため、記憶は検索補助ではなく、現在の個を構成する基盤として設計する。

## 記憶 docs の読む順

### まず全体像を掴むとき

1. `01_記憶システム概要.md`
2. `03_想起と判断.md`
3. `04_記憶更新と再整理.md`

### データ構造を詰めるとき

1. `01_記憶システム概要.md`
2. `02_記憶データモデル.md`
3. `04_記憶更新と再整理.md`

### 想起品質を詰めるとき

1. `03_想起と判断.md`
2. `05_想起と判断フロー図.md`
3. `06_想起と判断ユースケース.md`

## 各文書の責務

- `01_記憶システム概要.md`
  - 記憶システム全体の責務と境界
- `02_記憶データモデル.md`
  - `events / episode_digests / memory_units / affect_state` の構造
- `03_想起と判断.md`
  - `RecallHint`、構造レーン、連想レーン、`RecallPack`
- `04_記憶更新と再整理.md`
  - `turn consolidation` と `reflective consolidation`
- `05_想起と判断フロー図.md`
  - 現行設計の図解
- `06_想起と判断ユースケース.md`
  - 想起設計の補助検証

## 正本と補助文書

正本は次である。

- `01_記憶システム概要.md`
- `02_記憶データモデル.md`
- `03_想起と判断.md`
- `04_記憶更新と再整理.md`

補助文書は次である。

- `05_想起と判断フロー図.md`
- `06_想起と判断ユースケース.md`

補助文書は理解を助けるためのものであり、仕様変更の正本にはしない。

## 意図的に採らないもの

現時点では、次は採らない。

- ベクトル検索だけで記憶を成立させること
- なんでも 1 本の自由文 summary に潰すこと
- 生ログを捨てて派生記憶だけ残すこと
- 1 ターンごとに自己像全体を書き直すこと
- 最初から大規模知識グラフを前提にすること
