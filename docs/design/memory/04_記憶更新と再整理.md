# 記憶更新と再整理

## 位置づけ

この文書は `turn consolidation`、`vector_index_sync`、`reflective consolidation` を含む記憶更新と再整理の意味境界を定める。
実装状態は `src/` と smoke 結果を正とする。

## 更新の基本原則

OtomeKairo の記憶更新では、次を原則にする。

- 通常更新では生ログを消さない
- 経験のまとまりは `episodes` に束ねる
- 継続理解は `memory_units` に落とす
- 推測と確定を分ける
- 矛盾したら即削除ではなく、状態遷移で扱う
- 真偽に関わる強さは `confidence` で持つ
- 現在どれだけ前に出すかは `salience` で持つ
- 更新の根拠は必ず `events` に辿れるようにする

特に重要なのは、`confidence` と `salience` を混ぜないことである。

- `confidence`
  - それがどれだけ正しそうか
- `salience`
  - いまどれだけ思い出しやすく、前面に出すべきか

## `turn consolidation` の流れ

1 ターンごとの更新は、次の順で進める。

1. `events` を保存する
2. LLM が `events` と compact な cycle context を根拠に、`episode`、記憶候補メモ、`episode_affects` を 1 個の `memory_interpretation` として返す
3. コードが `episode` を正本構造へ変換する
4. コードが記憶候補メモを半構造化した `memory_units` 候補へ正規化する
5. 既存 `memory_units` の比較候補を探す
6. 各候補について更新操作を決める
7. `confidence` と `salience` を更新する
8. `episode`、`memory_units` への操作、`episode_affect` を同じ turn consolidation として保存する
9. `memory_links` を最小限だけ張る
10. `self` 由来の感情から `mood_state` を更新する
11. `memory_units` に変化があった場合だけ `revisions` を残す
12. `vector_index_entries` の差分更新を background job に要求する

この段階では「そのターンで確信できる範囲」に留める。
対象指向の `affect_state` の育成のような複数ターンをまたぐ整理は `reflective consolidation` に回す。
`vector_index_entries` 更新要求と `reflective consolidation` は background job として直列実行し、通常の会話応答を待たせない。

## `memory_interpretation` の出力契約

`turn consolidation` の入口で使う `memory_interpretation` は、厳密な構造化出力として扱う。
LLM への入力は JSON payload とし、`input_text / decision / reply_text / memory_context` を記憶化対象データとして渡す。
`memory_context` には `trigger_kind`、入力イベント種別、保留意図要約、観測要約、capability request 要約、ongoing action 遷移要約、compact events を含める。
compact events は補助文脈として扱い、件数と総文字数を上限内へ切り詰める。
`memory_context` には raw payload、画像本文、資格情報、内部 URL、配送先 client を含めない。
画像由来の記憶更新では raw image ではなく `visual_observation_record.detailed_summary_text` と `visual_observation_id` を根拠に使う。
1 日 1 回の background 整理では、視覚記録を保持、圧縮、検索除外に分け、反復した活動、環境、作業対象を記憶候補へ育てる。
現行実装では `daily_visual_digest` に記憶候補要約を保存し、2 日以上にまたがる類似候補だけを制限付きで `memory_unit` へ昇格する。

## `daily_visual_digest` からの長期記憶昇格

`daily_visual_digest.memory_candidate_summaries` は記憶候補であり、そのまま長期記憶ではない。
昇格処理は background worker で行い、通常応答を待たせない。

昇格対象は次をすべて満たすものに限る。

- `daily_visual_digest.result_status=succeeded`
- 同一または類似する `summary_text` が 2 日以上にまたがる
- `duplicate_group_id` または代表 `visual_observation_id` を根拠として参照できる
- `retention_status=compressed` の中間記録だけを根拠にしていない
- 秘密情報、画面上の一時的な文言、認証情報、個人情報を含む可能性が低い

昇格後の `memory_unit` は `status=inferred` とする。
初期 `confidence` は最大 0.55、初期 `salience` は最大 0.45 に制限する。
会話添付画像またはユーザーが後で確認した視覚記録が根拠に含まれる場合だけ、`confidence` を最大 0.65 まで上げる。

昇格先 scope は次を使う。

- 個人の習慣や作業傾向は `scope_type=user`
- 場所、部屋、机、画面、アプリ、作業対象の傾向は `scope_type=topic`
- OtomeKairo 自身の観測傾向や自律動作に関わるものは `scope_type=self`

昇格処理は同じ意味の `memory_unit` が既にある場合、新規作成ではなく revision 候補にする。
既存記憶と矛盾する場合は `conflict` 候補を残し、断定的に上書きしない。
`daily_visual_digest` 由来の記憶は、根拠として `daily_visual_digest.digest_id` と代表 `visual_observation_id` を payload に残す。
raw image、OCR 全文、画面全文は根拠に含めない。

昇格件数は 1 日あたり最大 3 件とする。
昇格 failure は該当候補だけを落とし、digest 全体や会話処理を failure にしない。

少なくとも次を守る。

- 返すトップレベルキーは `episode / candidate_memory_units / episode_affects` の 3 つだけにする
- `episode` は `episode_type / episode_series_id / primary_scope_type / primary_scope_key / summary_text / outcome_text / open_loops / salience` だけを持つ
- `candidate_memory_units` の各要素は、記憶候補メモとして必要な固定キーだけを持つ
- `episode_affects` の各要素も固定キーだけを持つ
- `episode.primary_scope_type` と `episode_affects[].target_scope_type` は [02_記憶データモデル.md](./02_記憶データモデル.md) で定義した `self / user / entity / topic / relationship / world` だけを使う
- 同じ `target_scope_type / target_scope_key / affect_label` を重複して返さない
- 1 `episode` で返す `episode_affects` は最大 4 件に制限する
- `episode` と `episode_affects` の `scope_key` は正規化済みの形だけを使う
- 余計なキー、説明文、Markdown、コードフェンスは入れない

`candidate_memory_units` はデータベース行候補ではない。
LLM は「何を覚える候補か」を自然文中心で出す。
コードはそれを `memory_units` の正本構造へ正規化する。

`candidate_memory_units` の各要素は次の key だけを持つ。

| 項目 | 役割 |
|------|------|
| `memory_type` | 候補の種類。値は [02_記憶データモデル.md](./02_記憶データモデル.md) の `memory_type` を使う |
| `scope` | 大まかな対象。`self / user / entity / topic / relationship / world` のいずれか |
| `subject_hint` | 主語候補。`self / user / person:<key> / place:<key> / tool:<key> / topic:<key>` または自然文の短い候補 |
| `predicate_hint` | 述語候補。短い snake_case または自然文の短い候補 |
| `object_hint` | 目的語または値の候補。参照文字列または自然文の短い候補 |
| `qualifiers_hint` | 条件、時期、頻度、極性、参加者などの補助候補 object |
| `summary_text` | 覚える候補の短い自然文 |
| `evidence_text` | なぜそう言えるかの短い自然文 |
| `confidence_hint` | `low / medium / high` のいずれか |

例:

```json
{
  "memory_type": "preference",
  "scope": "user",
  "subject_hint": "user",
  "predicate_hint": "likes",
  "object_hint": "food:spicy",
  "qualifiers_hint": {
    "polarity": "positive"
  },
  "summary_text": "ユーザーは辛い食べ物を好む。",
  "evidence_text": "ユーザーが辛い料理が好きだと話した。",
  "confidence_hint": "high"
}
```

LLM は `memory_unit_id`、`scope_type`、`scope_key`、`subject_ref`、`predicate`、`object_ref_or_value`、`status`、`commitment_state`、`confidence`、`salience`、`operation`、`evidence_event_ids`、`formed_at`、`updated_at` を返さない。
`subject_hint / predicate_hint / object_hint / qualifiers_hint` は正本値ではなく、コードが正規化するための意味ヒントである。
正本の `scope_type / scope_key / subject_ref / predicate / object_ref_or_value / qualifiers` はコードが validator と正規化規則を通して決める。

`episode_affects` の各要素は少なくとも `target_scope_type / target_scope_key / affect_label / vad / intensity / confidence / summary_text` を持つ。
`episode_affects[].vad` は `{v, a, d}` の object に固定する。
`episode_affects[].intensity / confidence` は `0.0` 以上 `1.0` 以下の数値に固定し、文字列や hint 値は使わない。
`mood_state` や `affect_state` 自体は LLM から直接返さない。
`mood_state` の deterministic 更新式は [../16_感情モデル.md](../16_感情モデル.md) を正とする。

validator 失敗時は同じ入力で 1 回だけ再生成する。
それでも契約を満たさない場合は、そのターンの `turn consolidation` を失敗として監査に残す。

## 非会話サイクルの記憶化

`wake`、background wake、capability result も判断サイクルである。
ただし、すべての非会話サイクルを長期記憶へ入れると、現在の個の輪郭がノイズ化する。

非会話サイクルでは、次のいずれかに当てはまる場合だけ `turn consolidation` の対象にする。

- 人や外部接点へ `reply` または通知を返した
- 新しい保留意図を作成または既存の保留意図を更新した
- capability の成功、失敗、timeout が以後の判断に効く
- 観測結果が `world_state`、`drive_state`、`ongoing_action`、記憶のいずれかを変える
- 自律判断の見送り理由が、次回以降の過剰介入抑制や再評価に効く

上の条件に当てはまらない純粋な `noop` は、`cycle_trace` と必要な `events` だけを残し、`episode` と `memory_units` を作らない。
非会話サイクルで `turn consolidation` を行う場合も、`1判断サイクル = 1episode` の原則を守る。

## `episode` の作り方

`turn consolidation` では、1 ターンにつき 1 件の `episode` を作る。

- 長い行動や継続中の確認でも、同じ継続を 1 件の `episode` にまとめ続けない
- そのターンで起きたことは、そのターンの新しい `episode` に残す
- 続き物である場合だけ、既存の `episode_series_id` を引き継ぐ

`episode_series_id` を引き継ぐのは、少なくとも次の場合に限る。

- 既存の `open_loops` に対する続きや確認
- 約束確認や未完了タスクの進行
- 実行中タスクの途中経過
- 過去ターンで開始した行動に対する完了結果や失敗結果の返却

単に同じ話題が続いているだけなら、新しい `episode` を独立に作る。

## 候補 `memory_units` を作る条件

毎ターンなんでも `memory_units` にしてはいけない。
少なくとも次のどれかに当てはまるものだけを候補にする。

- 今後の会話や判断に効きそうな継続理解
- 関係や約束の変化
- 自己像や他者像に効く解釈
- 高い感情強度を持つ出来事から生じた理解
- 明示的な好み、苦手、事実、否定、訂正

逆に、次のようなものは `events` や `episodes` に留める。

- その場限りの相槌
- 冗談の断片
- 後の会話や判断に効く理解へ育っていない軽い雑談
- 真偽も継続性も低い単発推測

## 記憶候補メモの正規化

LLM が返す `candidate_memory_units` は、半構造化済みの `memory_units` ではない。
コードは候補メモを読み、次を決めてから比較処理へ渡す。

- `scope_type / scope_key`
- `subject_ref / predicate / object_ref_or_value`
- `qualifiers`
- `commitment_state`
- `valid_from / valid_to`
- 初期 `confidence / salience`
- 根拠となる `evidence_event_ids`

`confidence_hint` は数値 `confidence` ではない。
コードは出来事の種類、明示性、既存記憶との関係を見て `confidence` と `salience` を決める。
`commitment` が AI 側の一時的な支援姿勢から来る場合、コードは `qualifiers.scope_duration=session` と `valid_to` を持つ弱い候補へ正規化する。
期限切れの `memory_units` は active recall と `drive_state` 再構築の候補にしない。

## 比較キー

正規化後の候補 `memory_units` を既存のものと比べるときは、次の比較キーで比べる。

- `memory_type`
- `scope_type`
- `scope_key`
- `subject_ref`
- `predicate`

必要なら補助キーとして次も見る。

- `object_ref_or_value`
- `qualifiers`
- `summary_text`

つまり、「何についてのどんな理解か」が同じなら比較候補とみなし、「中身が同じか違うか」を比較処理で判定する。
比較候補の入口は compare key の厳密一致を正とする。
一方で、`object_ref_or_value`、`qualifiers`、`summary_text` に自由文や表現差が残るため、
compare key が同じ候補に対する中身比較では埋め込み類似度を補助的に使う。

この補助判定では、少なくとも次を守る。

- compare key 自体を埋め込み類似度で置き換えない
- 埋め込み類似度だけで新しい比較候補を無制限に増やさない
- `vector-only` や embedding-only で `reinforce / refine / supersede / revoke` を確定しない
- 明示訂正、`commitment_state`、`qualifiers` の deterministic な境界はコードで守る
- semantic 補助の `refine` は、非明示の `interpretation` で片側だけ `object_ref_or_value` が未確定な場合に限定する
- コード側で `allow_parallel`、`negates_previous`、`replace_prior` を付けた候補には semantic 補助を使わない

要するに、比較候補の集合は構造で決め、
自由文項目の「同軸だが表現が違う」を埋め込み類似度で補助判定する。

## 比較結果の最小分類

比較結果は、少なくとも次の 5 つに分ける。

| 分類 | 意味 |
|------|------|
| `same` | 同じ理解の再確認 |
| `refine` | 同じ理解だが、精度や条件が増えた |
| `replace` | 同軸の新しい理解が旧理解を置き換える |
| `conflict` | 旧理解を否定する |
| `parallel` | 似ているが別理解として並存する |

この分類により、更新操作を安定させる。

## 採用する更新操作

記憶更新処理では、更新判定の結果として次の操作だけを持つ。

| 操作 | 内容 |
|------|------|
| `create` | 新しい `memory_units` を作る |
| `reinforce` | 既存理解を再確認し、`confidence` や `last_confirmed_at` を上げる |
| `refine` | 既存理解に条件や補足を加える |
| `supersede` | 旧理解を `superseded` にし、新理解を有効な理解にする |
| `revoke` | 旧理解を `revoked` にする |
| `dormant` | 保持はするが前面から下げる |
| `correct` | 直近の誤更新を消さず、訂正発話を根拠に現在の正本だけを補正する |
| `noop` | `events` と `episodes` は残すが、長期理解は更新しない |

ここで重要なのは、`noop` を `revisions.operation` に入れないことである。
`revisions` は `memory_units` の変化履歴だけを残す監査層として扱う。

これらの操作は、設定 API や外部 API から直接指定しない。
利用者からの訂正、否定、撤回要求は通常入力や出来事として受け取り、記憶更新が既存理解との関係を判断して操作へ落とす。
OtomeKairo の記憶を、利用者が行単位で編集するデータベースとして扱わない。

## 操作決定の基本ルール

### `create`

次のときに使う。

- 比較候補が見つからない
- 比較候補はあるが `parallel` と判断した
- 重要な新規約束、関係変化、強い出来事由来の理解

### `reinforce`

次のときに使う。

- 既存理解と内容がほぼ同じ
- 今回の出来事が、その理解の再確認になっている

### `refine`

次のときに使う。

- 理解の骨格は同じ
- ただし条件、頻度、程度、文脈が追加された

### `supersede`

次のときに使う。

- 同じ比較キーに対して、新しい理解が有効な理解を置き換える
- 並存させるより、有効な理解を入れ替えたほうが自然

### `revoke`

次のときに使う。

- ユーザーや当人が明確に否定した
- 旧理解が誤りだったと明言された

### `dormant`

次のときに使う。

- 間違いとは言えない
- ただし今は前面に出し続けるとノイズになる

### `correct`

次のときに使う。

- 利用者の訂正により、直近の `create`、`reinforce`、`refine`、`supersede`、`revoke`、`dormant` が誤更新だったと分かった
- 誤更新そのものは削除せず、補正元 revision と訂正発話 event を監査可能に残す

`correct` は DB 的な巻き戻しではない。
誤更新の `revision`、その根拠 `event`、訂正発話の `event` はすべて残す。
`correct` revision は `corrects_revision_id`、`correction_group_id`、`correction_basis_event_ids` を持ち、どの誤更新をどの訂正発話で補正したかを示す。

新規作成された誤記憶は、`correct` revision により同一 `memory_unit` の現在 status を `revoked` にする。
既存記憶の誤更新は、対象 revision の `before_snapshot` をベースに現在の `memory_unit` を補正する。
誤った `supersede` は correction group として扱い、置換元を補正し、置換先として作られた related unit を `revoked` にする。

## 訂正 reconciliation

利用者の発話が直近の誤解を訂正している場合、`turn consolidation` は通常の記憶候補生成に加えて訂正 reconciliation を行う。
訂正 reconciliation は、利用者が個別記憶行を直接編集する操作ではない。
訂正発話を通常入力として扱い、OtomeKairo の記憶更新処理が対象候補を作り、後段 worker の LLM 判定で `correct` revision へ落とす。
同期の `turn consolidation` では文字列一致や語彙重なりで訂正意図を判定しない。

訂正 reconciliation は次の順で進める。

1. 同期の `turn consolidation` は直近サイクルで変更された `revisions` を候補化する
2. 候補は `revision_id`、`memory_unit_id`、要約、現在 status、直近 operation、`before_snapshot` / `after_snapshot` の要約、根拠 cycle を compact にする
3. 候補と現在入力を `memory_postprocess_jobs` に積む
4. 後段 worker が `memory_correction_reconciliation` role で訂正意図と対象 revision を判定する
5. `last_operation=create` の誤記憶は `correct` で `revoked` にする
6. `last_operation` が `reinforce / refine / revoke / dormant` の誤更新は `correct` で `before_snapshot` ベースへ補正する
7. `last_operation=supersede` の誤置換は `correction_group_id` で束ねた `correct` 群にする
8. 実行結果を `memory_trace.correction_reconciliation` に残す

後段 worker の LLM 判定で対象が特定できない場合は、訂正 episode と events を残し、memory_unit の変更は行わない。
訂正対象を広く推測して一括変更しない。

## 並存を許すケース

OtomeKairo では、次のような理解は並存しうる。

- 関係の複数面
- 好みの条件差
- 話題ごとの感情傾向
- 期間差のある事実

違いが `object_ref_or_value` や `qualifiers` に自然に乗るなら、無理に 1 件へ潰さない。

## `confirmed` へ上げる条件

`confirmed` は、会話や判断に強く使う理解である。
そのため、`inferred` から上げる条件は厳しめにする。

次のどれかを満たしたときに `confirmed` へ上げる。

- 対象本人が明示的に述べた
- 対象本人が以前の推測を肯定した
- 独立した複数イベントで同じ理解が繰り返し支持された
- 行動や結果で強く裏づけられた
- `reflective consolidation` で十分な根拠があると再評価された

一方で、次のものは `confirmed` に上げない。

- 雰囲気だけの推測
- 単発の冗談
- 相槌からの過剰推定
- 他人づての曖昧な情報

## 格下げの条件

- `confirmed -> superseded`
  - 以前は正しかったが、今は更新された
- `confirmed / inferred -> revoked`
  - 明確な否定があった
- `confirmed / inferred -> dormant`
  - しばらく参照価値が低い

`dormant` は削除ではなく、「今は積極想起しない」状態である。

## `memory_type` ごとの更新注意

### `fact`

- 明示訂正が来たら `supersede` または `revoke` を優先する
- 期間差があるなら `parallel` ではなく `valid_from` / `valid_to` を伴う更新を優先する

### `preference`

- 推測だけなら `inferred`
- 本人の明示や反復で `confirmed`
- 反対極性の明示が来たら旧理解は `revoked` または `superseded`

### `relation`

- 1 イベントで決め打ちしすぎない
- 単発の摩擦や親密さより、複数イベントの傾向を重く見る
- `reflective consolidation` での再評価を前提にする

### `commitment`

- `status` ではなく `commitment_state` で履行状態を持つ
- 未完了のものは `salience` を高く保つ
- 完了、取消、保留解除が来たら `commitment_state` を優先的に更新する
- 同じ約束の進行更新はまず `refine` で扱い、別の約束へ置き換わるときだけ `supersede` を使う

### `interpretation`

- `confidence` を上げすぎない
- 明示根拠がない限り `confirmed` にしない
- `topic` や `relationship` に強く効くが、誤差も多いので `dormant` 化しやすくする

## `confidence` の更新原則

`confidence` は、最近性ではなく根拠の強さで動かす。
厳密な数式ではなく帯域ルールを使う。

| 帯域 | 目安 | 典型例 |
|------|------|------|
| 0.90 - 1.00 | 非常に強い | 対象本人の明示、強い再確認 |
| 0.75 - 0.89 | 強い | 複数回の一貫支持、結果でも裏づく |
| 0.55 - 0.74 | 中程度 | 間接証拠がある、妥当な解釈 |
| 0.35 - 0.54 | 弱い | 単発推測、軽い示唆 |
| 0.00 - 0.34 | 保存しないか `events` 側に留める | 根拠不足 |

時間で下がるのは `salience` であり、`confidence` ではない。

## `salience` の更新原則

`salience` は「想起の前面に出すべき強さ」である。
こちらは時間と文脈で動かす。

主な要因は次である。

- 最近性
- 感情強度
- 未完了性
- 関係重要度
- 反復回数
- 現在話題との一致
- 訂正や衝突の有無

次を守る。

- `active_commitments` は最低 salience を高めに保つ
- `relationship` は `topic` より落ちにくくする
- 強い感情を伴った出来事由来の理解は一時的に boost する
- `dormant` は `salience` を低く固定する

## 感情更新との境界

感情モデルの正本構造、更新式、`AffectContext` は [../16_感情モデル.md](../16_感情モデル.md) を正とする。
この文書では、記憶更新が感情層へ渡す境界だけを定める。

- `turn consolidation` は、その回の瞬間反応を `episode_affect` 候補として作る
- `mood_state` は `target_scope_type=self` の `episode_affect` からコードが deterministic に更新する
- `affect_state` は `reflective consolidation` で反復証拠から育てる
- 新規 `affect_state` は同じ `target_scope_type / target_scope_key / affect_label` が複数 episode で支持された場合だけ作る
- 新規 `affect_state` 作成時は、直前の reflection で単発消費された `episode_affect` を短い lookback で拾い直す
- 既存 `affect_state` は同じ対象感情の支持で更新し、支持が長く途切れた場合は強度と確度を弱める
- 1 対象につき強い `affect_state` だけを少数残す
- 感情は `salience` に効くが、感情だけで `confidence` を上げない

瞬間反応、現在機嫌、対象への持続感情を同じ保存先へ混ぜない。

## 長期記憶へ書かない条件

次のようなものは、`memory_units` を作らず `events` と `episodes` のみで扱う。

- 根拠が弱い単発推測
- 後の会話や判断に効かなそうな雑談断片
- 会話の流れ上だけの軽い反応
- 同じターン内で消えていく一時判断

この抑制がないと、記憶の増殖で現在の個の輪郭がノイズ化しやすい。

## `reflective consolidation` の役割

`reflective consolidation` は、1 ターンでは決めきれない理解を整える役割を持つ。

ここでは次をやる。

- 複数ターンにまたがる `episodes` と `memory_units` を横断し、selected `persona`、`mood_state`、`affect_state` を補助に使って `summary` を作る
- `inferred` を `confirmed` に上げるか見直す
- 近縁 `memory_units` を整理する
- `dormant` 化を判断する
- `drive_state` の見直しと更新を行う
- `affect_state` の育成と見直しを行う
- `relationship` と `self` の長期変化を再評価する

ここで重要なのは、「単発イベントではなく蓄積からしか分からないこと」を扱うことである。

## `memory_links` への反映

`memory_links` は理解同士の関係を少数だけ保存する補助構造である。
`revisions.related_memory_unit_ids` は更新履歴の監査関係を表し、`memory_links` は現在の理解グラフを表す。

現行実装では次の保存経路を使う。

- `create` が関連元を持つ場合、作成された理解から関連元へ `derived_from` を張る
- `create` の関連元が active な根拠理解である場合、関連元から作成された理解へ `supports` を張る
- `create` の関連元が `revoked / superseded` になった置換元である場合、作成された理解から置換元へ `affects` を張る
- `create` の source と target が同じ `scope_type / scope_key` を持つ場合、作成された理解から関連元へ `about_same_scope` を張る
- `revoke` が置換先を持つ場合、取り消された理解から置換先へ `contradicts` を張る
- `supersede` が置換先を持つ場合、置換された理解から置換先へ `contradicts` を張る
- `correct` は補正履歴であり、新しい意味関係を自動追加しない
- `reflective consolidation` が summary を作成または更新する場合、根拠になった同一 scope の active `memory_units` を `related_memory_unit_ids` に入れ、`derived_from / supports / about_same_scope` を張る
- 同じ `memory_set_id / source_memory_unit_id / target_memory_unit_id / label` の関係は 1 行に集約し、再発時は evidence revision と `updated_at` を更新する
- endpoint が `revoked / superseded / dormant` になっても link 行は削除しない。endpoint の `status / salience / confidence` を併記して弱化状態を判断する
- `clone/delete` は `memory_links` も対象にし、複製時は source、target、evidence revision の ID を張り替える

## `drive_state` への反映

`reflective consolidation` は、長期変化分析とは別に、中期の志向を `drive_state` として更新する。
`drive_state` は `memory_units` ではなく、人格設定と記憶から派生する状態である。
`drive_state` の意味境界と最小構造は [../05_判断と行動.md](../05_判断と行動.md) と [../06_状態モデル.md](../06_状態モデル.md) を正とする。

この文書では、記憶後段 worker が `drive_state` の材料を作る責務だけを定める。

- `memory_units` と `episodes` へ辿れる根拠を保持する
- 具体的な `fact` や `commitment` を `drive_state` へ複写して正本化しない
- 現在強い向きだけを少数出し、支持が弱くなった向きは `salience` 低下または失効の候補にする
- `drive_state` だけで能力実行を確定しない

現行実装では、`drive_state` 更新で次を行う。

- `commitment` は継続単位ごとに `follow_through` または `resume_when_ready` の `drive_kind` へ写す
- `qualifiers.scope_duration=session`、AI 応答由来、または AI 自身の待機姿勢を表す `commitment` は `drive_state` へ写さない
- `summary` は `self / user / relationship / topic` ごとに集約し、1 scope 1 drive を基本にする
- `relationship / self` summary の再構成に使った scope support を、`drive_state` 更新でも共有する
- `support_count` だけでなく、freshness、`scope_alignment`、`support_strength`、persona 整合、`mood_state / affect_state` の signal 強度も `salience` と `expires_at` に反映する
- `self` は `mood_state.current_vad`、`relationship / user` は `affect_state` の強さを `signal_strength` として使う
- `focus_scope_type / focus_scope_key`、`support_count / support_strength`、`freshness_hint`、`scope_alignment`、`signal_strength`、`persona_alignment`、`stability_hint`、`supporting_memory_types` を持たせ、initiative loop が中期の向きを比較しやすくする
- 相反する summary 候補が近い重みで混ざるときは `stability_hint=mixed` として `salience` を下げる
- stale かつ support が弱い summary drive は `stability_hint=weak` として `salience` と `expires_at` を落とし、必要なら active 集約から外す
- `memory_trace.drive_state_update.scope_supports` には、scope、support 種別、summary 更新有無、active drive との対応を残す

## 長期変化分析の出力は `summary` 系にする

長期変化の分析結果は、`memory_type=summary` の `memory_units` として保持する。

このときの基本原則は次である。

- 具体記憶を全部置き換えない
- 複数の具体記憶をまたぐ傾向だけを要約する
- 既存の `fact`、`preference`、`relation`、`commitment` の根拠を消さない
- 要約が変わったときだけ `summary` 系を更新する

`summary` は、具体的な `fact` や `commitment` の代替ではない。
位置づけとしては、「複数の具体記憶をまたいだ傾向の要約」である。

## `reflective consolidation` の主入力

`reflective consolidation` が主に見る入力は次である。

- 最近の `episodes`
- 既存の `memory_units`
- `revisions`
- 現在選択中の `persona`
- 関係する `episode_affect`、`mood_state`、`affect_state` の傾向

ここでは、毎回すべての `events` を読み直さない。
必要なら根拠確認のために狭く辿るが、主入力は圧縮済みの層に置く。

## `reflective consolidation` でやりすぎないこと

- 具体的事実を `summary` だけで確定すること
- 具体的な `commitment` を `summary` に吸収すること
- 短期の `pending_intent` をそのまま中期の `drive_state` に昇格させること
- 単発イベントから現在の個の長期変化を大きく断定すること
- 直近の強い感情だけで長期傾向を決めること

## 長期変化分析の更新ルール

- 新しい傾向要約がまだない
  - `summary` を `create` する
- 既存の長期傾向と整合し、補強される
  - `refine` または `reinforce` する
- 既存の傾向が明らかに変わった
  - 旧 `summary` を `superseded` にし、新しい `summary` を作る
- 今は前面に出しにくい古い傾向になった
  - `dormant` にする

この更新でも、根拠は `episodes`、`memory_units`、必要なら `events` に辿れるように保つ。

## `reflective consolidation` のトリガー

トリガーは次のどれかとする。

- chat 8 ターン経過
- 前回実行から 24 時間以上経過
- 高 salience の出来事が 3 件以上たまった
- 明示的な訂正イベントが入った
- `self` の変化を示すイベントが入った
- 関係変化を示すイベントが入った

毎ターン走らせる必要はない。
現在の個の構成に効くが、同期経路へ入れるべきではない。
OtomeKairo では、trigger 判定後の実処理を background の記憶後段 worker へ積み、inspection では queued / succeeded / failed を追えるようにする。

`summary_text` の自然文生成は、state 遷移ロジックとは分離して扱う。
`reflective consolidation` の `summary` 文面を LLM 生成で扱う詳細設計は [07_内省要約のLLM生成.md](07_内省要約のLLM生成.md) を正とする。
どの role を使ってどこまで実装済みかは `src/` と smoke 結果を正とする。
生成失敗は scope 単位で閉じ込め、`reflection_runs` と `memory_trace.reflective_consolidation.summary_generation` に件数要約と失敗一覧を残す。
