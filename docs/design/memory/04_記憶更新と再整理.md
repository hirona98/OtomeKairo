# 記憶更新と再整理

<!-- Role -->
## この文書の役割

この文書は、`memory_units` をどう更新し、いつ再整理するかを定める正本である。

ここで定めたいのは次の内容である。

- 新しい出来事からどの `memory_units` を作るか
- 既存 `memory_units` とどう比較するか
- いつ `confirmed` / `superseded` / `revoked` / `dormant` に動かすか
- `confidence` と `salience` を何で上下させるか
- `turn consolidation` と `reflective consolidation` をいつ走らせるか

ここでは DB や API の詳細は確定しない。
ただし、更新判断の原則はここで固定する。

<!-- Principles -->
## 更新の基本原則

OtomeKairo の記憶更新では、次を原則にする。

- 生ログは消さない
- 継続理解は `memory_units` に落とす
- 推測と確定を分ける
- 矛盾したら即削除ではなく、状態遷移で扱う
- 真偽に関わる強さは `confidence` で持つ
- 現在どれだけ前に出すかは `salience` で持つ
- 更新の根拠は必ず `events` に辿れるようにする

特に重要なのは、`confidence` と `salience` を混ぜないことである。

- `confidence` は「それがどれだけ正しそうか」
- `salience` は「いまどれだけ思い出しやすく、前面に出すべきか」

<!-- TurnFlow -->
## `turn consolidation` の流れ

1ターンごとの更新は、次の順で進める。

1. `events` を保存する
2. `episode_digests` を作る
3. 候補 `memory_units` を抽出する
4. 既存 `memory_units` の比較候補を探す
5. 各候補について更新操作を決める
6. `confidence` / `salience` を更新する
7. `memory_links` を最小限だけ張る
8. `affect_state` を更新する
9. `memory_units` に変化があった場合だけ `revisions` を残し、観測ログは毎ターン残す
10. `vector_index_entries` の差分更新を要求する

この段階では「そのターンで確信できる範囲」に留める。
複数ターンをまたぐ整理は `reflective consolidation` に回す。
再整理で本文が変わった `summary` 系は、あわせて再埋め込み対象に含める。

<!-- InterpretationContract -->
## `memory_interpretation` の出力契約

`turn consolidation` の入口で使う `memory_interpretation` は、厳密な構造化出力として扱う。

少なくとも次を守る。

- 返すトップレベルキーは `episode_digest / candidate_memory_units / affect_updates` の 3 つだけにする
- `episode_digest` は `episode_type / primary_scope_type / primary_scope_key / summary_text / outcome_text / open_loops / salience` だけを持つ
- `candidate_memory_units` の各要素は、更新判定に必要な固定キーだけを持つ
- `affect_updates` の各要素も固定キーだけを持つ
- `affect_updates.layer` は `affect_state` の時間層を表し、`surface / background` のどちらかだけを使う
- 余計なキー、説明文、Markdown、コードフェンスは入れない

MVP では、validator 失敗時は同じ入力で 1 回だけ再生成する。
それでも契約を満たさない場合は、そのターンの `turn consolidation` を失敗として監査に残す。

<!-- Candidate -->
## 候補 `memory_units` を作る条件

毎ターンなんでも `memory_units` にしてはいけない。
少なくとも次のどれかに当てはまるものだけを候補にする。

- 今後の会話や判断に効きそうな継続理解
- 関係や約束の変化
- 自己像や他者像に効く解釈
- 高い感情強度を持つ出来事から生じた理解
- 明示的な好み、苦手、事実、否定、訂正

逆に、次のようなものは `events` や `episode_digests` に留める。

- その場限りの相槌
- 冗談の断片
- 軽い雑談で、将来効く理解に育っていないもの
- 真偽も継続性も低い単発推測

<!-- CompareKey -->
## 比較キー

新しい候補 `memory_units` を既存のものと比べるときは、まず次の比較キーを見る。

- `memory_type`
- `scope_type`
- `scope_key`
- `subject_ref`
- `predicate`

必要なら補助キーとして次も見る。

- `object_ref_or_value`
- `qualifiers`

初期段階では、比較キーの意味は次のように扱う。

- 主キー相当:
  - `memory_type`
  - `scope_type`
  - `scope_key`
  - `subject_ref`
  - `predicate`
- 内容比較:
  - `object_ref_or_value`
  - `qualifiers`
  - `summary_text`

つまり、「何についてのどんな理解か」が同じなら比較候補とみなし、「中身が同じか違うか」を比較処理で判定する。

<!-- CompareResult -->
## 比較結果の最小分類

比較結果は、少なくとも次の 5 つに分ける。

| 分類 | 意味 |
|------|------|
| `same` | 同じ理解の再確認 |
| `refine` | 同じ理解だが、精度や条件が増えた |
| `replace` | 同軸の新しい理解が旧理解を置き換える |
| `conflict` | 旧理解を否定する |
| `parallel` | 似ているが別理解として並存する |

この分類により、更新操作を安定させる。

<!-- Operation -->
## 更新操作の最小セット

`turn consolidation` では、更新判定の結果として次の操作だけを持つ。

| 操作 | 内容 |
|------|------|
| `create` | 新しい `memory_units` を作る |
| `reinforce` | 既存理解を再確認し、`confidence` や `last_confirmed_at` を上げる |
| `refine` | 既存理解に条件や補足を加える |
| `supersede` | 旧理解を `superseded` にし、新理解を現行にする |
| `revoke` | 旧理解を `revoked` にする |
| `dormant` | 保持はするが前面から下げる |
| `noop` | `events` だけ残し、長期記憶は更新しない |

これ以上増やさなくても、MVP の更新は回せる。

ここで重要なのは、`noop` を `revisions.operation` に入れないことである。
`noop` は「更新判定としては何もしない」を表すが、`revisions` は `memory_units` の変化履歴だけを残す監査層として扱う。

<!-- Decision -->
## 操作決定の基本ルール

### `create`

次のときに使う。

- 比較候補が見つからない
- 比較候補はあるが `parallel` と判断した
- 重要な新規約束、関係変化、強い出来事由来の理解

### `reinforce`

次のときに使う。

- 既存理解と内容がほぼ同じ
- 今回の出来事が、その理解の再確認になっている

この場合は少なくとも次を更新する。

- `last_confirmed_at`
- `evidence_event_ids`
- `confidence`
- `salience`

### `refine`

次のときに使う。

- 理解の骨格は同じ
- ただし条件、頻度、程度、文脈が追加された

たとえば次のような場合である。

- 「辛いものが好き」→「特にカレー系の辛さが好き」
- 「最近疲れていそう」→「今週は睡眠不足が続いていて疲れていそう」

### `supersede`

次のときに使う。

- 同じ比較キーに対して、新しい理解が現行の理解を置き換える
- 並存させるより、現行版を入れ替えたほうが自然

例:

- 以前は「夜型」だったが、最近は「朝型に寄っている」
- 以前の約束Aは取り消され、新しい約束Bに置き換わった

この場合は次を行う。

- 旧理解を `superseded`
- 新理解を `create`
- 両者を `memory_links.label="contradicts"` または `derived_from` で繋ぐ

### `revoke`

次のときに使う。

- ユーザーや当人が明確に否定した
- 旧理解が誤りだったと明言された

例:

- 「辛いもの好きだっけ？」に対して「いや、むしろ苦手」
- 「昨日の話は勘違いだった」

### `dormant`

次のときに使う。

- 間違いとは言えない
- ただし、今は前面に出し続けるとノイズになる

例:

- 季節限定の関心
- 一時的な作業テーマ
- しばらく触れられていない低重要トピック

<!-- Parallel -->
## 並存を許すケース

OtomeKairo では、次のような理解は並存しうる。

- 関係の複数面
  - 例: 親しい / 仕事では厳密
- 好みの条件差
  - 例: 甘いものは好き / 生クリームは苦手
- 話題ごとの感情傾向
  - 例: 健康の話は不安になりやすい / 開発の話は前向き
- 期間差のある事実
  - 例: 以前はA、現在はB

そのため、違いが `object_ref_or_value` や `qualifiers` に自然に乗るなら、無理に 1 件へ潰さない。

<!-- Confirm -->
## `confirmed` へ上げる条件

`confirmed` は、会話や判断に強く使う理解である。
そのため、`inferred` から上げる条件は厳しめにする。

初期ルールとしては、次のどれかを満たしたときに `confirmed` へ上げる。

- 対象本人が明示的に述べた
- 対象本人が以前の推測を肯定した
- 独立した複数イベントで同じ理解が繰り返し支持された
- 行動や結果で強く裏づけられた
- `reflective consolidation` で十分な根拠があると再評価された

一方で、次のものは `confirmed` に上げない。

- 雰囲気だけの推測
- 単発の冗談
- 相槌からの過剰推定
- 他人づての曖昧な情報

<!-- Demotion -->
## 格下げの条件

### `confirmed` → `superseded`

次のときに使う。

- 以前は正しかったが、今は更新された

### `confirmed` / `inferred` → `revoked`

次のときに使う。

- 明確な否定があった
- 勘違いだったと判明した

### `inferred` / `confirmed` → `dormant`

次のときに使う。

- しばらく参照価値が低い
- 重要度が低い
- 主話題から外れ続けている

この `dormant` は削除ではなく、「今は積極想起しない」状態である。

<!-- TypeNotes -->
## `memory_type` ごとの更新注意

型ごとに、更新判断で少し癖が違う。
MVP では次の違いだけ押さえる。

### `fact`

- 明示訂正が来たら `supersede` または `revoke` を優先する
- 期間差があるなら `parallel` ではなく `valid_from` / `valid_to` を伴う更新を優先する

### `preference`

- 推測だけなら `inferred`
- 本人の明示や反復で `confirmed`
- 反対極性の明示が来たら旧理解は `revoked` または `superseded`

### `relation`

- 1イベントで決め打ちしすぎない
- 単発の摩擦や親密さより、複数イベントの傾向を重く見る
- `reflective consolidation` での再評価を前提にする

### `commitment`

- `status` ではなく `commitment_state` で履行状態を持つ
- `active_commitments` に入るのは `open` / `waiting_confirmation` / `on_hold`
- 未完了のものは `salience` を高く保つ
- 完了、取消、保留解除が来たら `commitment_state` を優先的に更新する
- 同じ約束の進行更新はまず `refine` で扱い、別の約束へ置き換わるときだけ `supersede` を使う
- 想起パックでは常に別枠で扱う

### `interpretation`

- `confidence` を上げすぎない
- 明示根拠がない限り `confirmed` にしない
- topic や relationship に強く効くが、誤差も多いので `dormant` 化しやすくする

<!-- Confidence -->
## `confidence` の更新原則

`confidence` は、最近性ではなく根拠の強さで動かす。

初期段階では、厳密な数式ではなく帯域ルールを使う。

| 帯域 | 目安 | 典型例 |
|------|------|------|
| 0.90 - 1.00 | 非常に強い | 対象本人の明示、強い再確認 |
| 0.75 - 0.89 | 強い | 複数回の一貫支持、結果でも裏づく |
| 0.55 - 0.74 | 中程度 | 間接証拠がある、妥当な解釈 |
| 0.35 - 0.54 | 弱い | 単発推測、軽い示唆 |
| 0.00 - 0.34 | 保存しないか `events` 側に留める | 根拠不足 |

更新時のルールは次とする。

- 明示確認で大きく上げる
- 再確認で少しずつ上げる
- 矛盾や否定で大きく下げる
- 最近触れられていなくても、`confidence` は自動では下げない

つまり、時間で下がるのは `salience` であり、`confidence` ではない。

<!-- Salience -->
## `salience` の更新原則

`salience` は「想起の前面に出すべき強さ」である。
こちらは時間と文脈で動かす。

初期要因は次とする。

- 最近性
- 感情強度
- 未完了性
- 関係重要度
- 反復回数
- 現在話題との一致
- 訂正や衝突の有無

### `salience` を上げる要因

- 直近の出来事で再確認された
- 強い感情が結びついた
- 約束や未完了として残っている
- 関係変化に直結する
- 現在の話題と直接一致する

### `salience` を下げる要因

- 長期間触れられていない
- 主話題から外れている
- 低重要の一時的関心だった
- `superseded` や `revoked` になった

### `salience` の扱い

初期段階では、次のように扱う。

- `active_commitments` は最低 salience を高めに保つ
- `relationship` は topic より落ちにくくする
- 強い感情を伴った出来事由来の理解は、一時的に boost する
- `dormant` は salience を低く固定する

<!-- AffectLink -->
## 感情と `salience` の関係

感情は独立層に置くが、更新時には `salience` に効かせる。

特に次は重要である。

- 喜怒哀楽が強かった出来事
- 関係に影響する出来事
- 未完了で気にかかり続ける出来事

ただし、感情が強いだけで `confidence` を上げない。
強い感情と正しさは別である。

<!-- NoWrite -->
## 長期記憶へ書かない条件

次のようなものは、`memory_units` を作らず `events` のみで扱う。

- 根拠が弱い単発推測
- 将来効かなそうな雑談断片
- 会話の流れ上だけの軽い反応
- 同じターン内で消えていく一時判断

この抑制がないと、記憶の増殖で人格がノイズ化しやすい。

<!-- FutureActBoundary -->
## `future_act` 候補キューは長期記憶と分ける

`future_act` 候補キューは、`13_自発判断と自発行動.md` で定めた短期の内部保留状態である。
そのため、候補を作った時点では、長期記憶へそのまま書き込まない。

MVP では、次の境界で扱う。

- `future_act` 候補の生成や更新
  - 内部候補キューだけを更新する
- 候補が `reply` に昇格した
  - 実際に起きた会話イベントとして `events` に返す
- 候補が失効した、見送られた、または明示的に捨てられた
  - その結果だけを必要なら `events` に返す

ここで重要なのは、保留候補そのものを継続知識として扱わないことである。
長期記憶へ落とすのは、実際の会話、結果、拒否、未完了、継続意図のように後の判断へ残す価値があるものだけにする。

<!-- ExternalActionBoundary -->
## 外界行動の実行計画は記憶責務に入れない

外界行動が入った場合でも、記憶が持つのは実行オーケストレーション本体ではない。

MVP では、記憶側が扱うのは次までとする。

- 何をしようとしたかという意図の要約
- 実際に成功したか失敗したか
- ユーザーが受け入れたか拒否したか
- 次回の抑制や再判断に効く結果要約

逆に、次は記憶責務に入れない。

- 実行計画の詳細手順
- 再試行ポリシーの本体
- スケジューリング制御
- 外部能力のオーケストレーション詳細

この境界により、記憶は「判断材料と結果の蓄積」に留まり、実行制御そのものを抱え込まないで済む。

<!-- Reflective -->
## `reflective consolidation` の役割

`reflective consolidation` は、1ターンでは決めきれない理解を整える役割を持つ。

ここでは次をやる。

- 複数 `episode_digests` を横断して `summary` を作る
- `inferred` を `confirmed` に上げるか見直す
- 近縁 `memory_units` を整理する
- `dormant` 化を判断する
- `relationship` と `self` の長期変化を再評価する

ここで重要なのは、「単発イベントではなく蓄積からしか分からないこと」を扱うことである。

<!-- LongTermChange -->
## 長期変化の分析は `reflective consolidation` に寄せる

MVP では、長期変化の分析を別の常時処理層に分けない。
この責務は `reflective consolidation` が持つ。

ここでいう長期変化とは、少なくとも次のようなものである。

- `self`
  - 最近の自分の姿勢や行動傾向の変化
- `user`
  - ユーザー理解の安定化や変化
- `relationship`
  - 親しさ、信頼、摩擦、支え方の流れ
- `topic`
  - 長く続いている関心や未完了テーマの推移

これらは、単発イベントだけでは決めにくく、複数の `episode_digests` と既存 `memory_units` を横断して初めて見える。

<!-- LongTermOutput -->
## 長期変化分析の出力は `summary` 系にする

長期変化の分析結果は、MVP では `memory_type=summary` の `memory_units` として保持する。

このときの基本原則は次である。

- 具体記憶を全部置き換えない
- 複数の具体記憶をまたぐ傾向だけを要約する
- 既存の `fact`、`preference`、`relation`、`commitment` の根拠を消さない
- 要約が変わったときだけ `summary` 系を更新する

つまり、長期変化分析は「新しい棚を増やすこと」ではなく、「既存記憶の上に長めの傾向要約を作ること」である。

<!-- LongTermInput -->
## 長期変化分析で主に見る入力

`reflective consolidation` が長期変化を見るときは、少なくとも次を主入力にする。

- 最近の `episode_digests`
- 既存の `memory_units`
- `revisions`
- 関係する `affect_state` の傾向

ここでは、毎回すべての `events` を読み直さない。
必要なら根拠確認のために狭く辿るが、主入力は圧縮済みの層に置く。

<!-- LongTermGuardrail -->
## 長期変化分析でやりすぎないこと

`reflective consolidation` は強いが、MVP では次をやりすぎない。

- 具体的事実を `summary` だけで確定する
- 具体的な `commitment` を `summary` に吸収する
- 単発イベントから人格変化を大きく断定する
- 直近の強い感情だけで長期傾向を決める

長期変化分析は、具体記憶の代替ではなく補助層である。
そのため、具体の根拠が必要な判断では、引き続き `fact`、`relation`、`commitment`、`episode_digests` を優先する。

<!-- LongTermUpdateRule -->
## 長期変化分析の更新ルール

MVP では、長期変化分析の更新を次のように扱う。

- 新しい傾向要約がまだない
  - `summary` を `create` する
- 既存の長期傾向と整合し、補強される
  - `refine` または `reinforce` する
- 既存の傾向が明らかに変わった
  - 旧 `summary` を `superseded` にし、新しい `summary` を作る
- 今は前面に出しにくい古い傾向になった
  - `dormant` にする

この更新でも、根拠は `episode_digests`、`memory_units`、必要なら `events` に辿れるように保つ。

<!-- ReflectiveTrigger -->
## `reflective consolidation` のトリガー

初期ルールとしては、次のどれかで走らせる。

- chat 8ターン経過
- 前回実行から 24 時間以上経過
- 高 salience イベントが 3 件以上たまった
- 明示的な訂正イベントが入った
- 関係変化を示すイベントが入った

一方で、毎ターン走らせる必要はない。
人格形成に効くが、同期経路へ入れるべきではない。

<!-- Revision -->
## `revisions` に残すべきこと

更新では、少なくとも次を残す。

- `revision_id`
- `occurred_at`
- 更新対象の `memory_unit_id`
- 操作種別
- 関連する `memory_unit_id` 群
- `before_snapshot`
- `after_snapshot`
- 理由
- 根拠 `event_id` 群

`revisions` に残すのは、`memory_units` が実際に変化した操作だけである。
つまり、`create`、`reinforce`、`refine`、`supersede`、`revoke`、`dormant` は残すが、`noop` は残さない。

`noop` だったターンの監査は、次で追う。

- `events`
  - そのターンの観測、返答、判断結果
- `episode_digests`
  - そのターンの要約
- デバッグ記録
  - そのターンで長期記憶更新に至らなかった理由要約

特に `supersede` と `revoke` は、あとから人格変化を追うために必須である。

MVP では、次だけ先に実装する。

## ここで固定したい設計判断

この文書で固定したいのは次である。

- 長期記憶へ落とすのは、将来の判断に効く継続理解だけにする
- 正確引用や詳細時系列の再現は、更新責務の中心に置かない
- `future_act` 候補キューは短期の内部保留状態であり、生成時点では長期記憶へ書かない
- 長期記憶へ返すのは、保留候補そのものではなく、実際の会話や結果として残る意味だけにする
- 外界行動が入っても、記憶が持つのは意図、結果、受容 / 拒否、次回抑制への影響までに留める
- 実行計画や再試行制御の本体は、記憶責務に含めない
- 長期変化の分析は別レイヤーを増やさず、`reflective consolidation` が `summary` 系として扱う
- `summary` は具体記憶の代替ではなく、複数記憶をまたぐ長めの傾向要約として使う
- `noop` は更新判定には存在するが、`revisions` には入れず、監査は観測ログとデバッグ記録で追う
