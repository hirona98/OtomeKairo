# 想起と判断ユースケース

## ここで検証する対象

この文書では、特に次を対象とする。

- `RecallHint` がどの程度自然に `primary_intent` を取り、`secondary_intents` を補助に使えるか
- 構造レーンと連想レーンの組み合わせで、必要な記憶を返答直前に集められるか
- `RecallPack` の section 分離で、現在の個の文脈と話題理解が崩れないか
- MVP の内部結果 `reply / noop / future_act` に持ち込んだとき、どこで分岐差分が効くか
- 現状設計で保証しづらいケースがどこにあるか

ここでは、実装性能やモデル選定ではなく、設計としての対応範囲を見ていく。

## 判定基準

この文書では、各パターンを次の 3 段階で判定する。

| 判定 | 意味 |
|------|------|
| `対応できる` | 現状設計の標準経路で安定して処理できる |
| `条件付きで対応できる` | 条件が揃えば処理できるが、誤想起や取りこぼしの余地がある |
| `現状では難しい` | 現状設計の責務外か、別の機構がないと不安定になる |

`対応できる` とみなす条件は次である。

- `RecallHint` が現在観測と直近文脈だけで十分な向き付けを作れる
- 構造レーンか連想レーンのどちらか、または両方で必要候補を取れる
- `RecallPack` に入った情報だけで、返答が断定しすぎず自然に組み立てられる

逆に、次のどれかを強く必要とする場合は `現状では難しい` に寄せる。

- `events` 全文の広域再読
- 複数の独立意図を同時に最適化すること
- 多段のグラフ探索や長い因果推論
- 長期記憶に存在しない暗黙情報の補完

## 対応できるパターン

### パターン A1: 未完了の約束確認

| 項目 | 内容 |
|------|------|
| 判定 | `対応できる` |
| 入力例 | 「この前の続き、また今度話すって言ってたよね」 |
| `RecallHint` | `primary_intent=commitment_check`, `time_reference=future`, `focus_scopes` に `relationship:self|user` と `user` が入りやすい |
| 主に効く section | `active_commitments`, `relationship_model`, `episodic_evidence` |
| 理由 | `commitment` は構造レーンで常に優先され、ベクトルに依存せず拾えるため |

このパターンは、現状設計が最も強く扱いたい領域である。
約束の継続性は現在の個の一貫性に直結するため、構造レーンを主役とする。

### パターン A2: 安定した嗜好の再利用

| 項目 | 内容 |
|------|------|
| 判定 | `対応できる` |
| 入力例 | 「辛いものの話してたけど、今度また食べに行きたい？」 |
| `RecallHint` | `primary_intent=preference_query`, `secondary_intents=[reminisce]`, `time_reference=past` または `persistent`, `focus_scopes` に `user` と `topic:food` が入りやすい |
| 主に効く section | `user_model`, `active_topics`, `episodic_evidence` |
| 理由 | 安定嗜好は `memory_units.preference` として残しやすく、話題一致で構造レーンから拾えるため |

ここでは、以前のエピソードを 1 件添えられれば、返答の自然さも出しやすい。

### パターン A3: 最近の体調や悩みの継続相談

| 項目 | 内容 |
|------|------|
| 判定 | `対応できる` |
| 入力例 | 「最近また眠れてなくて、どうしたらいいかな」 |
| `RecallHint` | `primary_intent=consult`, `time_reference=recent`, `focus_scopes` に `user` と `topic:health` が入りやすい |
| 主に効く section | `user_model`, `active_topics`, `relationship_model`, `episodic_evidence` |
| 理由 | 近況確認と相談は `user` / `topic` 軸で構造的に集めやすく、連想レーンも似た相談文脈を補助しやすいため |

このパターンでは、`relationship_model` が返答温度に効く。
同じ相談でも、踏み込み方は記憶された関係性で変わる。

### パターン A4: 関係そのものの確認

| 項目 | 内容 |
|------|------|
| 判定 | `対応できる` |
| 入力例 | 「最近ちょっと距離ある感じする？」 |
| `RecallHint` | `primary_intent=meta_relationship`, `time_reference=recent`, `focus_scopes` に `relationship:self|user` が入りやすい |
| 主に効く section | `relationship_model`, `episodic_evidence`, `conflicts` |
| 理由 | `relationship_model` を独立 section にした効果が最も出るパターンであり、返答の断定度も `conflicts` で抑制できるため |

このパターンは、OtomeKairo の「現在の個は人格設定と記憶から立ち上がる」という設計意図と相性がよい。

### パターン A5: 日をまたいだ明示的な話題継続

| 項目 | 内容 |
|------|------|
| 判定 | `対応できる` |
| 入力例 | 「昨日話してた仕事の件だけど」 |
| `RecallHint` | `primary_intent=reminisce`, `time_reference=past`, `focus_scopes` に `topic:work` や `user` が入りやすい |
| 主に効く section | `active_topics`, `episodic_evidence`, `user_model` |
| 理由 | 過去参照が明示されていれば、`active_topics` と evidence で自然につなぎやすいため |

ここで重要なのは、`events` 全文を読まずに `episode_digests` でつなげる点である。
現状設計はこの範囲なら十分に扱える。

### パターン A6: 明示された第三者を含む会話継続

| 項目 | 内容 |
|------|------|
| 判定 | `対応できる` |
| 入力例 | 「田中さんの件、その後どう思ってる？」 |
| `RecallHint` | `primary_intent=check_state`, `secondary_intents=[reminisce]`, `mentioned_entities` に `person:tanaka` 相当が入り、`focus_scopes` は `user` を中心に保つ |
| 主に効く section | `user_model`, `active_topics`, `episodic_evidence` |
| 理由 | 対象が明示されていれば、`mentioned_entities` を使う連想レーン補強と話題スコープで候補を狭められるため |

OtomeKairo は 2 者関係に閉じず、第三者を含む継続理解も扱いたい。
その意味で、このパターンは現状設計でも十分に射程内である。

## 条件付きで対応できるパターン

### パターン B1: 回想と相談が混ざった入力

| 項目 | 内容 |
|------|------|
| 判定 | `条件付きで対応できる` |
| 入力例 | 「前の続きなんだけど、また相談してもいい？」 |
| 強み | `RecallHint` が `primary_intent` と `secondary_intents` を返せるので、最低限の方向付けはできる |
| 弱み | `primary_intent` は 1 つだけなので、`commitment_check` と `consult` の完全な両立最適化は弱い |

このケースでは、`secondary_intents` に回った側の要素が薄くなりやすい。
会話としては破綻しなくても、最良の想起にはなりにくい。

### パターン B2: 参照語が曖昧で、直近窓の外に文脈がある

| 項目 | 内容 |
|------|------|
| 判定 | `条件付きで対応できる` |
| 入力例 | 「あの件、まだ気になってる？」 |
| 強み | 直近 `3 分以内` かつ `4-6 発話` に対象が残っていれば自然に補完できる |
| 弱み | 対象が直近窓の外にあると、`RecallHint` は長期記憶を読まないため、入口で対象特定に失敗しやすい |

この弱さは、`RecallHint` を軽く保った代わりの制約である。
曖昧参照を長期記憶で補完する設計にはまだしていない。

### パターン B3: 明示手がかりの薄い昔話の回想

| 項目 | 内容 |
|------|------|
| 判定 | `条件付きで対応できる` |
| 入力例 | 「なんか前にもこんな感じのことあったよね」 |
| 強み | 連想レーンが `episode_digests` をベクトル検索して、似た空気感のエピソードを拾える |
| 弱み | 手がかりが少なすぎると、ベクトル hit の質に強く依存し、外しやすい |

このパターンは、ベクトルレーンを入れた意義が出るが、同時に限界も出やすい。
「それっぽい思い出」は拾えても、「正しい思い出」を保証しにくい。

### パターン B4: 最近の矛盾を含む自己申告

| 項目 | 内容 |
|------|------|
| 判定 | `条件付きで対応できる` |
| 入力例 | 「前は辛いの好きって言ったけど、今はちょっと苦手かも」 |
| 強み | `conflicts` により断定を避け、確認寄りの返答に落とせる |
| 弱み | どちらを現行理解として採るかは、更新規則と再整理の質に依存する |

つまり、誤答を避けることはできるが、即座にきれいな再定義までできるとは限らない。

### パターン B5: 小さなやり取りの積み重ねで生じる関係変化

| 項目 | 内容 |
|------|------|
| 判定 | `条件付きで対応できる` |
| 入力例 | 「最近なんとなく話しにくい気がする」 |
| 強み | `relationship_model` と `episodic_evidence` があれば、確認寄りの返答はできる |
| 弱み | 微小な変化が複数ターンに分散している場合、`reflective consolidation` の質が重要になり、現時点ではまだ十分に検証できていない |

このケースでは、同期想起だけでなく、非同期再整理の設計が成否を大きく左右する。

### パターン B6: 複数人物が絡むが、主題は 1 つに絞れるケース

| 項目 | 内容 |
|------|------|
| 判定 | `条件付きで対応できる` |
| 入力例 | 「田中さんと佐藤さんの話、結局どっちがしんどそうだったと思う？」 |
| 強み | 主題が 1 本なら、片方を主軸に `entity` scope を引きつつ、もう片方を evidence で補助できる |
| 弱み | 人物が増えるほど `focus_scopes` 上限と `mentioned_entities` 上限に近づき、整理しきれなくなる |

対象数が少なく、今ターンの主題が明確なら処理できる。
ただし、多人数の関係図そのものをさばく設計にはなっていない。

### パターン B7: 場所や出来事を鍵にした「何を話していたか」確認

| 項目 | 内容 |
|------|------|
| 判定 | `条件付きで対応できる` |
| 入力例 | 「水族館に行ったとき、何のこと話してたんだっけ？」 |
| 強み | `episode_digests` に場所や出来事のアンカーが残っていれば、回想系として拾い、その `linked_event_ids` から例外的に細部確認できる |
| 弱み | 場所アンカーが digest に残っていない場合や、1 digest に複数話題が圧縮されている場合は粗くなる |

このケースでは、`episode_digests` が入口で、`events` は補助証拠である。
全文再読ではなく、選ばれた `linked_event_ids` だけを slot 型の `event_evidence` に短く圧縮して使う。

### パターン B8: 過去に決めた方針や軽い合意の確認

| 項目 | 内容 |
|------|------|
| 判定 | `条件付きで対応できる` |
| 入力例 | 「そのとき、どうするって言ったっけ？」 |
| 強み | `outcome_text` や `commitment` に上がっていれば強く、足りない場合も関連 `events` を例外参照して補える |
| 弱み | 軽い雑談レベルの合意で `outcome` や `commitment` に残っていない場合は、細部が欠けやすい |

このパターンは、継続理解が主、`events` は補助という設計と相性がよい。
強い約束なら構造レーン、軽い決定なら例外証拠で支える形になる。

## 判断分岐で見たときのパターン

この節では、D1 は MVP の内部結果 `reply / noop / future_act` を前提に見る。
D2 以降は将来の `act` 拡張を含む参考評価として読む。

### パターン D1: 定期起床からの自発会話提案

| 項目 | 内容 |
|------|------|
| 判定 | `対応できる` |
| 入力例 | 定期起床時に、未完了話題と前回会話から十分時間が空いている |
| 強み | `active_topics`、`active_commitments`、`relationship_model` を使って「今話しかけるべきか」を判断しやすい |
| 弱み | 起床時の観測が薄すぎると、想起はできても積極的に話しかける根拠が弱くなる |

このケースは、外界行動ではないが、MVP の `reply / noop / future_act` 分岐で見ると `reply` 側の自発判断である。

### パターン D2: 記憶と能力が両方そろった行動選択

| 項目 | 内容 |
|------|------|
| 判定 | `条件付きで対応できる` |
| 入力例 | 「前に頼まれたリマインドを今通知するべきか」を判断する |
| 強み | `active_commitments` が根拠を出し、能力マニフェストが実行可否を制限できる |
| 弱み | 現在の外界行動は MVP 主対象外なので、記憶側の設計は整っていても実行側の詳細が未確定である |

このケースでは、記憶だけでなく能力定義が必須になる。
これが会話分岐との大きな差である。

### パターン D3: 記憶が曖昧なため行動せず確認に倒す

| 項目 | 内容 |
|------|------|
| 判定 | `対応できる` |
| 入力例 | 「前にそうしてと言われた気がするが、確信がない」 |
| 強み | `conflicts` や低信頼を使って、実行より確認会話や `noop` に倒せる |
| 弱み | 行動に行かない判断はできても、なぜ止めたかの説明方針まではまだ細かく詰めていない |

このケースは、行動分岐で `conflicts` が会話より強い抑制になることを示す。

### パターン D4: 外界状態の継続追跡を要する複雑行動

| 項目 | 内容 |
|------|------|
| 判定 | `現状では難しい` |
| 入力例 | 複数ステップの外部操作や、継続監視が必要な行動を自律で行う |
| 難しい理由 | `world_state` と能力状態の境界は置けたが、継続監視、失敗回復、複数ステップ実行の実装方針がまだ不足しているため |

ここは記憶設計の次段ではなく、能力設計と実行設計も含めた次フェーズである。

## 現状では難しいパターン

### パターン C1: 正確な引用や詳細時系列の再現

| 項目 | 内容 |
|------|------|
| 判定 | `現状では難しい` |
| 入力例 | 「先々週、私が何て言ったかそのまま覚えてる？」 |
| 難しい理由 | `events` の例外参照は持つが、短い `event_evidence` へ圧縮して使うだけで、逐語再現や長い引用を保証しないため |

現状設計は「意味の継続」を優先しており、「逐語的な再現」は責務外に置いている。

### パターン C2: 1入力で複数の独立課題を同時に最適化すること

| 項目 | 内容 |
|------|------|
| 判定 | `現状では難しい` |
| 入力例 | 「体調の相談と、明日の約束の確認と、田中さんの件も一緒に話したい」 |
| 難しい理由 | `RecallHint` は `primary_intent` を 1 つに絞り、`secondary_intents` と `focus_scopes` にも上限があるため |

会話としては応答できるが、記憶想起を最適化するには分解機構が足りない。

### パターン C3: 手がかりの乏しい暗黙参照を長期記憶だけで解くこと

| 項目 | 内容 |
|------|------|
| 判定 | `現状では難しい` |
| 入力例 | 「あの感じ、また来てるかも」 |
| 難しい理由 | `RecallHint` は長期記憶を読まないため、入口時点で参照先がほぼ定まらない |

これは軽さと安全性の代償である。
曖昧参照を強く扱うなら、別の照応解決機構が要る。

### パターン C4: 多段の関係推論や因果推論

| 項目 | 内容 |
|------|------|
| 判定 | `現状では難しい` |
| 入力例 | 「田中さんとのやり取りと佐藤さんの反応を踏まえると、今の私は誰を避けてると思う？」 |
| 難しい理由 | 現状の想起は `RecallPack` の section 単位で止まり、多段グラフ推論を前提にしていないため |

現状設計は「必要な記憶を集める」ことには向くが、「集めた複数の記憶から新しい高次結論を導く」ことまでは責務化していない。

### パターン C5: 長期間の微小傾向から latent な自己変化理解を抽出すること

| 項目 | 内容 |
|------|------|
| 判定 | `現状では難しい` |
| 入力例 | 「ここ半年の会話を通して、私ってどう変わったと思う？」 |
| 難しい理由 | `reflective consolidation` の枠はあるが、長期統計的な自己変化分析はまだ具体化していないため |

ここは記憶設計の次段であり、MVP の同期想起に背負わせるべきではない。

### パターン C6: 1ターン内で独立した重要出来事が複数あるケース

| 項目 | 内容 |
|------|------|
| 判定 | `現状では難しい` |
| 入力例 | 1 回の長文で、関係変化、約束形成、第三者の相談が同時に強く出る |
| 難しい理由 | `episode_digests` は MVP の `turn consolidation` で `1 判断サイクル = 1 digest` を標準としているため、1 回の長いサイクルに独立した重要事項が多いと単一 digest が過密化しやすいため |

この制約は保存時にも想起時にも効く。
単一ターンを分割して理解する仕組みは、まだ次段の課題である。

## 現状設計で強い領域と弱い領域

要約すると、現状設計は次に強い。

- 約束の継続
- 安定嗜好の再利用
- 最近の相談や体調文脈
- 関係確認
- 明示手がかりのある日またぎ回想

逆に、次は弱い。

- 複合意図の同時最適化
- 曖昧参照の長期補完
- 正確引用や詳細時系列の再現
- 多段推論
- 長期間変化の統計的解釈

つまり、現状設計は「現在の個を支える継続理解」と「今必要な想起」を結ぶにはかなり良い。
一方で、「高度な照応解決」と「広い履歴からの高次推論」に加えて、「複雑な外界行動の実行設計」はまだ次段である。

## この検証で見えた主な弱点

この検証から、現状設計の弱点は次に集約される。

- 複合意図の同時最適化
- 直近窓の外にある曖昧参照
- 長期変化の分析
- 詳細時系列や正確引用の再現
- 複雑な外界行動へ広げたときの実行設計

これらは、MVP の保証範囲の外側にある課題として見るのが自然である。
今後の実装順と後段へ送る論点の一覧は、`docs/plan/01_MVP実装計画.md` を正とする。
