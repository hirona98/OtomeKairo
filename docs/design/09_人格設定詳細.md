# 人格設定詳細

<!-- Role -->
## この文書の役割

この文書は、OtomeKairo の人格設定をどの粒度で持つかを定める正本である。

ここで固定したいのは次の内容である。

- 人格設定をどう分割するか
- 性格と口調の関係
- 行動判断傾向をどこに持たせるか
- 記憶が人格設定にどう効いてよいか
- 人格更新をどの単位で扱うか

ここでは、実際の保存形式やプロンプト全文は定めない。
ただし、人格設定の責務と最小構造はここで固定する。

<!-- Premise -->
## 基本前提

OtomeKairo では、人格設定と記憶を分けて扱う。
人格設定は基本姿勢を与え、記憶はその時点の関係、話題、気分、継続理解を育てる。

そのため、人格設定は「その AI が元々どういう存在か」を持ち、記憶は「何を経験して、いま何をどう感じているか」を持つ。

この前提により、人格設定の内部も次の 2 層に分ける。

- `core_persona`
- `expression_style`

<!-- WhyTwoLayers -->
## 二層に分ける理由

人格設定を 1 枚の自由文にまとめると、次が混ざりやすい。

- 何を大事にするか
- どう判断しやすいか
- どう話すか
- どれくらい感情を出すか

これらは関連するが、同じものではない。

OtomeKairo では、会話だけでなく自発判断や将来の行動判断も扱うため、少なくとも次を分けるほうがよい。

- `core_persona`
  - 判断と姿勢の中核
- `expression_style`
  - その判断を言葉や表情としてどう出すか

この分離により、同じ人格の考え方を保ったまま、話し方だけを調整しやすくなる。
また、話し方の違いで判断傾向までずれてしまうのを防ぎやすくなる。

<!-- CorePersona -->
## `core_persona`

`core_persona` は、その人格がどう考え、どう判断し、どう他者と向き合うかの中核である。

ここには、少なくとも次を持つ。

| 項目 | 役割 |
|------|------|
| `self_image` | 自分をどういう存在として捉えるか |
| `core_values` | 何を大事にするか、何を避けるか |
| `judgement_tendencies` | 慎重寄りか、前向き寄りか、確認を挟みやすいか |
| `relation_baseline` | 他者へどう関わるのを基本とするか |
| `initiative_baseline` | 自分から話しかけるか、待つか、どの程度介入したがるか |

ここで重要なのは、`core_persona` を相手別の詳細設定にしすぎないことである。
MVP では、ユーザー専用の接し方を人格設定に大量に持ち込まず、基本姿勢だけを置く。

ユーザー固有の距離感や最近の空気感は、記憶側で扱う。

<!-- InitiativeBoundary -->
## `initiative_baseline` と `wake_policy` の境界

`initiative_baseline` は、判断機会が与えられたときに、その人格がどれくらい自分から前に出やすいかを表す。
つまり、人格側の「質」と「傾向」を持つ。

一方で `wake_policy` は、OtomeKairo にいつ判断機会を与えるかを決める。
つまり、設定側の「機会生成」と「頻度」を持つ。

この分離により、次のように扱える。

- `wake_policy`
  - 起床判断を止めるか
  - どの程度の間隔で判断機会を与えるか
- `initiative_baseline`
  - その判断機会で話しかけに行きやすいか
  - まず待つ傾向が強いか
  - 介入を慎重に抑えるか

つまり、`wake_policy` だけでは自発性は決まらない。
判断機会の上で、`initiative_baseline` と現在文脈が合わさって、実際に `reply` へ進むか `noop` に留まるかが決まる。

<!-- ExpressionStyle -->
## `expression_style`

`expression_style` は、`core_persona` が出した判断を、会話としてどう表出するかを決める層である。

ここには、少なくとも次を持つ。

| 項目 | 役割 |
|------|------|
| `tone` | 口調の基本傾向 |
| `sentence_length` | 返答を短めにするか、丁寧に広げるか |
| `emotional_expressiveness` | 感情をどれくらい表に出すか |
| `directness` | 率直に言うか、やわらかく包むか |
| `cadence` | 文のテンポや間の取り方 |
| `initiative_expression` | 自発提案をどう言い出すか |

`expression_style` は、判断そのものを決める主層ではない。
「何を返すべきか」よりも、「それをどの温度感でどう返すか」に強く効く。

<!-- AffectBoundary -->
## `emotional_expressiveness` と `affect_state` の境界

`emotional_expressiveness` は、その人格が感情をどれくらい表に出しやすいかを表す。
これは `expression_style` の安定した傾向である。

一方で `affect_state` は、その時点で実際にどういう気分や感情状態にあるかを表す。
これは記憶側の変動状態である。

この分離により、次のように扱う。

- `emotional_expressiveness`
  - 感情を大きく出しやすいか
  - 控えめに出しやすいか
  - 表出の振れ幅をどれくらい許すか
- `affect_state`
  - いま安心しているか
  - いま不安が強いか
  - いま親しみや苛立ちが前に出ているか

つまり、人格設定は「どう出しやすいか」を持ち、`affect_state` は「いま何が出ようとしているか」を持つ。
`affect_state` は表出の内容や強弱に影響してよいが、人格設定側の表出傾向そのものを定義し直さない。

<!-- RelationBetweenLayers -->
## `core_persona` と `expression_style` の関係

この 2 層は独立しているが、別人格として分離するわけではない。
1 つの人格設定の中に、両方を持つ。

役割分担は次のように考える。

- `core_persona`
  - 判断内容に強く効く
- `expression_style`
  - 表現の仕方に強く効く

たとえば、慎重で配慮を重視する人格であれば、その慎重さは `core_persona` に置く。
そのうえで、やわらかく話すか、簡潔に話すかは `expression_style` に置く。

<!-- ReplyActDifference -->
## 会話、自発判断、行動判断への効き方

人格設定は、会話だけでなく判断全体に効く。
ただし、2 層の効き方は同じではない。

| 場面 | `core_persona` | `expression_style` |
|------|------|------|
| 会話返答 | 強く効く | 強く効く |
| 自発会話判断 | 強く効く | 中程度に効く |
| 自発会話の言い出し方 | 中程度に効く | 強く効く |
| 将来の行動判断 | 強く効く | 弱く効く |

この整理により、行動判断傾向は `core_persona` 側に置くと固定する。
一方で、自発的な提案をどう言い出すかは `expression_style` 側で調整してよい。

<!-- RelationBoundary -->
## `relation_baseline` と関係記憶の境界

`relation_baseline` は、その人格が他者一般にどう接しやすいかを表す。
これは人格設定側の基本姿勢である。

一方で関係記憶は、「この相手と実際にどういう関係が育っているか」を表す。
これは記憶側の個別状態である。

この分離により、次のように扱う。

- `relation_baseline`
  - 基本的に世話焼きか
  - 一定距離を保ちやすいか
  - 親しみを出しやすいか
  - 配慮を優先しやすいか
- 関係記憶
  - このユーザーには特に親しいか
  - この人物には慎重になっているか
  - 最近距離が縮まったか、摩擦があったか

つまり、人格設定は「誰に対してもまずどう接しやすいか」を持ち、関係記憶は「この相手には実際どうなっているか」を持つ。
MVP では、ユーザー専用の基本姿勢を人格設定へ別レイヤーで持ち込まず、個別の関係変化は関係記憶へ寄せる。

<!-- MemoryBoundary -->
## 記憶との境界

記憶は人格設定を完全に上書きしない。
ただし、その時点の人格文脈として、表出や優先順位に強く効いてよい。

記憶が主に効くのは次である。

- どの話題を優先するか
- どの相手にどういう距離感で話すか
- どれくらい気を遣うか
- いま明るく出るか、静かに出るか
- 未完了や関係の重みをどれだけ前に出すか

逆に、記憶が直接書き換えないものは次である。

- `core_values`
- `self_image`
- `judgement_tendencies`
- `relation_baseline`
- `initiative_baseline`

つまり、記憶は現在文脈を育てるが、人格設定の中核を自動で再定義しない。
ただし、`relation_baseline` の上に関係記憶が重なることで、相手ごとの実際の距離感や言い回しは大きく変化してよい。
また、`emotional_expressiveness` の上に `affect_state` が重なることで、その瞬間の明るさ、静けさ、強さも変化してよい。

<!-- MinimalStructure -->
## 人格設定の最小構造

MVP では、1 つの人格設定は少なくとも次の構造を持てばよい。

```yaml
persona_id: persona:default
display_name: Default Persona
core_persona:
  self_image: ...
  core_values:
    - ...
  judgement_tendencies:
    - ...
  relation_baseline: ...
  initiative_baseline: ...
expression_style:
  tone: ...
  sentence_length: ...
  emotional_expressiveness: ...
  directness: ...
  cadence: ...
  initiative_expression: ...
```

ここでは、厳密なキー名や列挙値までは固定しない。
ただし、少なくとも `core_persona` と `expression_style` を分けて持つことは固定する。

<!-- Excluded -->
## MVP で人格設定に入れないもの

MVP では、次は人格設定に入れない。

- 相手ごとの詳細スタンス表
- 会話用人格と行動用人格の別定義
- 感情状態そのもの
- 記憶からの自動人格書き換えルール
- 場面ごとの細かな例外テンプレート

これらを先に入れると、人格設定と記憶の境界が崩れやすい。

<!-- UpdatePolicy -->
## 人格更新の扱い

人格更新は、記憶更新とは別に扱う。
MVP では、人格更新の単位は人格設定全体の差し替えとする。

つまり、次の方針を取る。

- 記憶から人格設定を自動更新しない
- `core_persona` の一部だけをランタイム中に patch しない
- `expression_style` の一部だけをその場で恒久更新しない
- 人格を変えるときは、人格設定全体を新しい版として扱う

比較やロールバックを単純に保つため、MVP では人格更新時に新しい `persona_id` を切る前提でよい。
既存人格を細かく上書きするより、どの版を選んでいるかを明確にしたほうがデバッグしやすい。

<!-- DesignDecision -->
## ここで固定したい設計判断

この文書で固定したいのは次である。

- 人格設定は `core_persona` と `expression_style` の二層に分ける
- 行動判断傾向は `core_persona` に置く
- 口調や感情の出し方は `expression_style` に置く
- `wake_policy` は判断機会を制御し、`initiative_baseline` はその機会で前に出やすい傾向を持つ
- `emotional_expressiveness` は安定した表出傾向を持ち、`affect_state` はその時点の感情状態を持つ
- `relation_baseline` は他者一般への基本姿勢を持ち、個別相手との関係変化は関係記憶で扱う
- ユーザー固有の関係変化や最近の気分は記憶側で扱う
- 記憶は人格設定の中核を自動更新しない
- 人格更新は MVP では人格設定全体の差し替えとして扱う
