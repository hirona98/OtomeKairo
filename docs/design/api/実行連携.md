# 実行連携

## 適用範囲

この文書は capability 実行要求と実行結果の wire 契約を扱う。
実行連携面の上位責務は [../integration/外部接点とAPI概念.md](../integration/外部接点とAPI概念.md) を正とする。
capability manifest の正本は [../capability/capability_manifest.md](../capability/capability_manifest.md) を正とする。
実装状態は `src/` と smoke 結果を正とする。

## 基本原則

- 実行連携の基本単位は capability とする
- capability manifest は server が持つ安定契約とする
- `hello.caps` は接続中 client の capability binding 候補だけを表す
- capability の request と result は別契約として扱う
- request を受ける client は manifest、`hello.caps` から導出した binding、`capability_state`、権限で決める
- 配送先の `target_client_id` は server の内部実行記録で持ち、wire request payload に重複して載せない
- `ongoing_action` は server 内部の継続単位として扱い、wire request payload に載せない
- 実行結果は新しい入力または観測結果として本体へ戻す

ここでいう capability には、各種センサを使った観測能力、物理デバイスへの作用能力、外部サービス利用能力を含む。
設計正本では capability 契約を固定し、具体的なハードウェア名や接続機器名は固定しない。

## manifest / binding / state

capability 実行では次を分ける。

| 項目 | 役割 |
|------|------|
| `CapabilityManifest` | capability の静的な契約 |
| `CapabilityBinding` | capability id / version と接続中 client の対応 |
| `CapabilityState` | 利用可否、権限不足、直近失敗、一時停止などの動的状態 |

これらの意味規則、`hello.caps` との関係、availability 判定は [../capability/capability_manifest.md](../capability/capability_manifest.md) を正とする。
この文書では、判定済みの capability request を stream で配送し、result endpoint で照合する wire 契約だけを定める。
LLM と外部 client には `target_client_id`、資格情報、内部配送情報を渡さない。

## capability 実行の最小概念

capability 実行要求の wire 契約は少なくとも次を持つ。

- `request_id`
- `capability_id`
- capability 固有の入力 payload
- `timeout_ms`

server の内部実行記録は少なくとも次を持つ。

- `request_id`
- `target_client_id`
- `capability_id`
- `timeout_ms`
- 継続中の能力実行に属する場合だけ `action_id`

capability 実行結果は少なくとも次を持つ。

- `request_id`
- `client_id`
- `capability_id`
- `result`
- `error`

能力そのものが観測方針や自律方針を内包しない。
result の照合では、`request_id` に紐づく内部実行記録の `target_client_id` と、返ってきた `client_id` を比較する。

## HTTP response への反映

会話面または自律面の判断で capability 実行を開始した場合、呼び出し元の HTTP response は `result_kind=capability_request` を返す。
実際の capability request は `events/stream` で配送する。

`decision_generation` が capability 実行を選ぶときの内部 decision は次の形を返す。

```json
{
  "kind": "capability_request",
  "reason_code": "capability:vision.capture",
  "reason_summary": "現在の画面状態を観測する必要がある。",
  "requires_confirmation": false,
  "pending_intent": null,
  "capability_request": {
    "capability_id": "vision.capture",
    "input": {
      "vision_source_id": "vision_source:main_display",
      "mode": "still"
    }
  }
}
```

- `capability_request` は `kind=capability_request` のときだけ object にする
- `capability_request` は `capability_id` と `input` だけを持つ
- `input` は manifest の `input_schema` に一致させる
- `target_client_id`、資格情報、内部 URL、transport 詳細は decision に含めない
- server は decision を受けた後、manifest、binding、`capability_state`、schema、`ongoing_action` を検証してから配送する
- `vision.capture` と `camera.ptz` は `input.vision_source_id` から実行先 client と対象 source を一意に決める
- `camera.ptz` は対象 source が `kind=camera` で、`supported_controls.camera.ptz` に requested operation / amount が含まれる場合だけ配送する
- source 指定を持たない capability は、受理済み binding が 1 件だけの場合に配送対象にする。複数 binding がある capability は配送先が一意に決まらないため実行しない
- `capability_state.paused=true` または動的一時 unavailable の capability は、decision view だけでなく server の dispatch 境界でも配送しない

HTTP response の `capability_request` 要約は次を持つ。

| 項目 | 役割 |
|------|------|
| `request_id` | capability request の識別子 |
| `capability_id` | 実行する capability |
| `status` | `dispatched` |
| `timeout_ms` | server が設定した timeout |
| `readiness_digest` | manifest 由来の readiness と、input key の対応成否 |

この要約には `target_client_id`、資格情報、内部 URL、transport 詳細を含めない。
配送先 client と内部実行記録は server 内部で保持する。
capability result は `request_id` を使って server が照合し、新しい入力または観測結果として後続の判断、状態、記憶、inspection へ戻す。
同期 result だけでなく accepted async result も、新しい入力または観測結果として shared pipeline へ戻す。

accepted async result は、`request_id`、`target_client_id`、`result_schema` を照合してから新しい判断サイクルとして扱う。
紐づく `ongoing_action` がある場合は、follow-up の結果に応じて `continued / on_hold / finished / interrupted` のいずれかへ遷移させる。
follow-up 判断には `CapabilityResultContext` を渡す。
source capability と異なる追加 `capability_request` は原則 repair 対象にするが、manifest の `state_policy.allow_followup_capability_requests` に一致する連鎖は許可する。
初期版では `camera.ptz` result follow-up から同じ `vision_source_id` の `vision.capture` request を出せる。
follow-up が `speech` になった場合は、source request chain に保存された外向き応答先 client へ `events/stream` の `assistant_message` event を `source_kind=capability_result` で送る。
外向き応答先 client は `event_subscriptions` に `assistant_message` を宣言している必要がある。
server は capability result を返した client を assistant message 表示先として使わない。

inspection には dispatch、result、follow-up、`ongoing_action` 遷移を比較できる短い要約を残す。
trace の field 名や表示項目の網羅は実装と inspection 文書を正とし、この文書では wire 契約と遷移語彙だけを固定する。
public result endpoint の通信仕様は [README.md](README.md) と `docs/design/api/` 配下を正とする。
実装状態は `src/` と smoke 結果を正とする。
dispatch failure、timeout、accepted async result は `state_policy` に従ってこの `capability_state` を更新する。

## capability state 操作

`capability_state.paused` は、server 側で新規 dispatch を止める runtime 操作である。
pause は manifest や binding を変更しない。
pause 中も既に配送済みの in-flight result は通常どおり照合し、follow-up pipeline へ戻す。

### `PATCH /api/capabilities/{capability_id}/state`

- 認証: 必要
- 役割: 指定 capability の runtime state を更新する
- 現行で更新できる field は `paused` だけである
- `capability_id` は path で指定し、request body へ重複して載せない

request:

```json
{
  "paused": true
}
```

response:

```json
{
  "ok": true,
  "data": {
    "generated_at": "2026-04-10T09:00:00+09:00",
    "capability": {
      "capability_id": "external.status",
      "manifest_version": "1",
      "kind": "external_service",
      "available": false,
      "unavailable_reason": "paused",
      "binding": {
        "status": "bound",
        "eligible_client_count": 1,
        "bound_client_ids": ["console-..."]
      },
      "permissions": {
        "required": [],
        "missing": []
      },
      "state": {
        "paused": true,
        "busy": false,
        "busy_request_id": null,
        "busy_action_id": null,
        "last_failure_at": null,
        "last_failure_summary": null,
        "last_result_at": null,
        "last_result_summary": null,
        "unavailable_active": false,
        "unavailable_reason": null,
        "unavailable_until": null,
        "parallel_blocked_by_action_id": null
      }
    }
  }
}
```

主な失敗:

| HTTP | `error.code` | 意味 |
|------|--------------|------|
| `400` | `invalid_capability_id` | `capability_id` が空または文字列ではない |
| `400` | `invalid_capability_state` | body が `paused` 以外の field を含む、または `paused` を含まない |
| `400` | `invalid_capability_paused` | `paused` が boolean ではない |
| `404` | `capability_not_found` | 指定した `capability_id` が manifest に存在しない |

## `ongoing_action` と capability request

`ongoing_action` は、最終目的と 1 回の capability request のあいだをつなぐ server 内部の継続単位である。
wire 契約の基本単位は capability request のまま固定する。

1 件の `ongoing_action` は、複数回の capability request を束ねる単位である。
request を発行するときは、必要時だけ内部実行記録へ `action_id` を紐づける。
result を受けたときは、`request_id` から内部実行記録を引き、その `action_id` がある場合だけ対応する `ongoing_action` を更新または終了する。
follow-up 判断が追加 request を返した場合は同じ `action_id` を継続し、terminal または hold に落ちた場合だけ `ongoing_action` を閉じる。
process startup 時点では capability request の内部照合表と event stream 配送状態が空になるため、未失効の `waiting_result` `ongoing_action` は結果を照合できない orphan として削除する。
削除した orphan は新規 capability request の並列制限に使わない。

## 現行の具体 capability: `vision.capture`

現行ラインの最初の具体 capability は `vision.capture` である。
現行ラインでは raw image payload 自体は保持せず、`input_interpretation` による詳細な視覚説明へ変換して判断、想起、視覚記録へ入れる。

### server -> client: `vision.capture_request`

- 経路: `GET /api/events/stream`
- 認証: `events/stream` の認証に従う
- 役割: `vision.capture` capability の実行要求を送る

message:

```json
{
  "event_id": 0,
  "type": "vision.capture_request",
  "data": {
    "request_id": "vision_capture_request:...",
    "capability_id": "vision.capture",
    "vision_source_id": "vision_source:main_display",
    "source_kind": "desktop",
    "source_label": "メイン画面",
    "mode": "still",
    "timeout_ms": 5000
  }
}
```

## client -> server: `POST /api/capability/result`

- 認証: 必要
- 役割: accepted async result を capability family 共通 outer shape で返す

request:

```json
{
  "request_id": "capability_request:...",
  "client_id": "console-...",
  "capability_id": "vision.capture",
  "result": {
    "images": ["data:image/png;base64,..."],
    "client_context": {
      "vision_source_id": "vision_source:main_display",
      "source_kind": "desktop",
      "source_label": "メイン画面",
      "active_app": "Slack",
      "window_title": "general | Slack",
      "locale": "ja-JP"
    },
    "error": null
  }
}
```

- `capability_id` は manifest に存在する capability id と一致させる
- `result` は capability ごとの `result_schema` に一致させる
- `request_id / client_id / capability_id` は outer shape にだけ置き、capability 固有の項目は `result` object 側へ寄せる
- `result.error` を持つ capability では、値がないときは `null` または省略を使う
- server は `request_id`、`target_client_id`、`capability_id`、`result_schema` を capability 共通経路で照合する

response:

```json
{
  "ok": true,
  "data": {}
}
```

遅延した capability result が来た場合は無視して成功で返す。
accepted された capability result は、HTTP response 自体は空のままでも、server 内部の共通 async capability-result pipeline を非同期実行する。

主な失敗:

| HTTP | `error.code` | 意味 |
|------|--------------|------|
| `400` | `invalid_request_id` | `request_id` が不正 |
| `400` | `invalid_client_id` | `client_id` が不正 |
| `400` | `invalid_capability_id` | `capability_id` が不正、または未知 |
| `400` | `invalid_result_payload` | `result` が object ではない |
| `400` | `invalid_capability_result` | `result` が capability の `result_schema` や追加検証に合わない |
| `409` | `capability_result_client_id_mismatch` | `request_id` に紐づく内部実行記録の `target_client_id` と `client_id` が一致しない |

### `vision.capture` result object 例

- `result.images` は 0 件以上、最大 1 件の画像 Data URI 配列とする
- `result.client_context` は result schema 上は object または `null` とする。値がないときは省略する
- `result.error` は string または `null` とする。値がないときは省略する
- `vision.capture` の accepted pending result では `result.client_context.vision_source_id / source_kind / source_label` を必須とし、request id に紐づく server 側 source metadata と完全一致させる
- `result.client_context.active_app / window_title / locale` は観測結果の判断補助に使う
- server は raw `images` を永続化せず、必要な場合だけ詳細な視覚説明へ変換して `observation_summary`、判断入力、視覚記録へ反映する

```json
{
  "status": "completed",
  "operation": "move_up",
  "amount": "small",

```

- `result.status` は `completed / rejected / failed` のいずれかにする

- server は `camera.ptz` result だけから `world_state.visual_context` を更新しない
- `camera.ptz` result follow-up では、同じ `vision_source_id` の `vision.capture` request を発行できる

## 拡張ルール

新しい capability を追加するときは、少なくとも次を同じ変更内で定める。

- `CapabilityManifest`
- `hello.caps` に出す capability 識別子と version
- `events/stream` で送る request message
- 対応する result request / response
- binding の選択規則
- `capability_state` に残す動的状態
- 必要権限
- capability 結果を観測面へ戻すか、実行結果として扱うか
- inspection に残す要約項目

センサや物理デバイスを追加するときも、個別機器の説明を上位設計へ直接入れず、まず capability 契約として定義する。
