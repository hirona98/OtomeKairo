# 実行連携

## 適用範囲

この文書は capability 実行要求と実行結果の wire 契約を扱う。
実行連携面の上位責務は [../11_外部接点とAPI概念.md](../11_外部接点とAPI概念.md) を正とする。
capability manifest の正本は [../17_capability_manifest.md](../17_capability_manifest.md) を正とする。
現在地は [../../plan/01_現行計画.md](../../plan/01_現行計画.md) を正とする。

## 基本原則

- 実行連携の基本単位は capability とする
- `desktop_watch` は capability ではなく観測方針とする
- capability manifest は server が持つ安定契約とする
- `hello.caps` は接続中 client の capability binding 候補だけを表す
- capability の request と result は別契約として扱う
- request を受ける client は manifest、`hello.caps` から導出した binding、`capability_state`、権限で決める
- 配送先の `target_client_id` は server の内部実行記録で持ち、wire request payload に重複して載せない
- `ongoing_action` は server 内部の継続単位として扱い、wire request payload に載せない
- 実行結果は新しい入力または観測結果として本体へ戻す

ここでいう capability には、各種センサを使った観測 capability、物理デバイスへの作用 capability、外部サービス利用 capability を含む。
設計正本では capability 契約を固定し、具体的なハードウェア名や接続機器名は固定しない。

## manifest / binding / state

capability 実行では次を分ける。

| 項目 | 役割 |
|------|------|
| `CapabilityManifest` | capability の静的な契約 |
| `CapabilityBinding` | capability id / version と接続中 client の対応 |
| `CapabilityState` | 利用可否、権限不足、直近失敗、一時停止、cooldown などの動的状態 |

これらの意味規則、`hello.caps` との関係、availability 判定は [../17_capability_manifest.md](../17_capability_manifest.md) を正とする。
この文書では、判定済みの capability request を stream で配送し、result endpoint で照合する wire 契約だけを定める。
LLM と外部 client には `target_client_id`、資格情報、内部配送情報を渡さない。

## capability 実行の最小概念

capability 実行要求の wire 契約は少なくとも次を持つ。

- `request_id`
- `capability_id`
- capability 固有の入力 payload
- `timeout_ms`

server の内部実行記録は少なくとも次を持つ。

- `request_id`
- `target_client_id`
- `capability_id`
- `timeout_ms`
- 継続中の能力実行に属する場合だけ `action_id`

capability 実行結果は少なくとも次を持つ。

- `request_id`
- `client_id`
- capability 固有の結果 payload
- `error`

`desktop_watch` のような観測方針は、観測実行条件を満たしたときに capability を呼ぶ。
能力そのものが観測方針や自律方針を内包しない。
result の照合では、`request_id` に紐づく内部実行記録の `target_client_id` と、返ってきた `client_id` を比較する。

## HTTP response への反映

会話面または自律面の判断で capability 実行を開始した場合、呼び出し元の HTTP response は `result_kind=capability_request` を返す。
実際の capability request は `events/stream` で配送する。

`decision_generation` が capability 実行を選ぶときの内部 decision は次の形を返す。

```json
{
  "kind": "capability_request",
  "reason_code": "capability:vision.capture",
  "reason_summary": "現在の画面状態を観測する必要がある。",
  "requires_confirmation": false,
  "pending_intent": null,
  "capability_request": {
    "capability_id": "vision.capture",
    "input": {
      "source": "desktop",
      "mode": "still"
    }
  }
}
```

- `capability_request` は `kind=capability_request` のときだけ object にする
- `capability_request` は `capability_id` と `input` だけを持つ
- `input` は manifest の `input_schema` に一致させる
- `target_client_id`、資格情報、内部 URL、transport 詳細は decision に含めない
- server は decision を受けた後、manifest、binding、`capability_state`、schema、`ongoing_action` を検証してから配送する
- 現行 dispatcher は受理済み binding が 1 件だけの capability を配送対象にする。複数 binding がある capability は配送先が一意に決まらないため実行しない

HTTP response の `capability_request` 要約は次だけを持つ。

| 項目 | 役割 |
|------|------|
| `request_id` | capability request の識別子 |
| `capability_id` | 実行する capability |
| `status` | `dispatched` |
| `timeout_ms` | server が設定した timeout |

この要約には `target_client_id`、資格情報、内部 URL、transport 詳細を含めない。
配送先 client と内部実行記録は server 内部で保持する。
capability result は `request_id` を使って server が照合し、新しい入力または観測結果として後続の判断、状態、記憶、inspection へ戻す。
同期 result だけでなく accepted async result も、新しい入力または観測結果として shared pipeline へ戻す。
accepted async result は、`request_id`、`target_client_id`、`result_schema` を照合し、紐づく `ongoing_action` をいったん `active` に戻したうえで新しい判断サイクルとして走る。
その follow-up が `capability_request` なら同じ `action_id` を `continued` として再 dispatch し、`pending_intent` なら `on_hold` として閉じる。
`reply / noop` なら terminal として閉じ、error result を受けた terminal は `interrupted` に寄せる。
`ongoing_action_transition_summary` では、少なくとも `transition_sequence / final_state / reason_code / reason_summary / transition_source` を共通項目として持つ。
follow-up 判断がある terminal では `decision_kind` を、result error が絡む遷移では `result_error` を、判断理由や failure 要約を残したいときは `detail_summary` を持ってよい。
inspection の `result_trace.capability_dispatch_summary` では、少なくとも `capability_id / capability_kind / request_summary / transition_summary` を共通 outer shape として持ってよい。
inspection の `result_trace.capability_result_followup_summary` では、少なくとも `capability_id / capability_kind / source_request_summary / observation_summary / decision_summary / followup_result_summary / transition_summary` を共通 outer shape として持ってよい。
その follow-up が `reply` になった場合は、同じ client へ `events/stream` の `capability_result` event を送る。
public result endpoint の具体一覧と現在地は `14_API仕様.md` と [../../plan/01_現行計画.md](../../plan/01_現行計画.md) を正とする。
dispatch failed や同期 wait の timeout は、対応する cycle がある場合 `capability_dispatch_summary.request_summary.status` を `dispatch_failed / request_timeout` として inspection から比較できるようにする。

## `ongoing_action` と capability request

`ongoing_action` は、最終目的と 1 回の capability request のあいだをつなぐ server 内部の継続単位である。
wire 契約の基本単位は capability request のまま固定する。

1 件の `ongoing_action` は、複数回の capability request を束ねる単位である。
request を発行するときは、必要時だけ内部実行記録へ `action_id` を紐づける。
result を受けたときは、`request_id` から内部実行記録を引き、その `action_id` がある場合だけ対応する `ongoing_action` を更新または終了する。
follow-up 判断が追加 request を返した場合は同じ `action_id` を継続し、terminal または hold に落ちた場合だけ `ongoing_action` を閉じる。

## 現行の具体 capability: `vision.capture`

現行ラインの最初の具体 capability は `vision.capture` である。
`desktop_watch` はこの capability を使って image payload を取得する。
現行ラインでは raw image payload 自体は保持せず、`input_interpretation` による短い観測要約へ変換して判断へ入れる。

### server -> client: `vision.capture_request`

- 経路: `GET /api/events/stream`
- 認証: `events/stream` の認証に従う
- 役割: `vision.capture` capability の実行要求を送る

message:

```json
{
  "event_id": 0,
  "type": "vision.capture_request",
  "data": {
    "request_id": "vision_capture_request:...",
    "capability_id": "vision.capture",
    "source": "desktop",
    "mode": "still",
    "timeout_ms": 5000
  }
}
```

### client -> server: `POST /api/vision/capture-response`

- 認証: 必要
- 役割: `vision.capture_request` の結果を返す

request:

```json
{
  "request_id": "vision_capture_request:...",
  "client_id": "console-...",
  "images": ["data:image/png;base64,..."],
  "client_context": {
    "active_app": "Slack",
    "window_title": "general | Slack",
    "locale": "ja-JP"
  },
  "error": null
}
```

- `images` は 0 件以上の Data URI 配列とする
- `client_context` は object または `null` とする。値がないときは省略する
- `error` は string または `null` とする。値がないときは省略する
- `client_context.active_app / window_title / locale` は観測結果の判断補助に使う
- server は `images` を永続化せず、必要な場合だけ短い観測要約へ変換して `observation_summary` と判断入力へ反映する

response:

```json
{
  "ok": true,
  "data": {}
}
```

遅延した capture response が来た場合は無視して成功で返す。
accepted された capture response は、HTTP response 自体は空のままでも、server 内部の共通 async capability-result pipeline を非同期実行してよい。

主な失敗:

| HTTP | `error.code` | 意味 |
|------|--------------|------|
| `400` | `invalid_request_id` | `request_id` が不正 |
| `400` | `invalid_client_id` | `client_id` が不正 |
| `400` | `invalid_images` | `images` が配列でない、または要素が不正 |
| `400` | `invalid_client_context` | `client_context` が object ではない |
| `400` | `invalid_capture_error` | `error` が string または `null` ではない |
| `409` | `capture_client_id_mismatch` | `request_id` に紐づく内部実行記録の `target_client_id` と `client_id` が一致しない |

## 現行の具体 capability: `external.status`

`external.status` は、外部サービスの現在状態を短い要約として返す external-service capability である。
raw external payload 自体は保持せず、`status_text` と必要な短い `client_context` summary だけを shared pipeline と inspection に戻す。

### server -> client: `external.status_request`

- 経路: `GET /api/events/stream`
- 認証: `events/stream` の認証に従う
- 役割: `external.status` capability の実行要求を送る

message:

```json
{
  "event_id": 0,
  "type": "external.status_request",
  "data": {
    "request_id": "external_status_request:...",
    "capability_id": "external.status",
    "service": "calendar",
    "timeout_ms": 5000
  }
}
```

### client -> server: `POST /api/external/status-response`

- 認証: 必要
- 役割: `external.status_request` の結果を返す

request:

```json
{
  "request_id": "external_status_request:...",
  "client_id": "console-...",
  "status_text": "calendar は正常に応答している。",
  "client_context": {
    "external_service_summary": "calendar は正常に応答している。"
  },
  "error": null
}
```

- `status_text` は空でない短い結果要約とする
- `client_context` は object または `null` とする。値がないときは省略する
- `error` は string または `null` とする。値がないときは省略する
- `client_context.external_service_summary` は外部サービス状態の `world_state` source pack に使ってよい

response:

```json
{
  "ok": true,
  "data": {}

```

## 拡張ルール

新しい capability を追加するときは、少なくとも次を同じ変更内で定める。

- `CapabilityManifest`
- `hello.caps` に出す capability 識別子と version
- `events/stream` で送る request message
- 対応する result request / response
- binding の選択規則
- `capability_state` に残す動的状態
- 必要権限
- capability 結果を観測面へ戻すか、実行結果として扱うか
- inspection に残す要約項目

センサや物理デバイスを追加するときも、個別機器の説明を上位設計へ直接入れず、まず capability 契約として定義する。
