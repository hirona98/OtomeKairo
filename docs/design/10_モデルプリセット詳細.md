# モデルプリセット詳細

## 基本前提

OtomeKairo では、モデル利用をその場の都度指定するのではなく、本体が保持する設定定義として管理する。

この前提により、次が成立する。

- モデル利用条件を一か所で読みやすい
- 人格設定や記憶と同じく、選択値で切り替えられる
- 役割ごとの責務を分けたまま設定できる

OtomeKairo では、モデルプリセットが役割ごとのモデル定義をそのまま持つ設定単位である。

## 論理役割

モデル役割は、少なくとも次の論理単位に分けて考える。

| 役割 | 主な責務 |
|------|------|
| 観測解釈 | 観測や想起入口を整理する |
| 意図判断 | 何をするべきかを判断する |
| 表現生成 | 伝達内容を整える |
| 記憶解釈 | 記憶更新や再整理に必要な解釈を行う |
| 埋め込み | ベクトル化を行う |

ここで重要なのは、論理役割とモデル実体を同一視しないことである。
同じモデルを複数役割で使いたい場合は、その role に同じ定義を書いてよい。

## `selected_model_preset_id`

現在どのモデルプリセットを使うかは `selected_model_preset_id` で表す。
これは現在設定に置き、選択された時点で現在使っているモデルプリセットとして扱う。

## モデルプリセットの最小構造

モデルプリセット（`model_preset`）は、少なくとも次を持つ。

```yaml
model_preset_id: model_preset:default
display_name: Default Preset
roles:
  observation_interpretation:
    kind: generation
    provider: provider_name
    model: model_name
    endpoint_ref: endpoint:primary
    api_key: ""
    reasoning_effort: low
  decision_generation:
    kind: generation
    provider: provider_name
    model: model_name
    endpoint_ref: endpoint:primary
    api_key: ""
  expression_generation:
    kind: generation
    provider: provider_name
    model: model_name
    endpoint_ref: endpoint:primary
    api_key: ""
  memory_interpretation:
    kind: generation
    provider: provider_name
    model: model_name
    endpoint_ref: endpoint:primary
    api_key: ""
  embedding:
    kind: embedding
    provider: provider_name
    model: embedding_model_name
    endpoint_ref: endpoint:primary
    api_key: ""
```

モデルプリセットは role ごとの定義を直接持つ。
役割ごとの接続先や資格参照を別概念へ逃がさず、この 1 件を見ればそのプリセットの意味が分かる形を優先する。

## role ごとのモデル定義

各 role は、少なくとも次の概念を持つ。

- `kind`
  - `generation` または `embedding`
- `provider`
- `model`
- `endpoint_ref`
- `api_key`

生成系 role では、必要に応じて `reasoning_effort` を持ってよい。

ここで重要なのは、実キー値をログやサンプルへ流し込まないことである。
`api_key` は model_preset の一部として持つが、表示や記録ではそのまま露出させない。

## role 直書きにする理由

モデルプリセットの内部にさらに別概念を置くと、次が重くなる。

- どの役割がどの設定で動くかを読むこと
- 安定したモデルプリセットを単位として切り替えること
- 比較実験を追うこと

そのため、OtomeKairo では role ごとのモデル定義を `model_preset` に直接持たせる。
重複よりも、読んだときの意味の明快さを優先する。

## validator の原則

モデルプリセットの validator では、少なくとも次を確認する。

- 必須 role が存在すること
- 各 role が object として成立していること
- 各 role の `provider`、`model`、`endpoint_ref`、`api_key` が有効であること
- role と `kind` が整合すること
- 秘密情報を露出しない扱いになっていること
- プリセット全体として意味が崩れていないこと

ここでは、モデル品質の優劣を validator の責務にしない。
契約が成立しているかを確認する。

## 更新

モデルプリセットの更新は、人格設定や記憶の更新と同様に明示的な設定変更として扱う。

ここでは次を守る。

- 選択変更と設定定義更新を分ける
- 中途半端な部分更新を正本にしない

変更単位の原則は `08_設定変更.md` を正とする。

## 役割と外部接点の境界

モデルプリセットは、外向き API の都合より先に、OtomeKairo の内部責務に合わせて決める。
詳細な wire 契約は `14_API仕様.md` と `design/api/` 配下で定める。

一方で、接続経路や資格情報の信頼境界は `12_接続と権限境界.md` を正とする。
