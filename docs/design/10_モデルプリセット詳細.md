# モデルプリセット詳細

## 基本前提

OtomeKairo では、生成系のモデル利用をその場の都度指定するのではなく、本体が保持する設定定義として管理する。

この前提により、次が成立する。

- 生成系の役割ごとの設定を一か所で読める
- 人格設定や記憶集合と同じく、選択値で切り替えられる
- 比較したい生成構成を明示的な設定単位として扱える

ここでいうモデルプリセットは、生成に関わる role 群だけを束ねる設定単位である。
埋め込みモデルは `model_preset` に含めず、`memory_set` 側に置く。

## 論理役割

生成系のモデル役割は、少なくとも次の論理単位に分けて考える。

| 役割 | 主な責務 |
|------|------|
| 入力解釈 | 入力や想起入口を整理する |
| 想起選別 | `RecallPack` 候補の section 配置、採用候補、競合要約を決める |
| 想起根拠圧縮 | 選択済み `event` 群から短い根拠表現を作る |
| 保留候補選別 | eligible な `pending_intent` 候補群から今扱う 1 件または `none` を決める |
| 意図判断 | 何をするべきかを判断する |
| 表現生成 | 伝達内容を整える |
| 記憶解釈 | 記憶更新や再整理に必要な解釈を行う |
| 内省要約 | `reflective consolidation` の `summary_text` を自然文で要約する |

同じモデルを複数役割で使う場合は、その role に同じ定義を書く。

## `selected_model_preset_id`

現在どのモデルプリセットを使うかは `selected_model_preset_id` で表す。
これは現在設定に置き、選択された時点で現在使っている生成モデル構成として扱う。

## モデルプリセットの最小構造

モデルプリセット（`model_preset`）は、少なくとも次を持つ。

```yaml
model_preset_id: model_preset:default
display_name: Default Preset
prompt_window:
  recent_turn_limit: 30
  recent_turn_minutes: 3
roles:
  input_interpretation:
    model: openrouter/google/gemini-3.1-flash-lite-preview
    api_key: ""
    reasoning_effort: low
    max_output_tokens: 3000
    web_search_enabled: false
  decision_generation:
    model: openrouter/google/gemini-3.1-flash-lite-preview
    api_key: ""
    max_output_tokens: 3000
    web_search_enabled: false
  expression_generation:
    model: openrouter/google/gemini-3.1-flash-lite-preview
    api_key: ""
    max_output_tokens: 4000
    web_search_enabled: false
  memory_interpretation:
    model: openrouter/google/gemini-3.1-flash-lite-preview
    api_key: ""
    max_output_tokens: 3000
    web_search_enabled: false
  memory_reflection_summary:
    model: openrouter/google/gemini-3.1-flash-lite-preview
    api_key: ""
    max_output_tokens: 2000
    web_search_enabled: false
  event_evidence_generation:
    model: openrouter/google/gemini-3.1-flash-lite-preview
    api_key: ""
    max_output_tokens: 1500
    web_search_enabled: false
  recall_pack_selection:
    model: openrouter/google/gemini-3.1-flash-lite-preview
    api_key: ""
    max_output_tokens: 2000
    web_search_enabled: false
  pending_intent_selection:
    model: openrouter/google/gemini-3.1-flash-lite-preview
    api_key: ""
    max_output_tokens: 1000
    web_search_enabled: false
```

`prompt_window` は会話履歴をどこまでプロンプトへ含めるかを決める。
`recent_turn_limit` は件数上限、`recent_turn_minutes` は時間窓の上限である。

各 role は必要な場合だけ `api_base` を持つ。
`api_base` が未指定または空欄のときは、`model` の先頭から導ける既定の接続先を使う。

## role ごとのモデル定義

各 role は、少なくとも次の概念を持つ。

- `model`
- `api_key`
- `max_output_tokens`
- `web_search_enabled`

必要な場合は次も持つ。

- `api_base`
- `reasoning_effort`

`provider` は独立項目として持たない。
どの provider 系を使うかは、`model` の先頭部分から導く。

`web_search_enabled` は、OtomeKairo 自身が検索処理を持つことを意味しない。
生成 request に対して provider 側の検索機能を有効化するための通過設定として扱う。

ここで重要なのは、実キー値をログやサンプルへ流し込まないことである。
`api_key` は model_preset の一部として持つが、通常の読み取りや記録ではそのまま露出させない。

## role 直書きにする理由

モデルプリセットの内部にさらに別概念を置くと、次が重くなる。

- どの役割がどの設定で動くかを読むこと
- 安定した生成構成を単位として切り替えること
- 比較実験を追うこと

そのため、OtomeKairo では role ごとの生成モデル定義を `model_preset` に直接持たせる。
重複よりも、読んだときの意味の明快さを優先する。

## 埋め込み設定との境界

埋め込みモデルは記憶検索と記憶更新のために使う。
そのため、OtomeKairo では埋め込み設定を `model_preset` ではなく `memory_set` に属させる。

これにより、同じ生成系モデルプリセットを使いながら、記憶集合ごとに異なる埋め込み設定を持てる。
逆に、生成系の切り替えだけで埋め込み設定まで暗黙に変えない。

## 更新

モデルプリセットの更新は、人格設定や記憶集合の更新と同様に明示的な設定変更として扱う。

ここでは次を守る。

- 選択変更と設定定義更新を分ける
- 中途半端な部分更新を正本にしない

変更単位の原則は `08_設定変更.md` を正とする。

## 役割と外部接点の境界

モデルプリセットは、外向き API の都合より先に、OtomeKairo の内部責務に合わせて決める。
詳細な wire 契約は `14_API仕様.md` と `design/api/` 配下で定める。

一方で、接続経路や資格情報の信頼境界は `12_接続と権限境界.md` を正とする。
