# 設定モデル

## 基本前提

OtomeKairo では、人格設定、記憶、モデルプリセット、方針に関する設定の正本を本体側に置く。
外部接点はそれらを読む、選ぶ、変更要求を送ることはできるが、正本にはならない。

設定の永続正本は `config.db` に置く。
記憶、events、cycle trace、world_state、activity_state は `memory.db` に置く。
設定定義を `memory.db` に保存しない。

ここでいう設定とは、OtomeKairo が現在どう動くかを決める `現在設定` と、その選択対象になる `設定定義` を含む。
状態のように自然変化するものではなく、明示操作で変える本体条件として扱う。
現在の個そのものは設定ではなく、人格設定と記憶からその都度成立する判断主体として扱う。
`drive_state` のような派生内部状態も設定ではなく状態として扱う。

## 設定の層

設定は、少なくとも次の 2 層に分ける。

- 現在設定
  - 現在の選択値と方針
- 設定定義
  - 選択対象となる定義本体

## 現在設定

現在設定は、OtomeKairo がいまどういう設定で動くかを表す。
ここには少なくとも次の系統を含める。

- 現在選択中の人格設定
- 現在選択中の記憶集合
- 現在選択中のモデルプリセット
- 観測源ごとの有効化と取得方針
- 自律判断機会の付与方針

ここで重要なのは、「長く残るもの」ではなく「明示操作で変える現在条件」を設定に入れることだ。
保存期間の長さではなく、意味上いま何を使うか、どう動くかを優先する。

## 設定定義

設定定義は、現在設定の選択値が指す定義本体である。
少なくとも次を含む。

- `persona`
  - 持続的な人格設定の定義
  - `persona_prompt`、`initiative_baseline`、`expression_addon` を含む
- `memory_set`
  - 人格設定と独立に選べる記憶全体の単位
  - 記憶検索と記憶更新に使う埋め込み設定もここに持つ
  - 埋め込み設定は `model`、`api_base`、`api_key`、`embedding_dimension` を含む
- モデルプリセット（`model_preset`）
  - 生成系 role ごとのモデル割り当てと接続定義
  - 直近会話候補の取得窓もここに含める
- カメラ source 定義（`camera_source`）
  - OtomeKairo の視覚として採用する camera source の定義
  - `vision_source_id`、connector の `client_id`、有効/無効、表示名を含む
  - 接続先 host と camera account を含む
  - camera account は connector ではなく OtomeKairo 本体が保持する
  - RTSP / ONVIF / PTZ の詳細値は connector 実装の既定値として扱う
- MCP server 定義（`mcp_server`）
  - 汎用 MCP client connector が接続する MCP server の定義
  - `mcp_server_id`、connector の `client_id`、有効/無効、表示名、transport、起動 command、args、cwd、env を含む
  - MCP server の API key や token は `env` の設定値として OtomeKairo 本体が保持する
  - MCP tool catalog は接続時の runtime metadata として扱い、設定定義の正本にしない

これらは「現在どれを選んでいるか」という現在設定そのものではない。
現在設定と設定定義を分けることで、同じ定義群の中からいま使うものだけを切り替えられる。

`memory_set` は人格設定に従属しない。
同じ人格設定に対して複数の `memory_set` を持つ。
同じ `memory_set` を異なる人格設定と組み合わせて選ぶ。

## 選択値

選択値は、現在設定のうち「現在どれを使うか」を表す部分である。

少なくとも次を持つ。

- 選択中の人格設定
- 選択中の記憶集合
- 選択中のモデルプリセット

選択値は、定義本体のコピーではなく参照である。

## 現在設定と状態

選択値は現在設定に置く。
設定された内容は、そのまま現在有効な条件として扱う。

一方で、状態側の実行状態は、接続状態や進行中の行動のような稼働状況だけを持つ。
設定の選択値を別名で重複保持する場所にはしない。

たとえば人格設定、記憶、モデルプリセットは、選択された時点で現在使っている設定として扱う。

## 設定定義の種類

人格設定、記憶集合、モデルプリセットはすべて設定定義だが、役割は同じではない。

- 人格設定
  - 持続的な姿勢、自発傾向、判断傾向を与える
- 記憶集合
  - どの経験蓄積全体を現在の個の構成要素として使うかを決める
  - その集合に対応する埋め込み設定も持つ
- モデルプリセット
  - どの生成 role をどのモデル定義で支えるかを決める

一方で、観測や自律判断の方針は `persona` のような選択対象ではなく、現在設定に直接入る項目として扱う。
定期起床処理は `enabled=true` の `camera_source` を実行時に観測対象へ含める。
`camera_source` の更新は `wake_policy.observations` を書き換えない。

`memory_set` は、空の集合として新規作成するか、既存の `memory_set` 全体を複製して作成する。
`memory_set` の削除は、集合全体を対象にする設定定義削除として扱う。
個別の `event`、`episode`、`memory_unit`、感情記録、派生状態を設定操作として確認、承認、訂正、撤回、削除しない。
`memory_set` の初期化と export は提供しない。

人格設定の詳細は [人格設定詳細.md](人格設定詳細.md)、モデルプリセットの詳細は [モデルプリセット詳細.md](モデルプリセット詳細.md) を正とする。

## 設定に含めないもの

次は、設定と混同しない。

- 外界の現在位置や一時的な接続可否
- 実行中かどうかの稼働状態
- `drive_state` のような中期内部状態
- 保留意図のような短期内部状態
- クライアントのローカル編集状態
- デバッグ表示用の派生要約

設定は「明示操作で変える本体条件」であり、自然変化するものや表示都合のものは含めない。
