# 接続と権限境界

## 基本前提

OtomeKairo の接続設計では、接続確立、相手識別、認証、権限を分けて考える。
外部接点は本体へ接続して観測や変更要求を送るが、判断や設定の正本にはならない。

少なくとも次を前提にする。

- 通信経路は暗号化する
- 接続先がどの OtomeKairo かを識別できる
- 通常操作に入る前に認証境界を越える
- 権限は人格設定や記憶の意味と混ぜない

## bootstrap と通常面の分離

接続確立のための bootstrap は、通常の観測、構成、inspection とは別の前段として扱う。

bootstrap 面では、少なくとも次を行う。

- 接続先へ到達できるかを確かめる
- 相手の安定識別子を読む
- 利用可能な接続条件を知る
- 通常面へ入るための資格を得る

この分離により、通常 API の意味単位へ接続確立都合を持ち込まずに済む。

## サーバ識別

OtomeKairo 側は、外部接点から安定して識別できる情報を持つ。
少なくとも次を含める。

- `server_id`
- 表示名
- 対応 API 版

証明書や到達 URL は接続手段として重要だが、存在そのものの識別子とは分ける。

## 認証

認証は、「どの外部接点が通常面へ入れるか」を判定する責務である。
方式そのものは token、certificate、その他の資格のいずれかとするが、少なくとも次を満たす。

- bootstrap 後に通常面へ入る手段がある
- 通常面では認証状態を要求できる
- 資格の発行、更新、失効を扱える

認証方式の exact な wire 契約は API 仕様で定める。
ここで固定するのは、認証を通常操作の前提条件として切り出すことだ。

## 権限

権限は、認証済みの相手に何を許すかを表す。
少なくとも概念上は次を分けられるように考える。

- 状態参照
- 設定変更
- inspection 参照
- 実行要求の送受信
- 資格管理

現在どこまで細かく分けるかにかかわらず、権限の責務を人格設定や記憶の意味と混ぜない。

## 資格情報のライフサイクル

資格情報には、少なくとも次のライフサイクルがある。

- 発行
- 保存
- 更新
- 失効
- 復旧

ここで重要なのは、資格を失ったときや切り替えたときに、どの境界まで再登録が必要かを明示できることである。

## 秘密情報の扱い

秘密情報は、接続設定の一部であっても、平文の設計正本やログへ流し込まない。

少なくとも次を守る。

- ログやサンプルへ秘密値を残さない
- 不要に外部接点間で複製しない
- 表示系 API では必要に応じて伏せる

この原則は、モデルプリセット内の各 role に置かれた `api_key` や、外部接続の資格にも同様に適用する。

## 信頼境界

信頼境界の中心は OtomeKairo 本体に置く。

- 外部接点は観測と要求を送る
- OtomeKairo は認証と権限を確認する
- 現在設定、設定定義、記憶の正本は本体に残す
- inspection やログは読み取り境界として分ける

詳細な API 面の分割は `11_外部接点とAPI概念.md` を正とする。
