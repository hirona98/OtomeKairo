# 接続と権限境界

## 基本前提

OtomeKairo の接続設計では、接続確立、相手識別、認証、権限を分けて考える。
外部接点は本体へ接続して観測や変更要求を送るが、判断や設定の正本にはならない。

少なくとも次を前提にする。

- 通信経路は暗号化する
- 接続先がどの OtomeKairo かを識別できる
- 通常操作に入る前に認証境界を越える
- 権限は人格設定や記憶の意味と混ぜない

## bootstrap と通常面の分離

接続確立のための bootstrap は、通常の観測、構成、inspection とは別の前段として扱う。

bootstrap 面では、少なくとも次を行う。

- 接続先へ到達できるかを確かめる
- 相手の安定識別子を読む
- 利用可能な接続条件を知る
- 未発行状態で通常面へ入るための初回資格を得る

この分離により、通常 API の意味単位へ接続確立都合を持ち込まずに済む。
bootstrap 面は発行済み資格の再表示には使わない。
資格発行済み状態では通常面へ入るために認証を要求する。

## サーバ識別

OtomeKairo 側は、外部接点から安定して識別できる情報を持つ。
少なくとも次を含める。

- `server_id`
- 表示名
- 対応 API 版

証明書や到達 URL は接続手段として重要だが、存在そのものの識別子とは分ける。

## 認証

認証は、「どの外部接点が通常面へ入れるか」を判定する責務である。
方式そのものは token、certificate、その他の資格のいずれかとするが、少なくとも次を満たす。

- bootstrap 後に通常面へ入る手段がある
- 通常面では認証状態を要求できる
- 資格の発行、更新、失効を扱える

認証方式の exact な wire 契約は API 仕様で定める。
ここで固定するのは、認証を通常操作の前提条件として切り出すことだ。

## 権限

現行設計では、通常 API に対して管理権限を分けない。
認証済み console token は、設定、inspection、`editor-state`、記憶管理操作を含む通常 API の操作主体として扱う。

一方で、capability 実行権限は通常 API の認証とは別の概念として扱う。
capability 実行可否は capability manifest の `required_permissions` と、認証済み client または接続主体に付与された capability 実行権限を照合して決める。

少なくとも次を分ける。

- 通常 API の認証済み操作
- capability ごとの実行権限
- 資格の発行、更新、失効

権限の責務を人格設定や記憶の意味と混ぜない。
権限不足の capability は `capability_state` で実行不可として扱う。
権限不足を LLM の判断だけに委ねない。

## 資格情報のライフサイクル

資格情報には、少なくとも次のライフサイクルがある。

- 発行
- 保存
- 更新
- 失効
- 復旧

ここで重要なのは、資格を失ったときや切り替えたときに、どの境界まで再登録が必要かを明示できることである。
初回資格は未発行状態の bootstrap でだけ発行する。
発行済み資格の更新は認証済みの通常 API で行う。
資格喪失時の復旧は、未認証 bootstrap endpoint で既存資格を再表示する形にしない。
復旧を提供する場合は、ローカルでの明示的な手順として定義する。

## 秘密情報の扱い

秘密情報は、接続設定の一部であっても、平文の設計正本やログへ流し込まない。

少なくとも次を守る。

- ログやサンプルへ秘密値を残さない
- 不要に外部接点間で複製しない
- 表示系 API では秘密値を伏せる
- bootstrap、inspection、status、logs で既存 token 実値を返さない

この原則は、モデルプリセット内の各 role に置かれた `api_key` や、外部接続の資格にも同様に適用する。

`GET /api/config/editor-state` と `PUT /api/config/editor-state` の response は、この原則の明示的な例外である。
`editor-state` は `CocoroConsole` の認証済み編集 UI が設定定義を全体編集するための設定編集 bundle であるため、`model_presets.roles.*.api_key` と `memory_sets.embedding.api_key` の実値を含む。
この例外は `editor-state` に限定し、bootstrap、inspection、status、logs、通常の設定読み取り API へ広げない。

この例外では、少なくとも次を守る。

- 認証済み主体だけが呼び出す
- read と write の両方を audit に残す
- audit、logs、inspection、cycle trace には秘密値の実値を残さない
- response body を server log へ出さない
- `editor-state` 以外の response へ秘密値を転写しない
- client は受け取った秘密値を通常ログや debug 表示へ出さない

現行設計では、`editor-state` のための追加管理権限を設けない。

## 信頼境界

信頼境界の中心は OtomeKairo 本体に置く。

- 外部接点は観測と要求を送る
- OtomeKairo は認証と権限を確認する
- 現在設定、設定定義、記憶の正本は本体に残す
- inspection やログは読み取り境界として分ける

詳細な API 面の分割は `11_外部接点とAPI概念.md` を正とする。
