# 視覚機能

## 目的

視覚機能は、OtomeKairo が必要なタイミングで指定された視覚 source を 1 回観測し、画像意味理解を通して判断、inspection、想起、記憶整理へ反映する能力である。
視覚機能の capability id は `vision.capture` とする。

この文書は、視覚 source、`vision.capture`、起床前観測、視覚前景の意味境界を正本として定める。
画像意味理解の LLM 契約は [23_vision_capture画像意味理解.md](23_vision_capture画像意味理解.md) を正とする。
通常会話の添付画像は [24_対話画像入力.md](24_対話画像入力.md) を正とし、この文書の視覚機能には含めない。
capability manifest は [17_capability_manifest.md](17_capability_manifest.md)、API wire は [api/05_実行連携.md](api/05_実行連携.md)、`world_state` 反映は [22_world_state.md](22_world_state.md) を正とする。

## 境界

視覚機能に入れるものは次である。

- `vision.capture` による能動的な 1 回の視覚観測
- カメラ、画面、仮想画面、定期観測対象を表す `VisionSource`
- `vision_source_id` による観測対象の指定
- `wake_policy.observations` による定期起床前の観測
- 画像意味理解で得た詳細な視覚説明
- 現在判断に効く短い視覚前景

次は視覚機能に入れない。

- 通常会話の添付画像
- 観測対象を指定しない常時監視
- raw image payload の永続保存
- OCR 全文抽出
- UI 要素、座標、クリック対象の構造化
- 複数画像、動画、音声入力
- カメラや画面ごとの capability id
- 旧 `desktop_watch` の独立機能

## 基本方針

`vision.capture` は capability id として 1 つだけ持つ。
カメラ、画面、定期観測対象ごとに capability id を増やさない。
観測対象は `vision.capture.input.vision_source_id` で指定する。

server は `VisionSource` の正本を持つ。
client は stream hello で自分が実行できる `vision.capture` と、実行可能な `vision_sources` を通知する。
server は hello を材料に `VisionSource` を導出し、decision view と dispatch 境界で `vision_source_id` を検証する。

デスクトップウォッチ相当の動作は `wake_policy.observations` として扱う。
`visual_watch` は独立設定として持たない。
定期観測も通常の `vision.capture` と同じ画像意味理解、視覚記録、`world_state` 反映の経路を使う。

desktop capture と camera capture は同じ visual capture として扱い、保存可否で分けない。
`source_kind` は入力元ラベルであり、視覚説明の保存、`world_state.visual_context` 更新、記憶整理は観測内容、会話との結びつき、重要度、反復性で決める。
`desktop / camera / virtual` の vision source から得た観測は `source_owner=user_environment` として判断文脈へ渡す。
`source_owner=user_environment` の内容はユーザー側の状況であり、発話ではユーザー側の見え方として表現する。

## `VisionSource`

`VisionSource` は server が扱う視覚 source の正本である。
1 件の `VisionSource` は、少なくとも次を持つ。

| 項目 | 役割 |
|------|------|
| `vision_source_id` | 視覚 source の識別子 |
| `kind` | `desktop / camera / virtual` のいずれか |
| `label` | 判断と inspection に出す短い表示名 |
| `aliases` | 自然言語指定の解決に使う別名 |
| `default_for` | 曖昧な指定を解決できる範囲 |
| `client_id` | 実行先 client。LLM decision には渡さない |
| `capability_id` | `vision.capture` |
| `required_permissions` | source ごとの必要権限 |
| `available` | 現在利用できるか |

`vision_source_id` は server 内で一意にする。
`required_permissions` は manifest の `observe_vision` と合わせて照合する。
source が一意に定まらない場合、server は `vision.capture` を dispatch しない。

## 起床前観測

`wake_policy.observations` は定期起床の前に実行する観測項目である。
observation は観測 source そのものではなく、`vision.capture` を発行する条件である。
周期は `wake_policy.interval_seconds` の 1 つだけを使う。

1 件の observation は、少なくとも次を持つ。

| 項目 | 役割 |
|------|------|
| `observation_id` | 観測項目の識別子 |
| `enabled` | 定期起床前に実行するか |
| `capability_id` | `vision.capture` |
| `input.vision_source_id` | 対象の視覚 source |
| `input.mode` | 現行では `still` |

`enabled=false` の observation は取得しない。
enabled observation は定期起床ごとに実行する。
複数 observation がある場合も、server は取得処理を終えたあとに 1 回だけ wake 判断を行う。

保存済み observation の `vision_source_id` が接続中 source に存在しない場合、server は source kind と接続中 source の `kind/default_for` から一意に再解決する。
再解決候補が 0 件または複数件の場合、server は observation を失敗として扱い、別 source へ推測配送しない。

observation result は wake 判断の前に画像意味理解へ通す。
画像意味理解後の詳細説明は `visual_observation_record` として保存する。
観測内容から現在判断に効く短い状態を抽出できる場合は `world_state.visual_context` 更新へ通す。

## 視覚変化

起床前観測では、短い observation signature を使って同じ観測の繰り返しを抑制する。
observation signature は変化判定専用であり、raw image、画面全文 OCR、UI 座標を含めない。
signature は `vision_source_id / source_kind / source_label / visual_summary_text` を材料にする。
`window_title` は変化判定の正本にしない。

視覚変化は次の `change_state` に正規化する。

| `change_state` | 意味 |
|------|------|
| `first_seen` | この source の観測が初めて前景化した |
| `changed` | 前回観測から意味上の変化がある |
| `stable` | 前回観測と大きく変わらない |
| `same_as_recent_speech` | 直近で発話済みの内容と同じ |

`first_seen / changed` は wake 判断へ進む前景シグナルとして扱うが、speech 義務にはしない。
`stable / same_as_recent_speech` は繰り返し発話を避ける材料として扱う。
定期起床の自律判断では、視覚変化だけを speech 義務にしない。

## 判断利用

`vision.capture` は、現在の視覚状態が判断に必要な場合だけ使う。

使う条件は次である。

- ユーザーが現在見えている内容について質問した
- ユーザーが特定の画面、カメラ、表示内容、作業中の対象について質問した
- 自律判断で強い `drive_state` があり、対象 source の視覚前景が無いか古い
- `wake_policy.observations` に enabled observation があり、定期起床の実行時刻に達した
- capability result follow-up で同じ source の視覚情報が必要になった

使わない条件は次である。

- ユーザーが添付画像について質問しており、`conversation_attachment` の要約で足りる
- 同じ `vision_source_id` の新鮮な `world_state.visual_context` があり、ユーザーが明示的に再観測を求めていない
- 現在の判断に視覚情報が不要
- 対象 source が未接続、権限不足、busy、一時 unavailable
- source が一意に決まらない
- capability result follow-up が source capability と異なる capability request を誘発する

自然言語で source が指定された場合、server は `label / aliases / default_for` から `vision_source_id` へ解決する。
該当 source が 1 件に定まる場合だけ実行する。
該当 source が 0 件の場合は、利用できる source が無い旨を speech する。
該当 source が複数あり default で決まらない場合は、source を確認する speech を返す。
LLM が label や alias を直接 input に入れた場合、server は repair または validation failure として扱い、`vision_source_id` 以外を実行対象にしない。

## 視覚前景

`vision.capture` result は `source_kind` に関係なく `world_state.visual_context` へ反映する。
詳細な視覚説明そのものは `visual_observation_record` に保存し、`world_state.visual_context` には現在判断に効く短い状態要約を入れる。
統合単位は `vision_source_id` とする。

`visual_context` の integration key は `visual_context:<vision_source_id>` とする。
同じ source の新しい観測は同じ key の state を置換する。
異なる source の視覚前景は併存する。
判断入力へ出す foreground は salience、鮮度、source の default 性で並べる。
result follow-up の判断入力では、対象 `vision_source_id` と異なる `visual_context` を前景から外す。

## 保存と inspection

保存するものは次である。

- user / observation / speech event の短い text
- `cycle_trace.input_trace.observation_summary`
- `world_state.visual_context` の短い `summary_text`
- `vision_source_id / source_kind / source_label`
- capability request / result の audit 用要約
- inspection の capability state、vision source state、world state trace、source pack hook 要約

保存しないものは次である。

- raw image Data URI
- base64 本文
- 画面の全文 OCR
- UI 座標
- credential、内部 URL、token、配送先 client の詳細

inspection では、source、availability、permission、起床前観測の有効状態、request、result、画像意味理解、`world_state.visual_context` 反映、視覚変化の判定結果を追えるようにする。

## failure の扱い

主な failure は次のように扱う。

| failure | 扱い |
|---------|------|
| source 未登録 | decision repair または `internal_failure` |
| source が複数候補に分かれる | source 確認 speech |
| source client 未接続 | `vision.capture` を source unavailable として decision view へ出す |
| source 権限不足 | source を unavailable にする |
| dispatch failure | capability state と source state に一時 unavailable を残す |
| timeout | ongoing action を失敗終了し、一時 unavailable を残す |
| result の source が request と一致しない | `400 invalid_capability_result` |
| image interpretation 失敗 | capability result cycle を failure として残す |
| `world_state` 更新失敗 | 判断サイクル本体は継続し、`world_state_trace.failure_reason` に残す |

silent fallback で画像なし成功へ丸めない。
空画像 result は client が観測できなかった明示結果として扱い、`image_count=0` を残す。

## 運用調整

source の `label / aliases / default_for`、観測頻度、定期起床で観測結果を根拠に介入する条件、視覚説明の詳細度、inspection 表示は実運用ログを根拠に調整する。
調整項目は `vision.capture` の意味境界や API wire 契約を変更しない。
