# 検証基盤

## 目的

この文書は、OtomeKairo の検証手順と検証層の正本である。
通常作業では短時間で回る検証を標準にし、時間がかかる検証は必要時の手動確認に分ける。

## 検証層

| 層 | 目的 | 標準コマンド | 位置づけ |
| --- | --- | --- | --- |
| fast test | 契約、validation、境界処理を決定的に確認する | `.venv/bin/python -m pytest tests` | 通常作業で毎回実行する |
| mock smoke | mock LLM でサーバ全体の代表フローを確認する | `.venv/bin/python scripts/run_long_smoke.py --profile smoke` | 大きな変更前後に手動で実行する |
| real LLM smoke | 実 LLM の判断構造と trace shape を確認する | `.venv/bin/python scripts/run_long_smoke.py --profile real-llm-smoke --keep-artifacts` | 実 LLM 設定がある環境で必要時に実行する |

## 通常検証

通常検証は fast test だけにする。
目標実行時間は 1 分以内とする。

```bash
.venv/bin/python -m pytest tests
```

fast test には、LLM 呼び出し、サーバ常駐起動、長時間待機、外部サービス接続を入れない。
fast test は、仕様が固まっている機械的な境界を確認する。

対象は次である。

- config validation
- LLM structured output の契約検証
- capability schema validation
- event stream hello / capability binding
- 文字列ぶつ切り禁止
- 秘密情報を保存、ログ、inspection へ出さない境界

## 重い検証

mock smoke と real LLM smoke は通常検証へ入れない。
これらはサーバ起動、scheduler、memory worker、capability result follow-up、inspection trace を含むため、実行時間が長い。

mock smoke は、次の代表フローをまとめて確認する。

- 通常会話
- 定期起床
- capability request / result follow-up
- memory worker
- vision capture
- 記憶と想起の代表ケース

real LLM smoke は自然文の完全一致を検証しない。
合否は `summary.json` の verified flag と trace shape で判定する。
API key、token、camera account は artifact、ログ、サンプルへ出さない。

## 代表ケースの割り当て

設計済みの代表ケースは、次の基準で検証層へ割り当てる。

- 入出力が決定的で、LLM やサーバ起動が不要なものは fast test に置く
- 判断サイクル、scheduler、memory worker、event stream をまたぐものは mock smoke に置く
- 実 LLM の判断構造を確認するものは real LLM smoke に置く

自律判断 matrix、capability result follow-up、記憶品質、視覚観測は、当面 `scripts/run_long_smoke.py` を正とする。
fast test へ移すのは、仕様が固まった deterministic な境界に限定する。

## 合否基準

fast test は pytest の exit code を合否にする。
mock smoke と real LLM smoke は exit code と `summary.json` の verified flag を合否にする。
LLM の自然文表現、完全な出力文面、偶発的な語彙選択を合否にしない。
