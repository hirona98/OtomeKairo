# 接続と認証

<!-- Role -->
## この文書の役割

この文書は、`CocoroConsole` が OtomeKairo へどう接続し、どう認証されるかを定める正本である。

ここで固定したいのは次の内容である。

- `CocoroConsole` 起点での接続登録フロー
- HTTPS をどう前提に置くか
- 初回接続をどう成立させるか
- 通常 API 呼び出しで何を認証情報として使うか
- bootstrap 面が受け付ける最小操作
- 接続 bootstrap と通常 API の境界

ここでは、証明書発行ツールの詳細までは定めない。
bootstrap 面と通常 API 面の厳密な request / response は `14_API仕様.md` と `design/api/` 配下を正とする。
ただし、接続の成立条件と最小の認証モデルはここで固定する。

<!-- Premise -->
## 基本前提

MVP では、接続設計を `CocoroConsole` 側の使い方に合わせる。

前提は次のとおりである。

- クライアントは `CocoroConsole` のみとする
- 利用範囲はローカルネットワーク内に限定する
- 通信経路は HTTPS に固定する
- 初回接続時の証明書フィンガープリント確認ダイアログは出さない
- 接続の主導権は `CocoroConsole` 側に置く

ここで重要なのは、接続 UX を `CocoroConsole` に合わせつつも、設定と判断の正本は OtomeKairo 側に残すことである。

<!-- ConsoleFirst -->
## `CocoroConsole` 起点の接続

MVP では、接続は `CocoroConsole` で接続先 URL を登録するところから始める。

`CocoroConsole` は少なくとも次をローカルに保持してよい。

- `server_url`
- `display_name`
- `server_id`
- `certificate_fingerprint`
- `console_access_token`
- `last_connection_state`

このうち、`certificate_fingerprint` は観測情報として保持してよいが、接続可否の判定条件にはしない。
MVP では、フィンガープリントは表示やデバッグ補助のための情報に留める。

<!-- Https -->
## HTTPS の前提

OtomeKairo と `CocoroConsole` の通信は HTTPS に固定する。

MVP では、少なくとも次を前提にする。

- 通常 API も bootstrap API も HTTPS を使う
- HTTP 平文では接続しない
- 証明書は自己署名でもよい
- `CocoroConsole` は提示された証明書をそのまま受け入れて接続を継続してよい

この設計では、HTTPS の主目的は通信経路の暗号化である。
MVP では、証明書フィンガープリントの手動確認や厳密なピンニングは行わない。

<!-- ServerIdentity -->
## サーバ識別

OtomeKairo 側は、少なくとも次の識別情報を持って返せるようにしてよい。

- `server_id`
- `server_display_name`
- `api_version`

`server_id` は、`CocoroConsole` が「どの OtomeKairo に接続しているか」を識別するための安定 ID である。
MVP では、証明書ではなく、この `server_id` と登録済み URL を接続先識別の主軸に置く。

<!-- FirstConnect -->
## 初回接続フロー

MVP では、初回接続は次の流れでよい。

1. `CocoroConsole` で `server_url` を登録する
2. `CocoroConsole` が HTTPS で OtomeKairo の bootstrap 面へ接続する
3. `server_id`、`server_display_name`、証明書フィンガープリントを受け取る
4. `CocoroConsole` が `register_first_console()` を呼ぶ
5. OtomeKairo が現在の `console_access_token` を返す
6. `CocoroConsole` がそのトークンを保存し、以後の通常 API 呼び出しで使う

ここでは、初回接続のために別ブラウザや外部認証基盤は使わない。
`CocoroConsole` 単体で接続登録が完結することを優先する。

補足すると、証明書フィンガープリントは OtomeKairo が payload として返す値ではなく、`CocoroConsole` が HTTPS 接続時に観測して保持する情報として扱ってよい。

<!-- AuthModel -->
## 認証モデル

MVP では、通常 API の認証は OtomeKairo が発行する `console_access_token` で行う。

このトークンの性質は次のとおりである。

- OtomeKairo 側が発行する
- `CocoroConsole` が保持する
- 通常 API 呼び出しごとに送る
- 権限は MVP ではフルアクセスでよい

MVP では、ユーザー単位や権限ロール単位には分けない。
最初のクライアントが `CocoroConsole` 1 つであることを前提に、単純な 1 トークン方式を採る。

<!-- TokenLifecycle -->
## トークンのライフサイクル

MVP では、OtomeKairo 側は少なくとも 1 つの `console_access_token` を保持できればよい。

扱いは次のとおりである。

- 未発行状態では、新しいトークンを発行して返す
- 発行済み状態では、通常 API はそのトークンを必須にする
- 発行済み状態でも、`register_first_console()` は現在のトークンを返してよい
- トークン再発行は、認証済みの `CocoroConsole` から行ってよい

トークンを失った場合の復旧は、MVP ではサーバ側で接続情報を初期化して再登録する形で十分である。
複数クライアントの同時登録や複数ロール管理は入れない。

<!-- BootstrapBoundary -->
## bootstrap 面と通常 API の境界

接続 bootstrap は、`08_API概念.md` で定義している 4 面とは別の前段として扱う。

つまり、MVP の構造は次のようになる。

- bootstrap 面
  - 接続確認
  - サーバ識別情報の取得
  - 初回登録
  - トークン再発行
- 通常 API
  - 観測面
  - 状態面
  - 設定面
  - 列挙面

この分離により、接続確立の都合を通常 API の意味単位へ持ち込まずに済む。

<!-- BootstrapOps -->
## bootstrap 面の最小操作契約

MVP では、bootstrap 面の最小操作は次の 4 つでよい。

- `probe_bootstrap()`
- `read_server_identity()`
- `register_first_console()`
- `reissue_console_access_token()`

ここで重要なのは、これらが接続成立のための前段操作であり、人格、記憶、モデル設定の通常操作とは別責務であることだ。

<!-- BootstrapCommonRule -->
## bootstrap 面の共通ルール

bootstrap 操作では、次を共通ルールにする。

- すべて HTTPS 上で行う
- `probe_bootstrap` と `read_server_identity` は未認証で呼べてよい
- `register_first_console` は未認証で呼べてよく、未発行なら発行済みにし、発行済みなら現在値を返してよい
- `reissue_console_access_token` は認証済み状態でのみ受け付けてよい
- bootstrap 操作は設定正本や `runtime_state` を直接書き換えない

このルールにより、接続確立のための最小操作だけを、通常 API から切り分けて保てる。

<!-- ProbeBootstrap -->
## `probe_bootstrap()`

`probe_bootstrap()` は、`server_url` に OtomeKairo が存在し、bootstrap 面へ到達できるかを確認する操作である。

この操作で変えないのは次である。

- `console_access_token`
- OtomeKairo の永続設定
- OtomeKairo の設定資源

少なくとも次を返せればよい。

- bootstrap 面が利用可能か
- HTTPS 接続が成立したか
- 現在の bootstrap 可否

この操作は、接続確認だけを担う。
サーバ識別や登録は別操作に分ける。

<!-- ReadServerIdentity -->
## `read_server_identity()`

`read_server_identity()` は、接続先がどの OtomeKairo かを識別するための情報を読む操作である。

この操作で変えないのは次である。

- `console_access_token`
- OtomeKairo の永続設定
- OtomeKairo の設定資源

少なくとも次を返せればよい。

- `server_id`
- `server_display_name`
- `api_version`
- `console_access_token` が発行済みかどうか

`CocoroConsole` は、この結果と登録済み URL を使って接続先を識別してよい。

<!-- RegisterFirstConsole -->
## `register_first_console()`

`register_first_console()` は、`CocoroConsole` が通常 API に入るための `console_access_token` を受け取る操作である。

この操作で変えるのは次である。

- `console_access_token`

この操作で変えないのは次である。

- 人格設定
- 記憶設定
- モデル設定
- `runtime_state`

少なくとも次を確認する。

- bootstrap 面が有効であること
- 未発行ならトークンを発行できること

成功時は、現在有効な `console_access_token` を返してよい。
未発行状態では新規発行し、発行済み状態では現在値をそのまま返してよい。

<!-- ReissueToken -->
## `reissue_console_access_token()`

`reissue_console_access_token()` は、認証済みの `CocoroConsole` が、現在のトークンに代わる新しい `console_access_token` を発行する操作である。

この操作で変えるのは次である。

- `console_access_token`

この操作で変えないのは次である。

- 人格設定
- 記憶設定
- モデル設定
- `runtime_state`

少なくとも次を確認する。

- 現在の `console_access_token` で認証済みであること
- 新しいトークンを発行できること

成功時は、新しい `console_access_token` を返す。
旧トークンの失効方式は実装で 1 つに固定する。
MVP では、即時失効でも短い重複猶予付きでもよい。

<!-- BootstrapFailure -->
## bootstrap 面の失敗条件

bootstrap 操作では、少なくとも次を失敗として返せればよい。

- HTTPS 接続を確立できない
- bootstrap 面が無効または未到達
- `server_id` など必要な識別情報を返せない
- `register_first_console` が発行済み状態で呼ばれた
- `reissue_console_access_token` が未認証で呼ばれた
- トークン発行または保存に失敗した

ここでは、失敗時に bootstrap 操作の途中状態を中途半端に残さないことを優先する。

<!-- BootstrapExit -->
## bootstrap 完了条件

`CocoroConsole` から見て、bootstrap が完了したとみなしてよい条件は次である。

- `server_url` が登録済みである
- `server_id` を取得できている
- `console_access_token` を保持している
- 通常 API を認証付きで呼べる

この条件を満たしたら、以後は bootstrap 面ではなく通常 API 面を主に使う。

<!-- CocoroConsoleFit -->
## `CocoroConsole` に合わせる理由

この設計を取る理由は次のとおりである。

- URL を入れれば接続開始できる
- 初回接続を `CocoroConsole` 単体で完結できる
- ローカルネットワーク前提の簡素な UX にしやすい
- モデル設定や人格設定の編集まで同じクライアント体験にまとめやすい

MVP では、厳密な PKI 運用や多要素認証よりも、`CocoroConsole` から迷わず接続できることを優先する。

<!-- Excluded -->
## MVP でまだ入れないもの

MVP では、次は入れない。

- 証明書フィンガープリントの手動確認
- 証明書ピンニングによる接続拒否
- mTLS
- 外部 IdP ログイン
- 複数ユーザー権限
- 複数 `CocoroConsole` の同時管理

これらを先に入れると、`CocoroConsole` 起点の最小接続 UX が重くなりやすい。

<!-- DesignDecision -->
## ここで固定したい設計判断

この文書で固定したいのは次である。

- 接続は `CocoroConsole` で URL を登録するところから始める
- 通信経路はローカルネットワーク内の HTTPS に固定する
- 証明書フィンガープリントは無条件で受け入れ、接続可否の判定条件にはしない
- 初回登録は未発行状態の OtomeKairo に対して `CocoroConsole` 単体で完結できるようにする
- 通常 API の認証は `console_access_token` で行う
- bootstrap 面の最小操作は `probe_bootstrap`、`read_server_identity`、`register_first_console`、`reissue_console_access_token` に限定する
- bootstrap 面は通常 API の 4 面とは別の前段として扱う
