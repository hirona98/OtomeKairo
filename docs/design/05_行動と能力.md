# 行動と能力

## 基本方針

OtomeKairo は、会話だけでなく将来的な行動も同じ意思決定の延長として扱う。
そのため、会話応答と外界行動を別系統の仕組みに分けるのではなく、「何をするべきかを判断する」というひとつの枠組みで捉える。

記憶利用の共通本体と、そこから `reply / act / noop` へどう分岐するかは、`memory/03_想起と判断.md` で統合して扱う。

## 行動判断に渡す入力

行動判断では、記憶だけを見て能力を選ばない。
少なくとも次を分けて渡す前提にする。

- 記憶文脈
- 現在観測
- `world_state`
- `runtime_state`
- 能力マニフェスト
- `capability_state`

このうち、能力選択と実行可否に直結する境界は `memory/03_想起と判断.md` を正とする。

## LLM と能力の関係

行動判断は LLM に寄せる。
ただし、LLM が何でも自由に実行できる形にはしない。

ここでの基本原則は次のとおりである。

- コード側が、現在利用可能な能力を定義する
- LLM は、その能力の中から何を使うかを判断する
- 実行そのものはコード側が担う

つまり、OtomeKairo は「何ができるか」をコード側から知らされ、その範囲の中で意図を形成する。

## 能力マニフェスト

能力は、個別の実装詳細ではなく、OtomeKairo にとっての「行動可能性」として外部化する。

この設計により、次が成立する。

- OtomeKairo は、今できることを明示的に把握できる
- 能力が増減しても、判断構造を変えずに済む
- 応答内容から能力を推測させるのではなく、宣言された能力に基づいて判断できる

## 能力マニフェストの最小契約

能力マニフェストは、少なくとも次の概念を持つものとして扱う。

- 能力の識別子
- その能力で何ができるかの説明
- 現在利用可能かどうか
- 実行に必要な入力の概念
- 実行結果として返るものの概念
- 実行できなかった場合の結果の概念

LLM には、この契約に基づいて「今使える能力」とその意味を渡す。
一方で、実装詳細や内部接続方法までは見せない。

## MVP で扱う行動

MVP の主対象は会話である。

この段階では、次の考え方を取る。

- 最も重要な行動は、適切な会話を返すこと
- 自発的な会話開始も行動の一種として扱う
- 会話以外の能力は、将来の拡張先として位置付ける

## 自発行動

OtomeKairo は、ユーザー入力がないときでも判断機会を持つ。
ただし、自発行動は単なる定期処理ではなく、その時点の文脈を踏まえて必要性を判断した結果として起こるものとする。

また、判断の機会を与える間隔は OtomeKairo の設定として保持し、`CocoroConsole` から調整可能であることを前提にする。
この設定は判断機会を作るためのものであり、実際に前へ出やすいかどうかの人格傾向は人格設定側で扱う。

## MVP における自発性の定義

MVP では、自発性をまず会話主体のものとして定義する。

この段階では、次の考え方を取る。

- 定期起床は、発話そのものではなく判断機会である
- 起床のたびに、内部的には `reply / noop / future_act` を判断する
- 自発性の対象は、自発会話に限定する
- `future_act` は、今すぐ外へ出さない保留候補として内部保持してよい
- 外界への自律行動は将来拡張として扱う

判断材料としては、少なくとも次を含む。

- 前回会話からの経過
- 未完了の話題
- 記憶から得た嗜好や関係
- 現在の軽量クライアント文脈

この定義により、MVP では「会話主体としての自発性」を成立させることを目指す。

## 将来拡張の方向

将来的には、次のような能力を扱えるようにする。

- 観測能力
  - 画面、カメラ、各種入力
- 情報取得能力
  - Web や外部システムの参照
- 作用能力
  - 通知、操作、移動、外部サービス利用

ただし、拡張後も次の構造は維持する。

- 能力は宣言される
- LLM は能力の中から選ぶ
- 実行はコードが担う
