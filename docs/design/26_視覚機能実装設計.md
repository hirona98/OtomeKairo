# 視覚機能実装設計

## 目的

視覚機能は、OtomeKairo が必要なタイミングで指定された視覚 source を 1 回観測し、短い意味要約として判断、`world_state`、inspection へ反映する機能である。

この文書は、複数のカメラ、複数の画面、デスクトップウォッチ対象を同じ `vision.capture` capability で扱うための実装設計を定める。
画像意味理解の LLM 契約は [23_vision_capture画像意味理解.md](23_vision_capture画像意味理解.md) を正とする。
通常会話の添付画像は [24_対話画像入力.md](24_対話画像入力.md) を正とし、この文書の視覚機能には含めない。
capability manifest は [17_capability_manifest.md](17_capability_manifest.md)、API wire は [api/05_実行連携.md](api/05_実行連携.md)、`world_state` 反映は [22_world_state.md](22_world_state.md) を正とする。

## 用語

| 用語 | 意味 |
|------|------|
| 視覚機能 | `vision.capture` capability を使って現在の視覚状態を観測する機能 |
| 視覚 source | カメラ、画面、仮想画面、定期観測対象を表す `VisionSource` |
| `vision_source_id` | 視覚 source を一意に指定する id |
| 視覚観測 | 1 回の `vision.capture` request と、その result follow-up |
| 視覚要約 | raw image payload から作る `visual_summary_text` |
| 視覚前景 | 判断に効く現在の見えている対象を `world_state.visual_context` として持つ短期状態 |
| wake observation | `wake_policy.observations` に定義する interval wake 前の自動観測項目 |
| 会話添付画像 | `POST /api/conversation.images` で渡される画像。視覚機能ではない |

## 基本方針

`vision.capture` は capability id として 1 つだけ持つ。
カメラ、画面、定期観測対象ごとに capability id を増やさない。
観測対象は `vision.capture.input.vision_source_id` で指定する。

server は `VisionSourceRegistry` を持つ。
client は stream hello で自分が実行できる `vision.capture` と、実行可能な `vision_sources` を通知する。
server は hello を材料に `VisionSource` を導出し、decision view と dispatch 境界で `vision_source_id` を検証する。

デスクトップウォッチは旧 `desktop_watch` 機能として復活させない。
デスクトップウォッチ相当の動作は `wake_policy.observations` として定義し、必ず特定の `vision_source_id` を指定して `vision.capture` を発行する。
観測 result、画像意味理解、`world_state.visual_context` 更新は通常の `vision.capture` と同じ経路を使う。
`visual_watch` は独立設定として持たない。

## 境界

この実装で行うこと:

- `vision.capture` を視覚機能の唯一の capability として使う
- `vision_source_id` で観測対象を指定する
- 複数カメラ、複数画面、定期観測対象を `VisionSource` として扱う
- client が `vision.capture_request` を受けたときだけ対象 source から 1 枚の画像を取得する
- server が image Data URI を検証し、raw image を保存せず LLM の multimodal input へ渡す
- server が `visual_observation` 契約で短い `visual_summary_text` を作る
- `vision.capture` result 由来の `visual_summary_text` だけを `world_state.visual_context` に反映する
- `world_state.visual_context` を `vision_source_id` 単位で統合する
- 判断時に同じ `vision_source_id` の新鮮な `visual_context` がある場合は同じ現在状態の再取得を抑制する
- `wake_policy.interval_seconds` ごとに `wake_policy.observations` の enabled 項目を取得し、取得結果を反映してから 1 回だけ wake 判断する
- inspection と smoke で source、request、result、要約、`world_state` 反映を追えるようにする

この実装で行わないこと:

- 旧 `desktop_watch` の復活
- 観測対象を指定しない常時監視
- raw image payload の保存
- OCR 全文抽出
- UI 要素、座標、クリック対象の構造化
- 会話添付画像から `world_state.visual_context` を更新すること
- 複数画像、動画、音声入力
- カメラや画面ごとの capability id 追加

## VisionSource

`VisionSource` は server が扱う視覚 source の正本である。
最小 shape は次とする。

```json
{
  "vision_source_id": "vision_source:main_display",
  "kind": "desktop",
  "label": "メイン画面",
  "aliases": ["画面", "デスクトップ", "メインモニタ"],
  "default_for": ["visual", "desktop"],
  "client_id": "console-...",
  "capability_id": "vision.capture",
  "required_permissions": ["observe_desktop"],
  "available": true
}
```

カメラ source は次の形を使う。

```json
{
  "vision_source_id": "vision_source:desk_camera",
  "kind": "camera",
  "label": "机のカメラ",
  "aliases": ["机", "手元", "デスクカメラ"],
  "default_for": ["camera"],
  "client_id": "console-...",
  "capability_id": "vision.capture",
  "required_permissions": ["observe_camera"],
  "available": true
}
```

`vision_source_id` は server 内で一意にする。
`kind` は `desktop / camera / virtual` のいずれかにする。
`label` は判断と inspection に出す短い表示名にする。
`aliases` は自然言語指定の解決に使う。
`default_for` は曖昧な指定を解決できる範囲を表す。
`client_id` は実行先 client を表すが、LLM decision には渡さない。
`required_permissions` は source ごとの権限であり、manifest の `observe_vision` と合わせて照合する。

## Wake Policy Observations

`wake_policy.observations` は interval wake の前に実行する観測項目である。
observation は観測 source ではなく、`vision.capture` を発行する条件である。
周期は `wake_policy.interval_seconds` の 1 つだけを使う。

最小 shape は次とする。

```json
{
  "observation_id": "observation:main_display",
  "enabled": true,
  "capability_id": "vision.capture",
  "input": {
    "vision_source_id": "vision_source:main_display",
    "mode": "still"
  }
}
```

`wake_policy.observations` は複数持てる。
`enabled=false` の observation は取得しない。
enabled observation は interval wake ごとに順番に実行する。
observation が発行する観測は通常の `vision.capture` request と同じ schema を使う。
observation result は wake 判断の前に画像意味理解と `world_state.visual_context` 更新へ通す。
複数 observation がある場合も、server はすべての取得処理を終えたあとに 1 回だけ wake 判断を行う。
observation は raw image を保存しない。
observation は `world_state.visual_context` を `vision_source_id` 単位で更新する。

## 完成形の流れ

視覚観測は次の順序で動く。

1. client が hello で `vision.capture` と `vision_sources` を通知する。
2. server が `VisionSourceRegistry` を更新する。
3. server が判断入力に `vision.capture` の decision view と短い source 候補を載せる。
4. LLM が `capability_request` として `vision.capture`、`vision_source_id`、`mode=still` を提案する。interval wake では `wake_policy.observations` が LLM decision の前に同じ request を発行する。
5. server が manifest、source、binding、availability、権限、ongoing action、parallel block を検証する。
6. server が `vision_source_id` に紐づく client へ `vision.capture_request` を送る。
7. client が指定された source から画像を 1 枚だけ取得し、画像 Data URI と短い `client_context` を `POST /api/capability/result` へ返す。
8. server が request id、client id、capability id、result shape、画像 Data URI、画像件数、source 対応を検証する。
9. server が `input_interpretation` の `visual_observation` で画像を 1 文の `visual_summary_text` へ圧縮する。
10. server が capability result follow-up を共有判断パイプラインへ入れる。
11. server が `visual_summary_text` を `world_state.visual_context` へ `vision_source_id` 単位で反映する。
12. server が follow-up の `foreground_world_state` から別 `vision_source_id` の `visual_context` を除外する。
13. decision / reply は対象 source の `foreground_world_state` と `observation_summary` を使って応答または次の行動を決める。

## Server 側責務

server は視覚機能の正本を持つ。

- `vision.capture` manifest を持つ
- hello から `vision.capture` binding と `VisionSource` を導出する
- `VisionSourceRegistry` を管理する
- source ごとの availability を `CapabilityState` と `VisionSource` に反映する
- decision view に利用可能な source 候補を短く載せる
- `vision_source_id` を input schema と registry で検証する
- `vision_source_id` から実行先 client を一意に決める
- `vision.capture_request` を stream へ dispatch する
- result endpoint で `images` を最大 1 件の画像 Data URI として検証する
- result の request id から元の `vision_source_id` を復元し、result の source 情報と照合する
- `visual_observation` source pack へ raw image を入れない
- raw image を store、cycle trace、inspection、log に保存しない
- `observation_summary.source=capability_result`、`capability_id=vision.capture`、`image_input_kind=vision_capture_result` を維持する
- `observation_summary.vision_source_id / source_kind / source_label` を維持する
- `visual_summary_text` を有効入力、decision context、`world_state.visual_context` へ渡す
- `vision.capture` result follow-up の判断入力には同じ `vision_source_id` の `visual_context` だけを渡す
- 会話添付画像の `observation_summary.source=conversation_attachment` を `world_state.visual_context` へ渡さない
- result が空画像の場合は `image_count=0` として扱い、視覚要約と `visual_context` 更新を行わない
- invalid result は `400 invalid_capability_result` で拒否する

## Client 側責務

client は `vision.capture` の実行体であり、manifest の正本を持たない。

- stream 接続の hello で `{ "id": "vision.capture", "version": "1" }` を通知する
- hello で実行可能な `vision_sources` を通知する
- source ごとの `vision_source_id / kind / label / aliases / default_for / required_permissions` を通知する
- source の利用者許可が無い場合は、その source を通知しない
- `vision.capture_request` を受けたときだけ画像を取得する
- request の `vision_source_id` と client が持つ source を照合する
- 取得画像は 1 件だけ返す
- 画像は `data:image/png;base64,...` または `data:image/jpeg;base64,...` の Data URI として返す
- `client_context.vision_source_id / source_kind / source_label` を返す
- desktop source では `client_context.active_app / window_title / locale` を取得できた field だけ短く返し、取得できない field は省略する
- camera source では `active_app / window_title` を返さず、`source_label` を判断補助に使う
- capture failure を `timeout_ms` 内に確定できた場合は `error` 付き空画像 result を返す
- `timeout_ms` を超える場合は result を返さず、server timeout に任せる
- capture 中に別 request を受けた場合は server の parallel block に従い、client 側で独自 queue を持たない
- raw image を client 側の永続ログに残さない

## Source 指定

ユーザーの自然言語指定は `vision_source_id` へ解決してから `vision.capture` を実行する。
解決規則は次とする。

| 入力例 | 解決 |
|--------|------|
| 「画面見て」 | `default_for` に `desktop` を持つ source |
| 「メイン画面見て」 | `aliases` または `label` が一致する desktop source |
| 「机のカメラ見て」 | `aliases` または `label` が一致する camera source |
| 「カメラ見て」 | `default_for` に `camera` を持つ source |
| 「今見えるものを見て」 | `default_for` に `visual` を持つ source |
| 「ウォッチ対象を見て」 | 対応する `wake_policy.observations[].input.vision_source_id` |

該当 source が 1 件に定まる場合だけ `vision.capture` を実行する。
該当 source が 0 件の場合は、利用できる source が無い旨を reply する。
該当 source が複数あり default で決まらない場合は、source を確認する reply を返す。
LLM が label や alias を直接 input に入れた場合、server は repair または validation failure として扱い、`vision_source_id` 以外を実行対象にしない。

## 判断ポリシー

`vision.capture` は、現在の視覚状態が判断に必要な場合だけ使う。

使う条件:

- ユーザーが現在見えている内容について質問した
- ユーザーが特定の画面、カメラ、表示内容、作業中の対象について質問した
- 自律判断で強い `drive_state` があり、対象 source の視覚前景が無いか古い
- `wake_policy.observations` に enabled observation があり、interval wake の実行時刻に達した
- capability result follow-up で視覚情報が必要だが、同じ source capability 以外へ横展開しない範囲で閉じる

使わない条件:

- ユーザーが添付画像について質問しており、`conversation_attachment` の要約で足りる
- 同じ `vision_source_id` の新鮮な `world_state.visual_context` があり、ユーザーが明示的に再観測を求めていない
- 現在の判断に視覚情報が不要
- 対象 source が未接続、権限不足、busy、一時 unavailable
- source が一意に決まらない
- capture result follow-up が source capability と異なる capability request を誘発する場面

## Data 形状

`vision.capture` の decision input は次を使う。

```json
{
  "capability_id": "vision.capture",
  "input": {
    "vision_source_id": "vision_source:main_display",
    "mode": "still"
  }
}
```

`vision.capture_request` は次を使う。

```json
{
  "event_id": 1,
  "type": "vision.capture_request",
  "data": {
    "request_id": "vision_capture_request:...",
    "capability_id": "vision.capture",
    "vision_source_id": "vision_source:main_display",
    "source_kind": "desktop",
    "source_label": "メイン画面",
    "mode": "still",
    "timeout_ms": 5000
  }
}
```

result は次を返す。

```json
{
  "request_id": "vision_capture_request:...",
  "client_id": "console-...",
  "capability_id": "vision.capture",
  "result": {
    "images": ["data:image/png;base64,..."],
    "client_context": {
      "vision_source_id": "vision_source:main_display",
      "source_kind": "desktop",
      "source_label": "メイン画面",
      "active_app": "Browser",
      "window_title": "docs | Browser",
      "locale": "ja-JP"
    },
    "error": null
  }
}
```

`images` は 0 件以上、最大 1 件とする。
画像がある場合は画像 Data URI にする。
`client_context` は視覚要約の補助であり、正本状態ではない。
accepted pending result では `client_context.vision_source_id / source_kind / source_label` を必須とし、request id に紐づく server 側 source metadata と一致させる。
正本 source は request id に紐づく server 側の `vision_source_id` である。

## world_state 反映

`vision.capture` result は `world_state.visual_context` へ反映する。
統合単位は `vision_source_id` とする。

`visual_context` の source pack は少なくとも次を持つ。

```json
{
  "visual_context": {
    "summary_text": "メイン画面ではブラウザの docs が前景にある。",
    "visual_summary_text": "メイン画面ではブラウザの docs が前景にある。",
    "image_interpreted": true,
    "visual_confidence_hint": "medium",
    "image_count": 1,
    "capability_id": "vision.capture",
    "vision_source_id": "vision_source:main_display",
    "source_kind": "desktop",
    "source_label": "メイン画面"
  }
}
```

`visual_context` の integration key は `visual_context:<vision_source_id>` とする。
同じ source の新しい観測は同じ key の state を置換する。
異なる source の視覚前景は併存する。
判断入力へ出す foreground は salience、鮮度、source の default 性で並べる。

## 保存と inspection

保存するもの:

- user / observation / reply event の短い text
- `cycle_trace.input_trace.observation_summary`
- `world_state.visual_context` の短い `summary_text`
- `vision_source_id / source_kind / source_label`
- capability request / result の audit 用要約
- inspection の capability state、vision source state、world state trace、source pack hook 要約

保存しないもの:

- raw image Data URI
- base64 本文
- 画面の全文 OCR
- UI 座標
- credential、内部 URL、token、配送先 client の詳細

inspection で最低限確認できる項目:

- `vision.capture` binding と availability
- `VisionSourceRegistry` の source 一覧
- source ごとの availability、permission、default
- wake observation の enabled、対象 `vision_source_id`
- wake observation の `last_run_at / last_status / last_summary / last_error`
- dispatch した request id
- request に紐づく `vision_source_id`
- accepted result の image count
- `image_interpreted`
- `visual_summary_text`
- `visual_confidence_hint`
- `cycle_trace.input_trace.wake_observation_summary`
- `cycle_trace.input_trace.wake_observations`
- `world_state_trace.source_pack_contexts.visual_context`
- `world_state_trace.source_pack_state_type_hooks.visual_context`
- `foreground_world_state` の `visual_context`

## Failure

主な failure は次のように扱う。

| failure | 扱い |
|---------|------|
| source 未登録 | decision repair または `internal_failure` |
| source が複数候補に分かれる | source 確認 reply |
| source client 未接続 | `vision.capture` を source unavailable として decision view へ出す |
| source 権限不足 | source を unavailable にする |
| dispatch failure | capability state と source state に一時 unavailable を残す |
| timeout | ongoing action を失敗終了し、一時 unavailable を残す |
| result の source が request と一致しない | `400 invalid_capability_result` |
| `images` が配列でない | `400 invalid_capability_result` |
| `images` が 2 件以上 | `400 invalid_capability_result` |
| `images` が画像 Data URI でない | `400 invalid_capability_result` |
| image interpretation 失敗 | capability result cycle を failure として残す |
| `world_state` 更新失敗 | 判断サイクル本体は継続し、`world_state_trace.failure_reason` に残す |

silent fallback で画像なし成功へ丸めない。
空画像 result は client が観測できなかった明示結果として扱い、`image_count=0` を残す。

## 実装順

### Phase 1: Source 指定の正本化

- `vision.capture.input.source` を廃止する
- `vision.capture.input.vision_source_id` を必須にする
- `VisionSourceRegistry` の shape を定義する
- decision view に source 候補を載せる
- `world_state.visual_context` の統合単位を `vision_source_id` にする
- manifest、API docs、画像意味理解 docs、world_state docs を同期する

### Phase 2: Server 境界固定

- `vision.capture` result の画像検証を Data URI / 最大 1 件に固定する
- `visual_observation` source pack に `image_input_kind=vision_capture_result` と `vision_source_id` を入れる
- `conversation_attachment` が `world_state.visual_context` を更新しないことを検証する
- result の source 情報を request 記録と照合する

### Phase 3: Client 実装

- stream hello で `vision.capture` と `vision_sources` を通知する
- `vision.capture_request` を受信する
- request の `vision_source_id` を client 側 source と照合する
- 指定 source から画像を 1 枚取得して Data URI 化する
- source metadata と desktop metadata を短く添える
- result endpoint へ返す
- timeout と capture failure を明示的に扱う

### Phase 4: Wake Policy Observation 実装

- `wake_policy.observations` 設定を追加する
- observation は必ず `vision_source_id` を持つ
- interval wake で enabled observation から `vision.capture` request を発行する
- observation result を wake 判断前の画像意味理解と `world_state.visual_context` 更新へ流す
- enabled と対象 source は設定 snapshot で確認し、取得結果は `cycle_trace.input_trace.wake_observations` と `world_state` で確認する

### Phase 5: 判断導線確認

- 「今画面に何が見える？」系の通常会話で default desktop source が選ばれることを確認する
- 「机のカメラ見て」系の通常会話で camera source が選ばれることを確認する
- source が曖昧な場合に確認 reply になることを確認する
- result follow-up で `visual_summary_text` が reply に使われることを確認する
- `world_state.visual_context` が `vision_source_id` 単位で作られることを確認する
- 新鮮な `visual_context` があるときに同じ source の再 capture が抑制されることを確認する

### Phase 6: 観測品質調整

- `visual_observation` prompt が対象 source の前景を 1 文に圧縮することを real LLM で確認する
- secret、base64、内部 URL、OCR 全文を出力しないことを確認する
- `client_context` だけに引きずられず、画像内容を優先して要約することを確認する
- confidence が不明確な画像で `low` になることを確認する

### Phase 7: Smoke と inspection

- long smoke で source 指定、invalid source、invalid result、timeout、follow-up、world_state 更新を確認する
- inspection API で source availability、request、result、world_state trace を確認する
- raw image が store、trace、log に残らないことを確認する

## 受け入れ条件

実装完了条件は次である。

- client 接続時に `vision.capture` と `vision_sources` が inspection へ出る
- source ごとの availability と permission が確認できる
- `vision.capture_request` が `vision_source_id` を持つ
- `vision_source_id` から実行先 client と source が一意に決まる
- `vision.capture_request` から result follow-up まで 1 cycle で閉じる
- valid result の raw image は永続化されない
- valid result は `visual_summary_text` を持つ
- valid result は `world_state.visual_context` を `vision_source_id` 単位で更新する
- 通常会話添付画像は `world_state.visual_context` を更新しない
- invalid source は実行されない
- invalid image result は `400 invalid_capability_result` になる
- timeout 後に ongoing action が残り続けない
- wake observation は指定 source だけを観測する
- interval wake は enabled wake observation を取得して `world_state.visual_context` へ反映してから 1 回だけ判断する
- inspection で source、request、result、要約、world_state trace を追える
- `python3 -m compileall src/otomekairo` が通る
- focused check と short smoke が通る

## 最初に実装する差分

server 側は `source=desktop` を廃止し、`vision_source_id` を input の正本にする。
client 側の画面取得は `vision.capture_request` を受信して実行する。
wake observation は `wake_policy.observations` として持ち、interval wake の判断前に enabled 項目を取得する。
判断導線と `world_state.visual_context` の動作確認は Phase 5 以降で継続する。
