# デバッグ可能性

## 基本前提

OtomeKairo では、後から「なぜその判断や変化になったか」を追える必要がある。
ただし、完全再生そのものより、判断と変化の説明可能性を優先する。

そのため、トークン単位の全文記録ではなく、判断サイクルごとの構造化要約をデバッグ正本として残す。

## 判断サイクルを記録単位にする

デバッグ記録の基本単位は 1 回の判断サイクルである。

1 判断サイクルとは、少なくとも次のいずれかを起点に始まる 1 回の処理を指す。

- 人からの入力
- 感覚入力や外界イベント
- 自律判断の起床
- 実行結果の受信

この単位ごとに `cycle_id` を払い、観測から結果までをひとまとまりで追えるようにする。

## 段階トレースに分ける

1 判断サイクルは、少なくとも次の段階に分けて記録する。

- `observation_trace`
- `recall_trace`
- `decision_trace`
- `result_trace`

この分離により、何を受け取り、何を想起し、どう判断し、何が起きたかを追いやすくする。

## `cycle_summary`

各判断サイクルには、一覧向けの `cycle_summary` を持つ。
少なくとも次を含める。

- `cycle_id`
- `server_id`
- `trigger_kind`
- `started_at`
- `finished_at`
- 判断開始時点の選択構成
- 最終結果種別
- 失敗有無

`cycle_summary` は、最近の判断を俯瞰するための最小要約である。

## `observation_trace`

`observation_trace` には、何を受け取って判断を始めたかを残す。
少なくとも次を含める。

- `trigger_kind`
- 入力やイベントの要約
- 正規化した観測の要約
- 観測時点の実行状態要約

ここで重要なのは、巨大な payload を丸ごと複製することではなく、後から読める意味単位へ要約して残すことである。

## `recall_trace`

`recall_trace` には、何を思い出そうとし、何を採ったかを残す。
少なくとも次を含める。

- 想起入口の要約
- 想起候補の件数
- 採用した記憶や状態の要約
- 採用理由の要約
- 代表的に落とした候補の要約

ここでは、全候補の完全再現よりも「なぜそれが前に出たか」を追えることを優先する。

## `decision_trace`

`decision_trace` には、なぜその結論になったかを残す。
少なくとも次を含める。

- 判断結果種別
- 判断理由の要約
- 強く効いた人格要素の要約
- 強く効いた記憶要素の要約
- 強く効いた現在文脈の要約
- 保留意図を残した場合の要約

自由文の長い思考ログではなく、後から追える短い構造化要約を優先する。

## `result_trace`

`result_trace` には、最終的に外へ出た結果と、その実行結果を残す。
少なくとも次を含める。

- 最終結果種別
- 外向きに出した内容または実行要求の要約
- 見送った理由の要約
- 保留意図へ回した場合の要約
- 内部失敗の要約
- 所要時間要約

返した全文や生ログだけを唯一のデバッグ正本にしない。
追跡上の意味を持つ要約を段階トレースとして残す。

## 判断開始時点の構成を残す

デバッグ記録では、判断開始時点の設定と反映状態を残す。
少なくとも次を開始時点のスナップショットとして記録する。

- `server_id`
- 現在の人格、記憶、モデルの選択値
- 自律判断方針
- 観測源方針
- 読み込み済みの人格参照、記憶参照、モデル参照

これにより、「何を選んでいたか」と「実際に何が反映されていたか」を切り分けて確認できる。

## デバッグ正本として保存しないもの

次は、標準のデバッグ正本として無条件には保存しない。

- LLM への完全プロンプト全文
- LLM の生レスポンス全文
- トークン単位の思考過程
- 想起候補の完全一覧
- 巨大 payload の完全複製

保存量と見通しを壊しやすいものは、必要時の補助情報とし、標準の追跡面にはしない。

## inspection 面との境界

デバッグ記録の参照は、通常運用とは別の inspection 面として扱う。

少なくとも次を分ける。

- 通常運用の面
  - 観測、状態、構成、列挙、実行連携
- inspection 面
  - `cycle_summary` 一覧
  - 特定 `cycle_id` の段階トレース
- debug stream 面
  - live の短い段階要約ログ

この分離により、通常運用の概念を崩さずに検証経路を追加できる。

## live 表示は派生情報として扱う

live の段階要約ログは、inspection 正本そのものではなく、追跡しやすくした派生表示である。
少なくとも次の種類を流しうる。

- Observation
- Recall
- Decision
- Result
- Memory

重要なのは、live stream を唯一の正本にしないことだ。

## 保存量の方針

保存量は無制限にしない。
最小方針は次のとおりである。

- `cycle_summary` は比較的長く残す
- 段階トレース詳細は要約中心に残す
- 古い詳細を削る場合でも、一覧要約は残す

具体的な保持件数や保持期間は実装と運用設定で決める。
