# デバッグ可能性

## 基本前提

OtomeKairo では、後から「なぜその判断や変化になったか」を追える必要がある。
ただし、完全再生そのものより、判断と変化の説明可能性を優先する。

そのため、トークン単位の全文記録ではなく、判断サイクルごとの構造化要約をデバッグ正本として残す。
段階トレースの責務、保存方針、標準的な含有内容は、この文書を正本にする。
`design/api/` 配下は inspection endpoint の wire 契約だけを定め、各 trace の意味規則を再掲しない。

## ログファイル

通常サーバ実行では、`debug_log` の出力を標準出力と `OTOMEKAIRO_DATA_DIR/server.log` の両方へ出す。
`OTOMEKAIRO_DATA_DIR` の既定値は `var/otomekairo` である。
ログファイルは実動作後に人間または開発補助が確認するための運用ログであり、inspection の正本ではない。

ログファイルは容量上限を持つ。
既定では `server.log` が 5MiB を超える前にローテーションし、`server.log.1` から `server.log.3` まで保持する。
既定の総保持量は `server.log` 本体と 3 世代を合わせて最大約 20MiB である。
`OTOMEKAIRO_DEBUG_LOG_MAX_BYTES` は 1 ファイルあたりの上限 byte 数を指定する。
`OTOMEKAIRO_DEBUG_LOG_BACKUP_COUNT` は保持する世代数を指定する。
`OTOMEKAIRO_DEBUG_LOG_BACKUP_COUNT=0` の場合、上限到達時に旧 `server.log` を破棄して新しい `server.log` を作る。

ログファイルには response body、raw image、base64 本文、API key、token を出力しない。
この境界は標準出力と `logs/stream` にも適用する。
`/api/inspection` 配下は inspection response 自体が確認正本であり、ポーリング量も大きいため HTTP access log を出さない。

## 判断サイクルを記録単位にする

デバッグ記録の基本単位は 1 回の判断サイクルである。

1 判断サイクルとは、少なくとも次のいずれかを起点に始まる 1 回の処理を指す。

- 人からの入力
- 感覚入力や外界イベント
- 自律判断の起床
- 実行結果の受信

この単位ごとに `cycle_id` を払い、観測から結果までをひとまとまりで追えるようにする。

## 段階トレースに分ける

1 判断サイクルは、少なくとも次の段階に分けて記録する。

- `input_trace`
- `recall_trace`
- `decision_trace`
- `result_trace`
- `memory_trace`
- `world_state_trace`

この分離により、何を受け取り、何を想起し、どう判断し、何が起きたかを追いやすくする。

## `cycle_summary`

各判断サイクルには、一覧向けの `cycle_summary` を持つ。
少なくとも次を含める。

- `cycle_id`
- `server_id`
- `trigger_kind`
- `started_at`
- `finished_at`
- 最終結果種別
- 失敗有無

`cycle_summary` は、最近の判断を俯瞰するための最小要約である。
`started_at` と `finished_at` は、OtomeKairo のローカルタイムゾーンに属する offset 付き timestamp とする。
timestamp の共通方針は `18_時刻モデル.md` を正とする。

## `input_trace`

`input_trace` には、何を受け取って判断を始めたかを残す。
少なくとも次を含める。

- `trigger_kind`
- 入力やイベントの要約
- 添付画像などの補助要約として内部補助文脈へ追加した差分要約
- 検索・想起用に拡張した `augmented_query_summary`
- 正規化した観測の要約
- 判断入力へ入った `world_state` の要約
- 入力時点の実行状態要約
- 入力時点の `ongoing_action` 要約
- 自律判断時の `initiative_context` 要約

ここで重要なのは、巨大な payload を丸ごと複製することではなく、後から読める意味単位へ要約して残すことである。

## `recall_trace`

`recall_trace` には、何を思い出そうとし、何を採ったかを残す。
少なくとも次を含める。

- 想起入口の要約
- 想起候補の件数
- 採用した記憶や状態の要約
- 採用理由の要約
- 代表的に落とした候補の要約

raw `events` を狭く直接参照する回答では、`recall_trace` に少なくとも次も含める。

- `answer_contract`
- `evidence_pack`
- `fact_resolution_trace`

`fact_resolution_trace` は、exact answer 系の根拠解決を後から追うための比較用 trace である。
少なくとも次を含める。

- 実行した `contract / boundary / target_actor / query_terms`
- 採用時に見た direct event 候補
- 採用済み recall section の代表項目
- 最終採用した `evidence_items`
- 境界 event と recall 側の明示 claim を比較した整合チェック

ここでは、全候補の完全再現よりも「なぜそれが前に出たか」を追えることを優先する。

mock smoke の `recall_quality_probe_digest` は、代表的な想起品質を一つの summary に残す。
対象は直近窓外の曖昧参照、`active_commitments`、`conflicts`、標準 / precise `event_evidence`、`decision_trace.internal_context_summary.recall_pack_summary` への反映である。
実 LLM smoke の `real_llm_memory_trace_shape_digest` は、同じ probe を意味判定ではなく trace shape の確認として残す。
対象は `memory_link_context`、`recall_pack_selection.memory_link_*`、decision trace の `memory_link_context`、`affect_state` 初回作成、`event_evidence_generation`、`memory_links` label 保存である。
実 LLM smoke では `real_llm_memory_trace_shape_verified` を合否に使い、`memory_quality_probe_verified` の semantic strict check は参考情報として扱う。

## `decision_trace`

`decision_trace` には、なぜその結論になったかを残す。
少なくとも次を含める。

- 判断結果種別
- 判断理由の要約
- 強く効いた人格設定要素の要約
- 強く効いた記憶要素の要約
- 強く効いた `drive_state` の要約
- 参照した `ongoing_action` の要約
- 強く効いた現在文脈の要約
- internal_context 専用 message と current_input 専用 message を分けて送ったことが分かる要約
- 添付画像などの補助要約として内部補助文脈へ追加した差分要約
- 検索・想起用に拡張した `augmented_query_summary`
- 保留意図を残した場合の要約

会話添付画像の raw image は decision trace に保存しない。
会話添付画像の解釈結果は、判断入力では `VisualObservationContext` として扱う。
`VisualObservationContext.source=conversation_attachment` かつ `image_interpreted=true` のとき、判断は添付画像要約が存在する前提で行う。
desktop `vision.capture` の解釈結果も、判断入力では `VisualObservationContext.source=vision_capture_result` として扱う。
desktop `vision.capture` の `VisualObservationContext.retention_policy` は `ephemeral_decision_only` とし、`world_state` と記憶更新の正本にしない。

自由文の長い思考ログではなく、後から追える短い構造化要約を優先する。
LLM 送信 preview では、reserved sentinel 名と message role の並びから内部注入 block の存在を追えることを優先する。

## `result_trace`

`result_trace` には、最終的に外へ出た結果と、その実行結果を残す。
少なくとも次を含める。

- 最終結果種別
- 外向きに出した内容または実行要求の要約
- `ongoing_action` の作成、継続、完了、中断の要約
- trigger をまたいだ比較用の `trigger_compact_summary`
- capability family 共通で比較できる `ongoing_action_transition_summary.reason_code / transition_source / detail_summary`
- capability dispatch が起きたサイクルでは、request と dispatch-side transition を束ねた `capability_dispatch_summary`
- `trigger_kind=capability_result` のときは、source request・observation・decision・follow-up result・transition を 1 つに束ねた `capability_result_followup_summary`
- 見送った理由の要約
- 保留意図へ回した場合の要約
- 内部失敗の要約
- 所要時間要約

返した全文や生ログだけを唯一のデバッグ正本にしない。
追跡上の意味を持つ要約を段階トレースとして残す。

`trigger_compact_summary` は、少なくとも `trigger_kind / trigger_family / current_input_summary / entry_summary / decision_summary / result_summary / transition_summary` を持つ。
`current_input_summary` には `sender / source_kind / response_target` と短い `text` 要約を含める。
`capability_result` では source request と observation を、`wake / background_wake` では `initiative_context` と pending-intent selection を主にこの summary へ束ね、trigger の違う cycle を inspection 上で並べて比較しやすくする。
soak observability は status 系 capability result の応答直後に `capability_result_followup_summary.source_request_summary.request_id` を照合し、最終 summary の cycle 回収件数制限に依存しない follow-up 到達記録を持つ。
soak observability の `max_memory_job_lag_seconds` は `memory_postprocess_jobs` の最古 `queued` job の年齢を測る。
initiative 系 trigger の `entry_summary` には、`selected_candidate_family` と `candidate_families` の要約も含める。
initiative 系 trigger の `input_trace.initiative_context` と `trigger_compact_summary.entry_summary` は、少なくとも `opportunity_summary / time_context_summary / foreground_signal_summary / candidate_families / suppression_summary` の比較軸を揃える。
initiative 系 trigger の `candidate_families` 要約には、少なくとも `priority_score / preferred_result_kind / preferred_result_reason_summary / blocking_reason_summary` を残し、`reply / noop / capability_request` の分岐理由を比較できるようにする。
`wake_policy.observations` を実行した cycle の `input_trace` には、`wake_observation_summary` と `wake_observations` を残す。
`wake_observations` には、少なくとも `observation_id / status / vision_source_id / visual_summary_text / image_count / request_id` を含め、観測結果が判断入力に渡ったことを trace で確認できるようにする。
desktop wake observation では `desktop_observation_signal.novelty_kind / reply_eligibility / reason_summary` を `wake_observations` と `initiative_context.foreground_signal_summary` に残す。
`reply_eligibility` が `eligible` ではない場合も、`current_input.response_target` と照合できるように trace へ残す。
inspection の `wake_policy_observations` には、process-local runtime の `last_scene_signature / same_scene_count / last_prompted_at / pending_novel_scene` を残す。
これらは novelty 判定用の短い runtime trace であり、raw image、画面全文 OCR、UI 座標、記憶候補を含めない。

## `memory_trace`

`memory_trace` には、そのサイクルが記憶へどう残り、記憶由来の派生状態へどう反映されたかを残す。
少なくとも次を含める。

- 保存した `events` の要約
- 生成した `episode` の要約
- `episode_series_id` がある場合はその値
- `open_loops` の要約
- `memory_units` 更新操作の要約
- `memory_link_update` の要約
- `drive_state` 更新要約
- 感情更新の要約
- `vector_index_sync` の状態要約
- `reflective consolidation` の状態要約
- `reflective consolidation.summary_generation` の件数要約と scope 単位失敗
- `reflective consolidation.affect_state_update` の作成、更新、弱化、整理の要約
- `reflective consolidation.memory_link_update` の件数と label 別要約
- 記憶更新失敗時の要約

ここでは、生の記憶表全件を並べるのではなく、そのサイクルで生じた意味変化と派生状態の更新を追えることを優先する。
`turn consolidation` の同期保存と、後段の embedding / reflection job は分けて追えるようにする。
`ongoing_action` の作成や終了は記憶更新ではないため、`memory_trace` へ混ぜず `result_trace` で追う。
`memory_link_update` は `revisions.related_memory_unit_ids` の再掲ではなく、理解同士の関係保存結果を表す。

## `world_state_trace`

`world_state_trace` には、そのサイクルで外界状態がどう判断へ入り、どう更新されたかを残す。
少なくとも次を含める。

- 判断入力へ入った `world_state` の件数
- 前景に出した state の要約
- 更新された state の件数
- 置換された state の件数
- 失効した state の件数
- source kind と source ref の要約
- `visual_context / external_service_context` など sanitized source-pack context の要約
- `state_type` ごとに、どの capability / summary field が source pack へ効いたかを示す hook 要約
- desktop 以外の `vision.capture` result follow-up で `foreground_world_state` を source 単位に絞った場合は `foreground_world_state_filter` と保存済み state の要約
- `schedule` では pending-intent 由来 slot の key 要約
- 候補ごとの `summary_source / effective_ttl_seconds / integration_key` 要約
- 更新失敗時の理由

ここでは、raw payload や生の画像、音声、長い外部サービス応答をそのまま並べない。
`world_state` の詳細な意味境界は `22_world_state.md` を正とする。

## デバッグ正本として保存しないもの

次は、標準のデバッグ正本として無条件には保存しない。

- LLM への完全プロンプト全文
- LLM の生レスポンス全文
- provider 側検索の内部結果全文
- トークン単位の思考過程
- 想起候補の完全一覧
- 巨大 payload の完全複製

保存量と見通しを壊しやすいものは、必要時の補助情報とし、標準の追跡面にはしない。
`web_search_enabled` は model role snapshot の設定値として追跡し、観測 capability の結果としては扱わない。

## inspection 面との境界

デバッグ記録の参照は、通常運用とは別の inspection 面として扱う。

少なくとも次を分ける。

- 通常運用の面
  - 観測、状態、構成、列挙、実行連携
- inspection 面
  - point-in-time の `current_state` snapshot
  - `cycle_summary` 一覧
  - 特定 `cycle_id` の段階トレース
  - 記憶更新の追跡
  - capability availability の確認
- debug stream 面
  - live の短い段階要約ログ

この分離により、通常運用の概念を崩さずに検証経路を追加できる。

## `current_state` snapshot

`current_state` snapshot は、判断サイクル trace とは別に「いま何が前景か」を見る inspection 用の現在値である。
少なくとも次を含める。

- `settings_snapshot`
- `runtime_summary`
- scheduler / memory worker / pending capability request の runtime detail
- 前景 `world_state`
- 現在の `drive_state`
- 現在の `ongoing_action`
- process-local の `pending_intent_candidates`
- `mood_state`
- 代表的な `affect_states`
- capability availability の snapshot

ここでは、cycle ごとの変化経路ではなく、現時点の成立状態を確認しやすくすることを優先する。
`pending_intent_candidates` と `pending_capability_requests` は process-local runtime state なので、再起動をまたぐ正本にはしない。
raw payload や credential は含めず、現在の判断に効く sanitized summary と source / updated_at / expires_at を主に返す。

## live 表示は派生情報として扱う

live の段階要約ログは、inspection 正本そのものではなく、追跡しやすくした派生表示である。
少なくとも次の種類を流しうる。

- Observation
- Recall
- Decision
- Result
- Memory

重要なのは、live stream を唯一の正本にしないことだ。

## 保存量の方針

標準のデバッグ正本は要約中心にし、保存時点で過剰な raw payload や秘密情報を入れない。
後追いの保持期限処理、伏せ字化、削除で整える設計にはしない。

少なくとも次を守る。

- `cycle_summary` は一覧確認に必要な最小要約にする
- 段階トレース詳細は要約中心にする
- raw payload、完全 prompt、LLM 生レスポンス全文を標準保存しない
- 保存済み trace を管理操作で削除または伏せ字化しない

個別記憶行や trace に対する削除、伏せ字化、保持期限処理、export の禁止境界は `memory/10_記憶管理と削除.md` を正とする。
