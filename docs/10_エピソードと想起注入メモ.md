## `episode_digests` の役割

`episode_digests` は、生ログ (`events`) と継続理解 (`memory_units`) の中間層である。

それぞれの役割は次のように分ける。

- `events`:
  - 何が入力され、何を返したかの正本
  - 完全な監査用
- `episode_digests`:
  - 「この出来事は何だったか」を思い出しやすい形にしたもの
  - 根拠エピソードとして返答へ添える
- `memory_units`:
  - 継続理解として残すもの
  - 人格形成の中心

OtomeKairo では、返答時に `events` 全文を読むのではなく、まず `memory_units` と `episode_digests` を使うほうがよい。

<!-- Block: EventExceptionRole -->
## `events` は例外的な証拠参照だけに使う

`events` は、通常の同期想起の主経路には入れないほうがよい。
生ログを広く読むと、直近会話と二重化しやすく、要約されていない細部が返答の軸を壊しやすいからである。

ただし、`events` を完全に読まない設計にも弱さがある。
特に次のような質問では、`episode_digests` だけだと粗くなりやすい。

- 「そのとき何のことを話していたか」
- 「そのときどうするって言ったか」
- 「どういう流れでそうなったか」

そのため、OtomeKairo では `events` を「細部確認のための例外的な証拠参照」としてだけ使うのがよい。
役割は次に限定する。

- 選ばれた `episode_digests` の細部確認
- 選ばれた `commitment` や `outcome` の由来確認
- 回想質問に対する短い補足証拠の抽出

ここで重要なのは、`events` を想起候補の主役にしないことである。
主役はあくまで `memory_units` と `episode_digests` に置く。

<!-- Block: WhyDigest -->
## `episode_digests` を分ける理由

`events` だけだと、生々しすぎて想起の単位として重い。
一方で、`memory_units` だけだと、答えの根拠になる具体的出来事が薄くなる。

そのため、`episode_digests` は「証拠」と「思い出しやすさ」のための層として置く。
ここで担わせたい役割は次である。

- 具体的な出来事の思い出し
- `memory_units` の根拠補強
- 返答時の自然な「そういえば」の材料
- 矛盾時の由来確認

<!-- Block: CutRule -->
## `episode_digests` の基本単位

MVP では、`episode_digests` は原則として **1イベント = 1件** でよい。

ここでいう 1 イベントとは、少なくとも次の単位を指す。

- 1 回の会話ターン
- 1 回の観測イベント
- 1 回の自発行動とその結果

初期段階では、無理に細かく割らないほうがよい。
監査しやすさと実装単純性を優先する。

ただし、次のような場合は将来的に複数 digest 化を検討してよい。

- 1ターンに全く別系統の重要話題が複数入っている
- 1イベント内で関係変化と約束形成が独立して強い
- 1イベント内で複数の対象が独立して記憶化される

MVP ではここまでやらず、まずは 1イベント 1digest を正とする。

<!-- Block: DigestFields -->
## `episode_digests` の最小項目

`episode_digests` は、少なくとも次を持つ前提にする。

| 項目 | 役割 |
|------|------|
| `episode_digest_id` | 識別子 |
| `event_ids` | どの `events` に由来するか |
| `episode_type` | chat / observation / proactive / action_result など |
| `primary_scope_type` / `primary_scope_key` | 主にどの対象領域の出来事か |
| `secondary_scopes` | 補助的に関わる対象領域 |
| `summary_text` | 何が起きたかの短い要約 |
| `outcome_text` | 何が決まったか、何が残ったか |
| `open_loops` | 未完了や保留 |
| `affect_hint` | 感情の手がかり |
| `salience` | 思い出しやすさ |
| `created_at` | その digest が生成された時刻 |
| `about_time` | 内容がいつの話か |

ここで重要なのは、`memory_units` のような継続理解ではなく、「その出来事で何が起きたか」を短く持つことである。

<!-- Block: DigestText -->
## `summary_text` と `outcome_text` の分け方

`episode_digests` では、本文を 2 種に分けるほうがよい。

- `summary_text`:
  - 出来事の要約
  - 例: 「あなたが最近の睡眠不足を気にしていると話した」
- `outcome_text`:
  - その結果どうなったか
  - 例: 「今週は無理をしないようにする話になった」

この分離により、同じ出来事でも「何が起きたか」と「何が残ったか」を分けて扱える。

<!-- Block: DigestAndUnit -->
## `episode_digests` と `memory_units` の関係

`episode_digests` から `memory_units` が育つ関係は、次のように考える。

- `episode_digests`:
  - 単発の出来事を持つ
- `memory_units`:
  - 複数 digest をまたいで再利用される理解を持つ

つまり、全ての digest が unit になるわけではない。
また、1つの digest から複数の unit が生まれてよい。

例:

- digest:
  - 「あなたが辛いもの好きだと話した」
- unit:
  - `preference(user, likes, food:spicy)`

- digest:
  - 「最近忙しくて眠れていないと話した」
- unit:
  - `interpretation(user, seems, tired_recently)`
  - `topic(topic:health, remains_open, true)`

<!-- Block: DigestUpdate -->
## `episode_digests` で持つべき未完了

人格形成では、出来事の未完了感が重要である。
そのため `episode_digests` には、少なくとも次の未完了を残せるほうがよい。

- 次回確認したいこと
- 約束したこと
- 感情が残っていること
- 解決していない話題

これにより、後の `memory_units.commitment` や `active_topics` への接続が安定する。

<!-- Block: InjectionPrinciple -->
## 想起注入の構造

返答生成では、役割の違う材料を混ぜずに渡すほうがよい。
OtomeKairo では、返答用入力を次の 5 層に分けるのを基本とする。

1. system prompt
2. 固定ヘッダ
3. 直近会話
4. 内部コンテキスト
5. 現在のユーザー入力

それぞれの役割は次のとおり。

- system prompt:
  - 人格設定
  - 発話ルール
  - 内部コンテキストを露出しない指示
- 固定ヘッダ:
  - 変わらない内部用メッセージ
  - キャッシュ効率のため先頭側に置く
- 直近会話:
  - 直前の流れ
  - 口調
  - いまの会話の接続
- 内部コンテキスト:
  - 時刻
  - 背景感情
  - 想起パック
- 現在のユーザー入力:
  - 今回の主観測

この分離で特に守りたいのは次である。

- 直近会話と長期記憶を二重注入しない
- 長期記憶をフラット一覧で混ぜない
- 生ログ全文を通常注入しない
- 背景感情と根拠事実を混同しない

<!-- Block: InternalContext -->
## 内部コンテキストの最小形

内部コンテキストの最小形は、次を推奨する。

```json
{
  "TimeContext": {},
  "AffectContext": {},
  "RecallPack": {
    "self_model": [],
    "user_model": [],
    "relationship_model": [],
    "active_topics": [],
    "active_commitments": [],
    "episodic_evidence": [],
    "conflicts": []
  }
}
```

この形なら、時刻、背景感情、長期記憶、根拠エピソードの責務を分けて扱える。
ここでの `episodic_evidence` には、通常の `episode_digests` 由来項目に加えて、例外的に `events` 由来の短い証拠項目が入ってよい。

<!-- Block: RecallItem -->
## `RecallPack` の各項目は短く揃える

返答モデルへ渡す各項目は、長文の生データではなく短い記憶文にする。

### `memory_units` 由来の項目

```json
{
  "memory_unit_id": 12,
  "summary_text": "あなたは辛い食べ物をかなり好む。",
  "status": "confirmed",
  "confidence": 0.94,
  "salience": 0.82,
  "why_now": "食事の話題が出ているため"
}
```

### `episode_digests` 由来の項目

```json
{
  "episode_digest_id": 41,
  "summary_text": "先週、あなたが睡眠不足を気にしていると話した。",
  "outcome_text": "今週は無理をしないようにする話になった。",
  "time_hint": "先週",
  "why_now": "体調の話題に直接関係するため"
}
```

### `events` 由来の補助項目

```json
{
  "event_id": 81,
  "excerpt_text": "その日は駅前のカフェで、仕事の締切と来週どう動くかを確認していた。",
  "speaker_hint": "dialogue",
  "time_hint": "先週の外出時",
  "why_now": "場所つきの回想質問で、digest だけでは細部が不足するため"
}
```

ここでの `excerpt_text` は生ログ全文ではなく、返答用に短く圧縮した証拠文である。
逐語引用や全文貼り付けにはしない。

### `conflicts` の項目

```json
{
  "left": "以前は辛いものが好きと言っていた",
  "right": "今回は辛いものが苦手だと明言した",
  "instruction": "断定せず確認寄りに話す"
}
```

<!-- Block: InjectionRule -->
## 注入時のルール

返答生成へ注入するときは、少なくとも次を守る。

- ID や内部キー名を本文に出さない
- `confirmed` は断定に使ってよい
- `inferred` は断定せず、推測表現か確認質問に寄せる
- `conflicts` がある場合は不確実さを明示する
- `AffectContext` はトーンに効かせるが、内部表現をそのまま言わない
- `episodic_evidence` は「思い出」として自然に使う
- `events` 由来項目は、必要時だけ短く使い、本文の主役にしない

直近会話は `RecallPack` に混ぜず、別メッセージ列で渡す。
また、生ログ全文は通常注入せず、必要なときだけ短い `event_evidence` に圧縮して補助利用する。
こうしておくと、口調の接続、長期記憶の参照、根拠エピソードの利用を別責務として保ちやすい。

<!-- Block: Selector -->
## 想起パックの組み立て手順

初期段階では、次の順でよい。

1. 現在入力から `RecallHint` を作る
2. 構造レーンで `memory_units` の厳密候補を集める
3. 連想レーンで `memory_units` と `episode_digests` の補助候補を集める
4. section ごとに再順位付けする
5. `episode_digests` から根拠候補を整える
6. `conflicts` を抽出する
7. セクションごとの上限で整形する
8. 必要なら LLM で最終圧縮する

重要なのは、最初から全てを LLM 任せにしないことである。
セクション分けと上限制御は、コード側で持ったほうがよい。

<!-- Block: EpisodeSelection -->
## `episodic_evidence` の選び方

`episodic_evidence` は、次のどれかに当てはまる digest を優先する。

- 今回の主 `scope` と一致する
- 現在の `memory_units` の根拠になっている
- `active_commitments` の由来になっている
- 感情や関係変化の起点になっている
- 連想レーンで上位に来ており、今回の本筋と整合する

逆に、似ているだけで本筋に効かない digest は落としてよい。

<!-- Block: PromptGuidance -->
## 返答モデルへの行動指針

返答モデルには、少なくとも次の指針を与えるのがよい。

- `RecallPack` は内部用であり、本文にキー名を出さない
- `self_model` は口調や姿勢の補助
- `user_model` は話題選択と理解の補助
- `relationship_model` は距離感や言い回しの補助
- `active_commitments` は忘れないための優先参照
- `episodic_evidence` は必要なときだけ自然に織り込む
- `conflicts` がある場合は断定を避ける
- 連想レーンだけで拾った項目は、断定根拠ではなく補助ヒントとして使う

<!-- Block: MVP -->
## MVP で先に入れる範囲

MVP では、次だけ先に成立すれば十分である。

- 1イベント 1digest
- `summary_text` と `outcome_text`
- `primary_scope`
- `open_loops`
- `RecallPack` の役割別構造
- 直近会話と内部コンテキストの分離
- `confirmed` / `inferred` / `conflicts` に応じた返答ルール

逆に、次は後回しでよい。

- 1イベントから複数 digest を切る高度化
- digest 同士の自動統合
- 複雑な multi-hop 選別
- プロンプトのキー圧縮最適化
