# 設定UI仕様

<!-- Block: Purpose -->
## このドキュメントの役割

- このドキュメントは、ブラウザの設定UIをどのような編集体験と保存モデルで実装するかを固定する正本である
- 目的は、現在の単一キー更新中心の初期実装から離れて、設定UI専用の編集モデル、プリセット、保存単位を整理することにある
- このドキュメントは、現行の `POST /api/settings/overrides` をそのまま延長するためではなく、設定UIを本来あるべき形に置き換えるための目標仕様である
- 現在の Web API 正本は `docs/35_WebAPI仕様.md` にあるが、設定UIの再設計はこのドキュメントを優先する
- 設定キーの個別制約は `docs/39_設定キー運用仕様.md` を見る

<!-- Block: Core Position -->
## 基本方針

- 設定UIは「単一キーを都度送る画面」ではなく、「編集用の全体モデルを読み、ローカルで編集し、保存時にまとまった単位で確定する画面」とする
- 設定UIの正本は、ランタイムの `runtime_settings` ではなく、設定UI専用の `settings_editor_state` と `settings_presets` とする
- ランタイムは、設定UIの保存結果から導出された「現在有効な設定」を materialize して使う
- 設定UIで複数保持したいものは、個別キーではなくプリセットとして扱う
- 設定UIは、運用上の全設定項目を保持する唯一の編集面とする
- API キーやトークンも、設定UIの通常項目として保持し、別の保管経路へ逃がさない
- 人格そのものの改変は設定UIに含めない
- ただし、人格そのものは変えずに、`CocoroConsole` 由来の会話指示と安全な範囲の行動傾向上書きは `behavior_preset` として扱ってよい

<!-- Block: Why This Shape -->
## なぜこの形にするか

- 現在の `settings_overrides` は、少数キーを変える API としてはよいが、UI で「まとまった設定群を比較・複製・保存」する用途には向いていない
- `CocoroGhost` に近い操作感を出すには、保存単位を「最終的に使いたい設定の完成形」に寄せるほうがよい
- プリセットを UI ローカルだけに置くと、別ブラウザや再起動で消えるため、サーバ正本として持つべきである
- 一方で、実行中の enable/disable や現在の間隔調整まで毎回プリセット化すると、操作が重くなるため、即時反映したい運用値はシステム設定として singleton で分ける
- この仕様では、API キーやトークンも設定UIの通常項目として扱い、別系統へ分離しない

<!-- Block: UI Model -->
## UI モデル

<!-- Block: Screen Layout -->
### 画面構成

- 右上の設定アイコンで設定画面を開く
- 設定画面は `CocoroConsole` の設定ウインドウに寄せた 1 枚のウインドウ風パネルにする
- 設定画面は `キャラクター`、`振る舞い`、`会話`、`記憶`、`システム` の 5 タブ構成にする
- 最上部には `設定` タイトルと短い説明文を置き、その下にタブ列を置く
- 各タブ本文は白背景の大きな編集領域にし、各設定ブロックは通常の文字色の見出しを持つ GroupBox 風セクションで囲う
- `振る舞い`、`会話`、`記憶` の各編集面は、先頭に `プリセット選択` セクションを置き、`ComboBox + 追加 / 複製 / 削除 + プリセット名` を並べる
- `キャラクター` タブには `キャラクター選択`、`基本設定`、`マテリアル・影設定`、`音声合成`、`音声認識` を並べ、`output` プリセットの選択とキャラクター関連設定の編集をここで行う
- `振る舞い` タブには `振る舞いプロンプト`、`追加プロンプト（任意）`、`行動傾向` を並べ、内部の `behavior` プリセットをここで編集する
- `システム` タブには `通知設定`、`システム設定`、`カメラ接続` を並べ、診断 JSON は `診断情報` の折りたたみ領域に入れる
- `システム` タブのカメラ接続は、接続一覧を表形式で並べ、その下でアクティブ接続を編集する
- 最下部には固定フッターを置き、左に `再読込`、中央に保存状態、右に `OK`、`キャンセル`、`適用` を置く

<!-- Block: Editing Rules -->
### 編集の基本ルール

- 設定画面を開いたとき、まずサーバ正本の `GET /api/settings/editor` 応答を取得してフォームへ描画する
- 編集中の変更は、即時保存せずブラウザ内の draft に保持する
- `OK` または `適用` を押したときだけ、現在の draft 全体をサーバへ送る
- `キャンセル` は未保存 draft を破棄して設定画面を閉じる
- `再読込` は未保存 draft を破棄し、`GET /api/settings/editor` と状態再取得をやり直す
- 保存成功時は、サーバから返った canonical な最新状態でフォームを再描画する
- `OK` は保存成功後に設定画面を閉じ、`適用` は開いたままにする
- 保存成功時と保存失敗時の結果表示は、設定画面内フッターの状態表示だけに出す
- 保存結果は、チャット欄や `notice` / `error` のストリーム通知へは出さない
- 保存失敗時は、どのブロックで失敗したかを設定画面内に出し、draft は失わない
- UI は、編集中かどうかを `dirty` フラグで明示する
- API キー、トークン、パスワードの入力欄はマスキングせず、通常の文字列入力欄で表示する

<!-- Block: Preset Policy -->
### プリセットとシステム設定の切り分け

- 複数保持・複製・切替をしたい設定群は、プリセットにする
- 現在の運用状態を直接切り替えたいものは、`settings_editor_state.system_values_json` に保持する
- システム設定は、原則として「その場の運用スイッチ」と「短い数値調整」に限定する
- カメラ接続はプリセット化せず、専用の `camera_connections` として別管理にする
- API キーやトークンも、必要なプリセットの中に通常項目として含める

<!-- Block: Preset Groups -->
## プリセットの種類

<!-- Block: Behavior Preset -->
### `behavior_preset`

- 役割: `CocoroConsole` 相当の会話指示と、OtomeKairo 独自の行動傾向をまとめて切り替える
- このプリセットは、`振る舞い` タブで編集する
- 複数保持、複製、アーカイブを許可する
- 含める項目:
  - `behavior.second_person_label`
  - `behavior.system_prompt`
  - `behavior.addon_prompt`
  - `behavior.response_pace`
  - `behavior.proactivity_level`
  - `behavior.browse_preference`
  - `behavior.notify_preference`
  - `behavior.speech_style`
  - `behavior.verbosity_bias`

- 注意:
  - これは `self_state.personality_json` を直接書き換えるものではない
  - `behavior.system_prompt` と `behavior.addon_prompt` は認知系 LLM の system prompt に加える
  - `behavior.response_pace` と `behavior.speech_style` は `selection_profile.interaction_style` を上書きして使う
  - `behavior.proactivity_level`、`behavior.browse_preference`、`behavior.notify_preference`、`behavior.verbosity_bias` は応答傾向のヒントとして使う

<!-- Block: Llm Preset -->
### `llm_preset`

- 役割: 会話生成と認知生成に使う LLM 関連設定をまとめる
- このプリセットは、`会話` タブで編集する
- 複数保持、複製、アーカイブを許可する
- 含める項目:
  - `llm.model`
  - `llm.temperature`
  - `llm.max_output_tokens`
  - `llm.api_key`
  - `llm.base_url`
- `llm.model` は `provider/model` 形式の単一文字列で持つ
- `llm.base_url` は任意で、空文字なら LiteLLM の既定接続先をそのまま使う

<!-- Block: Memory Preset -->
### `memory_preset`

- 役割: 記憶検索と文脈組み立ての傾向をまとめる
- このプリセットは、`記憶` タブで編集する
- 複数保持、複製、アーカイブを許可する
- 含める項目:
  - `llm.embedding_model`
  - `llm.embedding_api_key`
  - `llm.embedding_base_url`
  - `runtime.context_budget_tokens`
  - `retrieval_profile`
- `llm.embedding_model` は `provider/model` 形式の単一文字列で持つ
- `llm.embedding_base_url` は任意で、空文字なら LiteLLM の既定接続先をそのまま使う

- `retrieval_profile` の初期固定項目:
  - `semantic_top_k`
  - `recent_window_limit`
  - `fact_bias`
  - `summary_bias`
  - `event_bias`

- 注意:
  - `retrieval_profile` は、個別キーではなく `memory_preset.payload_json` の一部として扱う
  - これは、設定UI専用の構造化設定であり、単一キー更新 API には落とさない

<!-- Block: Server Canonical Model -->
## サーバ正本の編集モデル

<!-- Block: Editor State -->
### `settings_editor_state`

- 役割: 設定UIで現在編集中かつ現在有効な構成の正本を 1 件で持つ
- 単一行の singleton とする
- 持つ項目:
  - `active_behavior_preset_id`
  - `active_llm_preset_id`
  - `active_memory_preset_id`
  - `active_output_preset_id`
  - `active_camera_connection_id`
  - `system_values_json`
  - `updated_at`
  - `revision`

- `system_values_json` は、システム設定の完全オブジェクトとする
- `revision` は、設定UIの楽観ロック用に使う

<!-- Block: Presets Table -->
### `settings_presets`

- 役割: 設定UIで複数保持するプリセット群の正本をまとめて持つ
- `preset_kind` を持つ汎用テーブルにする
- 持つ項目:
  - `preset_id`
  - `preset_kind`
  - `preset_name`
  - `payload_json`
  - `archived`
  - `sort_order`
  - `created_at`
  - `updated_at`

- `preset_kind` の初期値:
  - `behavior`
  - `llm`
  - `memory`
  - `output`

- 削除は物理削除ではなく `archived=true` を基本にする
- UI は `archived=false` のものだけを標準表示する

<!-- Block: Camera Connections Table -->
### `camera_connections`

- 役割: 設定UIで複数保持するカメラ接続先の正本をまとめて持つ
- カメラ接続はプリセットではなく専用テーブルで管理する
- 持つ項目:
  - `camera_connection_id`
  - `display_name`
  - `host`
  - `username`
  - `password`
  - `sort_order`
  - `created_at`
  - `updated_at`

- `active_camera_connection_id` が指す接続先だけを、実際のカメラ制御と静止画取得に使う
- `host`、`username`、`password` は、接続前の下書き保存では空欄を許可する
- `システム` タブでは、各接続の `display_name`、`host`、`username`、`password` を表形式で一覧表示する

<!-- Block: Change Set -->
### `settings_change_sets`

- 役割: 設定UI保存の結果を、ランタイムへ 1 単位で渡すための制御面キュー
- 単一キー更新ではなく、「UI から保存された 1 回の変更全体」を 1 件として扱う
- 持つ項目:
  - `change_set_id`
  - `editor_revision`
  - `payload_json`
  - `created_at`
  - `status`

- `status` は、少なくとも `queued`、`claimed`、`applied`、`rejected` を区別する
- ランタイムは、この change set を 1 件ずつ claim して `runtime_settings` へ materialize する

<!-- Block: Derivation Rules -->
## ランタイムへの反映規則

- ランタイムは、`settings_editor_state` とアクティブなプリセット群から、現在有効な設定を導出する
- 導出順は、次の固定順にする
  1. システム既定値
  2. `behavior_preset`
  3. `llm_preset`
  4. `memory_preset`
  5. `output_preset`
  6. `system_values_json`

- 後に適用されるものほど優先する
- 同じ意味の値が重なるときは、システム設定を優先する
- `behavior_preset` は、`behavior.*` として `runtime_settings` に materialize しつつ、`selection_profile` と認知プロンプトの構成材料にも使う
- `llm_preset`、`memory_preset`、`output_preset`、システム設定は、必要なものを `runtime_settings` へ materialize する
- カメラ接続は `runtime_settings` へ混ぜず、アクティブ接続をカメラアダプタが直接参照する
- API キーやトークンも、アクティブなプリセットからそのまま導出して使う

<!-- Block: UI API -->
## 設定UI専用 API

<!-- Block: Editor Get -->
### `GET /api/settings/editor`

- 役割: 設定UIの描画に必要な全体状態を返す
- 返す内容:
  - `editor_state`
  - `preset_catalogs`
  - `camera_connections`
  - `constraints`
  - `runtime_projection`

- `runtime_projection` は、今のアクティブ構成から導出された現在有効値の要約である
- UI は、このレスポンスだけで設定画面を描ける状態にする
- 初期実装では、API キーやトークンもマスキングせずそのまま返してよい
- 初期実装では、API キーやトークンやパスワードも、入力欄でそのまま表示してよい

<!-- Block: Editor Put -->
### `PUT /api/settings/editor`

- 役割: 設定UIの draft 全体を 1 回で保存する
- リクエストは、`GET /api/settings/editor` と同じ形の「完成形」を送る
- サーバは、`settings_editor_state` と `settings_presets` と `camera_connections` を同じ transaction で更新する
- 更新後、必要なら `settings_change_sets` を 1 件 enqueue する
- 成功時は、サーバ正本を再構成した canonical な最新状態を返す

- この API を設定UIの主保存 API とする
- 単一キー更新 API を UI の主経路にしない

<!-- Block: Preset Convenience APIs -->
### 補助 API

- UI 実装を簡単にするため、次の補助 API を持ってよい
- `POST /api/settings/presets/{preset_kind}/duplicate`
- `POST /api/settings/presets/{preset_kind}/{preset_id}/activate`
- `POST /api/settings/presets/{preset_kind}/{preset_id}/archive`

- これらは、最終的には `PUT /api/settings/editor` の sugar として扱ってよい
- 正本は、常に「保存後の全体状態」である

<!-- Block: UI Operations -->
## UI 操作フロー

<!-- Block: Open Flow -->
### 画面を開くとき

1. `GET /api/settings/editor` を呼ぶ
2. `editor_state` をフォームへ反映する
3. `preset_catalogs` から各タブのプリセット選択 UI を描画する
4. `runtime_projection` を「現在有効な設定」として表示する
5. フッター中央に現在の保存状態を表示する

<!-- Block: Save Flow -->
### 保存するとき

1. ブラウザ内の現在 draft をそのまま保存対象にする
2. `OK` または `適用` から `PUT /api/settings/editor` へ全体を送る
3. 成功したら、返ってきた canonical 状態で再描画する
4. `OK` の場合は設定画面を閉じ、`適用` の場合は開いたままにする
5. `dirty=false` に戻す
6. ランタイム反映待ちなら、その状態も表示する

<!-- Block: Reload Flow -->
### 再読込するとき

1. 現在の未保存 draft を破棄する
2. `GET /api/settings/editor` と状態再取得をやり直す
3. サーバ正本でフォームを再描画する

<!-- Block: Cancel Flow -->
### キャンセルするとき

1. 現在の未保存 draft を破棄する
2. 設定画面を閉じてチャット画面へ戻る

<!-- Block: Preset Flow -->
### プリセットを扱うとき

- プリセット操作は各タブ先頭の `プリセット選択` セクションで行う
- `output` プリセットだけは `キャラクター` タブで `キャラクター選択` として見せる
- `追加`: 既定 payload から新しいプリセットを作る
- `複製`: 現在選択中のプリセットを複製し、新しい `preset_id` を割り当てる
- `切替`: アクティブ `preset_id` を切り替える
- `削除`: 物理削除ではなく `archived=true` にする
- `名前変更`: `preset_name` を変えて保存する

<!-- Block: Validation -->
## 検証方針

- 単一キーの型制約は、`docs/39_設定キー運用仕様.md` を再利用する
- `preset_kind` ごとの `payload_json` は、このドキュメントの固定形で検証する
- 未知のフィールドを黙って捨てない
- 型不一致や不正な列挙値は、必ず保存失敗にする
- サーバは、UI から送られた draft を勝手に補完せず、必要な値が欠けていれば明示的に失敗にする

<!-- Block: Constraints -->
## 制約と非目標

- API キーやトークンも設定UIで編集してよい
- API キーやトークンも、必要なプリセットの一部として通常項目で保持してよい
- この仕様では、設定UI用 API は API キーやトークンも通常項目として返してよい
- API キーやトークンやパスワードは、設定UI上で伏字にしない
- `Mic` のデバイス固有詳細設定は、初期段階では設定UIに出さない
- カメラについては、`システム` タブに接続先一覧とアクティブ接続の編集項目を出す
- 人格の核である `self_state.personality_json` を設定UIから直接編集しない
- 設定UIは、プリセットと運用値を管理するものであり、人格の生成器ではない

## 初期導入の優先順

1. `GET /api/settings/editor` と `PUT /api/settings/editor`
2. `behavior_preset`、`llm_preset`、`memory_preset`、`output_preset`
3. `settings_change_sets` によるランタイム反映
4. 補助 API（複製 / activate / archive）

- まずは設定UIの保存単位を正しく作ることを優先し、細かな補助 API は後から足してよい

## このドキュメントで確定したこと

- 設定UIは、単一キー更新ではなく全体モデルの保存を主経路にする
- 複数保持したい設定群は、サーバ正本のプリセットとして保持する
- プリセットは、`behavior`、`llm`、`memory`、`output` の 4 種を初期採用する
- 即時反映したい運用スイッチは、システム設定として `settings_editor_state` に持つ
- カメラ接続は、プリセットではなく `camera_connections` と `active_camera_connection_id` で持つ
- API キーやトークンも、設定UIの同一保存モデルの通常項目として扱う
- ランタイムへの反映は、UI 保存結果を `settings_change_sets` として渡し、ランタイムが materialize する
- 設定UIの本来あるべき保存 API は `GET /api/settings/editor` と `PUT /api/settings/editor` である
