# 設定UI仕様

<!-- Block: Purpose -->
## このドキュメントの役割

- このドキュメントは、ブラウザの設定UIをどの保存モデルと操作単位で実装するかを固定する正本である
- 設定UIは `CocoroConsole` の設定ウインドウをベースにしつつ、`CocoroGhost` が持っていた会話・記憶系の設定も OtomeKairo に統合して扱う
- 迷った場合は、`preset_kind` を持つ汎用プリセットではなく、用途別に分離した 5 種のプリセットを正本とする
- Web API の経路は `docs/35_WebAPI仕様.md`、JSON 形は `docs/36_JSONデータ仕様.md`、SQLite の保存先は `docs/34_SQLite論理スキーマ.md` を見る

<!-- Block: Core Policy -->
## 基本方針

- 設定UIは単一キー更新画面ではなく、全体 draft を読み、ブラウザ内で編集し、保存時にまとめて確定する画面とする
- UI の正本は `settings_editor_state` と 5 種のプリセットテーブルと `camera_connections` である
- 5 種のプリセットは `character`、`behavior`、`conversation`、`memory`、`motion` に固定する
- `preset_kind` を持つ汎用テーブルは採用しない
- システム設定だけは singleton の `settings_editor_state.system_values_json` に保持する
- API キー、トークン、パスワードも設定UIの通常項目として扱い、別保管経路へ逃がさない
- 後方互換の描画分岐は持たず、保存モデルは現在の正規形に統一する

<!-- Block: Current Effect Range -->
## current 実装で runtime に効く範囲

- current の設定 UI と `GET/PUT /api/settings/editor` は、このドキュメントどおりに実装済みである
- current の `browser_chat` ランタイムで直接効いているのは、`behavior.*`、`llm.*`、`llm.embedding_*`、`runtime.*`、`memory.*`、`retrieval_profile`、`speech.tts.*`、`speech.stt.*`、`sensors.camera.enabled`、`sensors.microphone.enabled`、`camera_connections` である
- `speech.stt.*` と `sensors.microphone.enabled` は current のブラウザ録音ボタンと `POST /api/microphone/input` に効き、転写後は `microphone_message` として runtime 入力へ流れる
- current のブラウザ録音ボタンは、`sensors.microphone.enabled` または `speech.stt.enabled` が無効なときに無反応にせず、押下時に不足条件をエラー表示する
- `integrations.sns.enabled`、`integrations.notify_route`、`integrations.discord.*` は保存・検証・materialize はするが、current の入出力アダプタまでは未接続である
- `motion` プリセットは UI と保存モデルには乗っているが、current の `browser_chat` ランタイムはモーション実行器をまだ持たない

<!-- Block: Window Layout -->
## 画面構成

- 設定画面は `CocoroConsole` 風のウインドウレイアウトにする
- 上部にタイトル、説明文、タブ列を置く
- タブは左から `キャラクター`、`振る舞い`、`会話`、`記憶`、`モーション`、`システム` の 6 個に固定する
- 各タブ本文は GroupBox 風のセクションを縦に並べる
- 下部フッターは `保存状態 / OK / キャンセル / 適用` に固定する
- 設定画面内の操作エラーは、隠れたチャット欄ではなく下部フッターの保存状態欄に表示する
- カメラ接続の `追加` は接続一覧の末尾へ空行を足して行う
- 既存のカメラ接続は接続一覧テーブルをそのまま編集し、`有効` は複数件を同時に選べる
- チャット欄の `Cam` は `camera_connections[].is_enabled=true` の一覧を selector に出し、利用時は 1 台を明示選択させる

<!-- Block: Canonical Model -->
## 保存モデル

<!-- Block: Editor State -->
### `settings_editor_state`

- singleton 1 行で持つ
- カラムは以下に固定する
  - `active_character_preset_id`
  - `active_behavior_preset_id`
  - `active_conversation_preset_id`
  - `active_memory_preset_id`
  - `active_motion_preset_id`
  - `system_values_json`
  - `revision`
  - `updated_at`
- API では `system_values_json` を `editor_state.system_values` として返す
- `revision` は楽観ロックに使う

<!-- Block: Preset Tables -->
### プリセットテーブル

- 汎用 `settings_presets` は作らず、用途別に 5 テーブルを持つ
- 各テーブルは同じ列構成に固定する
  - `preset_id`
  - `preset_name`
  - `payload_json`
  - `archived`
  - `sort_order`
  - `created_at`
  - `updated_at`
- テーブル名:
  - `character_presets`
  - `behavior_presets`
  - `conversation_presets`
  - `memory_presets`
  - `motion_presets`
- 削除は物理削除ではなく `archived=1` とする
- UI は通常 `archived=0` だけを選択候補へ出す

<!-- Block: Camera Table -->
### `camera_connections`

- カメラ接続はプリセットではなく専用テーブルで持つ
- `camera_connections[].is_enabled` が AI 利用候補フラグであり、複数件を許可する
- current の runtime は `is_enabled=true` の一覧を camera adapter へ渡して `camera_candidates` を組み、認知入力には live に解決した `presets[]` を含めてよい
- current の `look` では `camera_connection_id` を必須にし、広い視点変更や別方向確認では `camera_candidates[].presets[]` の `preset_name` 利用を優先してよい
- current の `direction` fallback は固定 PTZ step を 1 回だけ実行する方式であり、プリセットがない場合の確認に使ってよい
- 保存時も他の設定と同じ transaction で更新する

<!-- Block: Change Set -->
### `settings_change_sets`

- `PUT /api/settings/editor` 成功時は 1 回の保存を 1 件の change set として enqueue する
- change set には以下を含める
  - `editor_revision`
  - 5 種の `active_*_preset_id`
  - `system_values`
  - `preset_versions`
- `preset_versions` は `character / behavior / conversation / memory / motion` の 5 キーに固定する

<!-- Block: Runtime Materialization -->
## ランタイムへの反映規則

- ランタイムが使う `runtime_settings` は、設定UI保存結果から materialize する
- 導出順は次のとおりに固定する
  1. `config/default_settings.json`
  2. `character` プリセット
  3. `behavior` プリセット
  4. `conversation` プリセット
  5. `memory` プリセット
  6. `motion` プリセットの scalar 設定
  7. `system_values`
- `motion.animations` は `runtime_settings` のドット区切りキーへ落とさず、`runtime_projection.active_motion_preset` として別に扱う
- `integrations.notify_route` と Discord 認証情報はシステム設定であり、キャラクタープリセットへ混ぜない
- カメラ接続は `runtime_settings` へ混ぜず、カメラアダプタが直接読む
- `settings_editor_state`、5 プリセットテーブル、`camera_connections` は Web が更新する設定UI正本であり、ランタイムは書き戻さない
- current では `runtime_projection.active_motion_preset` は診断用の read model であり、モーション実行までは行わない

<!-- Block: Editor API -->
## 設定UI専用 API

<!-- Block: Editor Get -->
### `GET /api/settings/editor`

- 応答は次の top-level object に固定する
  - `editor_state`
  - `character_presets`
  - `behavior_presets`
  - `conversation_presets`
  - `memory_presets`
  - `motion_presets`
  - `camera_connections`
  - `constraints`
  - `runtime_projection`
- `runtime_projection` は次の 2 キーを持つ
  - `effective_settings`
  - `active_motion_preset`
- API キー、トークン、パスワードもそのまま返してよい

<!-- Block: Editor Put -->
### `PUT /api/settings/editor`

- リクエスト本文は次の top-level object に固定する
  - `editor_state`
  - `character_presets`
  - `behavior_presets`
  - `conversation_presets`
  - `memory_presets`
  - `motion_presets`
  - `camera_connections`
- `constraints` と `runtime_projection` は送らない
- 保存時は `revision` 一致を必須にする
- サーバは `settings_editor_state`、5 プリセットテーブル、`camera_connections` を同じ transaction で更新する
- 更新後は canonical な最新状態を `GET /api/settings/editor` と同じ形で返す

<!-- Block: UI Operations -->
## UI 操作フロー

- 画面を開いたら `GET /api/settings/editor` を読み、ブラウザ内 draft を組み立てる
- 変更は即時保存せず draft にだけ反映する
- `OK` は保存成功後に閉じる
- `適用` は保存成功後も開いたままにする
- `キャンセル` はサーバ正本から draft を作り直して閉じる
- `追加 / 複製 / 削除` はローカル draft 上の操作であり、保存時にまとめて確定する
- カメラ接続の `追加` は末尾へ既定値つきの空行を draft へ追加し、まだ保存はしない
- カメラ接続テーブルの入力変更、`有効` 切替、行削除は、その場で draft へ反映する

<!-- Block: Validation -->
## 検証方針

- `editor_state.active_*_preset_id` は、それぞれ対応する配列内に必ず存在しなければならない
- 5 種の `payload_json` は、それぞれ固定形で検証する
- `character`、`behavior`、`conversation`、`memory` の payload は登録済みの設定キーだけを持つ
- `motion` の payload は `motion.posture_change_loop_count_standing`、`motion.posture_change_loop_count_sitting_floor`、`animations` に固定する
- `animations[]` は `display_name`、`animation_type`、`animation_name`、`is_enabled` の固定形にする
- `camera_connections[]` は `camera_connection_id`、`is_enabled`、`display_name`、`host`、`username`、`password`、`sort_order`、`updated_at` の固定形にする
- `camera_connections[].is_enabled=true` の行は `host`、`username`、`password` を必須にする
- 未知のキーは黙って捨てず保存失敗にする
- Discord 通知を選んだ場合は `integrations.discord.bot_token` と `integrations.discord.channel_id` を必須にする

<!-- Block: Fixed Decisions -->
## このドキュメントで確定したこと

- 設定UIは `CocoroConsole` ベースの 6 タブ構成にする
- プリセットは `character`、`behavior`、`conversation`、`memory`、`motion` の 5 種に固定する
- `preset_kind` を持つ汎用プリセット管理は採用しない
- `CocoroConsole` と `CocoroGhost` に分かれていた設定項目は OtomeKairo の設定UIへ統合する
- 通知経路と Discord 設定はシステム設定へ置く
- モーションは独立プリセットで保持し、`active_motion_preset` を診断表示できる形にする
