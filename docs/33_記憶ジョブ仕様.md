<!-- Block: Core Rules -->
## 共通原則

<!-- Block: Job Design Principles -->
### `memory_jobs` の共通原則

- `memory_jobs` は、短周期で確定した事実から、長周期で安全に再開可能な仕事だけを切り出す
- 1 ジョブは 1 つの責務だけを持ち、`job_kind` をまたいで複数責務を混ぜない
- `payload_ref` は、巨大本文そのものを埋め込むためではなく、正本のどこを材料にするかを指す参照である
- claim 後に必要な材料が不足していても、暗黙の補完や別経路の推測で埋めない
- 同じ仕事を二重投入しないために、各 payload は idempotency 判定に使えるキーを持つ
- `write_memory` が長期記憶を更新し、`embedding_sync` は検索用の `vec_items` だけを更新する

<!-- Block: PayloadRef -->
### `payload_ref` の共通仕様

- `payload_ref` は、少なくとも `payload_kind`、`payload_id`、`payload_version` を持つ
- `payload_kind` は、初期段階では `memory_job_payload` を区別する
- `payload_id` は、そのジョブ専用の payload 本体を指す一意キーである
- `payload_version` は、payload スキーマの版であり、初期値は `1` に固定する
- `payload_ref` 自体には、業務データ本体を詰め込まない
- `payload_ref` が指す payload 本体は、`core.sqlite3` のジョブ payload 領域に保持し、`memory_jobs` 行本体とは分けてよい
- claim 側は、`payload_ref` を解決できない場合、そのジョブを失敗として扱い、推測再構成しない

<!-- Block: Common Payload -->
### 共通 payload ヘッダ

- すべての job payload は、少なくとも `job_kind`、`cycle_id`、`source_event_ids`、`created_at`、`idempotency_key` を持つ
- `job_kind` は、`memory_jobs.job_kind` と一致しなければならない
- `cycle_id` は、そのジョブを投入したランタイム側の起点サイクルを示す
- `source_event_ids` は、根拠となる `events.event_id` の配列であり、空にしない
- `created_at` は、payload 本体を確定した時刻である
- `idempotency_key` は、同一仕事の二重投入防止に使う安定キーである
- `idempotency_key` は、少なくとも `job_kind + cycle_id + source_event_ids + purpose suffix` で安定化する

<!-- Block: WriteMemory -->
## `write_memory` の仕様

<!-- Block: WriteMemory Purpose -->
### 役割

- `write_memory` は、確定済み `events` を材料に、長期記憶更新の本体を行う
- `write_memory` は、`MemoryWritePlan` の生成、検証、適用、必要な `embedding_sync` の followup 投入までを担当する
- current 実装では、`prepare -> plan -> apply -> followup -> complete` の orchestration は `usecase/run_write_memory_job.py` に置き、transaction 境界は `gateway/unit_of_work.py`、`infra/sqlite/unit_of_work.py`、`infra/sqlite/write_memory_execution_store.py` に置き、`write_memory_execution_store.py` は orchestration と followup enqueue を担当し、execution state load / state apply / preference apply / context apply は下位 helper module へ分けている。`state apply` は `write_memory_state_update_impl.py` と `write_memory_self_state_sync_impl.py`、`context apply` は `write_memory_event_affect_impl.py` と `write_memory_context_relation_impl.py` に分かれている

<!-- Block: WriteMemory Payload -->
### payload

- `write_memory` payload は、少なくとも `job_kind`、`cycle_id`、`source_event_ids`、`primary_event_id`、`reflection_seed_ref`、`event_snapshot_refs`、`idempotency_key` を持つ
- `primary_event_id` は、そのジョブで中心材料として扱う `event_id` である
- `primary_event_id` は、`source_event_ids` に含まれていなければならない
- `reflection_seed_ref` は、その短周期で確定した `reflection_seed` への参照である
- `event_snapshot_refs` は、更新時に参照する `events` のスナップショット参照である
- `event_snapshot_refs` の各要素は、少なくとも `event_id` と `event_updated_at` を持つ
- `event_snapshot_refs` は、`source_event_ids` と同じ件数、同じ順序で並ばなければならない

<!-- Block: WriteMemory Output -->
### 完了条件

- `write_memory` は、`MemoryWritePlan` を生成して妥当性検証を通し、`memory_states`、`preference_memory`、`stable_preference_projection`、`event_affects`、`event_links`、`event_threads`、`event_about_time`、`state_about_time`、`state_links`、`event_entities`、`state_entities` の更新を完了したときに `completed` になる
- 現在の実装では、`MemoryWritePlan` の fixed shape を検証したうえで、`state_updates.upsert`、`close`、`mark_done`、`revise_confidence`、`preference_updates`、`event_affects`、`context_updates`、`event_about_time`、`state_about_time`、`event_entities`、`state_entities` を同一 transaction で適用する
- `write_memory` は、新しく作った `memory_states` と更新した `event_affects` を対象に `embedding_sync` を followup enqueue してよい
- `write_memory` は、followup job の enqueue を含めて 1 つの完了条件とする

<!-- Block: EmbeddingSync -->
## `embedding_sync` の仕様

<!-- Block: EmbeddingSync Purpose -->
### 役割

- `embedding_sync` は、検索用の `vec_items` と `vec_items_index` だけを更新する
- `embedding_sync` は、`events` や `memory_states` の本文を書き換えない
- `embedding_sync` は、検索対象の追加と更新をベクトル索引へ反映する唯一の主体である
- current 実装では、claim 後の `embedding_sync` completion transaction は `infra/sqlite/memory_job_store.py` が担当し、job enqueue / idempotency は `infra/sqlite/memory_job_impl.py` が担当する

<!-- Block: EmbeddingSync Payload -->
### payload

- `embedding_sync` payload は、少なくとも `job_kind`、`cycle_id`、`source_event_ids`、`embedding_model`、`requested_scopes`、`targets`、`idempotency_key` を持つ
- `embedding_model` は、そのジョブで使う埋め込みモデル識別子である
- `requested_scopes` は、少なくとも `recent`、`global` の配列である
- `requested_scopes` は空にしない
- `targets` は、同期対象の列であり、各要素は `entity_type`、`entity_id`、`source_updated_at`、`current_searchable` を持つ
- `targets` は空にしない

<!-- Block: EmbeddingSync Completion -->
### 完了条件

- `embedding_sync` は、すべての `targets` に対して `requested_scopes` 分の `vec_items` を upsert または除外反映したときに `completed` になる
- `current_searchable=false` の対象は、新規埋め込みを作らず、既存の `vec_items.searchable=0` を反映する
- 一部だけ更新して終える部分成功は許可しない
- 同じ `entity_type`、`entity_id`、`embedding_model`、`requested_scopes`、`source_updated_at` の再同期は no-op 完了してよい

<!-- Block: Enqueue Rules -->
## 投入規則

<!-- Block: Enqueue Timing -->
### 短周期からの投入

- `write_memory` は、短周期で新しい `events` が確定したとき、同じ短周期の保存単位で投入する
- `embedding_sync` は、`write_memory` が検索対象の更新を確定したときだけ followup enqueue する

<!-- Block: Enqueue Idempotency -->
### 二重投入防止

- 同じ `idempotency_key` の `queued` または `claimed` ジョブがある場合、新規投入しない
- `completed` 済みの同一 `idempotency_key` がある場合、同一 payload なら再投入しない
- 内容が変わった再投入は、`payload_version` または `idempotency_key` を変える

<!-- Block: Fixed Decisions -->
## このドキュメントで確定したこと

- `payload_ref` は、payload 本体への参照であり、業務データ本体を直接持たない
- `write_memory` は、長期記憶更新の本体であり、`embedding_sync` の followup 投入までを担当する
- `embedding_sync` は、`vec_items` と `vec_items_index` の唯一の更新主体である
- `memory_jobs` の実装は、この payload 仕様に従って行う
