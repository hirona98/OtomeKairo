# 記憶検討メモ

<!-- Block: Role -->
## この文書の役割

この文書は、記憶設計を詳細仕様へ落とす前の検討メモである。

ここでやることは次の 3 点に絞る。

- OtomeKairo にとって、記憶が何を担うべきかを明確にする
- `CocoroGhost` と外部手法を見比べて、採るものと採らないものを決める
- 次の設計段階で何を固定すべきかの方向を出す

ここでは、DB スキーマや API をまだ確定しない。
ただし、後続設計の軸になる考え方までは明示する。

<!-- Block: Premise -->
## いま置く前提

OtomeKairo では「人格」と「記憶」を操作上は独立に扱う。
ただし、会話時に実際に表に出る人格は、人格設定ファイルだけで決まるわけではない。

現時点では、次のように考えるのが自然である。

- 人格設定は、初期の傾向と基本姿勢を与える
- 記憶は、自己像、他者理解、関係性、気分、未解決事項を育てる
- 実際の応答は、その時点の「人格設定 + 記憶から組み上がる現在の人格文脈」で決まる

つまり、操作単位としては分けるが、挙動としては「記憶が人格を構成する」を前提にしたほうがよい。

この意味で、OtomeKairo では単なる検索用ストレージではなく、人格形成の基盤として記憶を設計する必要がある。

<!-- Block: Requirements -->
## OtomeKairo で記憶に求めること

既存 docs から見ると、記憶には最低でも次が必要である。

- 日をまたいで会話が継続すること
- ユーザーの嗜好だけでなく、他者や出来事や関係も持てること
- 自発会話の根拠になること
- 訂正や関係変化を扱えること
- 後から「なぜそう答えたか」を追えること
- 同じ人格設定に対して、記憶だけ差し替えて挙動比較できること

ここから逆算すると、記憶は少なくとも次の性質を持つ必要がある。

- 生ログを残す
- 派生理解を更新できる
- 過去を消さずに現在の見解を持てる
- 想起時に「何のための記憶か」を分けて出せる

<!-- Block: CocoroGhost -->
## `CocoroGhost` から引き継ぎたいもの

`CocoroGhost` の設計と実装には、そのまま活かしたい強みが多い。

- 追記ログ (`events`) と更新される状態 (`state`) を分けている
- 非同期で記憶を育て、同期の会話速度を守っている
- `revisions` で変化の履歴と根拠イベントを追える
- `retrieval_runs` で検索理由を観測できる
- `event_links` / `event_threads` / `state_links` で芋づるの想起を作っている
- `event_affect` と `long_mood_state` を分け、瞬間感情と背景感情を分離している

このあたりは、OtomeKairo でもかなり重要である。
特に「消さないログ」と「更新される理解」を分ける構造は、そのまま中核にしてよい。

<!-- Block: CocoroGhostGap -->
## `CocoroGhost` をそのまま持ち込まないほうがよい点

一方で、`CocoroGhost` は主に「会話の質を上げる記憶」としては強いが、「人格を構成する記憶」として見ると足りない点がある。

- `state.kind` が汎用すぎて、自己像、他者像、関係、価値観、約束の区別が弱い
- `user_preferences` が「好み/苦手」に寄っており、人物理解全体には広がっていない
- `long_mood_state` は背景感情として有効だが、関係や話題ごとの感情傾向までは持たない
- 想起結果が基本的にフラットな `SearchResultPack` なので、「人格に効く記憶」と「根拠エピソード」が同列に混ざりやすい
- 1イベントごとの更新は強いが、複数の経験を横断して「私は最近どう変わったか」「この人との関係はどう育ったか」を統合する層が弱い

OtomeKairo では、ここを先に補っておくべきである。

<!-- Block: Research -->
## 外部手法から得た示唆

### 分類の軸

`CoALA` と LangGraph の整理は有用だった。
記憶を少なくとも次の型で見ると、設計の見通しがかなり良くなる。

- episodic: 体験そのもの
- semantic: 事実や理解
- procedural: 行動や判断のやり方

さらに LangGraph が整理しているように、記憶更新は「応答の最中に書くか」「背景で育てるか」を分けて考えるべきである。

### 参考にしたい手法

#### 1. `Generative Agents`

良い点:

- 体験の蓄積
- 重要度を使った想起
- 複数の記憶から反省を作る

OtomeKairo への示唆:

- 1ターンごとの記録だけでなく、複数イベントをまたいだ reflection 層が必要
- 「重要だった出来事」が、後の人格や行動に効く導線を作るべき

#### 2. `MemoryBank`

良い点:

- 長期対話で人格理解を深める方向を正面から扱っている
- 時間経過と重要度で記憶の強弱を変える発想がある

OtomeKairo への示唆:

- 記憶には salience が必要
- ただし、忘却曲線をそのまま採用して事実を自動で弱めるのは危険

特に OtomeKairo では、重要事実や関係は「削る」のではなく「想起順位を下げる」「休眠させる」方向のほうが合う。

#### 3. `MemGPT`

良い点:

- 速い作業文脈と遅い長期記憶を階層分離している

OtomeKairo への示唆:

- 短期会話文脈と長期記憶を明確に分けるべき
- 「今の会話の流れ」と「長く持つ人格記憶」を同じ器に入れないほうがよい

#### 4. `HippoRAG`

良い点:

- グラフと多段探索で、単純ベクトル検索より関係を辿れる

OtomeKairo への示唆:

- 人、物、話題、出来事のつながりは、軽量なグラフとして保持したほうがよい
- ただし、最初から大きい知識グラフを目指す必要はない

#### 5. 2025年の経験再利用の実証研究

良い点:

- 「似た入力に似た過去経験を当てると、似た出力が出やすい」こと
- その結果として、誤った経験の再生が将来の誤りを増幅しやすいこと

OtomeKairo への示唆:

- 記憶は量だけでなく品質管理が必要
- 未来の訂正や結果を、過去記憶の品質ラベルとして反映するべき

## 現時点の推奨方針

### 1. 記憶は「人格形成に効く層」で分ける

OtomeKairo では、少なくとも次の 6 層で考えるのがよい。

1. 生ログ
2. エピソード記憶
3. 意味記憶
4. 関係記憶
5. コミットメント記憶
6. 感情記憶

それぞれの役割は次のとおり。

- 生ログ: 観測された事実をそのまま残す
- エピソード記憶: 後から思い出せる単位に整えた出来事
- 意味記憶: 自分、ユーザー、他者、世界についての継続理解
- 関係記憶: 誰と誰がどういう関係か、何にどう結びつくか
- コミットメント記憶: 約束、未完了、保留、継続中の話題
- 感情記憶: 瞬間反応と持続的気分

加えて第 2 段階として、反省や行動スタイルを保持する「手続き記憶」を検討したい。

### 2. 長期記憶は自然文だけでなく半構造化にする

`CocoroGhost` の `state.body_text + payload_json` だけでは、人格形成に必要な比較や訂正がしづらい。
OtomeKairo では、少なくとも次の軸を各記憶単位に持たせる方向がよい。

- `memory_type`
- `scope`
- `subject`
- `predicate`
- `object` または `value`
- `status`
- `confidence`
- `salience`
- `valid_from` / `valid_to`
- `evidence_event_ids`
- 人間が読める短い本文

イメージは次のようなものになる。

```json
{
  "memory_type": "preference",
  "scope": "user",
  "subject": "user",
  "predicate": "likes",
  "object": "辛いもの",
  "status": "confirmed",
  "confidence": 0.94,
  "salience": 0.82,
  "evidence_event_ids": [120, 188],
  "text": "あなたは辛い食べ物をかなり好む。"
}
```

ここで重要なのは、自然文を捨てることではない。
比較、更新、検索のために、自然文に加えて骨格を持たせることである。

### 3. 記憶には「昇格」と「休眠」を持たせる

記憶は、最初から全部を同じ重みで持たないほうがよい。
少なくとも次のような段階が必要である。

- observed: 生ログ由来の観測
- inferred: 推測で得た理解
- confirmed: 明示発言や繰り返しで確定した理解
- superseded: 新しい理解に置き換えられたもの
- revoked: 明確に否定されたもの
- dormant: 削除はしないが、普段は前面に出さないもの

昇格条件として有効なのは次である。

- ユーザーが明示した
- 同種の観測が繰り返された
- 強い感情を伴った
- 約束や衝突など、将来行動に効く
- 複数の出来事をまとめた reflection が支持した

この設計なら、忘却曲線をそのまま使わなくても、人間らしい「残りやすさ」を表現できる。

### 4. 想起結果はフラット一覧ではなく役割別パックにする

人格を構成する記憶を本当に効かせるなら、応答モデルに渡す記憶も役割ごとに分けたほうがよい。

たとえば次のようなパックである。

```json
{
  "short_term_context": {},
  "self_model": [],
  "user_model": [],
  "relationship_model": [],
  "active_commitments": [],
  "affect_context": {},
  "episodic_evidence": []
}
```

これにより、次がやりやすくなる。

- 口調や姿勢に効く記憶
- 話題選択に効く記憶
- 訂正時に参照すべき記憶
- 根拠エピソード

`SearchResultPack` 的な一覧は残してよいが、最終的に返答へ注入する段階では役割ごとに整理し直したほうがよい。

### 5. 記憶更新は 2 つの時間粒度で行う

人格形成には、1ターンごとの更新だけでは足りない。

OtomeKairo では、次の 2 段階がよい。

- `turn consolidation`: そのターンから取れる事実、感情、関係、約束を更新する
- `reflective consolidation`: 数ターンや 1 日単位で横断的に見直し、自己像、他者像、関係の変化、行動方針の学びを作る

前者は `CocoroGhost` に近い。
後者が、OtomeKairo で特に重要になる。

「この人といると最近どう感じるか」「私はこの件をどう受け止めるようになったか」は、単発イベントではなく蓄積からしか作れないためである。

### 6. グラフは入れるが、最初から重くしない

関係想起のために、軽量なリンクは必要である。
ただし、最初から大規模知識グラフにする必要はない。

まずは次で十分である。

- event → entity
- memory → entity
- memory ↔ memory
- event ↔ event

これで、少なくとも次ができる。

- 人物経由の芋づる想起
- 同じ話題の再想起
- 矛盾や補足の追跡
- 関係の長期変化の把握

### 7. 品質管理を前提にする

記憶が人格を構成するなら、誤記憶も人格を壊す。
そのため、最初から次を入れるべきである。

- 根拠イベントの保存
- revision の保存
- 訂正時の格下げ
- 想起ミスの分離
- 将来の会話結果を使った品質ラベル更新

「一度覚えたら強い」ではなく、「覚えているが、いまどれだけ信じるか」を管理する設計が必要である。

## 現時点で有力な中核形

詳細設計前のメモとしては、OtomeKairo の長期記憶の中心は、`CocoroGhost` の単一 `state` をそのまま継承するより、次のような形が有力に見える。

- `events`
- `episode_digests`
- `memory_units`
- `memory_links`
- `event_affects`
- `mood_state`
- `revisions`
- `retrieval_runs`

ここでの考え方は次のとおり。

- `events`: 生ログの正本
- `episode_digests`: 想起しやすい出来事単位
- `memory_units`: 意味記憶 / 関係記憶 / コミットメント記憶の中核
- `memory_links`: 関係や補足や矛盾の接続
- `event_affects` + `mood_state`: 感情の短期と長期
- `revisions`: 変化の監査
- `retrieval_runs`: 想起の監査

この構成なら、人格形成に必要な区別を保ちつつ、まだシンプルさも守りやすい。

<!-- Block: Avoid -->
## 避けたい方向

現時点では、次は避けたほうがよい。

- ベクトル検索だけで記憶を成立させること
- なんでも 1 本の自由文 summary に潰すこと
- 重要事実まで忘却曲線で自動的に弱めること
- 生ログを捨てて派生記憶だけ残すこと
- 1ターンごとに自己像全体を書き直すこと

どれも短期的には楽だが、「記憶が人格を構成する」には弱い。
